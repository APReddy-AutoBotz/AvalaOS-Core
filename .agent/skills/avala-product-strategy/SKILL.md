# Avala Product Strategy

## Purpose

Keep work aligned to the AvalaOS thesis and source-of-truth product boundaries.

## When To Use

Use for roadmap, positioning, module scope, product naming, demo story, and feature-scope decisions.

## Non-Negotiable Rules

- AvalaOS is a governed operating layer for AI, automation, and agentic delivery.
- Do not turn AvalaOS into generic project management, process mining, a chatbot, or an agent builder.
- Preserve the flow: Assess -> Studio -> Govern Lite -> Delivery Lite -> Monitor.

## Inputs

- User request
- `docs/00_SOURCE_OF_TRUTH.md`
- Product, planning, and governance docs

## Outputs

- Scope decision
- In/out-of-scope notes
- Acceptance criteria

## Acceptance Criteria

- Scope matches the source of truth.
- Product language is current AvalaOS language.
- Unsupported claims are excluded.

## Common Failure Modes

- Adding features during a docs milestone.
- Repeating product thesis instead of referencing source-of-truth docs.
- Expanding Delivery into a Jira replacement.

## Verification Checklist

- Source-of-truth alignment checked.
- Naming scan completed when relevant.
- Scope boundaries documented.
