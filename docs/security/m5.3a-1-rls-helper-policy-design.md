# M5.3a-1 RLS Helper Policy Design

## Purpose

This security design records the future helper and SELECT policy posture for M5.3a-1. It is planning only. No SQL, helper functions, RLS policies, tests, seed data, backfill, constraint validation, or runtime behavior are implemented here.

## Security Scope

M5.3a-1 is limited to future SELECT policy design for:

- `profiles`
- `organizations`
- `workspaces`
- `roles`
- `organization_members`
- `workspace_memberships`

Deferred surfaces:

- Assess, project, generated document, and delivery work item artifact tables.
- Provider governance tables and existing provider policies.
- Legacy draft Delivery, Docs, timesheet, storage, export, Delivery Pack, Monitor, lineage, and handoff surfaces.
- All INSERT, UPDATE, DELETE, hard-delete, soft-delete, review, approval, delivery, export, or other mutation policies.

## Default Policy Posture

- Deny by default.
- SELECT-only for M5.3a-1.
- No broad `FOR ALL` policies.
- No mutation policies.
- No hard DELETE policies.
- No artifact policies.
- No provider governance policy changes.
- No browser service-role behavior.
- No secret values or personal hosted values in SQL, fixtures, logs, docs, or evidence.

## Helper Functions

Future required helpers:

- `public.is_active_org_member(p_org_id uuid)`
- `public.is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`

Required helper behavior:

- Return boolean only.
- Use `auth.uid()` as the identity anchor.
- Require `auth.uid()` to be present.
- Require matching membership `user_id = auth.uid()`.
- Require active, non-deleted membership.
- Require active, non-deleted parent organization.
- Require active, non-deleted parent workspace where workspace applies.
- Do not return tenant metadata.
- Do not return role or permission existence.
- Do not expose service-role behavior.

SQL posture:

- `LANGUAGE sql`.
- `STABLE`, not `IMMUTABLE`.
- Hardened `search_path`.
- Schema-qualified references wherever possible.
- `SECURITY DEFINER` only if required to break recursive RLS failure.

Grant posture:

- `REVOKE EXECUTE FROM PUBLIC`.
- `REVOKE EXECUTE FROM anon` where applicable.
- `GRANT EXECUTE` only to `authenticated`.

## SECURITY DEFINER Review Gate

`SECURITY DEFINER` is not a default approval. Before implementation, AP must confirm the helper design and static review must confirm:

- The helper returns only boolean output.
- The helper cannot leak membership or role existence across tenants.
- The helper cannot grant access from disabled, invited, archived, suspended, removed, or deleted membership.
- The helper cannot bypass disabled, archived, or deleted parent organization or workspace state.
- The helper has hardened `search_path`.
- Grants are narrow and do not expose behavior to anonymous browser contexts.

## SELECT Policy Matrix

| Table | SELECT direction | Required inputs | Recursion risk | Negative tests |
| --- | --- | --- | --- | --- |
| `profiles` | User can read own active, non-deleted profile only. | `auth.uid()`, `profiles.id`, profile status, `deleted_at`. | Low if direct `id = auth.uid()` predicate is used. | Anonymous denied; user cannot read another profile; disabled/deleted profile denied. |
| `organization_members` | User can read own active, non-deleted membership only, gated by active org helper. | `auth.uid()`, `user_id`, `org_id`, membership status, `deleted_at`. | Medium because helper may query membership table. | Non-member denied; disabled member denied; cross-org membership denied. |
| `workspace_memberships` | User can read own active, non-deleted workspace membership only, gated by active workspace helper. | `auth.uid()`, `user_id`, `org_id`, `workspace_id`, membership status, `deleted_at`. | Medium because helper may query workspace membership table. | Workspace non-member denied; disabled workspace member denied; cross-workspace membership denied. |
| `organizations` | Active org member can read authorized org metadata through `public.is_active_org_member(id)`. | `auth.uid()`, org status, `deleted_at`, active org membership. | Medium because helper resolves membership. | Non-member denied; disabled member denied; cross-org denied. |
| `workspaces` | Active workspace member can read authorized workspace metadata through `public.is_active_workspace_member(id, org_id)`. | `auth.uid()`, workspace status, workspace `deleted_at`, org state, active workspace membership. | Medium because helper resolves workspace membership. | Workspace non-member denied; disabled workspace member denied; cross-workspace denied. |
| `roles` | Org roles readable through active org membership; workspace roles readable through active workspace membership; system role visibility deferred. | `scope`, `org_id`, `workspace_id`, role status, membership helper. | Medium if role visibility depends on membership helpers. | Cross-org role denied; cross-workspace role denied; disabled membership denied; system visibility denied unless AP approves. |

## Recursion Controls

Recommended controls for future implementation:

- Keep helper functions minimal.
- Use direct predicates for own-profile reads.
- Avoid permission helpers in M5.3a-1.
- Avoid role helper dependencies in membership visibility.
- Avoid artifact table dependencies.
- Validate helper behavior locally before claiming tenant-isolation proof.

## Bootstrap Controls

Future policies should solve bootstrap access without cross-tenant exposure:

- A user can read their own active profile.
- A user can read their own active organization memberships.
- A user can read their own active workspace memberships.
- Organization metadata is readable only through active membership.
- Workspace metadata is readable only through active workspace membership.
- Role metadata is scoped and does not leak system or cross-tenant roles.

## Provider Governance Handling

Provider governance tables and existing provider policies are outside M5.3a-1. M5.3a-1 should leave them untouched and defer provider-governance RLS hardening to a separate AP-approved slice.

Provider secret values remain server-only. Browser contexts must not receive service-role keys, provider keys, raw private tokens, or secret reference values unless a later AP-approved server-only design permits a safe reference pattern.

## Static Review Requirements

Future implementation review must confirm:

- Approved helper names only.
- `STABLE`, not `IMMUTABLE`.
- Hardened `search_path`.
- Justified `SECURITY DEFINER`, if used.
- Execute grants restricted to `authenticated`.
- No broad `FOR ALL`.
- No mutation policies.
- No DELETE policies.
- No artifact policies.
- No provider governance policy changes.
- No legacy draft table policies.
- No service-role/browser exposure.
- No secret-looking literals.
- No personal hosted identifiers.

## Risks

- Recursive RLS failure could block legitimate membership bootstrap.
- Overbroad definer helpers could bypass row restrictions.
- Role metadata could leak cross-tenant existence if scoped incorrectly.
- Static review alone cannot prove isolation.
- Runtime may fail closed until later adapter and artifact policy milestones.
