# M5.2e Delivery RLS Readiness Gate

Milestone: M5.2e Delivery Ownership Columns Migration Plan

Branch: milestone/m5.2e-delivery-ownership-columns-migration-plan

Status: RLS readiness only; no RLS implementation

## Purpose

This gate defines what Delivery ownership must provide before future RLS policies and tenant-isolation tests can safely cover projects, work items, Delivery Pack records, Monitor signals, handoff, lineage, evidence, and exports.

## Readiness Boundary

M5.2e does not implement RLS, tenant isolation, schema, SQL, runtime guards, or database policies.

Delivery/project RLS is not ready until project authority exists in the applied migration chain and work-item ownership has an approved project/workspace model.

## Future Project And Work-Item Reads

Future Delivery policies should require active workspace membership for workspace-scoped project and work-item reads.

Organization membership alone should not be treated as enough for workspace-scoped Delivery artifacts once workspace ownership exists.

## Cross-Tenant Denial

Future policies must deny:

- cross-organization project reads and writes;
- cross-workspace project reads and writes;
- work-item reads or mutations outside the user's authorized workspace;
- source lineage references that point across tenant boundaries;
- handoff and evidence references that bypass project/workspace ownership.

## Role And Permission Direction

Buyer Viewer access should be read-only where AP approves it.

Delivery Lead roles should manage approved project and work-item fields within assigned workspace scope.

Contributors should update only permitted own-task fields when assigned or explicitly authorized.

Reviewer and approver gates should apply to review or approval fields, Delivery Pack readiness, and handoff status when those persistence models exist.

App-side view guards and Delivery policy checks are useful runtime controls, but they are not substitutes for database RLS.

## Source Lineage And Evidence Boundary

Future policies must treat source lineage and evidence references as tenant-scoped data. A Delivery work item that references Assess, Studio/Docs, handoff, or evidence records must not cross organization or workspace boundaries.

Embedded task metadata should not override database ownership fields.

Standalone lineage or evidence authority remains deferred until AP approves that persistence model.

## Docs-To-Delivery Handoff Boundary

Docs-to-Delivery imports should require same-tenant source and target ownership. The default policy direction is no cross-workspace handoff unless AP approves a specific governed transfer model.

Handoff ledger ownership remains deferred to a later lineage/handoff slice.

## Delivery Pack And Monitor Boundary

Delivery Pack records should remain derived until project and work-item authority exist. If persisted later, Delivery Pack records should be workspace-scoped and linked to source project, Assess, Studio/Docs, evidence, and handoff records.

Monitor project/task signals should remain derived until a persistence model is approved. M5.2e does not create live telemetry or runtime monitoring schema.

## Export Boundary

Export records, storage object references, and server-mediated download controls remain deferred to M5.5 export controls.

Browser export behavior used for local/demo workflows is not hosted export authority.

## Provider And Secret Boundary

Delivery records must not store provider secret values, service-role keys, database URLs, private tokens, personal hosted values, or raw secret material.

The browser must not receive service-role credentials or raw provider secrets. Service-role operations, where later approved, must be limited to trusted server or Edge paths with actor, tenant, correlation, and audit context.

## Negative Isolation Cases For Future Tests

Future RLS tests should prove:

- anonymous users cannot read Delivery tenant data;
- authenticated non-members cannot read projects or work items;
- disabled members cannot access Delivery data;
- workspace non-members cannot read workspace projects or work items;
- users in one organization cannot read or mutate another organization's Delivery records;
- users in one workspace cannot read or mutate another workspace's Delivery records;
- Buyer Viewer cannot mutate projects, work items, Delivery Pack records, Monitor signals, exports, or provider metadata;
- contributors cannot update unassigned work or restricted planning fields;
- source lineage, evidence, and handoff refs cannot cross tenant boundaries;
- archived or deleted records do not grant accidental access;
- browser paths cannot invoke service-role-only workflows.

## Positive Cases For Future Tests

Future RLS tests should prove:

- active workspace members can read authorized project and work-item records;
- Delivery Lead can update approved project/work-item fields;
- assigned contributors can update approved own-task fields;
- reviewers and approvers can update approved review/approval fields;
- Buyer Viewer can read approved records where AP approves that behavior and cannot mutate them.

## RLS Readiness Decision

M5.2e is not ready for Delivery RLS implementation. Project authority should be implemented first in M5.2f, followed by work-item ownership and then RLS policy/test implementation after ownership schema is sufficient.
