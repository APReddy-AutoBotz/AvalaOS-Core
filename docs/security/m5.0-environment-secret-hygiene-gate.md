# M5.0 Environment And Secret Hygiene Gate

Milestone: M5.0 Enterprise Supabase Readiness Plan
Status: Security planning gate only

## Purpose

This gate defines the environment and secret hygiene rules required before M5 implementation begins. It protects the move from buyer-demo RC readiness toward secure hosted pilot readiness without exposing local values or committing personal hosted-project details.

## Scope And Non-Scope

This document is policy and planning only. It does not create environment files, change runtime code, add secrets, connect to Supabase, deploy Edge functions, alter schema, or validate a live provider.

## Environment Config Policy

- Local, development, staging, and production must be treated as separate environments with separate ownership, values, storage, audit evidence, and promotion gates.
- Public browser config may include only non-secret values needed to initialize the client application.
- Secret values must live outside source control in approved environment storage or an approved secret manager.
- Personal hosted-project values may be used temporarily for local validation only after AP approval, and only when they are replaceable by governed environment values.
- No project URL, project reference, token, key, organization identifier, bucket name, or provider secret should be committed if it belongs to a personal or unmanaged environment.

## `.env*` Handling Policy

- `.env*` files remain ignored by source control.
- M5 work may inspect variable names referenced in code, sample files, or config contracts.
- M5 work must not read, print, summarize, copy, or paste actual local secret values.
- Sample config files may contain names and placeholders only.
- Verification output must never include secret values, full tokens, project references, personal URLs, or personal bucket names.

## Browser-Secret Prohibitions

- Pilot and production behavior must not place provider secrets, service-role credentials, database connection strings, management tokens, or storage-admin credentials in browser code or browser-accessible config.
- Browser code may receive public client config and user session state only.
- Any existing local-demo/internal-dev provider fallback must remain non-pilot behavior and must fail closed when pilot or production mode is selected.
- Browser storage must not become an enterprise authority for users, roles, evidence, approvals, exports, provider execution, or audit events.

## Service-Role And Server-Only Boundary

- Service-role credentials are server-only.
- Database owner credentials, management tokens, storage-admin credentials, and provider secrets are server-only.
- Edge functions or an AP-approved server runtime may use elevated credentials only for explicitly authorized persistence, storage, export, audit, or provider-governance operations.
- Server code must write audit metadata and references, not secret values.
- Any service boundary that touches elevated credentials must have least-privilege design, safe logging, and explicit failure behavior.

## Provider Key-Reference Direction

- Provider execution for pilot and production must use provider configuration records and key references.
- Provider key references should use an explicit product-owned prefix, provider class, and purpose name.
- The current naming direction is `AVALA_PROVIDER_SECRET_<PROVIDER>_<PURPOSE>`, with actual values stored outside source control.
- Database rows, logs, evidence bundles, and exported artifacts may store provider name, model, policy, key-reference name, status, and audit identifiers.
- They must not store raw provider secret values.

## No Secrets In DB, Logs, Evidence, Or Docs

- Do not store raw secrets in application tables.
- Do not store raw secrets in audit events, evidence references, lineage payloads, generated artifacts, export metadata, delivery packs, screenshots, test logs, or documentation.
- Do not include personal hosted-project values in planning docs, quality evidence, PR descriptions, issue comments, or commit messages.
- Error handling should classify failures without echoing secret values.

## Safe Verification Rules

Allowed verification:

- Confirm whether a required variable name is present or absent without printing the value.
- Confirm whether a value is classified as public client config, server-only credential, provider key reference, storage name, or deployment token.
- Confirm that ignored environment files are not staged.
- Confirm that no generated docs or diffs contain personal hosted-project values.
- Confirm that pilot/production modes fail closed when required server-side config is absent.

Disallowed verification:

- Printing, copying, or summarizing secret values.
- Printing full project URLs, project references, tokens, provider keys, or service-role credentials.
- Capturing screenshots or logs that expose secrets.
- Writing redacted values in a way that preserves enough characters to reconstruct the original.

## Accepted Variable Naming Approach

Accepted naming is by class:

- Public client config: names may start with `VITE_` only when the value is intentionally public and non-secret.
- Supabase server config: names may use `SUPABASE_` for server-only credentials and service integration.
- Provider secret references: names should use `AVALA_PROVIDER_SECRET_<PROVIDER>_<PURPOSE>` as references to server-side secret storage.
- Deployment and management credentials: names should be environment-owner controlled and server-only.

The name can be documented. The value cannot.

## Dev, Staging, And Prod Replacement Principle

Personal values can be temporary. Architecture must not be temporary.

Any environment value used during M5 validation must be replaceable by AP-owned development, staging, and production values without changing application behavior. A value is not pilot-ready until ownership, rotation, storage location, access path, safe verification, and replacement procedure are documented.

## M5.1 Gate Acceptance Direction

M5.1 should produce:

- Approved local/development/staging/production separation.
- Approved variable-name inventory by public, server-only, provider-reference, deployment, and storage class.
- Confirmation that `.env*` files remain ignored and unstaged.
- Safe verification steps that prove presence, absence, and class without printing values.
- Replacement procedure for personal validation values.
- Explicit prohibition of browser-side provider secrets and service-role credentials.
