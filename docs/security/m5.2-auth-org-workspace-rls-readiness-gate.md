# M5.2 Auth, Organization, Workspace, And RLS Readiness Gate

Milestone: M5.2 Supabase Auth / Organization / Workspace Schema Plan
Status: RLS readiness gate only

## Purpose

This gate defines what the Auth, organization, workspace, membership, role, and permission model must provide before M5.3 designs RLS policies and tests. It is a readiness handoff, not an RLS implementation.

## RLS Readiness, Not RLS Implementation

M5.2 does not create policies, migrations, helper functions, schema changes, Auth UI, runtime adapter changes, Edge Function changes, deployment changes, or live Supabase validation. M5.3 should use this gate to design deny-by-default RLS and regression tests unless AP separately approves implementation.

## Tables That Will Need RLS In M5.3

M5.3 should plan RLS for at least:

- `profiles` or `user_profiles`.
- `organizations`.
- `organization_members` or `organization_memberships`.
- `roles`.
- `permissions` and `role_permissions` if normalized.
- Role assignment tables.
- `workspaces`.
- `workspace_memberships`.
- `assess_processes`.
- `assessments`.
- `assessment_review_events`.
- Govern/Decision Pack persistence tables if introduced later.
- `document_generations`.
- Delivery projects, epics, sprints, tasks, task comments, task activity events, and timesheets.
- Handoff ledger entries.
- Evidence refs and source lineage tables if introduced later.
- Audit events.
- Export records and private storage object policies.
- Provider config, key-reference metadata, provider policy, generation job, usage, and provider audit tables.

## Required Policy Inputs

| Input | Required direction |
| --- | --- |
| `auth.uid()` | Primary authenticated user identity for RLS. |
| Active organization membership | User must have active membership in the record's organization for org-scoped access. |
| Active workspace membership | User must have active membership in the record's workspace for workspace-scoped access. |
| Role assignment | User role must be resolved from server-governed org/workspace role assignment. |
| Permission mapping | Permission checks must use canonical DB permission mapping or approved role helper functions. |
| Status fields | Suspended, archived, deleted, invited, or inactive records must not grant access unintentionally. |
| Actor fields | Owner, assignee, reviewer, approver, created_by, updated_by, and requested_by fields must be available for owner/reviewer/admin cases. |

## Expected Deny-By-Default Posture

- Unauthenticated requests should be denied except for explicit Auth entrypoints outside app data RLS.
- Authenticated users should see only their own profile and authorized org/workspace records.
- Active org membership should be required for every tenant record.
- Active workspace membership should be required for workspace-scoped work records.
- Role and permission checks should narrow access after membership is confirmed.
- Suspended or removed memberships should deny access.
- Cross-org and cross-workspace reads/writes should deny even when IDs are guessed.
- Mutations should require explicit write permissions and actor checks.
- Service-role access must bypass RLS only from server-side contexts with approved audit and failure handling.

## Service-Role And Server-Only Exception Boundaries

- Service-role credentials remain server-only.
- Browser code must never receive service-role credentials, database URLs, management tokens, storage-admin credentials, provider secrets, or provider secret reference values unless AP explicitly approves a later exception.
- Edge/server actions using elevated credentials must be limited to approved persistence, storage, export, audit, provider governance, or admin workflows.
- Server-side writes should record actor, correlation, and audit metadata where appropriate.
- Server-side errors must not echo secrets or personal hosted values.

## Anon And Authenticated Boundary

| Caller class | Expected M5.3 posture |
| --- | --- |
| Anonymous browser | No app data access. |
| Authenticated user without org membership | Own profile only, plus invite/onboarding records if AP later approves them. |
| Authenticated user with org membership | Org records allowed only according to membership, role, permission, and status. |
| Authenticated user with workspace membership | Workspace records allowed only according to workspace membership, role, permission, and status. |
| Suspended or removed member | Denied for tenant records. |
| Server-side service role | Allowed only for approved server operations with audit/failure boundaries. |

## Cross-Org Isolation Expectations

M5.3 tests should prove:

- A user in Org A cannot read Org B records.
- A user in Org A cannot write Org B records.
- A user cannot join or infer Org B role/membership rows unless authorized.
- Audit, evidence, export, provider, storage, handoff, and lineage records do not leak across orgs.
- Object storage paths are tenant-scoped and cannot be traversed across org IDs.

## Cross-Workspace Isolation Expectations

M5.3 tests should prove:

- A user in Workspace A cannot read Workspace B records without membership.
- A user with org membership but no workspace membership cannot access workspace-scoped work artifacts unless AP approves an org-admin inheritance rule.
- Workspace role permissions do not grant access to another workspace.
- Delivery, Studio, Assess, handoff, evidence, exports, provider policies, audit metadata, and monitor signals respect workspace boundaries.
- Workspace archive/suspension status is enforced.

## Negative Test Matrix Needed In M5.3

| Case | Expected result |
| --- | --- |
| Unauthenticated read of tenant table | Denied. |
| Authenticated user without org membership reads org data | Denied. |
| Active member reads own org records | Allowed by table-specific policy. |
| Active org member without workspace membership reads workspace record | Denied unless AP approves org-admin inheritance. |
| Active workspace contributor reads allowed workspace work | Allowed. |
| Contributor performs reviewer/admin mutation | Denied. |
| Reviewer performs approved review mutation | Allowed only for assigned scope. |
| Buyer viewer mutates work artifact | Denied. |
| Suspended member reads or writes tenant data | Denied. |
| Cross-org read/write attempt | Denied. |
| Cross-workspace read/write attempt | Denied. |
| Service-role operation from browser path | Prohibited by architecture and secret hygiene. |
| Server-side service operation without audit context where required | Denied or fail closed per policy. |
| Export artifact read from wrong org/workspace | Denied. |
| Provider config or audit read from wrong org/workspace | Denied. |

## No Browser Secrets

Browser code may receive public Supabase URL, public Supabase anon key, AI Edge enablement, runtime mode, and authenticated session state only. The anon key is acceptable in browser only under RLS-first design and is not a service credential.

## No Provider Secrets In Browser

Provider secrets and provider secret reference values remain server-only. Browser fallback provider behavior remains local-demo/internal-dev debt and is not pilot or production approved.

## No Personal Supabase Values Committed

M5.2 docs, evidence, commits, PR descriptions, and test output must not include personal hosted-project URLs, project IDs, keys, tokens, org IDs, bucket names, database passwords, or secrets. Verification may record names, classes, pass/fail status, and counts only.

## M5.3 Design/Test-Plan Handoff

M5.3 should produce:

- A table-by-table RLS matrix.
- Role and permission policy matrix.
- Org and workspace positive/negative regression plan.
- Service-role exception policy.
- Storage object policy direction.
- Provider governance RLS direction.
- Audit, evidence, export, and handoff denial cases.
- Hosted validation prerequisites.

M5.3 should not implement RLS unless AP separately approves an implementation milestone.

## AP Decisions Needed Before M5.3

- Whether workspace membership is mandatory for all work artifacts.
- Whether org admins inherit workspace admin authority.
- Whether roles and permissions are normalized or JSONB-based.
- How buyer-demo personas map to DB roles.
- Whether the existing guard permission strings become canonical DB permissions.
- Whether invite/SSO records are needed before RLS design.
- Whether provider governance workspace placeholders become strict workspace FKs.
