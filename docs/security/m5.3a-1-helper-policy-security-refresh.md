# M5.3a-1 Helper Policy Security Refresh

## Purpose

This document refreshes the security posture for the future M5.3a-1 RLS helper and SELECT policy implementation now that executable local assertion plumbing exists. It is docs-only and creates no SQL, helpers, policies, migrations, tests, runtime changes, or DB connections.

## Security Scope

Future implementation is limited to SELECT policy enablement for:

- `profiles`
- `organizations`
- `workspaces`
- `roles`
- `organization_members`
- `workspace_memberships`

Deferred surfaces:

- artifact authority tables
- provider governance tables and policies
- legacy draft tables
- storage/export
- Delivery Pack
- Monitor
- lineage/handoff
- all INSERT, UPDATE, DELETE, soft-delete, review, approval, export, delivery, or mutation policies

## Default Security Stance

- Deny by default.
- SELECT-only.
- No broad `FOR ALL`.
- No mutation policies.
- No hard DELETE.
- No artifact policies.
- No provider governance changes.
- No browser service-role behavior.
- No secret values, target values, hosted identifiers, or row payload dumps in SQL, fixtures, logs, docs, or evidence.

## Helper Security Plan

Future helpers:

- `public.is_active_org_member(p_org_id uuid)`
- `public.is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`

Required behavior:

- Return boolean only.
- Use `auth.uid()` as identity anchor.
- Require non-null authenticated subject.
- Require membership `user_id = auth.uid()`.
- Require active, non-deleted membership.
- Require active, non-deleted parent organization.
- Require active, non-deleted parent workspace where workspace applies.
- Return no tenant metadata.
- Return no role or permission existence.
- Expose no service-role behavior to browser contexts.

SQL posture:

- `LANGUAGE sql`.
- `STABLE`, not `IMMUTABLE`.
- Hardened `search_path`.
- Schema-qualified references wherever possible.
- `SECURITY DEFINER` only if AP approves it to avoid recursion.

Grant posture:

- `REVOKE EXECUTE FROM PUBLIC`.
- `REVOKE EXECUTE FROM anon` where applicable.
- `GRANT EXECUTE` only to `authenticated`.

## Recursion And SECURITY DEFINER Risk

Risks:

- `organization_members` policies may depend on a helper that queries `organization_members`.
- `workspace_memberships` policies may depend on a helper that queries `workspace_memberships`.
- `roles` policies may depend on membership checks.
- Parent `organizations` and `workspaces` checks may recursively depend on helpers.
- A weak `SECURITY DEFINER` helper can bypass RLS too broadly.
- A weak `search_path` can create hijack risk.
- Broad execute grants can expose helper behavior to anonymous or unintended browser contexts.
- Helper results can leak cross-tenant membership or role existence if they return anything other than boolean.

Safest future implementation pattern:

- Use direct `profiles.id = auth.uid()` for profile reads.
- Keep helper functions minimal and boolean-only.
- Use definer helpers only if AP approves and local assertions show direct policy joins recurse or fail closed incorrectly.
- Keep membership visibility constrained to own active, non-deleted rows plus active, non-deleted parent state.
- Keep role visibility dependent only on active scoped membership; do not add permission helpers.
- Do not query artifact, provider, export, storage, Delivery Pack, Monitor, lineage, or handoff tables.
- Validate every helper and policy path with the executable harness before claiming local tenant isolation.

## SELECT Policy Matrix

| Table | SELECT direction | Required checks | Negative checks |
| --- | --- | --- | --- |
| `profiles` | Own active, non-deleted profile only. | `auth.uid()`, `id`, `status`, `deleted_at`. | Anonymous denied; other profile denied; disabled/deleted profile denied. |
| `organization_members` | Own active, non-deleted membership only. | `auth.uid()`, `user_id`, `status`, `deleted_at`, active parent org. | Non-member denied; disabled member denied; cross-org membership denied. |
| `workspace_memberships` | Own active, non-deleted workspace membership only. | `auth.uid()`, `user_id`, `status`, `deleted_at`, active parent org/workspace. | Workspace non-member denied; disabled workspace member denied; cross-workspace membership denied. |
| `organizations` | Active org member can read authorized org metadata. | Active org helper. | Anonymous denied; non-member denied; disabled member denied; cross-org denied. |
| `workspaces` | Active workspace member can read authorized workspace metadata. | Active workspace helper. | Non-member denied; disabled workspace member denied; cross-workspace denied. |
| `roles` | Org roles through active org membership; workspace roles through active workspace membership. | Role scope, status, deleted state, active scoped membership. | Cross-org role denied; cross-workspace role denied; disabled membership denied; system role visibility denied unless AP approves. |

## Bootstrap Controls

Future policies must allow a user to bootstrap minimum visibility without exposing other tenants:

- own active profile
- own active organization memberships
- own active workspace memberships
- organization metadata through active membership
- workspace metadata through active workspace membership
- role metadata through active scoped membership

Future policies must not allow cross-tenant membership existence leaks, cross-tenant role existence leaks, disabled/deleted membership access, disabled/deleted parent org/workspace access, or system role visibility unless AP approves exact semantics.

## Harness Security Expectations

The executable harness must remain the proof path for any future local tenant-isolation claim. Future implementation must preserve explicit local-target proof before DB action, hosted target rejection before DB action, fail-closed behavior without target proof or without reachable local DB, counts-only sanitized output, and no target values, secret values, claim values, auth headers, or row payloads.

## Static Review Gates

Future implementation review must confirm approved helper names only, `STABLE` not `IMMUTABLE`, AP-approved `SECURITY DEFINER` if used, hardened `search_path`, schema-qualified references where possible, execute grants restricted to `authenticated`, no broad `FOR ALL`, no INSERT/UPDATE/DELETE policies, no artifact/provider/legacy policies, no service-role/browser exposure, no secret-looking literals, and no personal hosted identifiers.

## AP Decisions Needed

- Approve exact helper SQL.
- Approve or reject `SECURITY DEFINER`.
- Approve helper owner and grant posture.
- Approve exact SELECT policy predicates.
- Approve role visibility and system-role deferral.
- Approve membership visibility semantics.
- Approve whether fail-closed-only evidence is acceptable if no local DB is reachable.

## Risks

- Recursive RLS failure can block legitimate access.
- Definer helpers can become overbroad if grants or `search_path` are weak.
- Metadata policies can leak tenant existence if helper outputs or role policies are too broad.
- Static review alone cannot prove cross-tenant denial.
- Runtime may remain fail-closed until later approved runtime and artifact-policy work.
