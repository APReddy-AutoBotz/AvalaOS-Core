# M5.3a RLS Applied Authority Policy Matrix

## Purpose

This matrix refreshes RLS policy direction for applied authority tables only. It is a design artifact, not policy SQL. M5.3a creates no RLS policies, helper functions, migrations, tests, seed data, or runtime changes.

## Default Policy Posture

- Deny by default.
- Anchor user identity on `auth.uid()` through `profiles.id`.
- Require active organization membership for organization-scoped reads.
- Require active workspace membership for workspace-scoped artifact reads.
- Fail closed when workspace-scoped artifact rows have `workspace_id IS NULL`, unless AP explicitly approves a narrow exception.
- Use SELECT-only policies in the first implementation slice.
- Do not add broad `FOR ALL` policies.
- Do not add hard DELETE policies.
- Keep mutation policies denied until AP approves role and permission depth.
- Keep service-role behavior server-only and never browser-callable.

## Nullable Workspace Hardening

Workspace-scoped artifacts should not become broadly org-readable merely because `workspace_id` remains nullable during staged migration and backfill planning. The default future SELECT predicate for these tables should include a non-null workspace check:

- `assess_processes`
- `assessments`
- `assessment_review_events`
- `projects`
- `document_generations`
- `delivery_work_items`

Any exception must be AP-approved, narrow, time-bound, and explicitly tested.

## Helper Strategy

Future policy SQL should prefer small helpers once AP approves implementation:

- `is_active_org_member(p_org_id uuid)`
- `is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`

Optional helpers for permissions remain blocked pending AP role/permission decisions:

- `has_org_permission(p_org_id uuid, p_permission text)`
- `has_workspace_permission(p_workspace_id uuid, p_org_id uuid, p_permission text)`

Helper hardening requirements:

- Avoid recursive RLS failure modes.
- Use `SECURITY DEFINER` only if needed and with hardened `search_path`.
- Mark helpers `STABLE`, not `IMMUTABLE`.
- Grant execute only to intended authenticated roles.
- Do not expose service-role semantics to browser code.
- Do not leak cross-tenant membership or role existence.

## Provider Governance Handling

Provider governance tables from existing applied migrations are outside the first M5.3a policy implementation scope. They may already include read policies for provider configuration and workspace provider policy metadata. M5.3a leaves those policies untouched and defers provider-governance RLS hardening to a separate AP-approved slice.

Provider secret values remain server-only. Provider metadata policies must never expose provider keys, service-role values, raw private tokens, or personal hosted-project values to browser contexts.

## Legacy And Draft Table Exclusions

M5.3a does not secure:

- `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, `timesheet_entries`.
- Draft policy files under `docs/schema`.
- Storage/export records.
- Delivery Pack, Monitor, lineage, or handoff records.

Those surfaces require separate authority and migration decisions before RLS policy implementation.
