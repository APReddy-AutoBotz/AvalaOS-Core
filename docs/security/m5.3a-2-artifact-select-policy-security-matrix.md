# M5.3a-2 Artifact SELECT Policy Security Matrix

## Purpose

This matrix defines the planned SELECT-only RLS security posture for artifact authority tables after the closed M5.3a-1 membership helper and SELECT policy foundation.

This document is planning only. It does not implement policies.

## Global Security Rules

Future artifact SELECT policies must follow these rules:

- Deny by default.
- Use `public.is_active_workspace_member(workspace_id, org_id)` by default.
- Treat `org_id IS NULL` as fail-closed.
- Treat `workspace_id IS NULL` as fail-closed.
- Treat `deleted_at IS NOT NULL` as fail-closed where `deleted_at` exists.
- Do not use JSON payload fields for authorization.
- Do not use source references as authorization authority.
- Do not allow cross-org or cross-workspace existence leaks.
- Do not create broad `FOR ALL` policies.
- Do not create INSERT, UPDATE, DELETE, or mutation policies.
- Do not create service-role browser policies.
- Do not change provider, storage, export, or legacy draft-table policies.

`public.is_active_org_member(org_id)` is reserved for AP-approved org-only exceptions. No org-only artifact path is approved by this plan.

## Table Matrix

| Table | SELECT policy predicate direction | Positive scenario | Required negative scenarios |
| --- | --- | --- | --- |
| `assess_processes` | `org_id IS NOT NULL`, `workspace_id IS NOT NULL`, `deleted_at IS NULL`, active workspace membership | active workspace member reads same-workspace process | anonymous, non-member, disabled org member, disabled workspace member, cross-org, cross-workspace, null org, null workspace, deleted process |
| `assessments` | same base predicate plus same-org/same-workspace non-deleted `process_id` | active workspace member reads same-workspace assessment with valid process | all base negatives plus deleted process, orphan process, cross-workspace process, cross-org process |
| `assessment_review_events` | non-null org/workspace, active workspace membership, same-org/same-workspace non-deleted owning assessment and process | active workspace member reads same-workspace review event | all base negatives plus orphan assessment, orphan process, cross-workspace assessment/process, cross-org assessment/process |
| `projects` | `org_id IS NOT NULL`, `workspace_id IS NOT NULL`, `deleted_at IS NULL`, active workspace membership, populated source refs same workspace | active workspace member reads same-workspace project | all base negatives plus deleted project and cross-tenant source refs |
| `document_generations` | non-null org/workspace/project, `deleted_at IS NULL`, active workspace membership, same-workspace non-deleted project, populated source refs same workspace | active workspace member reads same-workspace document generation | all base negatives plus deleted doc generation, deleted project, orphan project, cross-workspace project, cross-tenant source refs |
| `delivery_work_items` | non-null org/workspace/project, `deleted_at IS NULL`, active workspace membership, same-workspace non-deleted project, populated document/source refs same workspace | active workspace member reads same-workspace work item | all base negatives plus deleted work item, deleted project, orphan project, cross-workspace project, cross-tenant document/source refs |

## Payload And Lineage Hardening

The following fields are never authorization inputs:

- `assessments.metadata`
- `assessments.responses`
- `assessments.evidence_items`
- `assessments.assumptions`
- `assessments.completion_by_section`
- `assessments.review`
- `assessments.scores`
- `assessment_review_events.payload`
- `projects.metadata`
- `document_generations.artifacts`
- `delivery_work_items.source_lineage`
- `delivery_work_items.metadata`

Future static review must confirm no SELECT policy uses these fields to grant access.

## Lifecycle Behavior

M5.3a-2 should not use `archived_at`, `status`, `lifecycle_stage`, `health_status`, `priority`, or `type` as authorization gates.

Rationale:

- Archived rows may remain evidence-bearing or history-bearing artifacts.
- Workflow status is not equivalent to tenant membership.
- Lifecycle-specific read restrictions need separate AP approval.

Only `deleted_at` is a planned SELECT denial field where present.

## Non-Scope Security Boundaries

M5.3a-2 must not create or modify:

- Mutation policies.
- Permission helpers.
- Provider governance policies.
- Storage/export policies.
- Legacy draft table policies.
- Runtime adapter behavior.
- Service-role or browser service-boundary behavior.
- Package scripts or dependencies.
- Supabase config.

## Risks

- Same-organization foreign keys do not prove same-workspace isolation.
- Nullable workspace rows can leak if org-only access is allowed accidentally.
- Source references can leak cross-tenant existence if treated as authority.
- JSON payload fields can create false authority if policies inspect them.
- Static review alone cannot prove artifact isolation.
