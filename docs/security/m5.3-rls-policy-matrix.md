# M5.3 RLS Policy Matrix

Milestone: M5.3 RLS Policy Design And Test Plan
Status: RLS policy matrix planning only

## Purpose

This matrix defines the future RLS direction for table classes needed for hosted pilot readiness. It is a design artifact only. It does not create SQL, policies, migrations, schema changes, or runtime behavior.

## Matrix Principles

- Deny by default.
- Use `auth.uid()` as caller identity.
- Require active org membership for tenant records.
- Require active workspace membership for workspace-scoped records after workspace schema exists.
- Require explicit role or permission authority for privileged reads and all mutations.
- Keep service-role exceptions server-only and audited.
- Keep raw secrets out of browser-visible records.

## Implementation Notes For Later Milestones

- M5.2a should establish canonical workspace and membership schema before workspace-aware policy SQL is written.
- M5.3a should implement policies and tests together so no policy ships without isolation proof.
- Org-only interim RLS should be treated as a temporary AP-approved exception, not workspace-ready isolation.
- Policy implementation should prefer small helpers over repeated complex joins when recursion and privilege boundaries are understood.
