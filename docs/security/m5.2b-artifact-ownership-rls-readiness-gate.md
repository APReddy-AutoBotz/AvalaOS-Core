# M5.2b Artifact Ownership RLS Readiness Gate

Milestone: M5.2b Artifact Ownership Columns Migration Plan

Status: Security readiness gate only

## Purpose

This gate defines how artifact ownership columns should prepare AvalaOS Core for future workspace-aware RLS. It does not implement policies, helper functions, migrations, tests, runtime authorization, or tenant isolation.

## Readiness Position

M5.2a created tenant authority tables and enabled RLS on them without policies. M5.2b determines how artifact tables should carry ownership so M5.3a can later implement deny-by-default policies and tenant-isolation tests.

Ownership columns are necessary for RLS readiness, but they are not RLS enforcement by themselves.

## Required Ownership Inputs For RLS

Future workspace-aware RLS needs these inputs:

- `auth.uid()` as the identity anchor.
- Active organization membership.
- Active workspace membership for workspace-scoped records.
- `org_id` on hosted tenant records.
- `workspace_id` on work artifacts where workspace is the intended boundary.
- Role and permission inputs from server-governed records.
- Actor fields such as `created_by`, `updated_by`, `owner_id`, `reviewer_id`, `approver_id`, and `actor_id` where policy behavior depends on them.
- Lifecycle fields such as `status`, `deleted_at`, and `archived_at`.
- `correlation_id` or `audit_correlation_id` for governed server-side actions.

## Cross-Org And Cross-Workspace Isolation

Future RLS implementation should use ownership columns to prove:

- Anonymous callers cannot read tenant data.
- Authenticated non-members cannot read org data.
- Disabled or removed members cannot read tenant data.
- Org A users cannot read or mutate Org B records.
- Workspace A users cannot read or mutate Workspace B records.
- Org membership alone does not grant workspace artifact access unless AP approves inheritance.
- Workspace archive or suspension status does not grant accidental access.

## Role And Actor Gate Readiness

Ownership planning should preserve enough actor fields for future policies:

- Buyer Viewer can read only approved artifacts and cannot mutate.
- Contributors can create or edit only allowed draft/work records.
- Reviewers can perform review actions only where assigned or permitted.
- Approvers can approve only where the role, permission, and workflow status allow it.
- Delivery leads can manage delivery records only in assigned workspace scope.
- Platform admins do not receive browser-visible service-role authority.

## Service-Role And Server-Only Boundary

Future service-role exceptions must remain server-only and audit-aware:

- No browser service-role keys.
- No provider secrets in browser code or tenant-visible records.
- No database URLs, management tokens, storage-admin credentials, raw provider secrets, or secret reference values in browser storage.
- Server-side writes should carry actor and correlation context where required.
- Errors and evidence must not print secret values or personal hosted values.

## FK And Constraint Readiness

Future implementation should:

- Add nullable `workspace_id` before policy enforcement.
- Use `NOT VALID` FKs where broad artifact data may need later backfill.
- Validate constraints after AP-approved data mapping.
- Add non-null constraints only after backfill and validation.
- Avoid constraints against draft-only tables.
- Avoid destructive backfills.

## RLS Policy Implementation Dependencies

M5.3a should wait for:

- Real artifact tables with ownership columns.
- AP-approved workspace mapping.
- Active org and workspace membership helper design.
- Role/permission helper design.
- Positive and negative fixture data.
- Clear server-only service-role exception model.
- Sanitized test strategy that records pass/fail and counts only.

## Not RLS Implementation

This gate does not:

- Create policies.
- Create helper functions.
- Validate tenant isolation.
- Change runtime guards.
- Replace app-side view guards.
- Make browser/demo storage authoritative.
- Make the product hosted-pilot ready.

## AP Decisions Needed

- Whether all hosted-pilot work artifacts require `workspace_id`.
- Whether org admins inherit workspace authority.
- Whether Assess/Govern is the first ownership migration slice.
- Whether provider-governance metadata remains org-scoped.
- Whether audit, evidence, lineage, and export records are implemented before or after initial RLS.
- Whether future RLS tests use SQL-level tests, JS integration tests, or both.
