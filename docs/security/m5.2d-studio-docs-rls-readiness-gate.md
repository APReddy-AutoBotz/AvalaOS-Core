# M5.2d Studio Docs RLS Readiness Gate

Milestone: M5.2d Studio Docs Ownership Columns Migration Plan

Branch: milestone/m5.2d-studio-docs-ownership-columns-migration-plan

Status: Readiness gate only; no RLS implementation

## Purpose

This gate defines the future RLS readiness expectations for Studio and Docs persistence. It does not create policies, implement schema, prove tenant isolation, or change runtime behavior.

## Readiness Boundary

Studio and Docs cannot claim workspace-aware RLS readiness until generated artifacts, document generation records, document repository records, source lineage, handoff context, and exports carry approved tenant ownership fields.

The current draft `document_generations` table is organization-scoped and project-linked, but not workspace-scoped. The project boundary is not applied migration authority, so a Studio/Docs policy cannot yet reliably enforce workspace isolation.

## Future Policy Direction

Generated artifact reads should be workspace-scoped. Active workspace membership should be required for generated artifact and document generation reads, unless AP approves an explicitly narrower server-mediated exception.

Cross-organization and cross-workspace reads must be denied by default. A row that links to a source assessment, process, project, handoff entry, evidence reference, or task must not point across tenant boundaries.

Buyer Viewer behavior should be read-only where approved. Buyer Viewer must not generate, mutate, review, approve, archive, delete, export, or alter provider-related metadata unless AP approves a specific permission.

Contributor generation rights, reviewer gates, approver gates, and owner/admin lifecycle actions should be modeled with explicit role or permission checks. App-side navigation guards are not a substitute for database RLS.

Draft, review, approved, rejected, archived, and deleted statuses should be considered in policies once status fields are approved. Archived or deleted rows should not grant accidental access.

## Source Assessment Boundary

Future Studio/Docs rows that reference Assess records should enforce same-tenant source relationships:

- `source_process_id` should match the same organization and approved workspace scope.
- `source_assessment_id` should match the same organization and approved workspace scope.
- source evidence and assumptions should stay within the same tenant boundary.
- source context embedded in generated artifact payloads should not override database ownership fields.

## Evidence, Lineage, And Handoff Boundary

Evidence references, source lineage, and handoff ledger records should not cross tenant boundaries. If they remain embedded payloads, policies should treat them as data under the parent row, not independent authority. If they become standalone tables, they need their own workspace ownership and append/update rules.

Docs-to-Delivery handoff should require same-workspace or explicitly approved cross-workspace transfer rules. Default policy direction is no cross-workspace lineage mutation.

## Export And Storage Boundary

Durable export records and storage objects are deferred to the export-controls milestone. Future Studio/Docs policies should assume document exports are private, tenant-scoped, server-mediated, and not readable through raw browser storage paths.

Browser download behavior used for local/demo flows is not export authority for hosted pilot persistence.

## Provider And Secret Boundary

Studio/Docs records must not store provider secret values, service-role keys, private tokens, database URLs, or personal hosted values. Provider execution, prompt templates, and job records must stay server-side where approved.

The browser must not receive service-role credentials or raw provider secrets. Secret-reference names are not approved as browser data unless a later AP decision explicitly permits them.

## Service-Role Boundary

Service-role operations, where later approved, must be limited to trusted server or Edge paths. They should require actor context, tenant context, correlation context, and auditability. Service-role access must not be callable directly from browser code.

## Negative Isolation Cases For Future Tests

Future RLS tests should prove that:

- anonymous users cannot read Studio/Docs tenant data;
- authenticated non-members cannot read generated artifacts;
- disabled members cannot access Studio/Docs records;
- workspace non-members cannot read workspace document records;
- users in one organization cannot read or mutate another organization's Studio/Docs rows;
- users in one workspace cannot read or mutate another workspace's generated artifacts;
- Buyer Viewer cannot mutate generated artifacts, document generations, exports, provider metadata, or review status;
- contributors cannot approve their own generated artifacts unless AP explicitly approves that permission;
- source assessment links cannot cross organization or workspace boundaries;
- embedded evidence, lineage, handoff, and source context cannot be used to bypass tenant scope;
- archived or deleted rows do not grant accidental access;
- browser paths cannot call service-role-only workflows.

## Positive Cases For Future Tests

Future RLS tests should prove that:

- active workspace members can read authorized generated artifact metadata;
- authorized contributors can create draft generated artifacts;
- reviewers can review records when permissions and status allow it;
- approvers can approve records when permissions and status allow it;
- Buyer Viewer can read approved records where AP approves that behavior and cannot mutate them.

## RLS Readiness Decision

M5.2d is not ready to implement Studio/Docs RLS. The next dependency is Delivery/project ownership planning, because current draft document generation rows point at project authority.

Tenant isolation remains unimplemented and unproven until ownership schema, RLS policies, and isolation tests are implemented in later approved milestones.
