# M5.3a-1c Auth Simulation And Secret Safety

## Purpose

This security plan defines the future local auth simulation and no-secret boundary for executable M5.3a-1 local DB assertions. It is docs-only. It does not implement DB assertions, RLS helpers, RLS policies, SQL migrations, package changes, local config, runtime behavior, or DB connections.

## Security Principles

- Local-only DB assertion execution.
- Fail closed before DB action if local proof is missing.
- Never silently accept `DATABASE_URL`.
- Never print target values.
- Never read or print `.env` values.
- Never print service-role values, provider keys, JWT/private tokens, hosted URLs, bucket names, personal identifiers, or row payloads.
- Use generated local UUIDs only.
- Keep server-only fixture setup separate from browser/client behavior.
- Do not claim tenant isolation until AP-approved helpers/policies exist and executable assertions pass.

## Auth Simulation Recommendation

Future implementation should use local SQL session claim simulation as the default plan.

Scenario mapping:

| Scenario | Future local simulation |
| --- | --- |
| Anonymous | No subject claim and no authenticated local role. |
| Authenticated active member | Local authenticated role/session with generated local subject and active membership. |
| Authenticated disabled organization member | Local authenticated role/session with generated local subject and disabled organization membership. |
| Authenticated disabled workspace member | Local authenticated role/session with generated local subject and disabled workspace membership. |
| Authenticated non-member | Local authenticated role/session with generated local subject and no matching membership. |
| Server-only fixture setup | Local privileged setup path only, never browser-callable and never secret-bearing. |

Candidate mechanisms:

- `SET LOCAL ROLE authenticated` where the local database role exists and is compatible.
- `SET LOCAL request.jwt.claim.sub` where compatible with local `auth.uid()`.
- AP-approved local claim/session helper only if direct session setup is insufficient.
- Supabase local Auth fixtures only if AP approves generic local config.

Real JWTs are not required for M5.3a-1d. Token generation and token printing should remain out of scope.

## Fixture Safety

Future fixtures must be:

- Generated.
- Local-only.
- Non-secret.
- Transaction-scoped or cleaned up deterministically.
- Neutral and not personally identifying.

Fixture records must not include:

- Personal IDs.
- Hosted URLs.
- Hosted project values.
- Provider keys.
- Service-role values.
- Tokens.
- Bucket names.
- `.env` values.
- Row payloads in output.

## Target Safety

The runner must prove local target before DB action.

Allowed target classes:

- `localhost`.
- `127.0.0.1`.
- Docker local network.
- AP-approved local-only token.

Rejected target classes:

- Hosted URLs.
- Remote hosts.
- Environment-only target values.
- Ignored secret-file fallback.
- Unknown target arguments.

The runner must never log target values, even when rejecting them.

## Secret-Safe Output

Allowed output:

- Scenario labels.
- Local target proof status without target value.
- Counts.
- Tenant isolation verified yes/no.
- Sanitized failure classifications.

Forbidden output:

- DB URL.
- `DATABASE_URL` value.
- `.env` values.
- Service-role values.
- Provider keys.
- JWT/private tokens.
- Hosted URLs.
- Bucket names.
- Personal organization, workspace, project, or user IDs.
- Claim values.
- Auth headers.
- Row payloads.

## SQL Assertion Safety

Future assertion SQL must:

- Run inside a transaction or isolated local state.
- Roll back or clean up.
- Use generated local UUIDs.
- Record scenario outcomes using sanitized identifiers.

Future assertion SQL must not:

- Define RLS helpers.
- Define RLS policies.
- Alter applied schema.
- Create SQL migrations.
- Create durable seed data.
- Validate constraints.
- Include artifact, provider, legacy, storage, export, Delivery Pack, Monitor, lineage, or handoff assertions.
- Include mutation or DELETE assertions for M5.3a-1.
- Include target values, tokens, hosted URLs, or personal identifiers.

## Node Runner Safety

Future runner must:

- Fail closed while executable assertions are not implemented.
- Fail closed when local-target proof is missing.
- Fail closed when a target appears remote.
- Fail closed when any scenario fails.
- Return zero only when AP-approved executable local assertions run and pass.
- Avoid printing raw DB errors if they contain target or secret-like values.
- Sanitize failure classes before output.

## `supabase/config.toml` Safety

Future `supabase/config.toml` remains AP-gated.

If AP approves it, the file must be:

- Generic.
- Portable.
- Local-only.
- Safe to commit.
- Free of project refs, hosted URLs, keys, tokens, service-role values, provider keys, bucket names, organization IDs, and personal values.

## `applySupabaseMigrations` Boundary

`scripts/applySupabaseMigrations.mjs` must not be the default assertion runner.

Future use requires AP approval for:

- A local-only wrapper.
- Target rejection before DB action.
- No secret output.
- No remote fallback.
- No target value logging.

## Evidence Rules

Evidence may record:

- Files inspected.
- Files changed.
- Scenario names.
- Counts.
- Sanitized failure classes.
- Local target proof status without value.
- Verification results.

Evidence must not record:

- DB URLs.
- `.env` values.
- Service-role values.
- Provider keys.
- JWT/private tokens.
- Hosted URLs.
- Bucket names.
- Personal identifiers.
- Row payload dumps.
- Claim values.

## Risks

- Environment target accidentally points to remote DB.
- Auth simulation drifts from Supabase behavior.
- Verbose DB errors leak target values.
- Fixture setup is mistaken for browser-callable behavior.
- Mocked tests are mistaken for RLS proof.
- Static review is mistaken for executable isolation proof.
