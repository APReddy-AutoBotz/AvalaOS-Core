# M5.1 Environment Config And Secret Hygiene Gate

Milestone: M5.1 Environment Config And Secret Hygiene Gate
Status: Security/readiness gate

## Purpose

M5.1 turns the M5.0 environment and secret-hygiene direction into a repository gate. It classifies environment variable names, records browser/server boundaries, adds a narrow static hygiene scan, and preserves the rule that hosted pilot readiness requires governed environment ownership before implementation milestones proceed.

This milestone improves repository safety. It does not make AvalaOS Core enterprise-production ready, does not connect to live Supabase, and does not implement hosted persistence.

## Strict Non-Scope

M5.1 does not implement Supabase schema, migrations, RLS, auth changes, workspace or organization runtime behavior, deployment, provider runtime changes, Edge Function changes, UI changes, routing changes, demo data changes, scoring changes, Reports/Teams/Admin behavior, M4.1g export policy, AGI roadmap items, Health repository changes, dependency changes, or CI changes.

M5.1 does not deploy, call providers, connect to live Supabase, read or expose local `.env*` values, print secrets, commit personal hosted-project values, claim regulatory certification, claim enterprise-production readiness, claim live provider readiness, assign approval authority to AI, or assign deterministic scoring decisions to AI.

## Environment Classification Table

| Class | Approved names or direction | Policy |
| --- | --- | --- |
| Client-public allowed variables | `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`, `VITE_AI_EDGE_FUNCTIONS_ENABLED`, `VITE_AVALA_AI_MODE` | These names may be browser-visible only when the value is intentionally public and non-secret. `VITE_SUPABASE_ANON_KEY` is client-public only under an RLS-first architecture and must not be treated as a service credential. |
| Local-demo/internal-dev debt | `VITE_GROQ_API&#95;KEY`, `VITE_GEMINI_API&#95;KEY` | Existing debt only. These names are not pilot or production approved. They may remain only in the known legacy demo/dev fallback surfaces until AP approves remediation timing. |
| Server-only variables | `SUPABASE_URL`, `SUPABASE_ANON_KEY` in Edge/server contexts, `SUPABASE_SERVICE_ROLE_KEY`, `DATABASE_URL`, `SUPABASE_ACCESS_TOKEN`, management tokens, database passwords, direct provider secret names, and storage-admin credentials | These values must not be placed in browser code, browser config, database records, logs, evidence, docs, screenshots, or exported artifacts. |
| CI/smoke variables | `SUPABASE_SMOKE_EMAIL`, `SUPABASE_SMOKE_PASSWORD`, `SUPABASE_SMOKE_ORG_ID` | These names may be used only through CI or smoke-test secret storage and sanitized verification. Values must not be printed. |
| Provider key-reference names | `AVALA_PROVIDER_SECRET_GROQ_<PURPOSE>`, `AVALA_PROVIDER_SECRET_GEMINI_<PURPOSE>` | Names only. They refer to server-side secret storage or an approved secret manager. The referenced secret value remains server-only. |
| Forbidden in browser | Service-role key, database URL, Supabase management tokens, direct provider API key values, JWT/private token values, storage admin secrets, raw provider secrets, and secret reference values unless AP explicitly approves a later exception | Browser code may receive public client config and user session state only. |

## `.env*` Handling Policy

- `.env*` files remain ignored by source control and must not be staged.
- M5.1 may inspect environment variable names referenced in code, config, docs, and scripts.
- M5.1 must not read, print, summarize, copy, or paste actual local `.env*` values.
- Sample config, docs, and evidence may include names and placeholders only.
- Verification output must never include secret values, full hosted-project URLs, project refs, tokens, keys, organization identifiers, bucket names, or provider secrets.

## Browser-Secret Prohibitions

- Pilot and production behavior must not put provider secrets, service-role credentials, database connection strings, management tokens, storage-admin credentials, raw private tokens, or provider secret references into browser code or browser-accessible config.
- Browser code may initialize Supabase with public client config and authenticated session state only.
- Existing local-demo/internal-dev provider fallback remains classified debt and must fail closed for pilot or production mode.
- Browser storage must not become an enterprise authority for users, roles, evidence, approvals, exports, provider execution, or audit events.

## Service-Role And Server-Only Boundary

- Service-role credentials are server-only.
- Database owner credentials, management tokens, storage-admin credentials, provider secrets, and raw private tokens are server-only.
- Supabase Edge functions or another AP-approved server runtime may use elevated credentials only for explicitly authorized persistence, storage, export, audit, or provider-governance operations.
- Server-side code must write audit metadata and references, not secret values.
- Any service boundary that touches elevated credentials must have least-privilege design, safe logging, and explicit fail-closed behavior.

## Provider Key-Reference Direction

- Pilot and production provider execution should use provider configuration records and key-reference names.
- Provider key references should use a product-owned prefix, provider class, and purpose name.
- The current naming direction is `AVALA_PROVIDER_SECRET_GROQ_<PURPOSE>` and `AVALA_PROVIDER_SECRET_GEMINI_<PURPOSE>`.
- Database rows, logs, evidence bundles, and exported artifacts may store provider name, model, policy, key-reference name, status, and audit identifiers.
- They must not store raw provider secret values.

## No Secrets In DB, Logs, Evidence, Or Docs

- Do not store raw secrets in application tables.
- Do not store raw secrets in audit events, evidence references, lineage payloads, generated artifacts, export metadata, delivery packs, screenshots, test logs, or documentation.
- Do not include personal hosted-project values in planning docs, quality evidence, PR descriptions, issue comments, commit messages, or release notes.
- Error handling should classify failures without echoing secret values.

## No Personal Supabase Values Committed

- Personal hosted-project URLs, project refs, anon keys, service-role keys, access tokens, organization IDs, bucket names, database passwords, and storage-admin credentials must not be committed.
- Personal values may support temporary local validation only after AP approval and only when they are replaceable by governed environment values.
- Any personal validation value is not pilot-ready until ownership, rotation, storage location, access path, safe verification, and replacement procedure are documented.

## Safe Verification Rules

Allowed verification:

- Confirm whether a required variable name is present or absent without printing the value.
- Confirm whether a value is classified as public client config, server-only credential, provider key reference, storage name, or deployment token without showing the value.
- Confirm that ignored environment files are not staged.
- Confirm that generated docs, diffs, logs, and evidence do not contain personal hosted-project values.
- Confirm that pilot and production modes fail closed when required server-side config is absent.

Disallowed verification:

- Printing, copying, summarizing, or screenshotting secret values.
- Printing full hosted-project URLs, project refs, tokens, provider keys, or service-role credentials.
- Writing redacted values in a way that preserves enough characters to reconstruct the original.
- Storing secret values in docs, DB rows, logs, screenshots, evidence, artifacts, or PR comments.

## Dev, Staging, And Prod Replacement Principle

Personal values can be temporary. Architecture must not be temporary.

Local, development, staging, and production must have distinct ownership, values, storage, audit evidence, and promotion gates. Any value used during validation must be replaceable by AP-owned environment values without code changes.

## CI Integration Decision

M5.1 adds the `test:secret-hygiene` package script but does not add it to CI and does not add it to the aggregate test script. CI integration is deferred unless AP separately approves it in a later milestone.
