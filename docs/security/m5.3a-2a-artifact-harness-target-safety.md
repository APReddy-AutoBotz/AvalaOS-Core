# M5.3a-2a Artifact Harness Target Safety

## Purpose

This document defines the target-safety and output-safety plan for a future local artifact assertion harness. It is planning only and does not implement a runner, SQL, package script, database command, or configuration file.

## Security Baseline

M5.3a-2 implemented artifact SELECT policies and assertion SQL. Artifact SELECT isolation is not verified because the artifact assertions have not run through an AP-approved local runner against a reachable local database.

The M5.3a-1 harness remains scoped to identity and membership assertions. It should not be silently expanded to artifact assertions.

## Local Target Gate

The future artifact harness must prove a local target before any database action.

Allowed target forms:

- Explicit `localhost`.
- Explicit `127.0.0.1`.
- AP-approved loopback alias.
- AP-approved Docker-local target that resolves to local-only behavior.

Rejected target forms:

- Hosted database URLs.
- Remote hosts.
- Implicit environment database target.
- Automatic `DATABASE_URL` execution.
- Ignored secret-file fallback.
- Unknown target arguments.
- Any value that cannot be proven local before connection.

The runner must fail before database action when proof is missing or rejected.

## No-Output Rules

The future harness must never print or persist:

- Database target values.
- Database URLs.
- Environment values.
- Service-role values.
- Provider keys.
- JWTs or private tokens.
- Hosted URLs.
- Bucket names.
- Personal organization, workspace, project, or user identifiers.
- Row payloads.
- Claim values.
- Auth headers.

Logs may show only:

- Scenario names or stable labels.
- Declared, executed, passed, failed, and blocked counts.
- Artifact SELECT isolation verified yes/no.
- Sanitized failure classifications.
- Local target proof status without the target value.

## SQL Static Guard Policy

Before execution, the future runner should inspect the artifact assertion SQL and fail closed on unsafe durable operations.

Allowed local assertion patterns:

- Transaction start and rollback.
- Transaction-scoped timeouts.
- Transaction-scoped session-claim setup.
- Temporary tables.
- Generated local fixture rows.
- SELECT-based assertions.
- Sanitized result summaries.

Blocked patterns unless AP explicitly approves a later exception:

- Durable policy creation, alteration, or deletion.
- Durable table creation, alteration, or deletion.
- Function creation.
- Extension creation.
- Durable grant or revoke changes.
- Commit.
- Fixture mutation outside transaction scope.
- Service-role behavior.
- Provider execution.
- Target, secret, row-payload, or claim-value output.

## Assertion File Boundary

The planned artifact assertion file is:

- `docs/schema/m5.3a_2_artifact_select_isolation_assertions.sql`

The assertion file should remain a local test contract. It must not define production policy behavior, helper functions, migrations, or durable data. It must not be treated as storage/export, provider, lineage, or mutation coverage.

## Failure Behavior

The future artifact runner must return non-zero when:

- Target proof is missing.
- Target proof is rejected.
- Local database connection is unavailable.
- The assertion SQL is missing.
- Static SQL guard fails.
- Scenario metadata is inconsistent.
- Any scenario fails.
- Any approved scenario remains blocked.
- Cleanup or rollback cannot be confirmed.
- Output sanitization fails.

Zero exit is acceptable only when AP-approved local artifact assertions execute and pass with sanitized output.

## Proof Boundary

Passing the future artifact harness can prove only local artifact SELECT behavior for the scoped tables and scenarios. It does not prove hosted runtime readiness, mutation safety, storage/export readiness, provider readiness, operations readiness, or legacy draft table coverage.

## AP Decisions Needed

AP must approve:

- Whether the runner is separate or generalized.
- Exact accepted local target forms.
- Static SQL guard allow and block lists.
- Whether a package script is added.
- Whether a local configuration file remains out of scope.
- Exact conditions for artifact SELECT isolation evidence.
