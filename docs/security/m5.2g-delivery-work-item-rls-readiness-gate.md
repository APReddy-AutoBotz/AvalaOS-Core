# M5.2g Delivery Work Item RLS Readiness Gate

Milestone: M5.2g Delivery Work Item Ownership Migration Plan

Branch: milestone/m5.2g-delivery-work-item-ownership-migration-plan

Status: RLS readiness only; no RLS implementation

## Purpose

This gate defines the future RLS-readiness expectations for Delivery work-item authority. It does not implement RLS policies, schema, runtime behavior, seed data, backfill, hosted validation, or tenant isolation tests.

## Readiness Boundary

M5.2g is docs-only. It does not create `delivery_work_items`, promote draft Delivery tables, or change runtime adapters.

Future M5.2g-a may create `delivery_work_items` and enable RLS with no policies. That would be fail-closed readiness only, not tenant-isolation proof.

Tenant isolation remains unproven until future AP-approved RLS policies and isolation tests are implemented.

## Recommended Authority Shape

The preferred future authority path is Option B: one narrow `delivery_work_items` table before any promotion of draft Agile-like tables.

This table should carry:

- organization ownership through `org_id`;
- nullable-first workspace ownership through `workspace_id`;
- project ownership through `project_id`;
- optional source links to `document_generations`, `assess_processes`, and `assessments`;
- actor links to profiles for created, updated, owner, assigned, and reporter roles;
- lifecycle fields such as status, archived marker, deleted marker, timestamps, and safe audit correlation.

## Future Workspace-Scoped Reads

Future RLS should require active workspace membership for workspace-scoped Delivery work-item reads.

Organization membership alone should not be enough for workspace-scoped Delivery artifacts after workspace ownership is present.

If AP approves any organization-level reader behavior, it must be explicit and tested.

## Cross-Tenant Denial

Future policies and tests must deny:

- anonymous reads of tenant Delivery work-item data;
- authenticated non-member reads;
- disabled-member reads;
- cross-organization reads and writes;
- cross-workspace reads and writes;
- writes to projects or work items outside authorized workspace scope;
- source refs to Assess, Studio/Docs, handoff, evidence, or document records outside the same tenant boundary.

## Role And Permission Direction

Future RLS should distinguish:

- Buyer Viewer read-only access where AP approves it;
- Delivery Lead permitted updates within authorized project/workspace scope;
- assigned contributor own-work updates only;
- reviewer or approver gates where later persistence models include review or approval state;
- admin/server workflows with explicit actor and correlation context.

App-side Delivery policy checks remain useful for buyer-demo behavior, but they are not substitutes for database RLS.

## App ID Hardening

If `app_id` is used in future `delivery_work_items`, it must remain a nullable compatibility bridge only.

`app_id` is not:

- tenancy authority;
- globally unique;
- a Supabase identifier;
- a replacement for `id`;
- an authorization input by itself.

Future uniqueness should be scoped to active, non-deleted rows by at least `(org_id, project_id, app_id)`.

## Metadata And Source Lineage Hardening

Future `metadata` and `source_lineage` fields are compatibility payloads only. They are not durable authority for evidence, approval, lineage, export, storage, review, or cross-tenant references.

They must not store:

- secrets;
- provider keys;
- service-role values;
- raw private tokens;
- personal hosted values;
- unsupported certification claims.

Extra assignees, dependencies, subtasks, user stories, and order rank may remain compatibility data only until AP approves normalized authority.

## FK And Delete Behavior Direction

Future M5.2g-a should avoid broad hard-delete cascade for tenant artifact history.

Tenant/source/project/document references should use default/no-cascade delete behavior:

- `org_id -> organizations(id)`;
- `(workspace_id, org_id) -> workspaces(id, org_id)`;
- `(project_id, org_id) -> projects(id, org_id)`;
- `(document_generation_id, org_id) -> document_generations(id, org_id)`;
- `(source_process_id, org_id) -> assess_processes(id, org_id)`;
- `(source_assessment_id, org_id) -> assessments(id, org_id)`.

Actor/profile references may use `ON DELETE SET NULL`.

Use staged `NOT VALID` FKs where existing/manual rows may exist. Do not validate constraints, seed data, backfill data, or perform destructive cleanup in the first implementation slice unless AP separately approves it.

## Legacy Draft Schema Risk Gate

Legacy draft/manual Delivery tables may exist in non-production databases:

- `epics`;
- `sprints`;
- `tasks`;
- `task_comments`;
- `task_activity_events`;
- `timesheet_entries`.

Option B should not alter, drop, or claim to secure those tables in M5.2g-a.

If AP later chooses draft-table promotion, a separate hardening plan must address:

- legacy organization-scoped policies;
- cascade FKs;
- single-column project FKs;
- timesheet/task cascade behavior;
- draft `app_id` uniqueness;
- migration away from metadata-only authority assumptions.

## Source, Evidence, And Handoff Boundaries

Future work-item policies must treat source refs as tenant-scoped.

A work item must not use `source_lineage`, `metadata`, or `app_id` to bypass same-organization and same-workspace ownership checks.

Source refs to Assess and Studio/Docs records should be same-organization at minimum and workspace-aware where AP approves workspace mapping.

Handoff, evidence, and lineage authority remain deferred. M5.2g does not create standalone authority for them.

## Delivery Pack, Monitor, And Export Boundaries

Delivery Pack persistence remains deferred until work-item authority and export/evidence boundaries are approved.

Monitor project/task signals remain derived until AP approves a persistence model. M5.2g does not create live telemetry or runtime monitoring schema.

Export records, storage object references, and server-mediated downloads remain deferred to export controls.

## Browser And Secret Boundary

Delivery work-item records must not store provider secrets, service-role keys, database URLs, private tokens, personal hosted values, raw secret material, or unsupported certification claims.

The browser must not receive service-role credentials or raw provider secrets. Any future service-role operation must be limited to trusted server or Edge paths with actor, tenant, correlation, and audit context.

## Future Negative Isolation Tests

Future RLS tests should prove:

- anonymous users cannot read Delivery work-item data;
- authenticated non-members cannot read work items;
- disabled members cannot access work items;
- workspace non-members cannot read workspace work items;
- users in one organization cannot read or mutate another organization's work items;
- users in one workspace cannot read or mutate another workspace's work items;
- Buyer Viewer cannot mutate work items, Delivery Pack records, Monitor signals, exports, or provider metadata;
- Delivery Lead can update only approved fields within authorized scope;
- assigned contributors can update only approved own-work fields;
- unassigned contributors cannot update another user's work;
- source lineage, evidence, handoff, document, process, and assessment refs cannot cross tenant boundaries;
- archived or deleted records do not grant accidental access;
- browser paths cannot invoke service-role-only workflows.

## Future Positive Tests

Future RLS tests should prove:

- active workspace members can read authorized work items;
- Delivery Lead can update approved work-item fields;
- assigned contributors can update approved own-work fields;
- Buyer Viewer can read approved records where AP approves that behavior and cannot mutate them.

## RLS Readiness Decision

M5.2g is not an RLS implementation milestone. M5.2g-a may create `delivery_work_items` and enable fail-closed RLS without policies, but M5.3a or a later approved RLS milestone must implement and test actual tenant-isolation policies.
