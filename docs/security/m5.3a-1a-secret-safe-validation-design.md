# M5.3a-1a Secret-Safe Validation Design

## Purpose

This security plan defines the secret-safe boundary for a future local-only M5.3a-1 RLS validation harness. It is planning only and does not create the harness, SQL, scripts, package commands, local config, tests, helper functions, or RLS policies.

## Security Principles

- Local-only validation by default.
- Fail closed when local proof is missing.
- Never connect to remote Supabase.
- Never print secret values.
- Never print database target values.
- Never read or expose `.env` values.
- Never commit local secret files.
- Never use personal hosted identifiers.
- Report counts and sanitized status only.

## No-Output Rules

Future harness logs must not print:

- `DATABASE_URL` values.
- Database target values.
- `.env` values.
- Service-role values.
- Provider keys.
- JWT or private token values.
- Hosted Supabase URLs.
- Bucket names.
- Personal org, workspace, project, or hosted project identifiers.
- Row payload dumps.
- Auth headers.
- Secret reference values that AP has not approved for display.

Allowed output:

- Scenario labels.
- Sanitized actor labels such as `active_member` or `cross_org_user`.
- Pass/fail status.
- Aggregate counts.
- Sanitized failure classifications.
- Command names without secret arguments.

## Remote-Target Rejection

Future local validation must reject remote targets before any database action.

Allowed local target classes:

- `localhost`.
- `127.0.0.1`.
- Docker local network.
- Another AP-approved local-only target.

Rejected target classes:

- Hosted Supabase URLs.
- Remote database hosts.
- Personal project refs.
- Targets loaded silently from ignored secret files.
- Environment values accepted without local proof.

The runner must not log the target value when rejecting it.

## `DATABASE_URL` Boundary

`DATABASE_URL` is server-only and environment-specific. Future harness work must not treat it as a default safe path.

Rules:

- Do not run `DATABASE_URL`-based commands in M5.3a-1a.
- Do not print `DATABASE_URL`.
- Do not silently accept `DATABASE_URL`.
- Do not rely on ignored local files to provide it.
- If AP later approves a `DATABASE_URL` strategy, it must be local-only, wrapped by target rejection, and no-value logged.

## `scripts/applySupabaseMigrations.mjs` Boundary

`scripts/applySupabaseMigrations.mjs` currently requires `DATABASE_URL`. It is not the default M5.3a-1 harness path.

Future use requires explicit AP approval for:

- Local-only wrapper behavior.
- Target rejection.
- No secret output.
- No remote fallback.
- Evidence that the DB URL value is never printed.

## `supabase/config.toml` Boundary

M5.3a-1a does not create or change `supabase/config.toml`.

If AP approves a future tracked config, it must:

- Be generic.
- Be portable.
- Be local-only.
- Be safe to commit.
- Contain no hosted URLs, project refs, keys, tokens, service-role values, provider keys, bucket names, org IDs, or personal values.

## Fixture Data Boundary

Future fixtures must:

- Use generated local UUIDs.
- Use neutral local labels.
- Avoid personal identifiers.
- Avoid hosted project values.
- Avoid provider keys.
- Avoid service-role values.
- Avoid bucket names.
- Avoid `.env` values.
- Avoid raw row dumps in output.

Fixture groups:

- Local organizations.
- Local workspaces.
- Local users.
- Local memberships.
- Local scoped roles.
- Deleted or disabled variants.

## Auth And Client Boundary

Future tests must separate:

- Anonymous local client behavior.
- Authenticated local active member behavior.
- Authenticated local disabled member behavior.
- Authenticated local non-member behavior.
- Server-only fixture setup behavior.

Server-only setup must never be represented as browser-callable behavior. Service-role behavior must remain server-only and must not be printed or exposed.

## Policy Scope Boundary

Future M5.3a-1 harness validation is limited to SELECT behavior for:

- `profiles`
- `organizations`
- `workspaces`
- `roles`
- `organization_members`
- `workspace_memberships`

The harness must not validate:

- Artifact table policies.
- Provider governance policies.
- Legacy draft table policies.
- Storage/export policies.
- Delivery Pack, Monitor, lineage, or handoff policies.
- INSERT, UPDATE, DELETE, hard-delete, soft-delete, review, approval, export, or delivery mutation policies.

## Static Review Gates

Future harness implementation must be blocked if static review finds:

- Tracked `.env*` files.
- DB URL output.
- Service-role output.
- Provider key output.
- Remote Supabase URL.
- Personal hosted identifier.
- Remote validation command.
- Default `applySupabaseMigrations` remote use.
- Broad `FOR ALL` policy acceptance.
- Mutation or DELETE policy validation in M5.3a-1.
- Artifact-table policy validation before M5.3a-2.
- Secret-looking literals.

## Evidence Rules

Evidence may record:

- Files inspected.
- Files changed.
- Scenario labels.
- Pass/fail counts.
- Sanitized failure categories.
- Local-only target proof status without target value.
- Confirmation no secrets were exposed.

Evidence must not record:

- DB URLs.
- `.env` values.
- Service-role values.
- Provider keys.
- Hosted project URLs.
- Bucket names.
- Personal hosted identifiers.
- Row payload dumps.
- JWT/private token values.

## Static-Review-Only Override

Static-review-only implementation is not the default. If AP overrides and permits it, the evidence must state:

- Local RLS behavior was not executed.
- Tenant isolation is not verified.
- Runtime hosted readiness is not claimed.
- M5.3a-1 remains risk-bearing until local validation is available.

## Risks

- Accidental remote targeting through environment-derived values.
- Secret leakage through verbose logs.
- Row payload dumps exposing tenant data or secret-like metadata.
- Static review creating false confidence.
- Server-only fixture setup being confused with browser-callable behavior.
- Future config accidentally containing personal hosted values.
