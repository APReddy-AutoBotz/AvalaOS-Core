# M5.3a-2b Local DB Execution Target Safety

## Purpose

This document defines the target-safety and secret-safety plan for future local DB execution evidence using the M5.3a-2 artifact assertion harness.

M5.3a-2b is docs-only. It does not run local assertions, connect to any database, change the runner, change assertion SQL, or create local configuration.

## Current Safety Baseline

The M5.3a-2a artifact runner is accepted as:

- Local-only.
- Explicit-target gated.
- Hosted-target rejecting.
- SQL-guarded.
- Fail-closed.
- Counts-only sanitized.

Current closure evidence shows no-target execution fails closed, hosted target input is rejected before database action, and `localhost` target proof is accepted but local DB execution is blocked when no reachable local DB is available.

Artifact SELECT isolation remains not verified.

## Target Proof Requirements

Future execution must prove a local target before any database action.

Allowed target forms:

- Explicit `localhost`.
- Explicit `127.0.0.1`.
- Another loopback-only target only if AP separately approves it.

Rejected target forms:

- Hosted database URLs.
- Remote hosts.
- Silent environment database targets.
- Automatic `DATABASE_URL` execution.
- Ignored secret-file fallback.
- Unknown target arguments.
- Any target that cannot be proven local before connection.

The runner must fail before DB action when target proof is missing or rejected.

## No-Output Rules

Future local execution evidence must never print or persist:

- DB URLs.
- Host values.
- Port values when treated as target-sensitive.
- `.env` values.
- Service-role values.
- Provider keys.
- JWT or private token values.
- Hosted identifiers.
- Bucket names.
- Personal local paths.
- Fixture UUIDs.
- Organization, workspace, project, or user IDs.
- Row payloads.
- Claim values.
- Auth headers.

Allowed output:

- Command names.
- Sanitized target proof status.
- Scenario counts.
- Pass, fail, and blocked counts.
- Artifact SELECT isolation verified yes/no.
- Sanitized failure classifications.
- Exit-code category.

## Migration And Bootstrap Safety

M5.3a-2b does not run migrations.

Future local execution evidence must identify the approved local schema preparation method without exposing target values or secrets. It must not use remote Supabase, hosted targets, or implicit environment-driven database access.

`scripts/applySupabaseMigrations.mjs` and `npm run db:migrate` remain out of scope unless AP later approves a local-only invocation path.

## Fixture Safety

Future assertion execution must keep fixtures local and rollback-scoped.

Fixture rules:

- Generated local IDs only.
- No production data.
- No durable fixture data.
- No row payload dumps.
- No fixture UUID output.
- No claim-value output.
- No service-role output.

## Evidence Safety Decision

Future evidence may mark local artifact SELECT isolation verified only if:

- The target proof is local and accepted.
- The local DB is reachable.
- Required migrations and policies are present locally.
- All 18 artifact scenarios execute and pass.
- Output remains sanitized.

Any target rejection, missing target proof, local DB unavailability, missing schema, assertion failure, blocked scenario, or unsafe output means artifact SELECT isolation is not verified.

## Proof Boundary

Passing future local artifact assertions proves only scoped local artifact SELECT behavior. It does not prove hosted readiness, production readiness, mutation safety, storage/export readiness, provider readiness, runtime readiness, operations readiness, certification, legacy draft table isolation, or full tenant isolation.
