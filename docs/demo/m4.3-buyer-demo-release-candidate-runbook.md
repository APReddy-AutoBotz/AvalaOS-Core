# M4.3 Buyer-Demo Release Candidate Runbook

## Purpose

This runbook packages the buyer-demo release candidate path for AvalaOS Core after M4.2h.

It is a demo evidence and review artifact only. It does not add product features, change data, change routes, call providers, deploy, or assert enterprise-production readiness.

Release candidate scope:

`Avala Demo Enterprise -> AP Invoice Exception Handling / AP Invoice Exception Workflow -> Avala Assess -> Avala Govern -> Avala Studio -> Avala Delivery -> Avala Monitor`

AvalaOS brand meaning:

`AvalaOS = Assess. Validate. Align. Launch. Audit.`

Core thesis:

- Evaluate before you automate.
- Govern before you execute.
- Humans approve risk.
- Evidence proves every decision.

Positioning:

- AvalaOS is buyer-demo ready for the governed AP workflow.
- AvalaOS is not yet enterprise-production deployed.
- Enterprise production readiness requires M5 Supabase/auth/RLS/audit/export/deployment hardening.

## Clean Browser Setup

Use a clean browser profile, private session, or cleared site data for the local app origin before every release-candidate rehearsal.

Setup checklist:

1. Close any stale local app tabs.
2. Clear site data for the local app origin, or use a fresh/private browser profile.
3. Reopen the app from the intended local URL.
4. Confirm no older user, view, scope, generated artifact, dashboard layout, handoff ledger, or module setting appears.
5. Sign in with the canonical first persona.
6. Confirm the active organization is `Avala Demo Enterprise`.
7. Confirm the canonical process is `AP Invoice Exception Handling`.
8. Confirm the canonical delivery project is `AP Invoice Exception Workflow`.

If old examples, unrelated generated documents, or stale view state appear, stop and clear site data again.

## Screen-By-Screen Flow

### 1. Login / First Impression

Expected:

- AvalaOS Core is presented as a governed enterprise workbench.
- The brand meaning is visible or available in the demo talk track: `Assess. Validate. Align. Launch. Audit.`
- The demo path is positioned around one AP Invoice Exception story.

Talk track:

- "AvalaOS helps teams evaluate automation opportunities before delivery work starts."
- "This walkthrough follows one governed AP Invoice Exception workflow."

### 2. Avala Assess

Expected:

- Open `Avala Assess`.
- Confirm `AP Invoice Exception Handling`.
- Confirm process id `proc-ap-invoice-exception`.
- Confirm the process is ready for assessment and review.

Talk track:

- "The demo starts with assessment, not automation."
- "We evaluate the process before any handoff."

### 3. Guided Assessment And Deterministic Score

Expected:

- Assessment id: `assess-proc-ap-invoice-exception`.
- Evidence, assumptions, and review context are present.
- Score and recommendation are deterministic outputs from the scoring engine.

Talk track:

- "This is deterministic scoring, not AI decisioning."
- "The score explains fit, value, risk, and readiness; it does not approve execution."

### 4. Decision Pack

Expected:

- Decision Pack explains rationale, controls, evidence, assumptions, risk, and handoff readiness.
- Human review remains required.

Talk track:

- "The Decision Pack makes the assessment explainable and reviewable."
- "Humans approve risk and delivery."

### 5. Avala Govern

Expected:

- Avala Govern is visible as the control-card step between Assess and Studio.
- It remains existing-context-backed: Decision Pack, control card, human review, evidence, assumptions, and governed handoff.
- It is not a new standalone route or expanded Govern product in this release candidate.

Talk track:

- "Avala Govern is the control moment before handoff."
- "Govern before you execute."

### 6. Avala Studio

Expected:

- Open the Assess-to-Studio handoff.
- Confirm AP source context is attached.
- Confirm generated documents are review artifacts.
- Do not rely on live provider execution for the standard external demo.

Talk track:

- "Studio drafts from approved source context."
- "Generated docs are review artifacts, not final approvals."

### 7. Avala Delivery

Expected:

- Open `AP Invoice Exception Workflow`.
- Confirm project id `proj-ap-invoice-exception`.
- Confirm work items are traceable to AP source context and evidence where the model supports it.

Talk track:

- "Delivery work is traceable to source context and evidence."
- "The backlog is the governed AP workflow execution plan, not a separate disconnected task list."

### 8. Delivery Pack

Expected:

- Open Delivery Pack `pack-ap-invoice-exception`.
- Confirm handoff ids `handoff-ap-assess-docs` and `handoff-ap-docs-delivery`.
- Confirm sources, evidence references, blockers, approval checklist, and audit summary are available for review.
- Export block/label enforcement remains deferred to M4.1g.

Talk track:

- "The Delivery Pack assembles the evidence trail."
- "Export enforcement is a separate policy-hardening milestone."

### 9. Avala Monitor

Expected:

- Review AP project/task/readiness signals.
- Treat Monitor Insights as partial if visible.
- Do not present Monitor as live production telemetry.

Talk track:

- "Monitor currently shows demo project/task/readiness signals, not live production telemetry."
- "The close is evidence-backed visibility into value, risk, blockers, and handoff status."

## Stop And No-Go Claims

Do not claim:

- HIPAA compliance.
- SOC 2 compliance.
- GDPR compliance.
- GxP compliance.
- certified compliance.
- autonomous approval.
- AI decides scores, gates, risks, or recommendations.
- live production monitoring.
- live provider execution unless explicitly enabled and verified.
- this is fully production deployed.

Stop the demo if:

- the AP process is missing;
- the AP delivery project is missing;
- the app opens on stale data that cannot be explained quickly;
- generated artifacts are unrelated to AP source context;
- Delivery Pack evidence or lineage appears disconnected;
- Monitor shows unrelated project/task signals.

| Export policy | "Delivery Pack exports are review artifacts until M4.1g export block/label policy is implemented." |
| Reports, Teams, Admin, AI Controls | "Those areas remain deferred or AP-decisioned and are outside the buyer-demo path." |
