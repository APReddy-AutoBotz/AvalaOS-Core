# M4.2d Buyer-Demo Golden Path Validation Runbook

## Purpose

This runbook validates the current canonical AP Invoice Exception Handling buyer-demo path after M4.2c-1 and M4.2c-2.

It is a rehearsal and reset-readiness checklist. It does not add features, change data, change routing, call providers, or deploy anything.

Canonical story:

`Avala Demo Enterprise -> AP Invoice Exception Handling / AP Invoice Exception Workflow`

Golden path:

Organization -> Avala Assess -> Deterministic Score -> Decision Pack -> Govern Lite control context -> Avala Studio -> Generated review artifacts -> Delivery Work Items -> Delivery Pack -> Avala Monitor project/task signals.

## Pre-Demo Clean State

Use a clean browser profile, private browser session, or clear site data for the local app origin before a buyer-demo rehearsal.

Minimum clean-state checklist:

1. Stop any running local app tab.
2. Clear site data for the local app origin used in rehearsal.
3. Reopen the app from the local dev URL.
4. Sign out if a previous user session appears.
5. Sign in with the canonical Process Analyst persona first.
6. Confirm the organization is `Avala Demo Enterprise`.
7. Confirm the configured modules include Assess, Studio, Delivery, and Monitor.
8. Confirm the first visible process story is `AP Invoice Exception Handling`.
9. Confirm the delivery project story is `AP Invoice Exception Workflow`.
10. Confirm generated docs and Delivery Pack records reference the canonical AP story.

Browser persisted state can mask new canonical defaults. If an older dashboard layout, user, scope, view, template set, handoff ledger, or organization-module setting appears, clear site data again and restart the rehearsal.

## Golden-Path Walkthrough

### 1. Dashboard Or Landing State

Expected validation:

- The app should open in a governed workbench shell.
- Sidebar and ModuleJourney should expose the enabled modules for the current persona.
- If starting from Monitor or Portfolio, frame it as a status view, then move to Avala Assess to begin the evidence-backed story.

Talk track:

- "This is one controlled buyer-demo story: AP invoice exception handling."
- "Evaluate before you automate. Govern before you execute."

### 2. Avala Assess

Steps:

1. Open `Avala Assess`.
2. Confirm the Process Catalog includes `AP Invoice Exception Handling`.
3. Open the process detail.

Expected validation:

- Process ID: `proc-ap-invoice-exception`.
- Process name: `AP Invoice Exception Handling`.
- Department: Finance Operations / Accounts Payable.
- The process is completed or review-ready for the demo path.

Talk track:

- "The demo starts by evaluating the business process before any delivery handoff."
- "The assessment captures evidence, assumptions, and human review context."

### 3. Guided Assessment

Steps:

1. Open the process assessment or Decision Pack from Process Detail.
2. Review assessment status, evidence, assumptions, and approval history.

Expected validation:

- Assessment ID: `assess-proc-ap-invoice-exception`.
- Evidence includes AP exception map, SOP, SAP sample summary, exception report, policy, and owner review note.
- The demo assessment is source context for downstream work, not a production approval.

Talk track:

- "Evidence proves every decision."
- "Humans approve risk and delivery."

### 4. Deterministic Score And Decision Pack

Steps:

1. Review score, recommendation, confidence, risk tier, and readiness.
2. Open the Decision Pack content in Process Detail or assessment export context.

Expected validation:

- Score and Decision Pack come from deterministic scoring outputs.
- No model-owned scoring, approval, risk, recommendation, or compliance claim is made.
- Decision Pack provides rationale, controls, evidence, assumptions, and handoff readiness.

Talk track:

- "This is deterministic scoring, not AI decisioning."
- "The score explains fit and readiness; it does not approve execution."

### 5. Govern Lite Control Context

Steps:

1. Review the Govern Lite card in Process Detail or Delivery Pack.
2. Confirm governance status, autonomy level, evidence policy, approval policy, blocked actions, and next governance action.

Expected validation:

- Govern Lite is visible in the top lifecycle as the control-card step between Avala Assess and Avala Studio.
- It remains attached to Decision Pack/Delivery Pack context rather than becoming a full standalone module or route.

Talk track:

- "Govern Lite keeps the AP story review-first and evidence-backed."
- "The demo allows drafting and planning, while finance owner review remains required for action."

### 6. Avala Studio Source Context

Steps:

1. Use `Open Avala Studio handoff` from Process Detail when available.
2. Confirm Studio source context references AP Invoice Exception Handling.

Expected validation:

- Source context label: `AP Invoice Exception Handling Assess source context`.
- Source context carries process id, assessment id, Decision Pack, Handoff Pack, Govern Lite summary, evidence refs, and assumptions.
- No live provider execution is required for this canonical demo data validation.

Talk track:

- "Studio drafts from approved source context."
- "Generated docs are review artifacts."

### 7. Generated Review Artifacts

Steps:

1. Open generated docs or the Workspace generated artifacts view.
2. Inspect the canonical document generation.

Expected validation:

- Document generation ID: `docgen-ap-invoice-exception`.
- Artifacts include BRD/PDD/FRD-style review content, quality gate content, diagrams, generated work items, and pending human review approvals.
- Generated documents are review artifacts, not approved compliance artifacts.

Talk track:

- "Generated documents are drafts for review."
- "They carry source context so reviewers can trace back to Assess evidence."

### 8. Delivery Work Items

Steps:

1. Open Avala Delivery Lite.
2. Select `AP Invoice Exception Workflow`.
3. Review work items, owners, status, priority, blockers, and lineage indicators.

Expected validation:

- Project ID: `proj-ap-invoice-exception`.
- Work items are AP-specific.
- All canonical delivery tasks carry complete source lineage where the current task model supports it.
- Evidence refs include AP evidence such as `ev-ap-owner-review`.

Talk track:

- "Delivery work is evidence-backed and traceable to the assessment."
- "The backlog is not a separate story; it is the AP workflow execution plan."

### 9. Delivery Pack

Steps:

1. Open Delivery Pack from the AP project.
2. Review sources, documents, work item lineage, approval checklist, evidence checklist, blockers, and audit summary.

Expected validation:

- Delivery Pack ID: `pack-ap-invoice-exception`.
- Handoff IDs: `handoff-ap-assess-docs`, `handoff-ap-docs-delivery`.
- Exports remain review artifacts and should not be presented as enforceable export-policy output before M4.1g.

Talk track:

- "The Delivery Pack assembles the evidence trail, not a final external approval."
- "Export policy hardening is a separate milestone."

### 10. Avala Monitor Project/Task Signals

Steps:

1. Open Avala Monitor, Portfolio, or dashboard widgets.
2. Confirm the AP project and task state drive readiness, blocker, and status signals.
3. Review Monitor Insights status if visible.

Expected validation:

- Monitor project/task signals derive from AP delivery state.
- Monitor Insights may be disabled or partial.
- Demo signals are review/readiness signals, not live production telemetry.

Talk track:

- "Monitor shows readiness and delivery signals from the governed AP workflow."
- "Live production monitoring is not part of this local demo."

## Stop Points And No-Go Points

Stop the demo and reset if:

- The app opens on an old process, old project, or unrelated generated document story.
- `AP Invoice Exception Handling` is missing from Avala Assess.
- `AP Invoice Exception Workflow` is missing from Delivery.
- Generated docs do not show AP source context.
- Delivery Pack lineage or evidence references are missing from canonical tasks.
- Monitor shows unrelated project/task status.

Do not claim:

- compliance certification;
- autonomous approval;
- live production monitoring;
- live provider execution if providers are not enabled;
- production SAP posting;
- external vendor communication;
- final risk approval without human review;
- export-policy enforcement before M4.1g.
