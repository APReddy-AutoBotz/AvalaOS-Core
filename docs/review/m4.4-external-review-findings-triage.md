# M4.4 External Review Findings Triage

Milestone: `M4.4 External Review Findings Triage`

Branch: `milestone/m4.4-external-review-findings-triage`

## Purpose

This document triages external review findings from the M4.3 buyer-demo release candidate review cycle.

M4.4 is triage only. It records which Claude UI/UX and Gemini product/governance findings should be fixed in the next buyer-demo release-candidate pass, which can be handled through talk track, which belongs to enterprise hardening, and which belongs only to the long-term roadmap.

No product fix is implemented in this milestone.

## Review Sources

- Claude UI/UX review: premium enterprise UI/UX review focused on visual trust, labels, flow clarity, demo interruptions, and polish.
- Gemini product/governance review: product, positioning, governance, enterprise-readiness, and buyer-objection critique.
- Gemini AGI-era strategic reflection: long-term strategic framing for future AI governance and alignment control-plane ideas.

## Overall External Verdict

The external reviews agree that AvalaOS Core has a credible buyer-demo story and a strong governance thesis.

- Claude rated the premium enterprise impression at `8.5 / 10`.
- Gemini rated enterprise buyer relevance at `8.5 / 10`.
- Gemini rated governance credibility at `8.5 / 10`.

The shared verdict is positive, but the reviews identify a small buyer-demo fix shortlist around label clarity, demo interruption risk, role wording, duplicated visual symbols, Monitor positioning, and Studio/human-review wording.

The main product risk is category confusion. AvalaOS must continue to be presented as a governed AI and automation delivery operating system: it evaluates, governs, drafts, hands off, and proves lineage. It must not be positioned as a runtime execution layer, generic PM tool, process mining platform, chatbot, or fully deployed enterprise control plane.

## Combined Must-Fix Shortlist

M4.4a should fix only confirmed buyer-demo issues:

1. Rename the header profile action from `Switch` to `Sign Out`.
2. Rename plain `Portfolio` in buyer-facing sidebar/header surfaces to `Avala Portfolio` if still visible.
3. Replace reviewer `View Only` access wording with review-capable wording.
4. Give `Document Vault` a distinct sidebar icon.
5. Clarify Monitor as readiness, lineage, blockers, value, and handoff status signals, not live production telemetry.
6. Clarify AvalaOS governs execution handoff but does not execute bots, agents, or external systems.
7. Clarify Avala Studio documents are editable review drafts requiring human sign-off.
8. Improve missing workspace/source-context fallback guidance if it can occur in the buyer-demo path.
9. Clarify `governed AI and automation delivery operating system` in plain buyer language.
10. Reinforce that generated artifacts need source context, SME/process-owner review, and human approval.

## Talk-Track Items

These items are acceptable for the buyer demo if handled honestly:

- Prepared AP demo data is used to keep the walkthrough controlled and coherent.
- AvalaOS is not a process mining platform and does not need live process mining integration for this demo.
- Reports is outside the primary AP path; use Avala Monitor and Delivery Pack for demo-ready evidence flow.
- Browser alerts are acceptable unless encountered in the core AP flow during rehearsal.
- WIP limits can be described as delivery-board guardrails, not a mature enforcement system.
- Provider selectors should not be used in the standard demo unless the environment is approved and verified.
- `OS` means governance and delivery operating system, not replacement runtime infrastructure.

## Enterprise-Hardening Backlog

These items should not be included in M4.4a:

- Replace browser alerts with custom toast/modal flows.
- Clean unused source files.
- Improve WIP limit warning styling and enforcement.
- Implement server-side AI execution through the governed backend path.
- Harden RBAC, SSO, Teams, workspace administration, and export policies.
