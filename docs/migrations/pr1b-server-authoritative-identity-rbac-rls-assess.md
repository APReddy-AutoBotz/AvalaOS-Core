# PR 1B Identity, RBAC, RLS, and Assess Migration

`20260712120000_pr1b_identity_rbac_rls_assess.sql` is the additive canonical migration for the PR 1B server-authority boundary.

It normalizes named capabilities into `capabilities` and `role_capabilities`, adds per-user/per-organization `authorization_versions`, and bumps those versions after membership, workspace-membership, role, or role-capability changes. `get_tenant_context(org, workspace)` derives the user exclusively from `auth.uid()` and returns a projection, never an authority token. The three mutation RPCs are an internal command boundary: execute is explicitly revoked from `PUBLIC`, `anon`, and `authenticated`, and granted only to `service_role`. Each accepts the actor derived and validated by Edge as its first parameter, then independently rechecks current organization membership, workspace membership, normalized capability, tenant ownership, authorization version, and expected aggregate version. The service credential is neither accepted from nor returned to a browser.

Assess process, assessment, and review-event lineage becomes workspace-complete. The migration refuses legacy rows with missing workspace authority using `PR1B_PREFLIGHT_ASSESS_WORKSPACE_REQUIRED`; organization memberships linked to absent, inactive, deleted, non-organization, or foreign roles fail with `PR1B_PREFLIGHT_ORGANIZATION_MEMBERSHIP_ROLE_INVALID`; workspace memberships with a role linked to the wrong organization/workspace or an inactive/deleted/non-workspace role fail with `PR1B_PREFLIGHT_WORKSPACE_MEMBERSHIP_ROLE_INVALID`. Legacy roleless workspace memberships remain supported consistently because capability may derive from the valid organization role; roleless organization memberships are rejected. Composite foreign keys prevent cross-workspace or cross-organization parentage. Permission-aware RLS exposes Assess rows only to active members with the required normalized capability, and cross-tenant filters return no rows or counts.

The create, response-upsert, and finalize RPCs use optimistic versions, scoped idempotency receipts, and required append-only audit events. State, successful receipt, and audit commit in one PostgreSQL transaction. Response-upsert rejects any aggregate that does not have exactly the typed `responses`, `metadata`, `evidenceItems`, and `assumptions` shape. Finalize accepts server-calculated deterministic results only across the server-only boundary and requires both score-version fields to equal the locked `assess-core-2026-05` value; the migration does not define or change scoring behavior.

RPC results are stable JSON envelopes. A first successful transaction returns `{ "ok": true, "outcome": "committed", "resource": { ... } }`; an identical idempotent replay returns the exact stored successful result. Expected failures return `{ "ok": false, "errorCode": "..." }`. Codes are `NOT_FOUND`, `AUTHORIZATION_STALE`, `VERSION_CONFLICT`, `IDEMPOTENCY_CONFLICT`, `INVALID_SCORE_VERSION`, and `INVALID_COMMAND`. Missing, revoked, unauthorized, and cross-tenant targets all collapse to `NOT_FOUND` and do not disclose existence. Any unexpected database exception rolls back the function subtransaction, emits only a sanitized server-side diagnostic, and returns `COMMAND_UNAVAILABLE`; raw SQL, tenant, credential, and infrastructure details are not returned.

Verification is split between `node scripts/checkPr1bMigrationContract.mjs` and the disposable PostgreSQL harness `PR1B_MIGRATION_DATABASE_URL=... node scripts/testPr1bMigrations.mjs`. The latter executes the real functions and verifies the privilege matrix; direct browser-role denial; forged score/version and malformed-response rejection; trusted create/upsert/finalize; exact replay and conflict behavior; concurrent identical commands; version and revocation rollback; non-disclosure; sanitized unexpected failure; atomic receipt/audit success; forced audit and receipt rollback; fresh application; populated supported upgrade; dirty-data transactional failure; migration reapplication; read-only fallback; and additive forward-fix preservation. It uses only disposable local databases and never contacts hosted infrastructure.

Rollback is operational, not destructive: disable the PR 1B command entry point or enter read-only maintenance mode while preserving Assess data, receipts, and audit history. Schema corrections use an additive forward-fix migration. Do not drop the normalized authorization or audit tables as a routine rollback.
