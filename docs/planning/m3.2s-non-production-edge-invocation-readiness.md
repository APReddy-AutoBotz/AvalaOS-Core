# M3.2s Non-Production Edge Invocation Readiness

## Milestone Purpose

M3.2s is a readiness-only checkpoint after M3.2r. Its purpose is to verify whether the approved non-production Edge invocation environment is available to the local execution process without exposing secrets, deploying Edge Functions, seeding data, invoking deployed functions, or calling provider APIs.

This milestone exists because M3.2r completed with a blocked outcome at deployed invocation credential readiness. M3.2s does not attempt the deployed smoke itself. It only determines whether M3.2t may proceed to AP/ChatGPT review for deployed blocked-smoke execution.

## Dependency Chain

M3.2s depends on the completed M3.2r milestone and its post-merge verification tag:

- `avalaos-core-m3.2r-non-production-deployed-edge-smoke-mocked-first`

M3.2r established that:

- Local resolver, integration, AI boundary, and prior mocked smoke checks were passing.
- Deployed smoke execution was blocked before seed, deploy, or invoke because required deployed invocation variables were unavailable.
- `SUPABASE_URL`, Supabase keys, and `AVALA_AI_RUNTIME_MODE` were not visible to the process used for M3.2r.

M3.2s narrows the next step to environment readiness only. The M3.2s hardening pass also verifies the same readiness check after loading `.env` into the current PowerShell process when the shell does not already expose the required variables.

## Strict Scope

M3.2s must not:

- Deploy Edge Functions.
- Invoke deployed Edge Functions.
- Seed the database.
- Call Gemini or Groq APIs.
- Use real provider keys.
- Print or store `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `DATABASE_URL`, provider keys, auth headers, or raw provider secret values.
- Use production, shared, customer, demo-customer, or unclear databases.
- Edit Edge Function entrypoints.
- Edit `_shared/ai.ts`.
- Edit runtime resolver or provider integration files.
- Change schema SQL, RLS, or migrations.
- Change Admin UI, product UI, Health, scoring, prompt versions, browser fallback, usage, or cost behavior.
- Add package dependencies.
- Make unsupported compliance claims.

## Required Environment Variables

Required for M3.2t deployed Edge invocation readiness:

| Variable | Required For M3.2s | Evidence Allowed |
| --- | --- | --- |
| `SUPABASE_URL` | Yes | Presence, parseability, host/project-ref classification only if safe. Full value must not be printed. |
| `SUPABASE_ANON_KEY` | Yes | Presence only. Value must not be printed. |
| `SUPABASE_SERVICE_ROLE_KEY` | Yes | Presence only. Value must not be printed. |
| `AVALA_AI_RUNTIME_MODE` | Yes | Presence and classification as `pilot` or `production`. |
| `DATABASE_URL` | No, unless local seed/verification becomes necessary in a later milestone | Presence only. Value must not be printed. |

Provider secret variables are explicitly out of scope for M3.2s. M3.2s must not check, print, or store any `AVALA_PROVIDER_SECRET_*` values. Live or mocked provider secret readiness remains future AP-gated work.

## Safe Environment Verification Method

The readiness check may run a local Node or Deno process that reads environment variable presence and emits only sanitized classifications. If the current shell does not already expose the required variables, the check may load `.env` into the current process first, without printing or storing values:

- Required variable visible: yes/no.
- Runtime mode visible: yes/no.
- Runtime mode classification: `pilot`, `production`, or `missing_or_invalid`.
- `SUPABASE_URL` parseability: yes/no.
- Safe host class only, such as `supabase-host` or `other-host`, without printing the full URL.
- Safe project-ref shape only, such as `standard-20-lower-alnum`, without printing the ref itself.
- Non-production marker result: pass/fail/not-confirmed.

If `AVALA_AI_RUNTIME_MODE` is `production`, M3.2s must stop and request AP approval before continuing. For this milestone, `pilot` is preferred.

## Non-Production Target Confirmation Method

M3.2s may confirm the target as non-production only when all of the following are true:

- `SUPABASE_URL` is visible and parseable without printing the full URL.
- The host/project-ref classification can be recorded without exposing secrets.
- The environment, host classification, runtime mode, and approved AP context are sufficient to rule out production, shared, customer, demo-customer, or live targets.
- `AVALA_AI_RUNTIME_MODE` is `pilot`.
- Disallowed target markers such as `prod`, `production`, `live`, `customer`, `shared`, or `demo-customer` are absent from the sanitized host classification check.
- No raw key, raw URL, auth header, database URL, provider key, or provider secret value is printed or stored.

If the target cannot be confirmed as non-production without exposing values, the milestone must record Outcome B blocked.

## Outcome Framework

### Outcome A - Ready

M3.2s may record Outcome A only if:

- Required Supabase invocation variables are visible to the process.
- `AVALA_AI_RUNTIME_MODE` is visible and equals `pilot`.
- The target identity is sufficiently confirmed as non-production without exposing secrets.
- No deploy, invoke, seed, provider call, schema change, UI change, Health change, or package change occurred.
- M3.2t deployed blocked-smoke execution may proceed after AP/ChatGPT review.

### Outcome B - Blocked

M3.2s must record Outcome B if:

- Any required Supabase invocation variable is unavailable.
- `AVALA_AI_RUNTIME_MODE` is missing, invalid, or set to `production` without AP approval.
- The target cannot be confirmed as non-production.
- Sanitized verification cannot run without risking secret exposure.

Outcome B keeps M3.2t blocked.

## M3.2t Go/No-Go Criteria

M3.2t may proceed to AP/ChatGPT review only when M3.2s records Outcome A.

M3.2t remains no-go when:

- Required invocation variables are missing.
- Runtime mode is missing, invalid, or `production` without explicit AP approval.
- Non-production target identity cannot be confirmed.
- Any readiness evidence would require printing or storing secrets.
- Any provider secret readiness, live provider call, mock switch, database seed, or deployed invocation requires AP approval not yet granted.

## Acceptance Criteria

- Exactly two docs are changed for M3.2s:
  - `docs/planning/m3.2s-non-production-edge-invocation-readiness.md`
  - `docs/quality/m3.2s-non-production-edge-invocation-readiness-evidence.md`
- Evidence records repo and remote safety.
- Evidence records sanitized env visibility without values.
- Evidence records runtime mode validation without broadening behavior.
- Evidence records non-production target confirmation or blocker.
- Required verification commands are run and recorded.
- Wording scan is run and reviewed.
- Readiness-only constraints are explicitly confirmed.

## Stop Conditions

Stop and record a blocked outcome if:

- `AVALA_AI_RUNTIME_MODE` is `production` and AP has not explicitly approved continuing.
- Any required Supabase invocation variable is unavailable.
- Non-production target identity cannot be confirmed without printing sensitive values.
- Any step would require deploy, deployed invocation, database seed, provider call, provider key use, schema change, UI change, Health change, scoring change, runtime code edit, package change, or unsupported compliance claim.
- Any source-of-truth conflict appears.

## Risks Carried Forward

- A safe deployed provider-handoff mock path is still not proven and remains out of scope for M3.2s.
- Live provider smoke remains blocked unless AP explicitly approves real provider key use and external provider calls.
- A controlled mock switch remains a separate AP-approved milestone if needed.
- M3.2t must re-run its own preflight before deploy, seed, invoke, or smoke execution because M3.2s proves only local process readiness at this point in time.
- Future deployed evidence must continue to exclude secrets, raw prompts, raw completions, provider payloads, full database rows, auth headers, service-role values, and raw `secret_ref` values beyond approved naming patterns.
