# M3.2p Non-Production Resolver Edge Smoke Mocked

## Milestone Purpose

M3.2p executes the AP/ChatGPT-approved next step after M3.2o: a controlled non-production resolver Edge smoke with mocked provider behavior first.

This milestone proves the resolver, audit, secret-reference, and provider-handoff ordering for the three M3.2n resolver-integrated operations without live provider network calls:

- `generate_document`
- `refine_section`
- `test_provider_connection`

## Scope

M3.2p is limited to a mocked smoke harness, supporting package script, and milestone evidence.

Allowed changes:

- Add `scripts/m3.2p-resolver-edge-smoke-mocked.mjs`.
- Add `npm run test:m3.2p-smoke`.
- Add M3.2p planning and evidence docs.

M3.2p does not:

- Call live Gemini or Groq APIs.
- Use real provider keys.
- Print or store `DATABASE_URL`, Supabase service-role keys, provider keys, auth headers, or Supabase secrets.
- Use production, shared, customer, or unclear databases.
- Deploy Edge Functions.
- Edit Edge Function entrypoints.
- Edit `_shared/ai.ts`.
- Edit runtime resolver or provider integration files.
- Change schema SQL, RLS, or migrations.
- Change Admin UI, product UI, Health, scoring, prompt versions, browser fallback, usage, or cost behavior.
- Add provider types beyond `gemini` and `groq`.
- Add package dependencies.
- Make unsupported compliance claims.

## Non-Production Guard

The smoke script requires `DATABASE_URL` to be visible to the process but does not print its value. It parses the URL only to apply a conservative non-production guard.

The guard blocks obvious production, shared, customer, or live markers in the database host and database name. The script prints only the visibility and guard result.

## Mocked Secret Boundary

The smoke uses only this mocked provider secret reference name:

- `AVALA_PROVIDER_SECRET_GROQ_SMOKE`

The script sets a fake in-process value for that reference and restores the prior environment state after execution. The fake value is not a real provider key and is not printed or written to evidence.

The smoke verifies that the wrong provider-prefix `secret_ref` blocks before any environment read.

## Database Row Strategy

The script opens one database transaction, seeds the minimum smoke-only rows, runs the scenarios, verifies audit rows, and rolls the transaction back.

Rows seeded only inside the transaction:

- One smoke auth user/profile mapping.
- Five harmless smoke organizations for allowed and blocked cases.
- One smoke role per organization.
- One active membership per organization.
- Smoke `ai_provider_key_refs` rows for active, disabled-config, manual-placeholder, wrong-prefix, and missing-policy contexts.
- Smoke `ai_provider_configs` rows with `pilot` mode and the three approved operations where applicable.
- Smoke `ai_workspace_provider_policies` rows for allowed operations only where required.
- Smoke `ai_provider_audit_events` rows written by the resolver audit path during the transaction.

Cleanup strategy:

- Roll back the transaction after success.
- Roll back the transaction after any blocked outcome.
- Do not delete unrelated rows.
- Do not retain smoke rows unless rollback itself cannot be confirmed, in which case evidence must stop and report Outcome B.

## Blocked Scenarios

The script runs blocked cases before allowed cases:

- Missing mode.
- Invalid mode.
- Missing policy.
- Disabled provider config.
- `manual_placeholder` key reference.
- Wrong provider-prefix secret reference.
- Allowed-path audit persistence failure.

Each blocked case verifies:

- The result is blocked.
- The expected failure class is returned.
- The mocked provider handoff is not called.
- The response is sanitized.
- The wrong provider-prefix case does not read the mocked secret environment reference.

## Allowed Mocked Scenarios

The script runs allowed mocked cases for:

- `generate_document`
- `refine_section`
- `test_provider_connection`

Each allowed case verifies:

- Resolver decision succeeds.
- Audit persistence happens before secret lookup.
- Secret lookup reads only the approved mocked provider reference.
- Mocked provider handoff is called exactly once.
- The handoff receives only the fake in-process value.
- The returned result is sanitized.
- The order is exactly:
  - resolver membership check
  - resolver policy check
  - resolver config check
  - resolver key-reference check
  - audit allowed event
  - secret lookup
  - mocked provider handoff

## Audit Sanitization

The smoke verifies the smoke audit rows inside the transaction and records only counts and pass/fail summaries.

Audit metadata must not contain:

- Raw provider keys.
- Secret reference values.
- `DATABASE_URL`.
- Supabase service-role keys.
- Auth headers.
- Raw prompts.
- Raw completions.
- Provider payloads.
- Stack traces.
- Raw database errors.

## Expected Outcome

Outcome A:

- Mocked smoke passes.
- `DATABASE_URL` is visible without value disclosure.
- Non-production guard passes.
- Governance tables are present.
- Seed rows are available only inside the transaction.
- Blocked cases pass.
- Allowed mocked cases pass.
- Audit sanitization passes.
- Transaction rollback confirms cleanup.

Outcome B:

- Smoke blocks before unsafe execution.
- Evidence records the safe stage and high-level failure class only.
- No secrets, database URL, provider payload, or raw database error is printed or stored.
