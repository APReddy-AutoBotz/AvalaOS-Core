# M4.1c Docs-to-Delivery Lineage / Evidence Refs Fix

## Purpose

M4.1c fixes the accepted M4.0a critical finding that generated work items imported from Avala Studio into Avala Delivery Lite did not retain source lineage or evidence references. The fix makes the Docs-to-Delivery handoff traceable when a valid Assess-to-Studio source context exists, while keeping docs-only imports visible as partial or missing lineage.

## Baseline

- Baseline milestone: `M4.1b Assess-to-Studio Full Handoff Payload Fix`
- Baseline tag: `avalaos-core-m4.1b-assess-to-studio-full-handoff-payload-fix`
- Baseline tag target: `684e7714dc8544350131aed59b71de6b8217ec50`
- Upstream source context: generated artifacts may carry `GeneratedArtifacts.sourceContext` from a valid Assess handoff.

M4.1b intentionally did not implement Delivery lineage. M4.1c consumes the artifact-level source context and attaches compact lineage to imported Delivery tasks without changing score logic, provider behavior, schema, RLS, migrations, Health, or Delivery Pack export policy.

## Scope

## Root Cause

Before M4.1c, `App.tsx` converted generated work item candidates into Delivery epics and tasks without copying document generation id, Assess source context, evidence refs, or handoff ledger ids onto the created tasks. The handoff ledger recorded generated work item titles as `evidenceRefs`, which were not actual evidence reference ids. Delivery Pack logic could read `task.sourceLineage`, but the import path did not populate or persist it.

## Minimal Fix Summary

- Added `TaskSourceLineageMetadata` fields for Docs-to-Delivery source module, generation, source artifact, source work item, upstream Assess context, evidence refs, assumption refs, completeness, notes, and creation timestamp.
- Added `services/docsToDeliveryLineage.ts` as a pure helper for building compact lineage from a generated artifact, document generation id, work item, and handoff ledger entry id.
- Updated `App.tsx` to persist temporary generated artifacts before import, attach source lineage to generated Story/Task records, and record a handoff ledger entry using real evidence ref ids.
- Updated `services/adapters/deliveryAdapter.ts` to round-trip `task.sourceLineage` through existing task metadata.
- Updated the generated work item review surface with a read-only lineage summary.
- Updated import modal wording to avoid implying autonomous generation ownership.
- Added focused regression coverage in `services/docsToDeliveryLineage.test.ts` and added `npm run test:docs-to-delivery-lineage`.

## Lineage Model

The imported Delivery task lineage is compact and intentionally non-authoritative. It points back to source records; it does not re-score, re-approve, or certify the work item.

| Field group | Purpose |
| --- | --- |
| `sourceModule`, `upstreamSourceModule` | Identifies Docs as the immediate source and Assess as the upstream source when present. |
| `documentGenerationId`, `sourceGenerationId` | Links the task to the saved document generation. |
| `processId`, `assessmentId` | Links to the Assess source context when generated from a valid Assess handoff. |
| `sourceArtifactKey`, `sourceArtifactTitle`, `sourceWorkItemTitle` | Identifies the generated artifact and candidate work item that produced the Delivery task. |
| `sourceDecisionPackRef`, `sourceHandoffPackRef` | Carries compact source pack reference ids when available. |
| `handoffLedgerEntryIds` | Links the task to the docs-to-delivery handoff ledger entry. |
| `evidenceRefs`, `assumptionRefs` | Carries upstream source evidence and assumption ids when present. |
| `lineageCompleteness`, `lineageNotes` | Labels complete, partial, or missing lineage without hiding gaps. |

## Completeness Rules

- `complete`: document generation id exists, Assess process/assessment ids exist, and evidence ref ids exist.
- `partial`: document generation id or Assess source context exists, but one or more lineage/evidence components are missing.
- `missing`: neither a document generation id nor Assess source context is available.

Docs-only/manual generated artifacts remain importable, but their imported tasks are marked partial or missing instead of receiving fake Assess lineage.

## Before And After

Before M4.1c:

- Generated work item import created epics and tasks without source lineage.
- Delivery tasks did not carry generation, assessment, or evidence refs from the Docs handoff.
- Handoff ledger `evidenceRefs` contained work item titles rather than evidence reference ids.
- Delivery Pack could display missing or partial lineage even when upstream source context existed.

After M4.1c:

- Generated Story/Task imports carry compact task-level lineage when source context exists.
- Existing task metadata persistence round-trips lineage.
- Handoff ledger events reference actual evidence ref ids when available and record flat lineage counts.
- Generated artifacts retain valid M4.1b `sourceContext`; M4.1c consumes it but does not remove it.
- Incomplete or docs-only imports are labeled partial or missing through lineage metadata.

## Epic Boundary

Generated Epic rows remain grouping records in this milestone. The current `epics` persistence path does not expose a metadata field comparable to tasks. M4.1c does not add schema, RLS, migration, or hidden local-only epic persistence behavior. If AP wants persisted epic-level lineage, that needs a separate schema/RLS-approved milestone.

## M4.1g Boundary

M4.1c makes lineage available to Delivery Pack surfaces. It does not block, label, or modify exports. `M4.1g Delivery Pack lineage/export guard` must decide and implement export behavior when lineage/evidence refs are incomplete.

## Acceptance Criteria

- Imported Story/Task records from Assess-backed generated artifacts include document generation id, process id, assessment id, evidence refs, assumption refs, and handoff ledger id when available.
- Imported tasks from docs-only artifacts do not receive fake Assess lineage and are marked partial or missing.
- Delivery adapter persists and reloads task lineage through existing metadata.
- Handoff ledger records real evidence ref ids rather than generated work item titles.
- Regression tests cover complete, partial, and missing lineage behavior.
- Existing M4.1a and M4.1b tests continue to pass.
- No scoring, provider, schema, Health, route/RBAC, Admin/AI Controls, buyer-demo safe mode, or export policy changes are introduced.
