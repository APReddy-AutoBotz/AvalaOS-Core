## Safety Rules

- Use only the approved non-production Supabase target.
- Confirm required env visibility with booleans only.
- Confirm `AVALA_AI_RUNTIME_MODE` is `pilot` locally before attempting deployed smoke.
- Do not print or store Supabase tokens, URLs, keys, database URLs, auth headers, project refs, hosts, usernames, passwords, IPs, provider keys, or secret values.
- Do not call Gemini or Groq live APIs.
- Do not use real provider keys.
- Do not exercise provider billing paths.
- Invoke only blocked or mocked deployed smoke scenarios that cannot call a live provider.
- Use explicit cleanup and confirm no retained smoke rows.

## Outcome Rule

Outcome A requires deployed blocked-smoke scenarios, deployed audit sanitization, cleanup, and no retained rows to pass without live provider calls or real provider keys.

Outcome B is used for readiness blockers such as missing env, unsafe target, CLI/listing failure, DB readiness regression, missing functions, deploy unavailability, deployed invocation unavailability, or deployed runtime readiness mismatch.

Outcome C is used when deployed functions are reachable but fail-closed behavior, audit sanitization, cleanup, or behavior expectations do not pass.

## Actual Outcome

Outcome B - readiness blocked.

The deployed function was reachable, but the first blocked-smoke invocation returned `mode_not_allowed`. Because the local process runtime mode was confirmed as `pilot`, this indicates the remote deployed Edge runtime is not exposing the required pilot runtime mode to the function. Per milestone stop rules, no further deployed scenarios were run and no remote runtime changes were made.

## M3.2z Recommendation

M3.2z should perform an AP-approved non-production remote runtime configuration readiness repair: verify and set only the required deployed Edge runtime mode for the approved target, without provider keys or live provider calls, then rerun the M3.2y blocked-smoke scenarios from a clean state.
