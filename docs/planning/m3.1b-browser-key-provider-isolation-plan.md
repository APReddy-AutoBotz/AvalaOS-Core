# M3.1b Browser Key / Provider Prop Isolation Plan

## Goal

Create the M3.1b planning record for browser key/provider prop isolation and `setup-api-key.js` retirement after the completed M3.1a mode boundary.

M3.1b future implementation should reduce and isolate browser raw-key/provider prop paths so pilot and production do not depend on `userApiKey` prop flow, localStorage API-key helpers, Vite provider secrets, browser-facing provider-secret injection, or active setup helpers. Browser provider classes may remain only for explicitly bounded `local-demo` and `internal-dev` fallback until later cleanup.

This document is planning-only. It does not implement M3.1b code, change AI behavior, change scoring, or approve M3.1b implementation.

## Scope

- Inventory current `userApiKey` prop flow, localStorage/API-key helper paths, Vite provider env references, `vite.config.ts` provider-secret injection, and browser provider class/factory paths.
- Plan how future M3.1b implementation should isolate or remove raw-key prop paths from active AI generation/refinement surfaces where safe.
- Plan retirement of `setup-api-key.js` so no active localStorage API-key setup helper remains.
- Plan how `StorageKeys.API_KEY` should remain temporarily as deprecated cleanup-only while `App.tsx` still removes old stored keys.
- Plan how browser provider factory/class paths should remain only behind explicit demo/dev fallback without broad file movement unless AP separately approves.
- Plan manual scan evidence requirements for every remaining key/provider boundary hit.
- Preserve M3.1a mode behavior and server-side Edge direction.

## Out Of Scope

- Implementing M3.1b code in this planning milestone.
- Edge provider config or key-reference resolution. That remains M3.2.
- Supabase Function changes.
- Schema/RLS changes.
- Admin AI Controls UI.
- Removing `GeminiProvider` or `GroqProvider` if doing so breaks explicit `local-demo` or `internal-dev` fallback.
- Moving or renaming provider factory files if it creates broad import churn.
- Static scan automation. That remains M3.1c unless separately approved.
- New AI features.
- Scoring changes.
- Health changes.
- Runtime agents.
- MCP/A2A.
- External execution.
- Compliance certification claims.
- Production customer data handling.
- Package or package-lock changes for this planning record.

## Current userApiKey Prop Inventory

- `App.tsx`
  - Keeps `userApiKey` in React state as an empty string.
  - Passes `userApiKey` to `CustomDashboardView`, `DocsForgeView`, `WorkspaceView`, and `ProjectView`.
- `components/docs/DocsForgeView.tsx`
  - Accepts `userApiKey` as a required prop.
  - Passes it into `aiOrchestrator.generateArtifacts`.
  - Uses `userApiKey` in a local error branch that rewrites API-key-like provider errors.
- `components/docs/RefineSectionModal.tsx`
  - Accepts `userApiKey` as a nullable prop.
  - Passes it into `aiOrchestrator.refineSection`.
- `components/delivery/WorkspaceView.tsx`
  - Accepts `userApiKey` as a nullable prop.
  - Passes it into `RefineSectionModal`.
- `components/delivery/ProjectView.tsx`
  - Accepts `userApiKey` as a nullable prop.
  - Passes it into `SprintPlanningView`.
- `components/delivery/SprintPlanningView.tsx`
  - Accepts `userApiKey` and `aiProviderType`.
  - Current AI sprint planning action throws a disabled-path error before provider execution.
- `components/shared/CustomDashboardView.tsx`
  - Accepts `userApiKey` and `aiProviderType`.
  - Passes all props through widget rendering, including to `AiInsightsWidget`.
- `components/shared/widgets/AiInsightsWidget.tsx`
  - Accepts `userApiKey` and `aiProviderType`.
  - Current AI insights path is disabled with a server-side AI requirement message.
- `services/aiOrchestrator.ts`
  - Accepts `userApiKey` in generation and refinement signatures.
  - Passes `userApiKey` to `getAiProvider(..., false)`.
  - Because `allowUserKey=false`, the current provider factory does not consume the `userApiKey` value.
- `services/geminiService.ts`
  - `getAiProviderApiKey` and `getAiProvider` accept `userApiKey`.
  - User key consumption is possible only when `allowUserKey=true`; current orchestrator calls pass `false`.

## Current localStorage / API-key Helper Inventory

- `setup-api-key.js`
  - Remains an active-looking helper.
  - Writes `autobotz_api_key` to localStorage if the hardcoded `apiKey` value is populated.
  - Mentions configuring keys through `.env` or UI settings.
  - Preferred future M3.1b implementation action is deletion.
  - If not deleted, it must become non-executable documentation only.
- `services/storage.ts`
  - Defines `StorageKeys.API_KEY` as `avalaos-core-v1-api-key`.
  - Also contains generic localStorage helpers for non-secret app state.
- `App.tsx`
  - Calls `localStorage.removeItem(StorageKeys.API_KEY)` on mount.
  - Per AP amendment, this cleanup path should remain until scan evidence proves it is safe to remove.
- `components/shared/CustomDashboardView.tsx`
  - Uses localStorage for `dashboardLayout` and `dashboardConfigs`.
  - These are not provider secrets and should be documented as allowed non-AI storage hits in future manual scan evidence.
- `services/adapters/authAdapter.ts`
  - Uses localStorage for current user state.
  - This is not provider-secret storage and should be documented separately from AI-key risks in future manual scan evidence.

## Current Vite Provider Env Inventory

- `services/geminiService.ts`
  - Reads `import.meta.env.VITE_GROQ_API_KEY`.
  - Reads `import.meta.env.VITE_GEMINI_API_KEY`.
  - These are client-exposed provider key paths in Vite builds.
- `vite-env.d.ts`
  - Still types `VITE_GEMINI_API_KEY`.
  - Does not type `VITE_GROQ_API_KEY`.
  - M3.1a added only `VITE_AVALA_AI_MODE` and `VITE_AI_EDGE_FUNCTIONS_ENABLED`.
- `services/aiEdgeClient.ts`
  - Reads `VITE_AI_EDGE_FUNCTIONS_ENABLED`, which is an Edge routing flag and not a provider secret.
- `services/aiOrchestrator.ts`
  - Reads `VITE_AVALA_AI_MODE`, which is the runtime mode selector and not a provider secret.

## Current vite.config.ts Provider-secret Injection Inventory

- `vite.config.ts`
  - Loads root env through `loadEnv(mode, '.', '')`.
  - Defines browser-facing `process.env.API_KEY` from `env.GEMINI_API_KEY`.
  - Defines browser-facing `process.env.GEMINI_API_KEY` from `env.GEMINI_API_KEY`.
- This injection is a client-provider-secret risk and should be removed during future M3.1b implementation.
- Future implementation must not replace it with another client-exposed provider-secret path.
- If removal breaks explicit `local-demo` or `internal-dev` fallback, implementation must stop and report.

## Current Browser Provider Class / Factory Inventory

- `services/geminiService.ts`
  - Imports `GeminiProvider` and `GroqProvider`.
  - Resolves keys through `getAiProviderApiKey`.
  - Instantiates `new GroqProvider(apiKeyToUse)` and `new GeminiProvider(apiKeyToUse)`.
  - Throws for OpenAI until server-side provider boundary is implemented.
- `services/geminiProvider.ts`
  - Performs direct browser `fetch` to Gemini.
  - Includes provider key in the Gemini request URL query string.
  - Supports document generation, section refinement, sprint planning, and dashboard insights methods at class level.
- `services/groqProvider.ts`
  - Performs direct browser `fetch` to Groq.
  - Uses provider key in the Authorization bearer header.
  - Supports document generation and refinement methods, plus current non-generative sprint/dashboard demo helpers.
- `services/aiOrchestrator.ts`
  - M3.1a resolves mode before calling these provider factory paths.
  - Pilot/production fail closed when Edge AI is unavailable.
  - Browser fallback can run only through explicit `local-demo` or `internal-dev` policy.

## Proposed Isolation / Retirement Approach

Future M3.1b implementation should:

- Remove `userApiKey` from active AI generation/refinement call signatures where safe.
- If broad prop-chain removal creates large unrelated UI churn, neutralize the value at the active AI boundary and record remaining `userApiKey` hits for M3.1c.
- Keep acceptance focused on pilot/production not depending on `userApiKey`.
- Remove user-key-specific error handling in Docs Forge when the raw key value is no longer active.
- Keep `StorageKeys.API_KEY` temporarily as deprecated cleanup-only while `App.tsx` removes the old stored key.
- Do not remove the cleanup path until manual scan evidence proves no active old-key storage remains.
- Delete `setup-api-key.js` as the preferred action.
- If deletion is not possible, replace it with non-executable documentation only and ensure it cannot write localStorage.
- Remove `vite.config.ts` browser-facing `process.env.API_KEY` and `process.env.GEMINI_API_KEY` injection.
- Do not add any replacement client-exposed provider-secret path.
- Keep `VITE_GROQ_API_KEY` out of `vite-env.d.ts`.
- Remove or isolate `VITE_GEMINI_API_KEY` typing only if it does not break explicit demo/dev fallback; if it does break fallback, stop and report.
- Do not move or rename provider factory files if doing so creates broad import churn.
- Mark/isolate browser provider paths through comments, naming, mode-boundary checks, and manual evidence instead of broad file movement.
- Stop and request AP approval if file movement becomes required.
- Keep browser provider classes only for explicit `local-demo` / `internal-dev` fallback until later cleanup.

## Risks

- `userApiKey` prop removal could create broad UI churn if done through every prop chain at once.
- Leaving `userApiKey` hits without clear evidence could make the boundary look incomplete.
- `vite.config.ts` provider-secret injection may be unused, but it is still a client-secret risk and should be removed in future implementation.
- Removing Vite provider env paths may break explicit demo/dev fallback if no alternate demo/dev key boundary exists.
- Moving provider factory files could create broad import churn and should not happen without AP approval.
- Keeping browser provider classes is acceptable only if scan evidence proves they are demo/dev-only and unreachable from pilot/production.
- Static scan automation is useful but belongs to M3.1c unless separately approved.
- Edge provider config/key-reference work belongs to M3.2 and must not be pulled into M3.1b.

## Split Recommendation

- neutralizing/removing active `userApiKey` generation/refinement dependency
- retiring `setup-api-key.js`
- removing browser-facing `vite.config.ts` provider-secret injection
- documenting remaining hits with manual scan evidence
- avoiding provider factory movement and static scan automation

- broad prop-chain refactor across unrelated UI surfaces
- provider factory file movement
- static scan automation
