# M3.2o Non-Production Resolver Edge Smoke Plan

## Milestone Purpose

M3.2o is a planning-only milestone for a controlled non-production Supabase Edge Function smoke after M3.2n Resolver Edge Function Integration Implementation.

M3.2n integrated resolver gates into:

- `ai-generate-document`
- `ai-refine-section`
- `ai-provider-test-connection`

M3.2o does not execute the smoke. It does not deploy Edge Functions, seed a database, call provider APIs, resolve real secrets, write audit rows, or change runtime files. It defines the safe execution plan that a later AP/ChatGPT-approved milestone can follow.

## Dependency Chain

M3.2o depends on the completed M3.2 provider-governance chain:

- M3.2h schema/RLS migration introduced provider governance tables, constraints, RLS enablement, and provider scope for `gemini` and `groq`.
- M3.2l resolver decision logic implemented deterministic allow/block resolver behavior, fail-closed gates, sanitized decision outputs, and audit event shells without runtime Edge integration.
- M3.2n Edge Function integration connected the resolver, audit persistence, secret lookup eligibility, and provider handoff gates into the three AI Edge Functions.
- M3.2n-dbval Outcome A validation confirmed the base schema plus M3.2h migration could be applied to a fresh, empty, AP-approved non-production Supabase database, with provider governance row counts remaining `0`.

## Non-Production Environment Requirements

M3.2p must use a clearly isolated non-production Supabase project or local Supabase environment. Production, shared, customer, demo-customer, or unclear databases are not allowed.

Required non-production variables must be configured without printing or committing values:

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `AVALA_AI_RUNTIME_MODE`
- `AVALA_PROVIDER_SECRET_GROQ_<SAFE_SUFFIX>` or `AVALA_PROVIDER_SECRET_GEMINI_<SAFE_SUFFIX>`
- `DATABASE_URL`, only if local DB validation or seed execution requires direct database access

Environment rules:

- Do not print variable values.
- Do not commit variable values.
- Do not paste production, shared, customer, provider, service-role, or database credentials into repository files.
- Rotate any credential pasted into chat.
- `AVALA_AI_RUNTIME_MODE` must be exactly `pilot` or `production`.
- M3.2p should prefer `pilot` unless AP/ChatGPT explicitly approves a production-mode non-production smoke.
- Provider secret variables must use only the approved provider-specific prefix:
  - `AVALA_PROVIDER_SECRET_GROQ_<SAFE_SUFFIX>`
  - `AVALA_PROVIDER_SECRET_GEMINI_<SAFE_SUFFIX>`
- The safe suffix must be uppercase letters, numbers, or underscores only.
- `DATABASE_URL` must point only to the approved non-production target and must never be printed.

## Seed Data Plan

M3.2p must seed only minimal, non-customer governance rows needed to exercise resolver-gated behavior. Seed execution must be explicit, reviewed, and limited to the approved non-production target.

Required safe rows:

- One test organization created only for this smoke.
- One test profile/user mapping if required by the existing auth model.
- One active organization membership linking the test user/profile to the test organization.
- One role compatible with the existing membership model and resolver role matching.
- One `ai_provider_key_refs` row:
  - `org_id` set to the test organization.
  - `provider` set to either `gemini` or `groq`.
  - `resolver_type` set to `server_reference`.
  - `secret_ref` set to the approved provider-specific reference naming pattern only.
  - `status` set to `active`.
  - No raw provider key value stored in any database column.
  - No auth header, encrypted key, browser-readable value, or secret payload stored in any database column.
- One `ai_provider_configs` row:
  - `org_id` set to the test organization.
  - `provider` matching the key reference.
  - `key_ref_id` linked to the seeded key reference.
  - `status` set to `active`.
  - `allowed_modes` including `pilot`.
  - `allowed_operations` including `generate_document`, `refine_section`, and `test_provider_connection`.
- Three `ai_workspace_provider_policies` rows:
  - One for `generate_document`.
  - One for `refine_section`.
  - One for `test_provider_connection`.
  - `org_id` set to the test organization.
  - `provider_config_id` linked to the seeded provider config.
  - `mode` set to the smoke runtime mode, preferably `pilot`.
  - `status` set to `active`.
  - `allowed_roles` containing the seeded test role.
  - `is_default` set to `true`.
  - Exactly one active default row per organization, operation, and mode.

Seed constraints:

- No customer data.
- No real source documents.
- No production or shared users.
- No raw prompts beyond minimal harmless smoke strings.
- No provider keys stored in the database.
- No raw provider response, completion, prompt body, auth header, provider payload, service-role key, or `DATABASE_URL` stored in evidence.
- Do not record the full `secret_ref` value in evidence if AP/ChatGPT treats it as sensitive; record only the approved naming pattern and provider family.
- Do not create migrations or edit schema SQL to make the seed possible.
- Stop if the seed requires schema changes.

## Smoke Scenarios

M3.2p must run blocked scenarios before allowed scenarios so fail-closed behavior is proven before any provider handoff path is exercised.

### Blocked Scenarios

Plan these blocked checks:

- Missing or invalid `AVALA_AI_RUNTIME_MODE` blocks before provider call.
- Missing policy for the requested org, operation, and mode blocks before provider call.
- Disabled provider config blocks before provider call.
- `manual_placeholder` key reference blocks before provider call.
- Wrong provider-prefix `secret_ref` blocks before environment read and provider call.
- Audit insert failure blocks allowed provider handoff, if safely mockable or plannable without corrupting the non-production database.

Each blocked check must verify:

- No provider API call occurs.
- No raw key material is resolved.
- No AI job is created when the current integration contract blocks before job creation.
- The response is sanitized.
- Any audit event written is sanitized and only written when safe context exists.

### Allowed Scenarios

Plan these allowed checks against seeded governance rows:

- Resolver allows `generate_document`.
- Resolver allows `refine_section`.
- Resolver allows `test_provider_connection`.

For allowed scenarios, M3.2p must choose one execution mode before any smoke runs:

- Mocked provider call first, preferred. This proves deployed Edge resolver, audit, and secret gates without consuming live provider network calls or provider credits.
- Live provider smoke only if AP/ChatGPT explicitly approves network calls and provider key use for the named non-production target.

If live provider smoke is approved, keep prompts minimal and harmless, use a dedicated low-risk provider key, and record only sanitized response classifications, not raw completions or provider payloads.

## Safe Expected Responses

Blocked Edge responses must expose only safe fields such as:

- `error` or `message`
- `correlationId`
- `safeUiMessageCategory`
- `retryCategory`
- `failureClass`

Blocked responses must not include:

- `secret_ref`
- Raw provider key.
- Auth headers.
- Service-role key.
- Policy, config, or key-reference rows.
- Raw prompt.
- Raw completion.
- Raw provider payload.
- Stack trace.
- Database error text.
- Supabase/PostgREST raw response text.
- `DATABASE_URL`.

Allowed responses must also be captured safely. Evidence may record operation name, HTTP status, sanitized classification, correlation ID presence, and whether the mocked or live provider boundary was reached. Evidence must not record raw completions, provider payloads, prompts, keys, headers, database URLs, or service-role values.

## Audit Verification Plan

M3.2p must verify `ai_provider_audit_events` through safe row-count and sanitized metadata checks only.

Audit verification must confirm:

- Allowed decisions write sanitized audit events before provider handoff.
- Blocked decisions write audit events only when safe resolver context exists.
- Allowed-path audit persistence failure blocks provider handoff.
- Audit metadata excludes secrets, prompts, completions, provider payloads, auth headers, service-role keys, `DATABASE_URL`, and raw database errors.
- Audit metadata does not include raw `secret_ref` values if AP/ChatGPT treats the reference names as sensitive.
- Row-count checks are safe to record.
- Correlation IDs are present and non-secret.
- Provider, operation, mode, status, failure class, and safe policy result are recorded only as sanitized evidence.

Audit verification must not dump full rows into evidence. Record only counts, allowed safe fields, and sanitized pass/fail summaries.

## Stop Conditions

Stop future smoke execution if any condition is met:

- Environment is not clearly non-production.
- `DATABASE_URL` points to production, shared, customer, or unclear infrastructure.
- Required environment variables are missing.
- `AVALA_AI_RUNTIME_MODE` is not `pilot` or `production`.
- A provider key would be printed, logged, committed, stored in the database, or exposed to the browser.
- Seed rows require schema changes.
- Seed rows require customer data, real source documents, production users, or shared tenant data.
- A live provider call is attempted without AP/ChatGPT approval.
- A blocked response leaks sensitive details.
- An audit event contains prohibited content.
- Scanner output has forbidden or stale hits.
- Wording scan reveals a newly introduced unsupported compliance claim or AI-decision wording.
- Health, browser fallback, Admin UI, product UI, scoring, prompt-version, usage/cost, dependency, schema, or migration changes are required.

## Acceptance Criteria

M3.2o is accepted when:

- This plan exists and is limited to future non-production smoke execution.
- The evidence file confirms only the two requested docs changed.
- No runtime, deploy, seed, provider-call, schema, migration, Admin UI, product UI, Health, browser fallback, scoring, prompt-version, usage/cost, dependency, or compliance-claim change is introduced.
- Required verification commands and scan reviews are recorded.
- M3.2p stop conditions and AP approval gates are explicit.
