# M5.3a-2n Redacted Unknown Local Startup Failure Signal Capture Plan

## Milestone Purpose

M5.3a-2n is a docs-only planning milestone for a future redacted signal-capture mechanism that can improve the `unknown-local-startup-failure` evidence path without copying raw local startup output into evidence.

The goal is to define a safe future mechanism for structured signal metadata only. M5.3a-2n does not implement that mechanism, run the classifier, run local startup, run Docker or Supabase stack commands, connect to a database, run schema bootstrap, run RLS harnesses, run artifact assertions, or infer root cause.

Recommended next milestone:

- `M5.3a-2o Redacted Unknown Local Startup Failure Signal Capture Implementation`, only after AP approval.

## Problem Statement

The expanded classifier still returned `unknown-local-startup-failure`.

Blind retries are no longer useful because they can repeat the same sanitized unknown outcome without adding evidence. Guessing additional raw-output patterns is also no longer enough because raw output, logs, local paths, target values, IDs, ports, and other local details cannot be copied into evidence safely.

The safer next step is to plan a future redacted signal-capture mechanism that can produce structured, non-sensitive metadata during an approved controlled classifier run.

## Redacted Signal-Capture Objective

Future M5.3a-2o implementation should produce only structured signal categories that help classify an unknown local startup failure without exposing local or sensitive values.

The mechanism should:

- Preserve scratch-only raw-output handling.
- Derive metadata categories in memory.
- Emit only approved field names and approved enum values.
- Record counts and buckets instead of raw text.
- Fail closed when sanitization cannot be proven.
- Leave local DB availability, schema availability, artifact SELECT isolation, tenant isolation, hosted-pilot readiness, and production readiness unclaimed.

## Proposed Future Implementation Scope

Future implementation may update the existing classifier only after AP approval.

Allowed future scope:

- Add a redacted signal metadata object to the classifier result.
- Add strict field allowlisting for the signal object.
- Add enum validation for each signal field.
- Add count-only pattern-family metadata.
- Add output safety validation for the signal object and emitted lines.
- Add tests or static checks that prove forbidden fields and values are rejected.
- Preserve scratch lifecycle and cleanup.

Future implementation must not:

- Change package scripts unless AP separately approves.
- Change SQL, migrations, RLS policies, assertion SQL, RLS harnesses, artifact harnesses, runtime, UI, CI, provider, deployment, export, scoring, demo, Reports/Teams/Admin, or Health files.
- Add DB execution, schema bootstrap, RLS harness execution, artifact assertion execution, remote validation, or provider calls.

## Allowed Signal Schema

Future signal capture may emit only these fields:

| Field | Allowed values |
| --- | --- |
| `startupAttemptStatus` | `attempted`, `not-attempted` |
| `exitResultClass` | `success`, `nonzero-exit`, `timeout`, `spawn-error`, `killed`, `unknown` |
| `commandPhaseClass` | `preflight`, `supabase-cli-invocation`, `image-resolution`, `image-pull`, `container-create`, `service-start`, `health-wait`, `cleanup`, `unknown` |
| `outputSizeBucket` | `none`, `small`, `medium`, `large`, `oversized`, `unknown` |
| `lineCountBucket` | `none`, `small`, `medium`, `large`, `oversized`, `unknown` |
| `timeoutFlag` | boolean |
| `knownPatternFamilyCounts` | approved family labels mapped to counts only |
| `sanitizedCategoryCandidates` | category labels only, no matched text |
| `confidence` | `high`, `medium`, `low` |
| `outputSafetyResult` | `passed`, `failed` |
| `scratchCleanupResult` | `removed`, `not-created`, `failed` |
| `safetyBlockReasonCategory` | `none`, `raw-output-unsafe`, `secret-like`, `local-path-like`, `target-like`, `container-image-id-like`, `raw-log-like`, `output-too-large`, `unknown` |
| `noSecretsConfirmation` | category only |
| `noLocalPathConfirmation` | category only |
| `noRawLogConfirmation` | category only |

`knownPatternFamilyCounts` may include only AP-approved family labels. It must never include matched substrings, raw lines, raw tokens, target values, path values, IDs, or local output.

## Redaction And Safety Gates

Future implementation must enforce all of these gates:

- Field allowlist validation before output.
- Enum validation before output.
- Count-only pattern family reporting.
- Bucket-only output-size and line-count reporting.
- No raw text in `sanitizedCategoryCandidates`.
- No raw line samples.
- No raw command output.
- No target values.
- No local paths.
- No IDs from local tooling.
- No scratch-path output.
- No fallback that copies raw output into evidence.

If any gate fails, the classifier must fail closed with a sanitized safety-block category.

## Output Safety Verification

Future output safety verification must scan emitted classifier output and evidence-ready metadata before any evidence is written.

The verification should confirm:

- Only approved field names appear.
- Only approved enum values appear.
- Counts are numeric.
- Buckets are category labels only.
- No forbidden local or sensitive fields appear.
- No raw-output fragments appear.
- No local path or target-like value appears.
- No secret-like value appears.

Output safety failure must prevent any readiness or classification confidence increase.

## Scratch Lifecycle

Future implementation must preserve scratch-only raw-output handling:

- Raw startup output may exist only in generated scratch during classification.
- Signal metadata may be derived from scratch in memory only.
- Scratch output must not be copied into docs, evidence, logs, commits, or final output.
- Scratch cleanup must run before final scans.
- Scratch cleanup result must be reported as a category only.

If scratch cleanup fails, the future implementation must fail closed.

## Fail-Closed Behavior

Future implementation must fail closed if:

- A disallowed field appears.
- Raw output would be emitted.
- Safety validation fails.
- Scratch cleanup fails.
- Evidence would require copying local output.
- Signal metadata cannot be produced without leaking local or sensitive values.
- A value cannot be categorized without exposing the original value.
- A category cannot be assigned safely.

The fail-closed result must not infer root cause.

Future M5.3a-2o implementation should verify:

- Syntax or static validation for changed classifier code.

- Unit or static checks that raw text is not emitted.

- Restricted wording scan.

A controlled classifier run may occur only if AP explicitly approves it for M5.3a-2o.

## Rollback And Abandon Criteria

Future implementation should be abandoned or rolled back if:

- Redacted metadata is insufficient to improve classification safely.
- Useful classification would require raw output.
- Useful classification would require local paths, target values, IDs, ports, host values, logs, or secrets.
- Output safety cannot be proven.
- Scratch cleanup cannot be proven.
- The implementation encourages manual inspection of raw output.
- Evidence would overclaim readiness, isolation, availability, schema status, or root cause.

The safe fallback remains `unknown-local-startup-failure` with low confidence.
