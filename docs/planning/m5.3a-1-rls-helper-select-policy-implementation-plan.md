# M5.3a-1 RLS Helper And SELECT Policy Implementation Plan

## Purpose

M5.3a-1 defines the first RLS implementation slice after the M5.3a refresh. It is a docs-only planning milestone for helper function design, SELECT policy direction, isolation fixture/test contracts, and the local validation gate.

This milestone does not implement SQL, helper functions, RLS policies, tests, runtime adapters, seed data, backfill, constraint validation, package changes, CI changes, deployment changes, or provider execution.

## Strict Non-Scope

M5.3a-1 does not:

- Create SQL migrations, RLS policy SQL, helper functions, test files, seed data, backfills, or constraint validation.
- Edit existing migrations, schema contracts, package files, CI, deployment config, runtime adapters, UI, routing, view keys, Sidebar behavior, demo data, scoring, provider execution, export behavior, Reports/Teams/Admin behavior, or the separate Health repository.
- Connect to remote Supabase, run remote validation, run `scripts/applySupabaseMigrations.mjs`, run `npm run db:migrate`, or run `npm run test:rls:assess`.
- Read, print, summarize, copy, expose, or commit `.env` values, secrets, personal hosted values, provider keys, service-role values, tokens, bucket names, or project-specific identifiers.
- Claim tenant isolation is implemented or proven.
- Claim hosted-pilot readiness.

The future M5.3a-1 implementation should be limited to:

- Helper function design for active organization membership and active workspace membership.
- SELECT policy design for identity, organization, workspace, role, and membership authority tables.

- A local validation gate that must be approved before verified tenant isolation can be claimed.

## Current Authority Baseline

M5.2a created and enabled RLS on the identity, organization, workspace, role, and membership authority tables. Those tables currently have no applied M5.3a RLS policies. That means browser/authenticated access remains fail-closed until an approved RLS implementation slice adds policy SQL and validates behavior.

M5.2c, M5.2f, M5.2d-a, and M5.2g-a created applied artifact authority tables with RLS enabled and no policies. Those artifact tables are intentionally outside M5.3a-1 and should remain fail-closed until M5.3a-2 or a later AP-approved artifact policy slice.

## Helper Function Proposal

Future required helpers:

- `public.is_active_org_member(p_org_id uuid)`
- `public.is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`

Helper SQL direction:

- Use `LANGUAGE sql`.
- Mark helpers `STABLE`, not `IMMUTABLE`.
- Use `SECURITY DEFINER` only if required to avoid recursive RLS failure.
- Harden `search_path`.
- Prefer schema-qualified table and function references.
- Return boolean membership status only.
- Never return tenant metadata, membership lists, role existence, permission payloads, or service-role behavior.

Access grants:

- Revoke execute from `PUBLIC`.
- Revoke execute from `anon` where applicable.
- Grant execute only to `authenticated`.
- Do not expose service-role behavior to browser code.

Helper predicates must require:

- `auth.uid()` is present.
- Membership `user_id = auth.uid()`.
- Membership `status = 'active'`.
- Membership `deleted_at IS NULL`.
- Parent organization `status = 'active'`.
- Parent organization `deleted_at IS NULL`.
- Parent workspace `status = 'active'` where workspace applies.
- Parent workspace `deleted_at IS NULL` where workspace applies.

## SECURITY DEFINER Hardening

`SECURITY DEFINER` is provisional, not automatic. Future implementation must prove:

- No recursive RLS failure.
- No broad role grants.
- No `search_path` hijack risk.
- No cross-tenant information leakage.
- No data-returning helper behavior.
- No helper path grants access from disabled, invited, archived, suspended, removed, or deleted membership.
- No helper path bypasses disabled, archived, or deleted parent organization or workspace state.

The safest direction is to use minimal definer helpers only for membership resolution if direct policy joins cannot avoid recursion.

## SELECT Policy Plan

Future M5.3a-1 policy SQL should be explicit SELECT-only policy SQL.

Default stance:

- Deny by default.
- No broad `FOR ALL` policies.
- No INSERT policies.
- No UPDATE policies.
- No DELETE policies.
- No hard DELETE.
- No mutation policies.
- No artifact policies.
- No provider governance policy changes.
- No legacy draft table policies.

Table directions:

| Table | SELECT policy direction |
| --- | --- |
| `profiles` | User can read own active, non-deleted profile only. |
| `organization_members` | User can read own active, non-deleted membership only, gated by active org helper. |
| `workspace_memberships` | User can read own active, non-deleted workspace membership only, gated by active workspace helper. |
| `organizations` | Readable only through `public.is_active_org_member(id)`. |
| `workspaces` | Readable only through `public.is_active_workspace_member(id, org_id)`. |
| `roles` | Organization roles readable through active org membership; workspace roles readable through active workspace membership; system role visibility deferred to AP. |

## Recursion Risk Analysis

Main recursion risks:

- Helper functions may query `organization_members` while `organization_members` has RLS enabled.
- Helper functions may query `workspace_memberships` while `workspace_memberships` has RLS enabled.
- Role policies may depend on membership visibility.
- Membership policies may depend on helper functions that query membership tables.

Recommended controls:

- Use minimal helper functions only for membership resolution.
- Do not add permission helpers in M5.3a-1.
- Do not make membership visibility depend on role helpers.
- Do not make helper functions depend on artifact tables.
- Require local validation before implementation can claim verified isolation.

## Bootstrap And Access Paradox

Users need enough visibility to operate without exposing other tenants. Future policy design should allow:

- Own active, non-deleted profile read.
- Own active, non-deleted organization membership read.
- Own active, non-deleted workspace membership read.
- Organization metadata read through active organization membership.
- Workspace metadata read through active workspace membership.
- Role metadata read only through active scoped membership.

Future policy design should not allow:

- Cross-tenant role existence leaks.
- Cross-tenant membership existence leaks.
- Disabled, deleted, invited, archived, suspended, or removed membership to bootstrap access.
- System role visibility unless AP approves exact semantics.

## Isolation Test Contract

Future M5.3a-1 tests should cover only identity, org, workspace, role, and membership policy behavior.

Negative tests:

- Anonymous user is denied.
- Authenticated non-member is denied.
- Disabled org member is denied.
- Disabled workspace member is denied.
- Org A user cannot read Org B org, workspace, membership, or role metadata.
- Workspace A user cannot read Workspace B membership or role metadata.
- Service-role behavior remains server-only and not browser-callable.

Positive tests:

- Own active profile is readable.
- Own active organization memberships are readable.
- Own active workspace memberships are readable.
- Organization metadata is readable only through active membership.
- Workspace metadata is readable only through active membership.
- Role metadata is scoped to active membership.

No artifact-table tests belong in M5.3a-1 except explicit deferral confirmations.

## Runtime And Hosted Readiness Posture

M5.3a-1 planning does not make runtime hosted-ready. Future M5.3a-1 implementation will not make runtime hosted-ready by itself.

Artifact tables remain fail-closed until M5.3a-2 or later. Runtime adapters may still fail closed or target legacy draft tables. Hosted pilot readiness still requires later adapter, export, storage, deployment, operations, evidence, and audit work.

- Recursive RLS failure if helpers and membership policies depend on each other incorrectly.
- Overbroad `SECURITY DEFINER` behavior if grants or `search_path` are not hardened.

- Disabled/deleted membership accidentally granting access.
- Bootstrap policy gaps that prevent legitimate users from reading their own memberships.
- Static review false confidence without repeatable local isolation tests.
- Runtime fail-closed behavior after future policies are added.
