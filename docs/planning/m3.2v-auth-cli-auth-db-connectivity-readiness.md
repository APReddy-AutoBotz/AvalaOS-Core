# M3.2v-auth Supabase CLI Auth And DB Connectivity Readiness

## Milestone Purpose

M3.2v-auth verifies Supabase CLI auth and database connectivity readiness after M3.2u repaired the missing `supabase-go` local CLI blocker.

This milestone is readiness-only. It does not deploy Edge Functions, invoke deployed Edge Functions, seed the database for a new deployed smoke, or call provider APIs.

## Dependency Chain Through M3.2u

M3.2v-auth depends on the prior M3.2 readiness chain:

- M3.2p proved resolver sequencing with mocked provider behavior, transaction rollback cleanup, sanitized audit evidence, and no live provider calls when database connectivity was available.
- M3.2t stopped before deployed blocked-smoke execution because target functions were unreachable and the local Supabase CLI could not find `supabase-go`.
- M3.2u repaired the missing `supabase-go` blocker with a verified official Supabase CLI `v2.101.0` Windows release companion binary.
- M3.2u remained blocked because Supabase CLI auth was unavailable and the existing M3.2p mocked smoke was blocked by database DNS/connectivity.

## Readiness Scope

In scope:

- Confirm repository and remote safety.
- Confirm `.env` is not tracked.
- Load `.env` into process memory only for non-printing checks.
- Confirm required environment variables are process-visible without printing values.
- Confirm `AVALA_AI_RUNTIME_MODE` is `pilot`.
- Confirm Supabase and database targets pass non-production guards without printing full URL, project ref, token, keys, host, or database URL.
- Run a sanitized Supabase CLI `functions list` check using `SUPABASE_ACCESS_TOKEN` from process environment.
- Run a sanitized database connectivity probe.
- Run the existing M3.2p mocked smoke regression and record pass or sanitized blocker.
- Create planning and evidence documentation only.

Out of scope:

- Edge Function deploy.
- Deployed Edge Function invocation.
- New database seed for deployed smoke.
- Gemini or Groq provider API calls.
- Real provider keys.
- Runtime, schema, RLS, migration, Admin UI, product UI, Health, scoring, prompt-version, browser fallback, dependency, usage, or cost changes.
- Unsupported compliance claims.

## Environment Readiness Checks

Required variables must be process-visible without printing values:

- `SUPABASE_ACCESS_TOKEN`.
- `SUPABASE_URL`.
- `SUPABASE_ANON_KEY`.
- `SUPABASE_SERVICE_ROLE_KEY`.
- `DATABASE_URL`.
- `AVALA_AI_RUNTIME_MODE`.

`AVALA_AI_RUNTIME_MODE` must be `pilot`.

If any required variable is missing or runtime mode is not `pilot`, M3.2v-auth records Outcome B and stops before target checks that require the missing variable.

## Target Safety Checks

Supabase target checks:

- Parse `SUPABASE_URL` without printing it.
- Confirm Supabase host class.
- Confirm project-ref shape without printing the project ref.
- Confirm disallowed markers such as production, shared, customer, or live are absent.

Database target checks:

- Parse `DATABASE_URL` without printing it.
- Confirm disallowed markers such as production, shared, customer, or live are absent from sanitized target classification.
- Do not print host, database name, username, or URL.

If target safety is production, shared, customer, live, or unclear, M3.2v-auth records Outcome B and stops before CLI listing or database connectivity.

## Supabase CLI Auth Check

The CLI auth check uses only process environment credentials:

```powershell
supabase functions list --project-ref <project-ref-redacted>
```

The command is read-only for Edge Functions. It must not deploy or invoke functions.

Evidence records only:

- CLI version.
- Sanitized command status.
- Exit code.
- Whether listing reached the approved non-production target.

Evidence must not record:

- `SUPABASE_ACCESS_TOKEN`.
- Full Supabase URL.
- Project ref.
- Supabase keys.
- Auth headers.

## Database Connectivity Check

The database connectivity check verifies:

- `DATABASE_URL` is parseable without printing value.
- Target safety passes without printing host or database name.
- A sanitized connection probe can reach the database.

If connectivity fails, evidence records only sanitized error class, code, and redacted message.

## M3.2p Mocked Smoke Check

The existing M3.2p mocked smoke command remains required:

```powershell
npm run test:m3.2p-smoke
```

This command is an existing regression check. It is not deployed smoke and does not invoke deployed Edge Functions.

If database connectivity fails before seed, evidence records the sanitized blocker only.

## Outcome Framework

Outcome A - ready:

- Supabase CLI auth works.
- `functions list` or equivalent reaches the approved non-production target.
- `DATABASE_URL` connectivity works.
- M3.2p mocked smoke passes.
- M3.2w deployed blocked-smoke retry may proceed after AP review.

Outcome B - blocked:

- CLI auth is unavailable or fails.
- Database connectivity is unavailable or fails.
- Target is unsafe or unclear.
- M3.2p mocked smoke is blocked.
- M3.2w remains blocked.

## M3.2w Go/No-Go Criteria

M3.2w is a go only if AP review accepts evidence that:

- Supabase CLI auth works without exposing token values.
- The approved non-production target can be reached by read-only CLI listing.
- Database connectivity works.
- The existing M3.2p mocked smoke passes.
- No M3.2v-auth deploy, invoke, seed, provider call, or runtime behavior change occurred.

M3.2w is a no-go if:

- Database DNS/connectivity remains blocked.
- Supabase CLI auth fails.
- Target safety is unclear.
- AP requires another readiness step before deployment is retried.

## Stop Conditions

M3.2v-auth stops before deploy, invoke, seed, or provider call if:

- Repository or remote safety fails.
- Required env visibility is blocked.
- Runtime mode is not `pilot`.
- Supabase or database target safety is blocked.
- A check would require printing or storing a secret.
- A check would require changing runtime, schema, UI, Health, scoring, fallback, package, usage, or cost behavior.

## Risks Carried Forward

- Deployed target functions remain not proven reachable until a future milestone performs an approved deployed check.
- Deployed blocked scenarios remain unexecuted.
- Deployed runtime-mode failure behavior remains unproven.
- Safe deployed mock behavior remains unconfirmed.
- Live provider network calls and real keys remain blocked unless AP explicitly approves them.
