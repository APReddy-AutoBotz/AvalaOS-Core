# M5.3a-2c Local DB Availability And Schema Bootstrap Plan

## Milestone Objective

M5.3a-2c is a planning-only milestone for the local DB availability and schema bootstrap gap discovered in M5.3a-2b.

The objective is to define what AP must approve before a future implementation can make a local database available, prepare the required schema safely, and collect local artifact assertion evidence without exposing secrets or overclaiming isolation.

M5.3a-2c does not implement bootstrap logic, change runners, change assertion SQL, change RLS policies, run any RLS harness, run artifact assertions, run migrations, connect to any database, or authorize M5.3a-2d execution.

Recommended next milestone:

- `M5.3a-2d Local DB Availability And Schema Bootstrap Implementation`, only after AP approval.

## Local DB Availability Planning

Future local DB availability work must remain local-only and AP-approved before any assertion execution.

Planning requirements:

- Define the approved local target form without relying on secret-file fallback.
- Prove a target is local before any DB action.
- Fail closed if target proof is missing, rejected, ambiguous, or unavailable.
- Fail closed if a local DB is not reachable.
- Preserve sanitized output that reports only labels, gate status, counts, and classifications.
- Do not print target values, DB URLs, host values, port values, database names, user names, service-role values, provider keys, hosted identifiers, fixture IDs, row payloads, auth headers, claim values, private tokens, or `.env` values.

AP decisions needed before implementation:

- Whether a local DB availability implementation is approved.
- Which local target form is acceptable.
- Whether a generic local config is permitted.
- Whether any local-only privileged setup path is permitted.
- What output is safe to capture in implementation evidence.
- Whether implementation must stop if local DB availability cannot be proven.

## Schema Bootstrap Planning

Future schema bootstrap work must be planned and approved separately before any local artifact assertion can verify scoped behavior.

Planning boundaries:

- M5.3a-2c may inventory possible future approaches and approval gates.
- M5.3a-2c must not provide executable bootstrap steps.
- M5.3a-2c must not run migration or bootstrap commands.
- M5.3a-2c must not create or modify local Supabase config.
- M5.3a-2c must not change SQL migrations, RLS policies, assertion SQL, or runners.

Future bootstrap must ensure the local schema contains, at minimum:

- Applied authority schema required by the artifact assertion contract.
- M5.3a-1 helper and membership SELECT policy foundation.
- M5.3a-2 artifact SELECT policies.
- The M5.3a-2 artifact assertion SQL contract.

Possible future approach categories to evaluate in M5.3a-2d:

- Generic local stack configuration approved by AP.
- Local-only migration/bootstrap wrapper approved by AP.
- Local-only schema preparation using existing tracked migrations, with no remote fallback.
- Static schema readiness check before any assertion execution.

These categories are planning inputs only. M5.3a-2c does not choose or execute a bootstrap path.
