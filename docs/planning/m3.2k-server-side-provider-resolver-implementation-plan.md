# M3.2k Server-Side Provider Resolver Implementation Plan

## Milestone Purpose

M3.2k is a planning-only milestone that translates the M3.2j server-side provider resolver contract into a future implementation plan. It defines the proposed file layout, function signatures, data flow, safety gates, database query plan, secret-reference adapter plan, audit event write plan, test strategy, scanner classification, and AP approval checklist needed before resolver implementation can begin.

M3.2k does not authorize implementation. M3.2l Server-Side Provider Resolver Implementation may begin only after AP/ChatGPT reviews and approves the M3.2k PR.

## Dependency Chain Through M3.2j

1. M3.2h implemented the approved provider configuration governance schema/RLS migration.
2. M3.2i verified provider config RLS evidence and recommended resolver contract planning.
3. M3.2j defined the resolver contract only.
4. M3.2j was post-merge verified and tagged as `avalaos-core-m3.2j-server-side-provider-resolver-contract-planning`.
5. M3.2k converts the M3.2j contract into an implementation plan without creating resolver code.

M3.2k is constrained by the M3.2j contract:

- Eligible providers: `gemini`, `groq`
- Resolver-governed modes: `pilot`, `production`
- Resolver-governed operations: `generate_document`, `refine_section`, `test_provider_connection`
- Enforceable tenant boundary: `org_id`
- Deferred authority: `workspace_id`
- Required next-step approval: AP/ChatGPT approval before M3.2l implementation

## Proposed Resolver File/Module Location

Future implementation should keep resolver logic inside Supabase Edge shared modules so it cannot enter the browser bundle.

Proposed files for M3.2l, subject to AP approval:

- `supabase/functions/_shared/providerResolver.ts`
  - Owns resolver input normalization, gate orchestration, policy/config/key-reference lookup, failure mapping, and allowed/blocked decision output.
- `supabase/functions/_shared/providerSecretAdapter.ts`
  - Owns future server-only `secret_ref` lookup adapter behavior after all resolver gates pass.
- `supabase/functions/_shared/providerResolverAudit.ts`
  - Owns sanitized `ai_provider_audit_events` write helpers, or this may be a narrow AP-approved extension to `supabase/functions/_shared/audit.ts`.

Future integration points, subject to AP approval:

- `supabase/functions/ai-generate-document/index.ts`
- `supabase/functions/ai-refine-section/index.ts`
- `supabase/functions/ai-provider-test-connection/index.ts`
- `supabase/functions/_shared/ai.ts`

M3.2k must not create or edit any of these implementation files.

## Proposed Resolver Function Signatures

Future types should be local to the server-side resolver module unless AP later approves shared type export.

Proposed provider, mode, and operation types:

```ts
type ProviderResolverProvider = 'gemini' | 'groq';
type ProviderResolverMode = 'pilot' | 'production';
type ProviderResolverOperation = 'generate_document' | 'refine_section' | 'test_provider_connection';
```

Proposed resolver input:

```ts
type ProviderResolverInput = {
  mode: string;
  operation: string;
  requestedProvider?: string;
  requestedProviderConfigId?: string;
  orgId: string;
  workspaceId?: string;
  actorId: string;
  correlationId?: string;
  evidenceRef?: string;
  promptReference?: {
    promptKey?: string;
    promptVersion?: string;
    templateId?: string;
  };
};
```

Proposed dependency boundary:

```ts
type ProviderResolverDeps = {
  now: () => Date;
  queryMembershipAndRoles: (input: { orgId: string; actorId: string }) => Promise<MembershipRoleContext>;
  queryProviderPolicy: (input: PolicyLookupInput) => Promise<ProviderPolicyRow[]>;
  queryProviderConfig: (input: ConfigLookupInput) => Promise<ProviderConfigRow | null>;
  queryProviderKeyRef: (input: KeyRefLookupInput) => Promise<ProviderKeyRefRow | null>;
  createCorrelationId: () => string;
};
```

Proposed resolver entrypoint:

```ts
async function resolveProviderForOperation(
  input: ProviderResolverInput,
  deps: ProviderResolverDeps,
): Promise<ProviderResolverDecision>;
```

Proposed decision shapes:

```ts
type ProviderResolverDecision =
  | {
      status: 'allowed';
      providerCallAllowed: true;
      provider: ProviderResolverProvider;
      providerConfigId: string;
      keyRefId: string;
      operation: ProviderResolverOperation;
      mode: ProviderResolverMode;
      orgId: string;
      workspaceId?: string;
      actorId: string;
      correlationId: string;
      evidenceRef?: string;
      policyResult: 'allowed';
      auditEvent: ProviderResolverAuditEventShell;
    }
  | {
      status: 'blocked';
      providerCallAllowed: false;
      failureClass: ProviderResolverFailureClass;
      safeUiMessageCategory: ProviderResolverSafeUiCategory;
      retryCategory: ProviderResolverRetryCategory;
      orgId?: string;
      workspaceId?: string;
      actorId?: string;
      correlationId: string;
      evidenceRef?: string;
      policyResult: 'blocked';
      auditEvent: ProviderResolverAuditEventShell;
    };
```

Proposed future secret lookup entrypoint:

```ts
async function resolveProviderSecretForCall(
  allowedDecision: Extract<ProviderResolverDecision, { status: 'allowed' }>,
  deps: ProviderSecretAdapterDeps,
): Promise<ProviderSecretMaterial>;
```

Proposed future audit entrypoint:

```ts
async function recordProviderResolverDecision(
  decision: ProviderResolverDecision,
  deps: ProviderResolverAuditDeps,
): Promise<void>;
```

No proposed signature should return `secret_ref`, raw provider keys, auth headers, service-role keys, provider payloads, raw prompts, or raw completions.

## Resolver Gate Sequence

Future implementation must fail closed in this order:

1. Normalize and validate `mode`; only `pilot` and `production` are resolver-governed.
2. Require server-authenticated `actorId`.
3. Require trusted `orgId`.
4. Verify active organization membership.
5. Resolve role context from `organization_members.role_id` and `roles`.
6. Normalize and validate operation.
7. Normalize and validate provider.
8. Resolve active provider policy for `org_id`, operation, mode, and role.
9. Resolve provider config from the policy or explicit selector.
10. Verify provider config belongs to the same `org_id`.
11. Verify provider config is active and non-deleted.
12. Verify provider config allows the requested mode and operation.
13. Verify provider config provider matches the selected provider.
14. Resolve `key_ref_id`.
15. Verify key ref belongs to the same `org_id` and provider.
16. Verify key ref is active, non-deleted, non-expired, and not `manual_placeholder`.
17. Verify `secret_ref` is reference-only and not exposed beyond the server boundary.
18. Prepare sanitized audit event shell.
19. Confirm scanner classification for any new implementation paths.
20. Permit provider-call handoff only after all gates pass.

Any missing, ambiguous, inactive, deleted, expired, unsupported, unsafe, or unclassified prerequisite must return a blocked decision before provider-call handoff.

## Database Query Plan

Future implementation should use the existing `postgrest` helper in `supabase/functions/_shared/supabase.ts` or a narrow wrapper around it. No new database objects are authorized by M3.2k.

Membership and role lookup:

- Query `organization_members` by `org_id`, `user_id`, and `status = active`.
- Join or follow `role_id` to `roles`.
- Treat missing membership, missing role, or unreadable role context as fail closed.
- Match `ai_workspace_provider_policies.allowed_roles` against approved role names and/or AP-approved role identifiers. M3.2l must not invent a new role authority.

Policy lookup:

- Query `ai_workspace_provider_policies` by `org_id`, `operation`, `mode`, `status = active`, and `deleted_at IS NULL`.
- Prefer exactly one active default for `(org_id, operation, mode)` when no explicit config selector is supplied.
- If multiple active matching policies are returned, fail closed as `provider_policy_ambiguous`.
- Validate actor role against `allowed_roles`.

Provider config lookup:

- Query `ai_provider_configs` by `id`, `org_id`, `status = active`, and `deleted_at IS NULL`.
- Validate provider is `gemini` or `groq`.
- Validate `allowed_modes` contains the normalized mode.
- Validate `allowed_operations` contains the normalized operation.
- Use only `key_ref_id`; ignore deprecated prior-context `key_reference`.

Key-reference lookup:

- Query `ai_provider_key_refs` by `id`, `org_id`, provider, and `deleted_at IS NULL`.
- Require `status = active`.
- Reject `expires_at` in the past.
- Reject `resolver_type = manual_placeholder`.
- Treat `secret_ref` as server-only reference metadata, not secret material.

Audit write:

- Insert sanitized rows into `ai_provider_audit_events` only after AP approves M3.2l implementation.
- Do not write raw `secret_ref`, raw prompts, raw completions, auth headers, service-role keys, secret-manager payloads, or provider payloads.

## Service-Role Boundary

Future resolver implementation may use Supabase service-role access only inside Supabase Edge/server shared modules.

Service-role access must not bypass:

- server-authenticated actor resolution
- active org membership
- role authorization
- provider policy
- provider config status
- mode authorization
- operation authorization
- key-reference eligibility
- secret-safety checks
- audit redaction checks
- scanner classification

Service-role keys must never be logged, returned, stored in audit metadata, sent to provider APIs, or exposed to browser/client code.

## `secret_ref` Lookup Adapter Plan

`providerSecretAdapter.ts` should remain isolated from general resolver decision output.

Future adapter rules:

- Accept only an allowed resolver decision.
- Lookup only `server_reference` or AP-approved `external_secret_reference` values.
- Reject `manual_placeholder`.
- Return raw provider key material only to the immediate server-side provider-call function.
- Never include raw key material or `secret_ref` in logs, audit metadata, errors, browser responses, or general business logic.
- Map lookup failures to sanitized failure classes such as `secret_reference_unsafe` or `key_reference_ineligible`.

M3.2k does not choose a secret manager or implement any lookup. M3.2l must stop if AP has not approved the exact adapter and deployment secret source.

## Audit Event Write Plan

Future implementation should add a sanitized resolver audit writer.

Event type:

- `ai_provider_resolver_decision`

Allowed event fields:

- `schema_version`
- `event_type`
- `org_id`
- nullable `workspace_id`
- `provider`
- `provider_config_id`
- `key_ref_id`
- `operation`
- `mode`
- `policy_result`
- `status`
- `failure_class`
- `actor_id`
- `service_context`
- `correlation_id`
- `evidence_ref`
- `metadata`

Allowed metadata examples:

- role check result
- policy lookup result
- provider config status
- key-reference eligibility status
- retry category
- scanner classification reference
- sanitized timing information

Audit writes should be required for resolver decisions if the audit context can be made safe. If audit context cannot be made safe, the resolver must block before provider-call handoff with `audit_context_unsafe`.

## Failure Class Mapping

Future implementation should preserve M3.2j failure classes:

| Failure class | Future trigger |
| --- | --- |
| `mode_not_allowed` | Mode is not `pilot` or `production`. |
| `unauthenticated` | Server-authenticated actor is absent. |
| `org_missing` | Trusted org context is missing or ambiguous. |
| `membership_denied` | Actor is not an active member of the org. |
| `role_not_allowed` | Actor role does not satisfy policy. |
| `operation_not_allowed` | Operation is unsupported or denied by policy/config. |
| `provider_not_supported` | Provider is not `gemini` or `groq`. |
| `provider_policy_missing` | No active matching policy exists. |
| `provider_policy_ambiguous` | Policy lookup returns ambiguous active defaults or matches. |
| `provider_config_missing` | Policy or selector cannot resolve config. |
| `provider_config_ineligible` | Config is inactive, deleted, wrong org/provider, or disallows mode/operation. |
| `key_reference_missing` | Config has no `key_ref_id` or no matching key row. |
| `key_reference_ineligible` | Key ref is inactive, deleted, expired, wrong org/provider, placeholder, or unsafe. |
| `secret_reference_unsafe` | `secret_ref` shape or adapter classification is unsafe. |
| `audit_context_unsafe` | Safe audit shell cannot be prepared. |
| `scanner_classification_missing` | Future implementation cannot be safely classified by static scan controls. |
| `provider_call_blocked` | Generic fail-closed class for blocked handoff after a prior failure. |

Errors returned to browser/client callers should use safe categories and correlation IDs only.

## Scanner Classification Plan

M3.2l implementation must update static scan allowlist only if new server-side resolver references trigger scanner patterns.

Expected classification:

- `providerResolver.ts`: server-side Edge/Supabase control path
- `providerSecretAdapter.ts`: server-side secret-reference adapter path
- `providerResolverAudit.ts`: server-side audit/evidence path

Scanner requirements:

- New paths must not be browser-bundled.
- Any allowlist update must be exact, minimal, and include future action text.
- Browser `VITE_GEMINI_API_KEY`, `VITE_GROQ_API_KEY`, browser provider classes, and demo/dev fallback remain unchanged.
- Direct `GEMINI_API_KEY` and `GROQ_API_KEY` references in `_shared/ai.ts` may be reclassified or replaced only in the AP-approved implementation milestone.
- Static scan must pass with `0` forbidden hits and `0` stale allowlist entries.

## Test Plan

Future M3.2l test coverage should include unit-style resolver tests, static scan coverage, and non-production integration validation.

Resolver tests:

- allowed decision for active member, allowed role, active policy, active config, eligible key ref, allowed mode, and allowed operation
- fail closed for invalid mode
- fail closed for unauthenticated actor
- fail closed for missing org
- fail closed for inactive membership
- fail closed for missing or denied role
- fail closed for unsupported provider
- fail closed for unsupported operation
- fail closed for missing policy
- fail closed for ambiguous policy
- fail closed for inactive/deleted config
- fail closed for disallowed mode or operation
- fail closed for missing key ref
- fail closed for inactive/deleted/expired/wrong-provider key ref
- fail closed for `manual_placeholder`
- fail closed when audit shell would contain prohibited fields
- fail closed when scanner classification is missing

Secret adapter tests:

- accepts only allowed resolver decisions
- rejects blocked decisions
- rejects `manual_placeholder`
- never returns `secret_ref` to resolver output
- maps lookup failures to sanitized classes

Audit tests:

- writes safe resolver audit shape
- excludes raw prompt, completion, provider payload, auth header, service-role key, `secret_ref`, and raw secret fields
- preserves correlation ID and failure class

Verification commands for M3.2l should include the M3.2k command suite plus any AP-approved resolver-specific tests.

## `DATABASE_URL` Validation Plan

M3.2k carries forward the non-production database validation risk.

Future M3.2l should:

- use only safe non-production `DATABASE_URL`
- validate existing M3.2h tables and RLS are present before resolver integration testing
- never run validation against production or shared customer data without AP approval
- record whether `DATABASE_URL` is set or absent
- stop if the validation database does not include the M3.2h provider governance objects

No database validation is required or performed by M3.2k because no schema or runtime behavior changes are made.

## Rollback / No-Runtime-Safety Plan

M3.2k is docs-only and has no runtime rollback path beyond reverting the two planning docs.

Future M3.2l implementation should be reversible by:

- keeping resolver integration behind the server-side Edge path only
- preserving current behavior until resolver allow/deny paths pass tests
- avoiding schema changes in the resolver implementation slice unless AP separately approves them
- isolating secret adapter code from provider-call code
- keeping audit writes sanitized and non-blocking only where AP approves non-blocking behavior
- not touching browser fallback, Health, scoring, package files, or Admin UI
