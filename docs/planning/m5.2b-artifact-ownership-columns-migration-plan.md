# M5.2b Artifact Ownership Columns Migration Plan

Milestone: M5.2b Artifact Ownership Columns Migration Plan

Branch: milestone/m5.2b-artifact-ownership-columns-migration

Status: Docs-only artifact ownership and backfill planning pack

## Purpose

M5.2b plans how AvalaOS Core should add consistent artifact ownership columns before workspace-aware RLS is implemented. M5.2a added the tenant authority foundation for profiles, organizations, workspaces, roles, organization members, and workspace memberships. It did not add ownership columns to Assess, Govern, Studio, Delivery, handoff, evidence, lineage, audit, export, or provider artifact records.

The goal of this milestone is to decide which domain records need `org_id`, `workspace_id`, actor, lifecycle, and audit-linkage fields, and to define a safe implementation sequence. This milestone does not implement schema changes.

## Strict Non-Scope

M5.2b does not create migrations, alter schema, implement RLS policies, change runtime adapters, change UI, change routing, alter demo data, change scoring, deploy, connect to live Supabase, call providers, change provider runtime, change CI, add dependencies, update package scripts, or modify a separate Health repository.

M5.2b does not read, print, summarize, copy, expose, or commit `.env*` values, Supabase tokens, service-role credentials, provider keys, personal hosted-project values, database passwords, storage credentials, or other secrets.

M5.2b does not claim hosted pilot readiness, enterprise-production readiness, provider readiness, certification, self-running approval authority, or current AGI capability.

## Recommended Implementation Shape

Recommended shape: **A. M5.2b docs-only artifact ownership and backfill plan**.

M5.2b should not implement a broad additive migration. The current applied migration chain has M5.2a authority tables and provider-governance tables, while most artifact tables still exist as draft SQL contracts under `docs/schema`. A single migration for every artifact domain would force table existence, workspace mapping, foreign key, and backfill decisions before AP has approved the canonical artifact authority model.

Recommended split:

1. M5.2b Artifact Ownership Columns Migration Plan: docs-only inventory, ownership model, FK/backfill plan, and RLS readiness gate.
2. M5.2c Assess Govern Ownership Columns Migration: first additive schema slice for the highest-priority Assess/Govern surfaces.
3. Later Studio/Docs ownership slice.
4. Later Delivery/Delivery Pack ownership slice.
5. Later Monitor, evidence, lineage, handoff, audit, export, and provider workspace-scope slices as AP approves persistence authority.

Do not proceed to M5.3a RLS Policy Implementation And Isolation Tests until enough artifact ownership exists to support workspace-aware positive and negative isolation tests.

## Current Schema And Persistence Posture

M5.2a created the first real tenant authority tables:

- `profiles`
- `organizations`
- `workspaces`
- `roles`
- `organization_members`
- `workspace_memberships`

RLS is enabled on those six tables, but M5.2a intentionally created no RLS policies and no seed data.

Current artifact persistence is uneven:

- The applied migration chain includes provider-governance records with org-scoped ownership and a deferred nullable workspace placeholder on provider audit events.
- `docs/schema` contains draft SQL contracts for Assess, Delivery, Docs, timesheets, handoff ledger, audit, storage, AI governance, and RLS regression checks.
- Runtime adapters still target org-scoped draft table names such as `assess_processes`, `assessments`, `assessment_review_events`, `projects`, `tasks`, `document_generations`, `handoff_ledger_entries`, and `timesheet_entries` when Supabase is configured.
- Browser/demo storage remains buyer-demo persistence and is not hosted pilot authority.

## Artifact Ownership Model

Use a standard vocabulary, but do not force every field onto every table:

- `org_id`: tenant root. Required for hosted pilot tenant records.
- `workspace_id`: hosted pilot work boundary. Required or derived for most work artifacts after AP approves workspace mapping.
- `created_by`: actor that created the record.
- `updated_by`: actor that last changed mutable metadata.
- `owner_id`: business owner or accountable user.
- `reviewer_id`: review actor where a workflow requires review.
- `approver_id`: approval actor where a workflow requires approval.
- `source_org_id` and `source_workspace_id`: only for approved cross-source lineage or import cases.
- `status`, `deleted_at`, `archived_at`: lifecycle and soft-delete fields where needed.
- `correlation_id` or `audit_correlation_id`: audit linkage for server-side transitions, exports, provider actions, handoffs, and review events.

Per-domain defaults:

- Assess/Govern: `org_id`, nullable `workspace_id`, owner/reviewer/approver/created/updated actors, lifecycle, and correlation fields.
- Studio/Docs: `org_id`, nullable `workspace_id`, created/reviewed actor fields, source Assess/project refs, generation/review status.
- Delivery: `org_id`, nullable `workspace_id`, project owner, task owner, assignee/reporter direction, source lineage refs, lifecycle fields.
- Handoff/evidence/lineage/audit/export: workspace ownership plus append/status/audit linkage after persistence authority is approved.
- Provider governance: keep key refs and config metadata org-scoped by default; add workspace ownership only if AP approves workspace-specific provider policy authority.

## FK Strategy

Future implementation should:

- Add nullable `workspace_id` before enforcing non-null ownership.
- Use additive `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` patterns.
- Prefer `NOT VALID` foreign keys for broad artifact tables, then validate after AP-approved backfill.
- Add `NOT NULL` only after backfill is verified.
- Reference `organizations(id)` and `workspaces(id)` only where the target table is part of the real migration chain.
- Avoid FKs for draft-only, demo-only, or not-yet-canonical tables.
- Avoid destructive data rewrites.

## Backfill Strategy

M5.2b recommends deferring backfill execution.

Future backfill should:

- Use no seed or broad update in the initial ownership-column migration.
- Map non-production demo records only after AP approves the canonical demo workspace.
- Map the AP Invoice Exception Workflow to an AP-approved non-production workspace.
- Use manual AP-approved data mapping for any hosted database that already has rows.
- Avoid personal organization IDs, workspace IDs, project IDs, bucket names, or hosted values.
- Avoid committing remote/live identifiers.
- Avoid destructive updates.

## Migration Safety

M5.2b creates no migration and performs no schema validation against a database.

Future M5.2c implementation should be:

- Additive only.
- Non-destructive.
- Free of dropped columns.
- Free of behavior or semantic changes to current runtime flows.
- Free of RLS policies.
- Free of seed data.
- Free of provider execution.
- Free of runtime dependency on new columns until adapters are separately approved.
- Local-only for validation unless AP approves a separate hosted strategy.

Do not run remote Supabase validation for M5.2b or M5.2c without explicit AP approval.

## RLS Readiness Mapping

Ownership columns prepare M5.3a for:

- Workspace-scoped reads by active workspace members.
- Cross-org denial through `org_id`.
- Cross-workspace denial through `workspace_id`.
- Buyer Viewer read-only policies.
- Reviewer and approver role gates.
- Export, evidence, lineage, handoff, and audit tenant boundaries.
- Provider metadata boundaries without exposing secret values.
- Server-side audit append behavior with actor and correlation context.

## Risks

- Broad schema blast radius if every artifact domain is migrated in one PR.
- FK and backfill failure if workspace mapping is inferred instead of AP-approved.
- Buyer-demo compatibility risk if runtime depends on new columns too early.
- Ambiguity between draft SQL contracts and the real migration chain.
- Existing provider-governance tables are mostly org-scoped.
- False confidence if ownership columns are treated as RLS enforcement.
- Migration ordering and hosted migration-history risk.
- Personal value leakage risk if validation uses local ignored files or remote projects.
