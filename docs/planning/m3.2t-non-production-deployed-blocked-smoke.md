# M3.2t Non-Production Deployed Blocked-Smoke Execution

## Milestone Purpose

M3.2t attempts the first deployed non-production blocked-smoke execution for the resolver-gated Edge Functions after M3.2s confirmed local invocation readiness.

The purpose is to prove deployed Edge Functions fail closed safely before any live provider call can occur.

Target Edge Functions:

- `ai-generate-document`
- `ai-refine-section`
- `ai-provider-test-connection`

## Dependency Chain Through M3.2s

M3.2t depends on the prior M3.2 chain:

- M3.2n integrated resolver, audit, secret eligibility, and provider handoff gates into the target Edge Functions.
- M3.2p proved resolver sequencing with mocked provider behavior, sanitized audit events, transaction rollback cleanup, and no live provider calls.
- M3.2q planned deployed Edge smoke with blocked scenarios first and live provider smoke blocked.
- M3.2r stopped before deployed smoke because deployed invocation credentials were incomplete.
- M3.2s confirmed the required local process variables were visible without printing values, runtime mode classified as `pilot`, the Supabase target passed sanitized non-production confirmation, and `.env` was not committed.

## Deployed Blocked-Smoke Scope

M3.2t is blocked-scenario execution only.

In scope:

- Confirm required environment variable visibility without printing values.
- Confirm runtime mode is `pilot`.
- Confirm the target is a sanitized non-production Supabase target.
- Check deployed function reachability.
- Deploy only the three existing Edge Function directories to the confirmed non-production project if required and if the local Supabase CLI context is safe.
- Seed only harmless smoke rows if function availability and deploy safety pass.
- Invoke only blocked scenarios before any possible provider handoff.
- Verify blocked response sanitization, audit sanitization, and cleanup.

Out of scope:

- Allowed live provider smoke.
- Gemini or Groq provider API calls.
- Real provider keys.
- Runtime, schema, RLS, migration, Admin UI, product UI, Health, scoring, browser fallback, prompt-version, dependency, usage, or cost changes.

## Preflight Summary

The M3.2t preflight must run in a process that loads `.env` without printing values.

Required visibility:

- `SUPABASE_URL`: process-visible.
- `SUPABASE_ANON_KEY`: process-visible.
- `SUPABASE_SERVICE_ROLE_KEY`: process-visible.
- `AVALA_AI_RUNTIME_MODE`: process-visible.

Required classification:

- `AVALA_AI_RUNTIME_MODE`: `pilot`.
- Supabase host class: `supabase-host`.
- Project-ref shape: `standard-20-lower-alnum`.
- Disallowed target markers: absent.
- `.env`: not tracked by git.

If any required variable is missing, runtime mode is not `pilot`, or target identity is production/shared/customer/live/unclear, M3.2t stops before deploy, seed, auth, or invoke.

## Edge Function Availability Strategy

The three target functions are checked with sanitized `OPTIONS` requests. The check records function name and HTTP status only; it does not print the Supabase URL, auth keys, or headers.

If a function returns `404`, it is treated as not deployed/reachable.

If a function is not deployed/reachable, deployment may proceed only when:

- The sanitized target remains non-production.
- The project ref is derived from the confirmed non-production `SUPABASE_URL` without printing it.
- The local Supabase CLI has a usable deploy binary and credentials.
- Deployment is limited to the three current Edge Function directories.

If CLI/project context is incomplete, M3.2t records Outcome C and stops before seed or invocation.

## Seed Strategy

Seed runs only after function availability or safe deploy passes.

Allowed seed rows:

- One temporary non-production smoke auth user.
- One profile mapping for the smoke auth user.
- Smoke organizations for blocked scenarios.
- Smoke roles and active memberships.
- Smoke provider key-reference rows.
- Smoke provider config rows.
- Smoke workspace provider policy rows where required by the blocked case.

Seed rows must not include:

- Real provider keys.
- Raw provider payloads.
- Customer data.
- Raw prompt or completion bodies.
- Auth headers.
- Service-role values.
- Production/shared data.

## Invocation Strategy

Invocation uses a temporary non-production smoke auth user token only if seed succeeds.

The token is:

- Created only for this milestone.
- Kept in process memory.
- Not printed.
- Not stored.
- Deleted during cleanup with the smoke auth user.

Each deployed request must include a safe correlation ID and evidence reference. Evidence records only function name, scenario name, sanitized HTTP status, expected failure class, correlation presence, and sanitizer checks.

## Blocked Scenario Summary

M3.2t planned blocked scenarios:

| Scenario | Expected failure |
| --- | --- |
| Missing policy | `provider_policy_missing` |
| Disabled config | `provider_config_ineligible` |
| `manual_placeholder` key reference | `key_reference_ineligible` |
| Wrong provider-prefix smoke reference | `secret_reference_unsafe` |
| Missing/invalid runtime mode | Deferred unless the deployed environment can be safely and reversibly toggled. |

The first four scenarios are planned across all three target functions when deployed invocation is available.

For every blocked response M3.2t verifies:

- HTTP response is received.
- No provider call occurs.
- No real provider key is used.
- No raw provider key appears.
- No raw secret reference appears.
- No auth header appears.
- No service-role key appears.
- No raw prompt or completion appears.
- No provider payload appears.
- No stack trace appears.
- No raw database or PostgREST error appears.
- `correlationId` exists.
- `safeUiMessageCategory`, `retryCategory`, and `failureClass` appear where expected.

## Audit Verification Strategy

Audit verification runs only if deployed invocation proceeds.

It records:

- Row counts only.
- Sanitized metadata checks.
- Correlation presence.
- Failure-class coverage.
- Zero retained row confirmation after cleanup.

It does not record full database rows, raw secrets, raw prompt/completion text, provider payloads, auth headers, service-role keys, database URLs, or raw database errors.

## Cleanup Strategy

Cleanup deletes only M3.2t smoke data:

- Provider audit events for the M3.2t evidence reference.
- Workspace provider policies for M3.2t smoke organizations/configs.
- Provider configs and key refs created by M3.2t.
- Organization memberships, roles, organizations, and profile created by M3.2t.
- The temporary auth user created by M3.2t.

Cleanup must verify zero retained smoke rows. If cleanup fails, M3.2t records Outcome C.

## Outcome Decision

Outcome C - blocked.

Reason:

- Sanitized preflight passed.
- All three target Edge Functions returned `404` to sanitized reachability checks.
- Deployment was required but could not safely proceed because the local Supabase CLI shim could not find its co-located `supabase-go` binary.
- M3.2t stopped before seed, auth user creation, deployed invocation, blocked scenario execution, audit writes, or cleanup deletes.

## Strict Non-Goals

M3.2t does not:

- Run allowed live provider smoke.
- Call Gemini or Groq APIs.
- Use real provider keys.
- Print or store Supabase keys, database URLs, provider keys, auth headers, or tokens.
- Commit `.env`.
- Store secrets in docs, evidence, logs, or git.
- Use production/shared/customer data.
- Change schema SQL, RLS, or migrations.
- Change Admin UI or product UI.
- Touch Health.
- Change scoring formulas, score weights, thresholds, hard stops, or recommendation logic.
- Change browser fallback behavior.
- Change prompt versions.
- Expand usage/cost behavior.
- Add dependencies.
- Edit Edge Function entrypoints.
- Edit `_shared/ai.ts`.
- Edit runtime resolver or provider integration files.
- Make unsupported compliance claims.

## Risks Carried Forward

- The target Edge Functions remain not deployed/reachable in the confirmed non-production target.
- The local Supabase CLI installation is incomplete and cannot deploy until `supabase-go` is restored or another approved deploy path is provided.
- Deployed blocked scenarios remain unexecuted.
- Deployed runtime-mode behavior remains unproven.
- Allowed provider-handoff smoke remains blocked unless AP approves live provider use or a controlled mock strategy.

- Confirm the same non-production target without printing values.
- Deploy only the three current Edge Function directories.
- Re-run M3.2t blocked-smoke execution from preflight.
