# M3.2n Resolver Edge Function Integration Implementation

## Milestone Purpose

M3.2n integrates the M3.2l provider resolver decision logic into the three Supabase Edge Functions that can hand off to AI providers. The implementation keeps the first runtime integration narrow, server-side only, and governed by resolver, audit, and secret lookup gates before provider calls.

## Integration Behavior Summary

The target Edge Functions now use a resolver-governed sequence:

1. Handle CORS preflight.
2. Authenticate the caller through the existing Edge auth helper.
3. Resolve trusted `org_id` through the existing org membership helper.
4. Resolve server-owned runtime mode from server environment only.
5. Treat request provider fields as preferences/selectors only.
6. Call `resolveProviderForOperation` before job creation, secret lookup, and provider handoff.
7. Persist sanitized resolver audit evidence.
8. Return sanitized fail-closed responses for blocked decisions.
9. For allowed decisions, resolve server-side key material only after audit persistence succeeds.
10. Create AI jobs and call provider helpers only after resolver, audit, secret lookup, and scanner classification gates pass.

Client/request mode does not weaken controls. Missing or invalid server-owned mode fails closed through the resolver.

## Per-Function Integration Summary

`ai-generate-document` maps to `generate_document`.

- Resolves provider governance before `createAiJob`.
- Blocks before job creation, secret lookup, and provider call when resolver blocks.
- Calls `generateDocumentWithProvider` with explicit server-side key material only after gates pass.

`ai-refine-section` maps to `refine_section`.

- Preserves existing refinement instruction validation.
- Resolves provider governance before `createAiJob`.
- Blocks before job creation, secret lookup, and provider call when resolver blocks.
- Calls `refineSectionWithProvider` with explicit server-side key material only after gates pass.

`ai-provider-test-connection` maps to `test_provider_connection`.

- Resolves provider governance before any provider test call.
- Keeps the existing `connected: false` response style for blocked and failed connection tests.
- Calls the provider helper only with explicit server-side key material after resolver, audit, and secret gates pass.

## Secret Lookup Safety Summary

`providerSecretAdapter` now supports server-side secret lookup for allowed resolver decisions only.

- Blocked resolver decisions are not eligible for lookup.
- Only `server_reference` key references are eligible.
- `manual_placeholder` is blocked.
- `external_secret_reference` is blocked and deferred.
- Key references must match allowed decision org, provider, key reference ID, active status, and expiry rules.
- `secret_ref` is treated as an environment variable name, not key material.
- `secret_ref` must match a safe environment-variable identifier shape.
- Raw provider key material is returned only to the immediate server-side provider-call boundary.
- `secret_ref` and raw key material are not returned to clients, audit metadata, logs, docs, or tests.

## Audit Persistence Safety Summary

`providerResolverAuditDb` persists sanitized M3.2l audit event shells into `ai_provider_audit_events`.

- Metadata is rechecked with the existing audit metadata guard before insert.
- Raw prompts, completions, provider payloads, auth headers, service-role values, raw keys, and secret references are not inserted.
- Allowed decisions require audit persistence before secret lookup and provider handoff.
- Audit failure on allowed decisions fails closed before secret lookup and provider handoff.
- Blocked decisions attempt sanitized audit persistence when safe org/provider context is available; if not available or persistence fails, the request still fails closed with a sanitized response.

## Safe Failure Response Summary

Blocked resolver responses expose only:

- stable safe error text
- `correlationId`
- `safeUiMessageCategory`
- `retryCategory`
- `failureClass`

They do not expose policy rows, config rows, key reference rows, secret references, provider keys, auth headers, provider payloads, raw prompts, raw completions, raw provider errors, stack traces, or service-role details.

## Test Coverage Summary

The focused integration test covers:

- `generate_document`, `refine_section`, and `test_provider_connection` block before job creation and provider handoff when resolver blocks.
- Allowed path order is resolver, audit persistence, secret lookup, job creation, provider handoff.
- Blocked path does not call secret lookup.
- Blocked path does not call provider helper.
- Audit failure blocks secret lookup and provider handoff.
- Secret lookup failure blocks job creation and provider handoff.
- Secret adapter rejects `manual_placeholder`.
- Secret adapter rejects `external_secret_reference`.
- Secret adapter rejects unsafe environment reference shape.
- Safe failure responses do not include secret references, raw keys, or provider payloads.
- Mocked allowed path receives explicit key material at the server-side provider boundary.
- Tests do not make real provider network calls.
