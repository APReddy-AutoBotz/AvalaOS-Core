# M3.1 Browser AI Boundary Lockdown Implementation Plan

## Goal

Plan the M3.1 browser AI boundary lockdown slice so pilot and production behavior cannot use browser-side provider execution, browser-stored raw API keys, localStorage API-key helpers, or Vite-exposed provider secrets.

M3.1 is a planning-only record at this stage. It does not implement code, change AI behavior, change scoring, or approve M3.1 implementation.

## Scope

- Inventory current browser AI execution paths, provider classes, key handling, localStorage usage, and Vite provider env references.
- Define the mode contract needed before M3.1 implementation.
- Plan fail-closed pilot/production behavior when Edge AI is unavailable or controls are missing.
- Preserve explicitly labeled `local-demo` and `internal-dev` behavior where needed during transition.
- Plan tests and static scans proving pilot/production cannot reach browser provider execution or browser raw-key paths.
- Record UI/copy impacts for unavailable Edge AI states.
- Record PDF/DOCX browser extraction mismatch for later source-safety remediation.

## Out Of Scope

- Implementing M3.1 code in this planning milestone.
- Edge provider config or server-side key-reference resolution. That belongs to M3.2.
- Supabase Function changes, except future needs identified for later approval.
- Schema or RLS migrations.
- Admin AI Controls UI.
- New AI features.
- New model providers.
- Scoring changes.
- Health changes.
- Package or package-lock changes.
- Dependency additions.
- Runtime agents, MCP/A2A, external execution behavior, or compliance certification claims.
- Production customer data handling.

## Current Browser AI Path Inventory

- `services/aiOrchestrator.ts`
  - Calls `aiEdgeClient.generateDocument` and `aiEdgeClient.refineSection` only when `isAiEdgeEnabled()` is true.
  - Falls back to browser provider execution when Edge AI is not enabled.
  - Uses `getAiProvider(providerType, userApiKey, false)` for generation and refinement fallback.
  - On transient generation errors, tries fallback browser providers through `getAiProviderApiKey(fallbackProviderType, null, false)` and `getAiProvider(...)`.
  - Current comments label fallback as transitional dev/demo behavior, but no explicit runtime mode contract enforces that boundary.
- `services/aiEdgeClient.ts`
  - Enables Edge AI only when `VITE_AI_EDGE_FUNCTIONS_ENABLED === 'true'` and Supabase is configured.
  - Invokes `ai-generate-document`, `ai-refine-section`, extraction, and export Edge Functions.
  - Does not yet enforce the M3 mode contract by itself.
- `services/geminiService.ts`
  - Creates `GeminiProvider` or `GroqProvider` in the browser.
  - Reads provider keys from `import.meta.env.VITE_GROQ_API_KEY` and `import.meta.env.VITE_GEMINI_API_KEY`.
  - Has an `allowUserKey` parameter but current orchestrator calls pass `false`.
  - OpenAI provider is disabled until the server-side provider boundary is implemented.
- `services/geminiProvider.ts`
  - Performs browser `fetch` calls directly to Gemini with the provider key in the request URL.
  - Supports document generation, section refinement, sprint planning, and dashboard insights at the provider class level.
- `services/groqProvider.ts`
  - Performs browser `fetch` calls directly to Groq with a bearer token.
  - Supports document generation and section refinement, and includes non-generative sprint/dashboard helper behavior.
- `App.tsx`
  - Keeps `userApiKey` in React state and passes it into Docs Forge, Workspace, dashboard, and project views.
  - Clears `StorageKeys.API_KEY` from localStorage on mount.
  - Persists `StorageKeys.AI_PROVIDER` for the selected provider.
- `components/docs/DocsForgeView.tsx`
  - Reads uploaded files in the browser with `file.text()`.
  - Passes `userApiKey`, provider type, source text, and file name into `aiOrchestrator.generateArtifacts`.
- `components/docs/RefineSectionModal.tsx`
  - Passes `userApiKey` and provider type into `aiOrchestrator.refineSection`.
- `components/delivery/WorkspaceView.tsx`
  - Passes `userApiKey` and provider type into the section refinement modal.
  - Uses Edge export paths when available, with local export fallback for document export. This is not provider AI execution but should remain in the M3.1 scan evidence.
- `components/shared/LandingPage.tsx`
  - Shows an "Avala AI Controls" advanced area with provider selection.
  - Accepts `.txt`, `.md`, `.pdf`, and `.docx` uploads even though current browser extraction uses `file.text()`.
- `components/shared/widgets/AiInsightsWidget.tsx`
  - Receives `userApiKey` and provider type props, but currently disables AI insights in the pilot-safe client path.
- `components/delivery/SprintPlanningView.tsx`
  - Receives `userApiKey` and provider type props, but current "Plan with AI" behavior throws a disabled-path error.
- `setup-api-key.js`
  - Is an API-key/localStorage setup helper and should not remain as an active helper for pilot or production.

## Current Key / Storage / Env Risk Inventory

- `services/geminiService.ts` references `VITE_GROQ_API_KEY` and `VITE_GEMINI_API_KEY`, which are client-exposed in Vite builds.
- `vite-env.d.ts` declares `VITE_GEMINI_API_KEY`; `VITE_GROQ_API_KEY` and `VITE_AI_EDGE_FUNCTIONS_ENABLED` are referenced without the same type declaration.
- `services/storage.ts` still defines `StorageKeys.API_KEY` as `avalaos-core-v1-api-key`.
- `App.tsx` clears the legacy API-key storage key, but `userApiKey` state and props still exist.
- `setup-api-key.js` writes an API key to localStorage when populated and references an older `autobotz_api_key` key.
- `StorageService.clear()` clears all localStorage, which is broad but not specifically a provider-key path.
- Dashboard layout/config localStorage keys are not provider secrets, but future static scan evidence must distinguish allowed non-AI storage from AI-boundary risks.

## Mode Contract Design

Allowed modes are exactly:

- `local-demo`
- `internal-dev`
- `pilot`
- `production`

Client-side mode gating is only a product, UI, and orchestrator guard. It is not the final security boundary. Server-side Edge enforcement remains mandatory for pilot and production.

| Mode | Defaulting Rule | AI Execution Rule | Key Boundary | Failure Behavior |
| --- | --- | --- | --- | --- |
| `local-demo` | Local development may default to this mode only when Vite/dev context is explicit. | Prepared demo data and synthetic-only fallback are allowed. | No real customer data. Browser provider fallback must not be represented as pilot-safe. | Demo fallback may run only when clearly local-demo and synthetic/prepared. |
| `internal-dev` | Must be explicit outside local demo assumptions. | Transitional fallback may exist only when clearly labeled. | Must never be used for real customer data. | Warn or fail clearly when behavior is not pilot-safe. |
| `pilot` | Must be explicit. | Server-side Edge AI only. | No browser raw keys, browser-side provider execution, or Vite provider secrets. | Fail closed if Edge AI is unavailable or required controls are missing. |
| `production` | Must be explicit. | Server-side Edge AI only. | Same as pilot, with production deployment controls. | Fail closed if Edge AI is unavailable or required controls are missing. |

Additional mode rules:

- Invalid mode must fail closed.
- Deployed or non-dev builds must fail closed or require an explicit mode.
- Pilot and production must use Edge AI only.
- Pilot and production must fail closed if Edge AI is not enabled/configured.
- M3.1 should not implement Edge provider config/key-reference resolution. Missing provider config or key-reference readiness remains a fail-closed condition until M3.2 implements server-side resolution.

## Safe Browser Fallback Sequence

1. Inventory all browser AI paths and key paths before changing behavior.
2. Add or plan a narrow mode contract that separates `local-demo`, `internal-dev`, `pilot`, and `production`.
3. Gate pilot/production orchestrator paths so they cannot instantiate browser provider classes or read browser/Vite provider keys.
4. Preserve demo fallback only through explicit `local-demo` behavior using prepared or synthetic-only data.
5. Preserve transitional fallback only through explicit `internal-dev` behavior with clear labeling and no real customer data.
6. Add static scan evidence and focused tests for forbidden pilot/production browser provider and key paths.
7. Retire `setup-api-key.js` by removing, archiving, or clearly marking it inactive during implementation.

## Test Plan

Future implementation should verify:

- TypeScript typecheck passes.
- Existing deterministic scoring and delivery/govern tests continue to pass.
- `pilot` mode generation fails closed when Edge AI is disabled.
- `production` mode generation fails closed when Edge AI is disabled.
- `pilot` and `production` modes do not instantiate `GeminiProvider` or `GroqProvider`.
- `pilot` and `production` modes do not read `VITE_GEMINI_API_KEY`, `VITE_GROQ_API_KEY`, `StorageKeys.API_KEY`, or user-provided raw provider key state for provider execution.
- `local-demo` behavior is clearly limited to prepared demo data and synthetic-only fallback.

- Disabled AI surfaces such as AI Insights and AI sprint planning remain pilot-safe.
- Docs Forge surfaces a controlled unavailable state when Edge AI is required but unavailable.
- PDF/DOCX unsupported extraction fails gracefully or is restricted in pilot when later source-safety work is approved.

## UI / Copy Impact

- Pilot/production UI should show server-side AI unavailable states without asking users for browser raw keys.
- The Avala AI Controls area should not expose raw-key entry.
- Provider selection copy should avoid framing provider configuration as a user-side key setup task for pilot/production.
- AI generated draft copy must continue to state that outputs are reviewable handoff artifacts.
- Copy must not imply AI determines scores, approvals, risk gates, recommendations, or compliance status.
- Local-demo/internal-dev fallback labels should be visible enough that they cannot be mistaken for pilot-safe behavior.

## PDF / DOCX Upload Mismatch

- Current UI accepts PDF and DOCX uploads.
- Current Docs Forge browser extraction reads files with `file.text()`, which is not production-ready for binary formats.
- M3.1 inventories this mismatch so pilot/production browser AI lockdown can avoid unsafe source handling assumptions.
- Remediation belongs to M3.4 Prompt Registry + Source Safety unless AP separately approves earlier work.
- Pilot behavior must restrict unsupported binary extraction or fail gracefully when M3.4 source safety is implemented.
