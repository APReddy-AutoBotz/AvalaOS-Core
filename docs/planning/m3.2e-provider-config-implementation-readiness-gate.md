# M3.2e Provider Config Implementation Readiness Gate

Status: Planning/readiness-gate only

## Goal

Create a final implementation-readiness decision pack before any real schema/RLS, Supabase Function, Edge resolver, provider config, key-reference, or audit/evidence implementation begins.

This milestone reconciles M3.2, M3.2a, M3.2b, M3.2c, and M3.2d into one AP-reviewable implementation plan.

M3.2e does not create migrations, edit schema SQL, change Supabase Functions, change runtime behavior, change product code, change tests, change package files, change dependencies, change Admin UI, touch Health, change scoring, add runtime agents, add MCP/A2A behavior, add external execution, remove browser demo/dev fallback, or make compliance claims.

## Source-Of-Truth Alignment

This gate follows `docs/00_SOURCE_OF_TRUTH.md`:

- Evaluate before you automate. Govern before you execute.
- Deterministic scoring remains deterministic.
- AI must not decide final scores, gates, risk tiers, recommendations, approvals, or compliance status.
- Browser-stored provider keys and browser-side AI are not acceptable for pilot or production behavior.
- Server-side AI through Supabase Edge Functions remains the pilot and production direction.
- Evidence proves every governed decision.

## Consolidated Architecture Decision

The final proposed architecture across M3.2 through M3.2d is server-side, fail-closed, and evidence-backed. It is not approved for implementation by this readiness gate.

| Component | Consolidated decision |
| --- | --- |
| Provider registry | Use a server-side registry for supported provider IDs, display labels, status semantics, provider capabilities, supported operations, and safe model metadata. Initial runtime provider IDs remain `groq` and `gemini` because `supabase/functions/_shared/ai.ts` currently supports those providers. |
| Provider config record | Use tenant/org and optionally workspace-scoped provider config metadata for provider ID, display name, status, default model, model policy, allowed modes, allowed operations, key-reference binding, and audit metadata. Existing `ai_provider_configs` is prior context only and is not sufficient as-is. |
| Provider key-reference model | Store only reference metadata that points to server-side secret material or secret manager identifiers. Do not store raw provider secrets in ordinary application DB rows, even encrypted, unless AP separately approves a dedicated secret-storage model. |
| Workspace/tenant provider policy | Enforce a server-side policy that resolves whether a tenant/workspace may use a provider config for the requested mode and operation. Missing, ambiguous, disabled, or denied policy fails closed. |
| Edge resolver contract | Future resolver receives mode, provider/config selector, operation type, tenant/workspace ID, actor ID, prompt/version reference, requested model metadata, correlation/evidence ID, and key-reference selector. It returns safe metadata, policy result, prompt binding, evidence context, usage shell, and a server-side-only provider-call handoff. |
| Audit/evidence event contract | Future evidence events prove gates were checked. They record result, reason class, safe identifiers, timestamps, sanitized status, and usage metadata shell. They never store raw secrets, prompt bodies, response bodies, raw provider errors, hidden chain/payload, or unredacted stacks. |
| Prompt/version binding | Every pilot/production provider call should bind to a governed prompt/template/version reference before provider execution. Existing `prompt_templates` is prior context only and may need alteration or a separate prompt version model. |
| Usage metadata | Usage must come from server-side provider responses. Safe usage may include provider ID, model, token counts, output reference, operation type, status, correlation ID, and cost estimate only when safe and non-misleading. |
| Sanitized failure classes | M3.2c classes are the canonical starting point: `mode_not_allowed`, `unauthenticated`, `unauthorized_workspace`, `provider_policy_missing`, `provider_disabled`, `provider_not_allowed`, `operation_not_allowed`, `key_reference_missing`, `key_reference_disabled`, `key_resolution_failed`, `prompt_version_missing`, `provider_call_blocked`, and `provider_call_failed_sanitized`. |
| M3.1c scanner interaction | Future implementation must update `docs/quality/ai-boundary-static-scan-allowlist.json` only with exact, evidence-backed classifications. Direct server `GEMINI_API_KEY` and `GROQ_API_KEY` hits should be reclassified or removed after resolver implementation. Browser `VITE_*` provider secrets remain forbidden outside approved `local-demo` / `internal-dev` fallback. |

## Current Repository State

The current repo still has these implementation facts:

- `docs/schema/ai_governance_jobs.sql` contains prior-context `ai_provider_configs`, `prompt_templates`, `ai_generation_jobs`, and `ai_usage_events`.
- `supabase/functions/_shared/ai.ts` reads `Deno.env.get('GROQ_API_KEY')` and `Deno.env.get('GEMINI_API_KEY')` through `getProviderKey`.
- `supabase/functions/_shared/ai.ts` reads `GROQ_MODEL` and `GEMINI_MODEL` for model selection.
- `services/aiMode.ts` keeps pilot/production behind Edge and allows browser fallback only for `local-demo` / `internal-dev`.
- `services/geminiService.ts` still contains browser VITE provider fallback references classified by M3.1c as explicit local-demo/internal-dev fallback.
- `scripts/check-ai-boundary.mjs` scans tracked and untracked files and fails on forbidden hits or stale allowlist entries.

M3.2e does not change any of these files.

## Conflict And Gap Review

No direct contradiction was found across M3.2 through M3.2d on the main product law: provider resolution must be server-side, raw secrets must not enter browser/client paths, and missing/unsafe controls must fail closed.

The remaining gaps are AP decisions rather than contradictions.

| Area | Current decision | Conflict/gap? | Required AP decision | Recommended next action |
| --- | --- | --- | --- | --- |
| Schema shape | M3.2b proposes provider configs, key refs, workspace policies, provider audit events, prompt templates/versions, and usage events. | Gap: final table/column shape is not approved. | Approve whether to alter existing objects, add new objects, or defer schema. | Do not create migrations until AP approves final object list and columns. |
| RLS ownership | RLS must isolate tenant/workspace metadata and keep sensitive fields service-side. | Gap: exact actor model and workspace membership source are not final. | Approve read/write roles, workspace ownership model, and service-role bypass posture. | Require RLS design review before migration PR. |
| Service-role boundary | Edge/service-role may resolve key refs only after auth, membership, policy, provider, operation, prompt, and evidence gates pass. | Gap: exact service-role function/module boundary is not approved. | Approve which server paths may read config, key refs, prompts, and write audit/usage. | Keep first implementation schema-only or skeleton-only until boundary is explicit. |
| Key-reference storage model | Store server-side references only; raw secrets never in ordinary DB rows, even encrypted, without separate AP approval. | Gap: secret manager vs server env reference strategy is still undecided. | Approve initial key-reference resolver type and whether a dedicated secret-storage model is out of scope. | Prefer reference rows pointing to server env or secret manager identifiers; no raw secret columns. |
| Tenant/workspace policy source | Future resolver must enforce tenant/workspace policy before provider calls. | Gap: policy source table and org-level fallback precedence are not final. | Approve whether policy is separate table, provider config columns, or both. | Treat workspace policy as separate object in migration plan unless AP chooses simpler scope. |

| Browser demo/dev fallback cleanup | Browser fallback remains approved only for `local-demo` / `internal-dev`. | Gap: cleanup is intentionally later and must not be bundled with provider config implementation. | Approve separate fallback cleanup proposal if needed. | Do not remove browser demo/dev fallback in first implementation PR. |

## Recommended First Implementation PR

Recommended option: no implementation yet; more AP decision work is needed.

Reason:

- The schema shape is not final.
- RLS ownership is not final.
- Service-role resolver boundary is not final.
- Tenant/workspace policy source is not final.
- Key-reference storage and resolver type are not final.
- Audit/evidence table reuse versus a new table is not final.

The safest first real implementation PR after AP decisions should be a schema/RLS migration PR with no runtime provider calls.

Why schema/RLS first after AP approval:

- It is smaller and more reversible than changing runtime provider behavior.
- It can be tested with migration/RLS verification without enabling new AI execution paths.
- It can preserve current direct Edge env-key behavior until a later resolver PR.
- It can avoid exposing raw secrets by using reference-only fields.
- It gives scanner evidence a stable DB contract before runtime files move.

## Migration Execution Checklist

If AP approves a schema/RLS migration as the first implementation PR, plan only:

- Exact migration purpose: establish reference-only provider config, key-reference, workspace policy, and provider audit evidence structures without enabling provider calls.
- Candidate tables/fields: `ai_provider_configs`, `ai_provider_key_refs`, `ai_workspace_provider_policies`, `ai_provider_audit_events`, prompt/template version references, and safe `ai_usage_events` fields only as approved.
- RLS assumptions: tenant/workspace members read safe metadata only; write paths are admin/control-plane server or service-role only; browser/client cannot mutate provider config/key refs/policies/audit events.
- Service-role assumptions: service-role may read config/key-reference metadata and write audit/usage only through approved server-side workflows after resolver gates pass.
- Browser/client non-access assumptions: browser cannot read raw secrets, resolved key values, secret manager payloads, service-role-only fields, or sensitive key-reference internals.
- Rollback strategy: rollback must preserve fail-closed behavior, leave existing unrelated data intact, and remove only AP-approved new structures or columns.
- Seed/demo strategy: seed data may use safe provider labels, statuses, and non-secret references only; never raw provider secrets.
- Local-dev strategy: local examples may use placeholder references to server env names or secret-manager identifiers, never committed secret values.
- Verification strategy: run typecheck, tests, static scan, build, audit, diff check, wording scan, and any AP-approved migration/RLS verification.
- Data-safety checks: confirm no raw secret columns, no prompt/response payload storage in evidence, no browser write policy, no unsafe member-create policy for audit events, and no compliance claim text.
- AP approval needed: migration file creation must not begin until AP approves final schema/RLS object list, columns, policies, rollback, and verification plan.

## Secret-Safety Gate

Future implementation must follow these rules:

- raw provider secrets must never be stored in ordinary application DB rows, even encrypted, unless AP separately approves a dedicated secret-storage model
- raw provider secrets must never be returned to browser/client
- raw provider secrets must never be logged
- raw provider secrets must never appear in evidence
- key references must point to server-side secret material or secret manager identifiers
- provider-call layer may receive secret material only inside Edge/server boundary
- no secret material may cross into general business logic

## Evidence-Safety Gate

Future evidence must follow these rules:

- evidence proves gate checks
- evidence does not store sensitive payloads
- evidence records result, reason class, safe identifiers, and timestamps
- evidence must not store raw secret, prompt body, response body, raw provider error, hidden chain/payload, `localStorage` values, `userApiKey`, or unredacted stack traces

## M3.1c Scanner Readiness

Future implementation must update scanner evidence this way:

- Static scanner allowlist changes must be exact and evidence-backed.
- Direct `GEMINI_API_KEY` and `GROQ_API_KEY` hits should be reclassified or removed after an approved resolver replaces direct server env-key reads.
- Resolver file classifications must prove the path is server-side and cannot run in browser/client bundles.
- Audit/evidence classifications must prove events store safe metadata only.
- Stale allowlist entries must be removed when implementation moves or deletes direct references.
- Browser `VITE_GEMINI_API_KEY`, `VITE_GROQ_API_KEY`, and other VITE provider secrets remain forbidden outside approved `local-demo` / `internal-dev` fallback.
- Evidence required before allowlist updates: path, pattern, line or anchor, server-only proof, secret posture, reason, future action, static scan pass, and wording scan review.
