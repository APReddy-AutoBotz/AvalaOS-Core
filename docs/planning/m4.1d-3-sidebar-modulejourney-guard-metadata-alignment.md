# M4.1d-3 Sidebar And ModuleJourney Guard Metadata Alignment

## Purpose

M4.1d-3 aligns Sidebar and ModuleJourney navigation decisions with the central View/RBAC guard metadata introduced in M4.1d-1 and integrated into the App shell in M4.1d-2.

The milestone keeps the existing navigation layout and product semantics. It does not redesign Sidebar, redesign ModuleJourney, change route behavior, or implement deferred decision areas.

## Baseline

- Baseline milestone: `M4.1d-2 App.tsx Central Guard Integration`
- Baseline tag: `avalaos-core-m4.1d-2-app-central-guard-integration`
- Prior accepted state: App-level view/scope transitions use `resolveViewAccess`; Sidebar and ModuleJourney still had separate local navigation metadata.

## Strict Scope

In scope:

- Use central View/RBAC guard metadata for Sidebar navigation access decisions.
- Use central View/RBAC guard metadata for ModuleJourney clickable/disabled/hidden state.
- Preserve Sidebar labels, icons, sections, and current disabled-with-tooltip scope behavior.
- Preserve ModuleJourney operating lifecycle layout as much as possible.
- Keep App changes minimal and limited to prop wiring required by ModuleJourney.

Out of scope:

- Sidebar redesign.
- ModuleJourney redesign.
- App guard behavior changes.
- Route/product semantic changes.
- Deterministic scoring changes.
- Schema, RLS, migrations, or Supabase Edge Function changes.
- Provider runtime or provider governance changes.
- Health changes.
- Mock/demo data changes.
- M4.1e buyer-demo safe mode.
- M4.1f Admin/AI Controls.

## Root Cause

M4.1d-2 made App-shell transitions guarded, but navigation display still had independent rules:

- Sidebar stored local module, scope, and permission arrays for each nav item.
- Sidebar evaluated local permission access rather than the central guard result.
- ModuleJourney displayed enabled modules and navigated directly to `MODULE_HOME_VIEW`.
- ModuleJourney did not know whether a module home was allowed for the current user, organization, module set, and scope.

This could let navigation visibility and clickability drift from the central guard contract.

## Sidebar Alignment Plan

Sidebar keeps only presentation metadata locally:

- view
- label
- icon
- section grouping

Sidebar delegates access decisions to `resolveViewAccess` with the current user, auth/org loading state, organization, enabled modules, target view, and current scope.

Sidebar behavior by central guard result:

| Guard result | Sidebar behavior |
| --- | --- |
| `allowed` | Show enabled nav item. |
| `invalid_scope` | Show disabled nav item with available-scope tooltip. |
| `disabled_module` | Hide nav item. |
| `missing_permission` | Hide nav item. |
| `deferred_view` | Hide nav item. |
| `auth_loading` | Hide nav item while loading. |
| `unauthenticated`, `no_organization`, `setup_required` | Hide nav item. |
| `stale_persisted_view`, `admin_decision_pending` | Hide nav item. |

The Avala Admin organization-scope button remains a preserved behavior path. Workspace/Admin semantics remain an AP decision area and are not broadened in M4.1d-3.

## ModuleJourney Alignment Plan

ModuleJourney keeps its module labels, icons, outcomes, and lifecycle layout.

For each enabled module, ModuleJourney resolves the module home view through `resolveViewAccess`:

- Allowed module homes remain clickable.
- Invalid-scope module homes remain visible but disabled, with a title that names the valid scope.
- Missing-permission, disabled-module, deferred, loading, setup, and decision-pending results are hidden.
- ModuleJourney no longer navigates blindly to `MODULE_HOME_VIEW`.

The active module indicator reads from `VIEW_ACCESS_METADATA` instead of a separate view-to-module map.

## App Wiring

App only passes `currentScope` into ModuleJourney so ModuleJourney can resolve target module homes against the active scope.

M4.1d-3 does not change:

- `resolveAppViewAccess`
- `applyGuardedView`
- persisted/default view fallback
- Assess-to-Studio source-context behavior
- Docs-to-Delivery behavior
- organization-scope Admin Workspace behavior

## Tests And Verification Approach

No new UI test framework is introduced. Because the alignment stays inside existing React components without a pure navigation helper, verification is structural plus existing guard and regression gates:

- TypeScript validates the new props and central guard calls.
- `services/viewAccessGuard.test.ts` remains the source for guard semantics.
- `npm run test:view-access-guard` verifies Admin bypass and any-one-permission semantics remain intact.
- Full app build verifies Sidebar, ModuleJourney, and App wiring compile together.
