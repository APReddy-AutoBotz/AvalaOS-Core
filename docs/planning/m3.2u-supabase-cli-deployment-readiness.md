# M3.2u Supabase CLI Deployment Readiness

## Milestone Purpose

M3.2u is a tooling/readiness milestone to diagnose and restore local Supabase CLI deployment capability for the approved non-production target.

This milestone exists because M3.2t stopped before deployed blocked-smoke execution when the local Supabase CLI could not find its co-located `supabase-go` binary.

M3.2u does not deploy Edge Functions and does not run deployed smoke.

## Dependency Chain Through M3.2t

M3.2u depends on the prior M3.2 chain:

- M3.2n integrated resolver, audit, secret eligibility, and provider handoff gates into the target Edge Functions.
- M3.2p proved resolver sequencing with mocked provider behavior, sanitized audit events, transaction rollback cleanup, and no live provider calls.
- M3.2q planned deployed Edge smoke with blocked scenarios first and live provider smoke blocked.
- M3.2r stopped before deployed smoke because deployed invocation credentials were incomplete.
- M3.2s confirmed local invocation environment readiness and non-production target classification.
- M3.2t confirmed the target Edge Functions were not reachable, attempted deployment only to the sanitized non-production target, and stopped when the local Supabase CLI could not find `supabase-go`.

## CLI Diagnosis Plan

M3.2u diagnosis is local-tooling only.

Allowed checks:

- Confirm repository and remote safety.
- Confirm `.env` is not tracked.
- Load `.env` into process memory only if needed for non-printing visibility checks.
- Confirm required Supabase invocation variables are process-visible without printing values.
- Confirm `AVALA_AI_RUNTIME_MODE` is `pilot`.
- Confirm the Supabase target can be parsed as an approved non-production target without printing URL or project-ref values.
- Inspect local Supabase CLI resolution with `where.exe supabase`, `Get-Command supabase`, `supabase --version`, and user-level CLI file metadata.
- Inspect whether the local CLI directory contains the `supabase-go` companion binary.
- Check available local installer sources only if useful and without changing repo files.

Disallowed checks:

- Edge Function deploy.
- Deployed Edge Function invocation.
- Database seed.
- Provider API call.
- Real provider key use.
- Secret value printing.

## Repair Plan

If diagnosis confirms the CLI binary is incomplete, M3.2u may perform a local-only repair when all conditions are true:

- The repair touches only user-level tooling or temporary files outside the repo.
- The repair does not modify package manifests.
- The repair does not add package dependencies.
- The repair does not modify runtime, schema, UI, Health, scoring, prompt, browser fallback, usage, or cost behavior.
- The repair uses a verifiable Supabase CLI release asset.
- No secrets are printed, stored, or committed.

Preferred repair path:

- Use an official Supabase CLI Windows release asset matching the installed CLI version.
- Verify the downloaded asset against the official release checksum.
- If the installed `supabase.exe` already matches the verified release asset, add only the missing `supabase-go.exe` companion binary beside it.
- Do not commit repair artifacts.

## Readiness Checks

After diagnosis or repair, M3.2u checks:

- `supabase --version` works.
- The local CLI no longer fails solely because `supabase-go` is missing.
- The approved non-production target can be parsed without printing values.
- Optional `supabase functions list` may be run only against the approved non-production target and only without printing secrets.
- No deploy, invoke, seed, or provider call occurs.

## Outcome Framework

Outcome A - ready:

- Supabase CLI works.
- Missing `supabase-go` blocker is resolved.
- Non-production target readiness is confirmed safely.
- M3.2v may attempt deployed blocked-smoke execution after AP review.

Outcome B - blocked:

- CLI is still broken.
- `supabase-go` is still missing.
- Required CLI auth credentials are unavailable.
- Target readiness cannot be safely confirmed.
- M3.2v remains blocked.

## M3.2v Go/No-Go Criteria

M3.2v is a go only if AP review accepts evidence that:

- The local Supabase CLI is complete.
- `supabase --version` works.
- The missing `supabase-go` blocker is resolved.
- The approved non-production target is confirmed without printing values.
- CLI auth is available without exposing secrets.
- No M3.2u deploy, invoke, seed, provider call, or runtime behavior change occurred.

M3.2v is a no-go if:

- CLI auth remains unavailable.
- The target cannot be confirmed safely.
- The local CLI again fails on `supabase-go`.
- AP requires a separate tooling approval path before deployment is retried.

## Stop Conditions

M3.2u stops before any deploy, invoke, seed, or provider call if:

- Repository or remote safety fails.
- Required environment visibility is blocked.
- Runtime mode is not `pilot`.
- The target appears production, shared, customer, live, or unclear.
- CLI repair would require repo changes or dependency changes.
- CLI repair cannot be verified.
- CLI auth is unavailable for target readiness checks.
- Any secret would need to be printed or stored.

## Risks Carried Forward

- The target Edge Functions remain not deployed/reachable until a future milestone deploys them.
- Deployed blocked scenarios remain unexecuted.
- Deployed runtime-mode failure behavior remains unproven.
- Safe deployed mock behavior remains unconfirmed.
- Live provider network calls and real keys remain blocked unless AP explicitly approves them.
- CLI auth may remain a separate AP/ops prerequisite for M3.2v.
