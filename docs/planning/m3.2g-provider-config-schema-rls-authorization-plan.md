# M3.2g Provider Config Schema/RLS Authorization Plan

Status: Planning/authorization only

Branch: `milestone/m3.2g-provider-config-schema-rls-authorization-plan`

## Milestone Purpose

M3.2g is the final planning and AP authorization gate before the first real provider config schema/RLS migration PR.

This milestone converts M3.2f AP direction into an exact schema/RLS migration blueprint. It does not implement schema/RLS, create migrations, edit schema SQL, change Supabase Functions, change runtime behavior, change Admin UI, change product code, change tests, change package files, touch Health, change scoring, change browser fallback behavior, add runtime agents, add MCP/A2A behavior, add external execution, or introduce unsupported compliance claims.

Blueprint details in this document are table, column, RLS, rollback, and verification guidance only. They are not migration-ready SQL.

- Evaluate before you automate. Govern before you execute.
- Deterministic scoring remains deterministic.

## Exact First Migration Object List

The first schema/RLS migration PR after AP approval should create or alter only these proposed objects:

- `ai_provider_configs`
- `ai_provider_key_refs`
- `ai_workspace_provider_policies`
- `ai_provider_audit_events`

Deferred objects, features, and behavior:

- prompt version table changes
- usage/cost expansion
- Admin UI
- runtime resolver
- provider-call integration
- external secret-manager lookup
- Supabase Function changes
- browser fallback removal
- any Health work

`ai_workspace_provider_policies` remains the future-compatible name for provider policy. Because the current repo schema has `organizations`, `profiles`, `roles`, and active `organization_members`, but no canonical `workspaces` table, the first migration blueprint must use `org_id` as the enforceable tenant boundary. A strict `workspace_id` foreign key remains deferred until AP approves a workspace authority.

## Proposed Table Blueprint

### `ai_provider_configs`

Purpose:

- Store safe provider configuration metadata for future server-side resolver decisions.
- Describe provider status, allowed modes, allowed operations, model policy, and optional key-reference binding.

Ownership model:

- Tenant-owned through `org_id`.
- Workspace-specific scope is deferred until AP approves a canonical workspace authority.

Required columns:

- `id`
- `org_id`
- `provider`
- `display_name`
- `status`
- `default_model`
- `model_policy`
- `allowed_modes`
- `allowed_operations`
- nullable `key_ref_id`
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`
- `deleted_at`

Optional or deferred columns:

- strict `workspace_id`
- provider capability version
- provider registry version
- prompt/template/version references
- usage/cost references
- secret-manager lookup fields

Sensitive fields prohibited:

- raw provider keys
- encrypted provider keys
- provider auth headers
- browser-readable secret values
- secret-manager values or payloads
- resolved key material

Expected indexes:

- `org_id`
- `provider`
- `status`
- `key_ref_id`
- active records by `org_id`, `provider`, and status

Expected uniqueness constraints:

- active provider config identity should be unique by tenant, provider, and display label or approved provider config key.
- default-provider uniqueness should live in `ai_workspace_provider_policies`, not in provider configs.

Expected foreign-key relationships:

- `org_id` references `organizations.id`.
- `created_by` and `updated_by` reference `profiles.id`.
- `key_ref_id` may be a nullable FK to `ai_provider_key_refs.id`.

Soft-delete or status strategy:

- Use status for lifecycle: active, disabled, pending review, retired.
- Use `deleted_at` for soft-delete when records should no longer resolve.

Audit/evidence role:

- Provider config changes should be represented by safe events in `ai_provider_audit_events`.
- Provider configs are metadata inputs to future resolver evidence, not proof that provider calls are allowed.

### `ai_provider_key_refs`

Purpose:

- Store reference-only key metadata and lifecycle state for future server-side resolver work.

Ownership model:

- Tenant-owned through `org_id`.
- Service-owned for sensitive resolution behavior.
- No FK back to provider configs in the first migration blueprint, to avoid circular FK complexity.

Required columns:

- `id`
- `org_id`
- `provider`
- `resolver_type`
- `secret_ref`
- `safe_label`
- `safe_fingerprint`
- `status`
- `rotation_status`
- `last_rotated_at`
- `expires_at`
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`
- `deleted_at`

Optional or deferred columns:

- strict `workspace_id`
- external secret-manager provider metadata
- resolver configuration details
- connection-test evidence references

Sensitive fields prohibited:

- raw provider keys
- encrypted provider keys
- provider auth headers
- browser-readable secret values
- raw secret-manager values or payloads
- resolved key material

`secret_ref` definition:

- `secret_ref` means a service-only reference identifier.
- `secret_ref` must never contain raw provider keys, encrypted provider keys, provider auth headers, browser-readable secret values, raw secret-manager payloads, or values that browser/client code could use to resolve provider credentials.
- Browser/client responses must not expose `secret_ref` unless AP later proves it is non-sensitive and safe for that specific surface.

Expected indexes:

- `org_id`
- `provider`
- `status`
- `resolver_type`
- rotation and expiration timestamps

Expected uniqueness constraints:

- active key-reference labels or fingerprints should be unique per tenant/provider when AP approves the exact safe identifier rule.

Expected foreign-key relationships:

- `org_id` references `organizations.id`.
- `created_by` and `updated_by` reference `profiles.id`.
- No FK back to `ai_provider_configs` in the first migration blueprint.

Soft-delete or status strategy:

- Use status for active, disabled, expired, pending rotation, and retired states.
- Use `deleted_at` for soft-delete when a reference must no longer resolve.

Audit/evidence role:

- Key-reference lifecycle changes and future resolver checks should write safe audit events.
- Audit events may store key-reference ID, safe label, status, or AP-approved safe fingerprint only.

### `ai_workspace_provider_policies`

Purpose:

- Define the canonical tenant/provider policy for future resolver decisions by mode, operation, role, default selection, and enablement.

Ownership model:

- First migration blueprint is tenant-owned through `org_id`.
- The table name stays future-compatible for later workspace authority.
- Strict workspace FK is deferred until AP approves a canonical workspace table and membership model.

Required columns:

- `id`
- `org_id`
- `provider_config_id`
- `operation`
- `mode`
- `allowed_roles`
- `is_default`
- `status`
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`
- `deleted_at`

Optional or deferred columns:

- strict `workspace_id`
- policy precedence
- org-to-workspace inheritance
- advanced role/action matrix
- policy versioning

Sensitive fields prohibited:

- raw provider secrets
- encrypted provider secrets
- secret references
- provider auth headers
- prompt bodies
- provider request or response payloads

Expected indexes:

- `org_id`
- `provider_config_id`
- `operation`
- `mode`
- `status`
- default policy lookup by tenant, operation, and mode

Expected uniqueness constraints:

- at most one active default policy per tenant, operation, and mode.
- active policy identity should be unique by tenant, provider config, operation, and mode.

Expected foreign-key relationships:

- `org_id` references `organizations.id`.
- `provider_config_id` references `ai_provider_configs.id`.
- `created_by` and `updated_by` reference `profiles.id`.

Soft-delete or status strategy:

- Use status for active, disabled, pending review, and retired.
- Use `deleted_at` to remove policies from resolver eligibility.

Audit/evidence role:

- Policy records are the source of future provider allow/deny evidence.
- Missing, disabled, conflicting, or ambiguous policy must fail closed.

- `failure_class`
- `actor_id`
- `service_context`
- `correlation_id`
- `evidence_ref`

- `created_at`

Optional or deferred columns:

- prompt template reference
- prompt version reference
- generation job reference
- usage event reference
- retention classification
- immutable event hash

Sensitive fields prohibited:

- raw provider keys
- encrypted provider keys
- provider auth headers
- service-role keys
- browser/client provider secrets
- raw prompts
- raw model responses
- raw source document bodies
- raw provider request or response payloads
- raw provider errors when sensitive
- secret-manager values or payloads
- hidden payloads

- `provider_config_id`
- `key_ref_id`
- `event_type`

Soft-delete or status strategy:

Audit/evidence role:

- This is the primary provider governance evidence table.
- It proves gates were checked without storing sensitive payloads.

## RLS Blueprint

### `ai_provider_configs`

- RLS posture: enabled, tenant-scoped, fail-closed.
- Allowed readers: active tenant members may read safe metadata only when AP-approved policy allows it.
- Allowed writers: AP-approved server/control workflows only; future Admin UI writes must be mediated by server/control workflows, RLS, audit evidence, and secret-safety gates.
- Service-role use: service role may read metadata only inside future approved server-side resolver/control workflows.
- Isolation logic: `org_id` must match an active `organization_members` row for user-visible safe reads.
- Fail-closed behavior: missing org ownership, inactive membership, disabled config, deleted config, or unclear RLS policy blocks access.
- Browser/client must not access raw secrets, secret-manager values, key material, service-only fields, or direct write paths for sensitive provider objects.

### `ai_provider_key_refs`

- RLS posture: enabled, tenant-scoped, service-sensitive, fail-closed.
- Allowed readers: service-role future resolver paths and AP-approved admin/control evidence paths; member/browser reads of sensitive internals are prohibited.
- Allowed writers: AP-approved server/control workflows only.
- Service-role use: service role may read key-reference metadata only after future resolver gates are satisfied.
- Isolation logic: `org_id` must remain the tenant boundary; strict workspace isolation is deferred until AP approves workspace authority.
- Fail-closed behavior: missing, disabled, expired, malformed, or unauthorized key reference blocks provider use before provider calls.
- Browser/client must not resolve provider keys, receive `secret_ref` if sensitive, or directly write key-reference metadata.

### `ai_workspace_provider_policies`

- RLS posture: enabled, tenant-scoped, fail-closed.
- Allowed readers: active tenant members may read safe policy metadata only if AP approves visibility.
- Allowed writers: AP-approved server/control workflows only.
- Service-role use: service role may read policy records for future resolver decisions.
- Isolation logic: first migration uses `org_id` as the enforceable boundary. Strict `workspace_id` FK is deferred.
- Fail-closed behavior: missing, disabled, conflicting, ambiguous, or deleted policy blocks provider use.
- Browser/client must not directly write sensitive policy records; future Admin UI writes must go through AP-approved server/control mediation.

### `ai_provider_audit_events`

- RLS posture: enabled, tenant-scoped, append-oriented, fail-closed.
- Allowed readers: AP-approved admin/control evidence readers; normal member visibility remains deferred.
- Allowed writers: service-side resolver/control workflows only.
- Service-role use: service role may write safe evidence events and read evidence needed for AP-approved control workflows.
- Isolation logic: `org_id` must scope events; future workspace reference remains nullable until workspace authority is approved.
- Fail-closed behavior: provider calls remain blocked if future evidence preparation or write posture is unavailable.
- Browser/client must not directly write audit events or use audit rows to bypass policy.

## Service-Role Boundary

Future service-role workflows may:

- read safe provider config metadata
- read key-reference metadata without exposing secret material
- read tenant provider policy
- write append-oriented provider audit events
- later resolve server-side key material only after AP approves resolver implementation

Service role cannot bypass:

- runtime mode checks
- authentication
- tenant membership
- provider policy
- provider status
- operation authorization
- key-reference eligibility
- prompt/version binding
- evidence preparation
- scanner classification
- secret-safety gates

Browser/client code must never:

- use service-role credentials
- resolve provider keys
- receive raw provider keys
- receive encrypted provider keys
- receive provider auth headers
- receive secret-manager values or payloads
- directly write sensitive provider/key/audit objects
- bypass server/control mediation for future Admin UI writes

Resolver-related reads remain server-side. Raw secret exposure remains prohibited in DB rows, browser code, logs, evidence, responses, and UI-facing payloads.

## Key-Reference Safety Blueprint

- Key-reference records store metadata only.
- `secret_ref` is a service-only reference identifier, not raw secret material.
- No raw provider keys, encrypted provider keys, auth headers, browser-readable secret values, raw secret-manager values, logs, or evidence payloads are allowed.
- No key material is returned to browser/client code.
- No runtime resolver implementation is created in M3.2g.
- No provider calls are added in M3.2g or in the first schema/RLS migration PR.
- `ai_provider_configs` stores only `key_ref_id` linkage and safe metadata. It must not store raw secrets, encrypted secrets, auth headers, browser-readable secret values, or secret-manager values.

## Rollback And Reversibility Plan

Future migration rollback should:

- be reversible for the first schema/RLS objects while tables are empty
- drop or disable only AP-approved provider config, key-reference, policy, and audit objects
- preserve fail-closed behavior if config/policy/key-ref tables are absent
- avoid deleting unrelated prior-context job, usage, prompt, or audit data unless AP separately approves
- include policy rollback considerations for each RLS policy created
- include object dependency order to avoid FK rollback failures
- stop if rollback cannot be proven before merge

Because the first PR should be schema/RLS-only and empty-table, data migration risk should be minimal. Any seed data, runtime dependency, or backfill requirement is scope creep and requires AP approval.
