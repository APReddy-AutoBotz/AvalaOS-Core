# M3.2n-dbval Provider Governance Bootstrap And DB Validation

## Milestone Purpose

M3.2n-dbval validates the provider governance schema path against the AP-approved fresh, empty, non-production Supabase database after M3.2n-readiness confirmed that database validation had previously been blocked.

This milestone is validation and evidence only. It bootstraps the minimum approved prerequisite schema sequence, applies the existing M3.2h provider governance migration, and records table, column, RLS, and provider-scope validation evidence.

## Dependency Chain

- M3.2h introduced the provider governance schema/RLS migration at `supabase/migrations/20260607150600_m3_2h_provider_config_governance.sql`.
- M3.2n-readiness confirmed validation was blocked until a safe non-production `DATABASE_URL` was available.
- AP/ChatGPT approved applying the prerequisite schema sequence only to a fresh, empty, non-production Supabase DB.
- M3.2n-dbval may proceed only if the target DB remains fresh/empty for AvalaOS app tables before applying schema.

## AP-Approved Prerequisite Sequence

Apply exactly this sequence before M3.2h:

1. `docs/schema/initial_schema.sql`
2. `docs/schema/platform_rls_membership_policies.sql`
3. `docs/schema/ai_governance_jobs.sql`
4. `supabase/migrations/20260607150600_m3_2h_provider_config_governance.sql`

Use the existing migration runner:

```powershell

```

The files may be run one by one to isolate failures.

## Safety Rules

- Do not print `DATABASE_URL`.
- Do not store `DATABASE_URL` or any secret in files.
- Do not commit secrets.
- Do not touch production, shared, or customer databases.
- Confirm `DATABASE_URL` is visible to Node without exposing its value.
- Confirm the target is intended non-production through the AP/ChatGPT prompt approval and the fresh-empty catalog gate.
- Run a read-only catalog check before applying any SQL.
- Stop if existing AvalaOS app tables are present unexpectedly.
- Stop if any apply or validation command fails.
- Do not insert provider secrets, raw prompts, completions, provider payloads, auth headers, secret refs, or test customer data.

## Stop Conditions

Stop and report Outcome B if:

- `DATABASE_URL` is not visible to Node.
- The target cannot be confirmed as the AP-approved non-production validation DB.
- The pre-apply catalog check finds existing AvalaOS app tables unexpectedly.
- Any SQL apply command fails.
- Any required post-apply table or column is missing.
- RLS is not enabled on required provider governance tables.
- Provider constraints are not limited to `gemini` and `groq`.
- Verification commands fail.
- The work would require runtime integration, provider calls, secret resolution, schema edits, Admin/UI changes, Health changes, scoring changes, browser fallback changes, package changes, seed/test customer data, or compliance claims.

## Outcome Framework

Outcome A - validated:

- Prerequisite sequence applied successfully to the safe non-production DB.
- M3.2h migration applied successfully.
- Required table, column, RLS, and provider-scope checks passed.
- M3.2n implementation may proceed after AP/ChatGPT review.

Outcome B - blocked:

- DB safety could not be confirmed, apply failed, or validation checks failed.
- M3.2n implementation remains blocked.

## M3.2n Go/No-Go Criteria

Go:

- Evidence confirms Outcome A.
- AP/ChatGPT reviews and accepts the bootstrap/validation evidence.
- No runtime integration, provider call, provider-secret resolution, schema-file edit, Admin/UI, Health, scoring, or browser-fallback change is introduced.

No-go:

- Any stop condition is met.
- AP/ChatGPT does not accept the validation evidence.
- Additional schema, RLS, runtime, provider, or product-scope decisions are needed.
