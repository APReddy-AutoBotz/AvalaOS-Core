# M5.3a-2h Local Supabase Stack Availability Remediation Implementation

## Milestone Objective

M5.3a-2h implements a narrow, status-only local remediation diagnostic for the local Supabase stack availability gap carried from M5.3a-2f and planned in M5.3a-2g.

The implementation adds sanitized availability diagnostics only. It does not start a local Supabase stack, connect to any database, bootstrap schema, run readiness SQL, run RLS harnesses, run artifact assertions, use personal or hosted Supabase, call providers, or claim local DB availability, schema readiness, artifact SELECT isolation, tenant isolation, hosted-pilot readiness, or production readiness.

The diagnostic script performs command-availability checks only. Raw command output is suppressed. The script emits only sanitized category names and approved status values.

No package script was added. The diagnostic is invoked directly as:

- `node scripts/m5.3a-2h-local-stack-availability-diagnostics.mjs --static`

## Diagnostic Categories Implemented

The script reports only these approved categories:

- Docker availability: `yes`, `no`, or `unknown`.
- Supabase CLI availability: `yes`, `no`, or `unknown`.
- Local config presence: `present`, `missing`, or `unknown`.
- Local stack startup readiness: `ready`, `blocked`, or `unknown`.
- Diagnostic output safety: `passed`, `failed`, or `unknown`.
- Remediation readiness: `ready`, `blocked`, or `unknown`.

The readiness categories indicate whether the status-only prerequisites appear sufficient for a future AP-approved local stack attempt. They do not prove local DB availability or schema readiness.

## Diagnostic Command Categories Used

The diagnostic uses only suppressed command-availability probes:

- Docker command availability probe with raw output suppressed.
- Supabase CLI command availability probe with raw output suppressed.
- Tracked repository local config presence check with no path output.

The script does not print command output, versions, executable locations, local paths, target values, project refs, host values, port values, database names, user names, IDs, logs, row payloads, claims, `.env` values, tokens, provider keys, service-role values, or secrets.

## Diagnostic Command Categories Not Used

The implementation does not run:

- `supabase start`
- `supabase stop`
- `supabase status`
- `supabase db reset`
- `supabase db push`
- `supabase db pull`
- `supabase migration up`
- `supabase link`
- `docker ps`
- `docker info`
- `docker compose`
- Any command that lists containers, images, ports, networks, volumes, local paths, logs, project refs, target values, or raw local output.
- Any DB command.
- Any DB connection.
- Any schema bootstrap or schema readiness command.
- Any `npm run test:rls:*` command.
- Any RLS harness.
- Any artifact assertion.
- Any remote Supabase validation.
- Any provider call.

## Sanitized Output Model

Output is category-only and counts/status-safe. The script emits:

- Command label.
- Static no-DB mode.
- Approved category names.
- Approved category status values.
- No-go action status as `not-run` or `not-made`.
- Proof boundary outcomes as `no`.
- Sanitized failure classifications.

The script includes an output safety guard. If generated output matches prohibited value patterns, the script reports diagnostic output safety as failed and exits non-zero without printing raw diagnostic values.

## Root-Cause Boundary

M5.3a-2h does not infer the root cause of the prior local stack failure.

The script can report only sanitized category status. It must not claim Docker, Supabase CLI configuration, local config, ports, containers, images, migrations, local paths, environment setup, or local machine state caused the prior failure unless a future AP-approved milestone proves that safely without prohibited output.

## Proof Boundary

- M5.3a-2h may prove only sanitized diagnostic readiness categories.
- M5.3a-2h does not resolve local DB availability.
- M5.3a-2h does not prove schema availability.
- M5.3a-2h does not verify artifact SELECT isolation.
- M5.3a-2h does not newly verify tenant isolation.
- M5.3a-2h does not claim hosted-pilot readiness.
- M5.3a-2h does not claim production readiness.

## Future Use Boundary

The diagnostic can inform whether a future AP-approved milestone should attempt local stack startup or continue remediation planning. It does not authorize local stack startup, DB execution, schema bootstrap, RLS harness execution, artifact assertion execution, hosted validation, or provider calls.
