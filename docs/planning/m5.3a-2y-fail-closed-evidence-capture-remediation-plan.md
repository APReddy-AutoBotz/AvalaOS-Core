# M5.3a-2y Fail-Closed Evidence Capture Remediation Plan

## Purpose

M5.3a-2y plans a compact remediation for the M5.3a-2x evidence capture failure. This milestone is planning-only and does not rerun the classifier or perform any startup, Supabase, Docker, DB, schema, RLS, artifact, migration, hosted validation, or provider command.

## M5.3a-2x Baseline

M5.3a-2x attempted exactly one AP-approved classifier evidence run.

- Evidence capture failed closed.
- No classifier category was proven.
- No rerun was performed.
- Root cause is not inferred.
- Local DB availability remains unresolved.
- Schema availability is not proven.
- Artifact SELECT isolation is not verified.
- Tenant isolation is not newly verified.
- Hosted readiness is not proven.
- Production readiness is not proven.

## Problem

The M5.3a-2x wrapper did not safely capture the approved classifier fields after the single approved run. Recovering the result would have required either a second classifier run or raw output inspection, both outside the approved scope. A future remediation must improve capture reliability without weakening the redaction, command-label, and fail-closed boundaries.

## Recommended Future Implementation

Recommended next milestone: M5.3a-2z Fail-Closed Evidence Capture Remediation Implementation, only after AP approval.

Future implementation should:

- Add safe parser/capture helpers for approved classifier output.
- Use synthetic fixtures only.
- Never record literal command strings.
- Never inspect, store, summarize, or copy raw output.
- Fail closed on missing, extra, unsafe, malformed, or unapproved fields.
- Preserve counts/status/category-only evidence.
- Preserve no-root-cause and no-readiness boundaries.
- Require a separate AP approval before any future real classifier rerun.

## Planned Capture Helper Boundary

The future helper should accept only already-redacted synthetic classifier result shapes. It should validate allowlisted fields, exact known categories, and required status values. It should reject unknown fields and return a sanitized failure category instead of partial evidence.

Allowed future output from the helper should be limited to command label, run count, sanitized category fields, confidence, safety status, scratch cleanup status, fail-closed status, and proof-boundary outcomes.

## Explicit Non-Goals

- No classifier rerun.
- No real evidence run.
- No startup execution.
- No Supabase stack execution.
- No Docker command or listing.
- No DB command or DB connection.
- No schema readiness command.
- No RLS harness.
- No artifact assertion.
- No migration or bootstrap command.
- No remote, hosted, or personal validation.
- No provider call.
- No raw output, scratch output, generated log, `.env`, local path, or local machine state inspection.
- No root-cause inference.
- No hosted, production, artifact isolation, tenant isolation, local DB, or schema availability claim.

## Acceptance Criteria For Future M5.3a-2z

- Synthetic safe parser tests cover approved-field success and fail-closed rejection for missing, extra, unsafe, malformed, and unapproved fields.
- Tests prove literal command strings are not accepted as evidence fields.
- Tests prove raw-output-like fields are rejected.
- Evidence remains compact and sanitized.
- Any future real classifier rerun remains blocked until a separate AP approval explicitly authorizes it.

## Proof Boundary

M5.3a-2y is planning-only. It does not fix the M5.3a-2x evidence capture behavior, does not rerun evidence, does not establish a classifier result, does not infer root cause, and does not advance local DB, schema, artifact SELECT isolation, tenant isolation, hosted readiness, or production readiness.
