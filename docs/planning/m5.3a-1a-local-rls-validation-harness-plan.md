# M5.3a-1a Local RLS Validation Harness Plan

## Purpose

M5.3a-1a plans a repeatable, local-only validation harness for the future M5.3a-1 RLS helper and SELECT policy implementation. It is a docs-only planning milestone. It does not implement the harness, SQL migrations, helper functions, RLS policies, test files, package scripts, CI, runtime adapters, seed data, backfill, constraint validation, or deployment changes.

The goal is to define the validation gate required before M5.3a-1 can claim verified local tenant-isolation behavior for identity, organization, workspace, role, and membership SELECT policies.

## Strict Non-Scope

M5.3a-1a does not:

- Create SQL migrations, RLS policy SQL, helper functions, validation scripts, tests, seed data, backfill, or constraint validation.
- Change package scripts, package dependencies, CI, deployment config, runtime adapters, UI, routing, view keys, Sidebar behavior, demo data, scoring, provider execution, export behavior, Reports/Teams/Admin behavior, or the separate Health repository.
- Create or change `supabase/config.toml`.
- Connect to remote Supabase.
- Run `scripts/applySupabaseMigrations.mjs`, `npm run db:migrate`, `npm run test:rls:assess`, any `DATABASE_URL`-based command, remote Supabase validation, or provider calls.
- Read, print, summarize, copy, expose, or commit `.env` values, secrets, personal hosted identifiers, provider keys, service-role values, tokens, bucket names, or project-specific values.
- Claim tenant isolation is implemented or proven.
- Claim hosted-pilot readiness.

## Current Validation Inventory

Current repository inspection found:

- The tracked Supabase file inventory does not include `supabase/config.toml`.
- Local Supabase startup cannot be assumed from tracked config.
- `package.json` includes `db:migrate` and `test:rls:assess`.
- Both `db:migrate` and `test:rls:assess` call `scripts/applySupabaseMigrations.mjs`.
- `scripts/applySupabaseMigrations.mjs` requires `DATABASE_URL` from the shell environment.
- `docs/schema/rls_regression_assess_handoff.sql` is an existing Assess/handoff regression contract, not the M5.3a-1 local membership harness.
- Existing `docs/schema` role-policy files include historical/draft policy material and do not define the approved M5.3a-1 harness.
- Safe validation for this docs-only milestone is limited to static repo checks, TypeScript/test/build/audit commands, `supabase --version`, and wording scans.

Unsafe paths for M5.3a-1a:

- Any `DATABASE_URL` command.
- `scripts/applySupabaseMigrations.mjs`.
- `npm run db:migrate`.
- `npm run test:rls:assess`.
- Existing Supabase delivery smoke scripts that depend on environment-specific hosted or database state.
- Remote Supabase validation.

## Harness Decision

Recommended future work:

- Create SQL test-contract files only after AP approves local-only validation scope.
- Create a JS smoke runner only after AP approves local target detection and no-secret output rules.
- Create a package script only after AP approves package-script impact.
- Track `supabase/config.toml` only if AP approves a generic, portable, secret-free local config.

Default AP posture:

- M5.3a-1b should implement only the local validation harness, not RLS helper functions or RLS policies.
- Static-review-only implementation is not the default.
- If AP overrides and allows static review only, evidence must state tenant isolation is not verified.

## Local-Only Validation Design

Future harness requirements:

- Never connect to remote Supabase.
- Never read, print, or summarize `.env` values.
- Never print `DATABASE_URL`, DB target values, service-role values, provider keys, JWT/private tokens, bucket names, hosted project URLs, personal org/workspace/project IDs, or row payload dumps.
- Never silently accept `DATABASE_URL`.
- Never fall back to ignored local secret files.
- Use generated local UUIDs only.
- Report scenario labels, pass/fail status, counts, and sanitized diagnostics only.
- Separate server-only fixture setup from anonymous/authenticated client behavior.
- Be safely rerunnable through transaction rollback, isolated schema/state, or explicit AP-approved cleanup.
- Fail safely if required local config or target proof is missing.

Local targets may be only:

- `localhost`.
- `127.0.0.1`.
- Docker local network.
- Another AP-approved local-only target.

## Remote-Target Rejection

The future runner must reject remote targets by design:

- It must fail if the DB target is not explicitly local.
- It must not log the DB target value.
- It must not accept hosted Supabase URLs.
- It must not use personal hosted project values.
- It must not use ignored local secret files as fallback configuration.
- It must not use a default environment variable path that can accidentally point at remote state.

## `supabase/config.toml` Decision

If future work proposes `supabase/config.toml`, AP must approve it before implementation.

The file must be:

- Generic and portable.
- Safe to commit.
- Local-only.
- Free of personal project refs, hosted Supabase URLs, keys, tokens, org IDs, bucket names, service-role values, provider keys, and personal values.

M5.3a-1a does not create or change `supabase/config.toml`.

## `scripts/applySupabaseMigrations.mjs` Posture

`scripts/applySupabaseMigrations.mjs` is inspect-only for this planning milestone. It must not be run in M5.3a-1a.

It should not become the default harness runner unless AP later approves:

- A local-only wrapper.
- DB target rejection.
- No secret output.
- No remote fallback.
- Evidence that no `DATABASE_URL` value is printed.

Any future `DATABASE_URL` strategy must be local-only and must never print the value.

## Fixture Design

Future M5.3a-1 fixtures should be deterministic, generated, local-only, and non-secret:

- Two local organizations.
- Two local workspaces per organization.
- Active member.
- Disabled organization member.
- Disabled workspace member.
- Non-member.
- Org-scoped roles.
- Workspace-scoped roles.
- Same-org and cross-org membership rows.
- Same-workspace and cross-workspace membership rows.
- Deleted-state variants where useful.

Fixtures must not include personal IDs, hosted URLs, real project IDs, provider keys, service-role values, tokens, bucket names, or `.env` values.

## Auth Simulation Design

Future implementation should evaluate and document one AP-approved local-only method for:

- Anonymous client behavior.
- Authenticated active member behavior.
- Authenticated disabled organization member behavior.
- Authenticated disabled workspace member behavior.
- Authenticated non-member behavior.
- Server-only fixture setup.

Candidate methods:

- SQL-level local role/session claim simulation for `auth.uid()`.
- Supabase local auth fixtures if tracked local config is approved.
- Local anon/authenticated client simulation if the local stack is repeatable.
- Another AP-approved local-only method.

Server-only fixture setup may use local privileged access only if AP approves it and values are never printed or committed.

## Isolation Scenarios

M5.3a-1 validation scenarios should cover identity, organization, workspace, role, and membership behavior only.

Negative scenarios:

- Anonymous user is denied.
- Authenticated non-member is denied.
- Disabled organization member is denied.
- Disabled workspace member is denied.
- Org A user cannot read Org B organization metadata.
- Org A user cannot read Org B membership metadata.
- Workspace A user cannot read Workspace B workspace membership metadata.
- Workspace A user cannot read Workspace B scoped role metadata.
- System role visibility remains denied unless AP approves it.
- Service-role behavior remains server-only and not browser-callable.

Positive scenarios:

- Own active profile is readable.
- Own active organization membership is readable.
- Own active workspace membership is readable.
- Organization metadata is readable only through active organization membership.
- Workspace metadata is readable only through active workspace membership.
- Role metadata is scoped to active membership.

Explicitly deferred artifact tests:

- `assess_processes`
- `assessments`
- `assessment_review_events`
- `projects`
- `document_generations`
- `delivery_work_items`

## Runtime And Hosted Readiness Posture

Hosted pilot readiness still requires later adapter, export, storage, deployment, operations, evidence/audit, backup/restore, and provider-control milestones. Remote Supabase validation remains out of scope.
