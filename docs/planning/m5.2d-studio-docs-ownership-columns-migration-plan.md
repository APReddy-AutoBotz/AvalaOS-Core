# M5.2d Studio Docs Ownership Columns Migration Plan

Milestone: M5.2d Studio Docs Ownership Columns Migration Plan

Branch: milestone/m5.2d-studio-docs-ownership-columns-migration-plan

Status: Docs-only planning pack

Recommended next milestone: M5.2e Delivery Ownership Columns Migration Plan

## Purpose

M5.2d defines the Studio and Docs ownership-column strategy needed before future workspace-aware RLS can safely cover generated artifacts, document generation records, document repository concepts, templates, source lineage, and Docs-to-Delivery handoff metadata.

The milestone does not implement schema. It records why a Studio/Docs migration should wait until the Delivery/project workspace boundary is planned, because the current draft `document_generations` contract links to `projects`, and `projects` is not applied migration-chain authority yet.

## Strict Non-Scope

M5.2d does not create migrations, alter schema, add RLS policies, add SQL, change runtime adapters, change UI, change routing, change guard semantics, change demo data, change scoring, change exports, change provider execution, change Supabase configuration, change CI, change dependencies, deploy, connect to hosted Supabase, call providers, seed data, backfill data, or modify the separate Health repository.

M5.2d does not claim tenant isolation, hosted-pilot readiness, production readiness, live-provider readiness, certification status, automated approval authority, machine-owned deterministic decisions, or current AGI capability.

No `.env` values, local secret files, provider keys, Supabase tokens, service-role values, hosted project identifiers, or personal environment values are needed for this plan.

## Current Dependency Finding

Studio/Docs ownership cannot be implemented safely as a first migration slice without settling the project and workspace boundary.

The current draft `document_generations` table is defined in `docs/schema`, not in the applied migration chain. It has `org_id` and `project_id`, but no `workspace_id`, creator, updater, reviewer, approver, source assessment, source process, lineage, lifecycle, or durable export authority. Runtime adapters resolve document generations through `projects`, and `projects` is also draft schema rather than applied migration authority.

Because of that dependency, a Studio/Docs migration now would either:

- create document ownership before the canonical Delivery/project boundary exists;
- introduce weak or nullable references that cannot yet prove workspace isolation;
- force a premature decision about whether Studio artifacts are project-owned, workspace-owned, assessment-owned, or some mixed model;
- create false confidence before RLS policies and isolation tests exist.

The safe path is to keep M5.2d docs-only, then plan Delivery/project ownership in M5.2e before implementing Studio/Docs ownership columns.

## Current Studio/Docs Schema Inventory

`document_generations` exists as a draft SQL contract and runtime adapter target. It has organization scope through `org_id`, project linkage through `project_id`, template identity through `template_id`, generation time through `generated_at`, and generated content through `artifacts`. It does not have workspace scope or durable actor/governance/source columns.

Generated artifacts currently live as runtime structures attached to document generation records and app state. They may carry Assess-to-Studio source context, evidence references, assumptions, approvals, and generated work items, but those fields are embedded artifact payloads, not standalone persistence authority.

The Document Vault is currently a product surface and browser/export workflow, not an applied database authority. Document versioning, review state, repository membership, retention, and export records are not implemented as durable tenant tables.

Docs templates are app/runtime data and browser persistence today. Draft AI generation job tables exist in schema docs, but they are not a Docs template authority and should not be promoted in M5.2d.

Docs-to-Delivery lineage is runtime metadata built from generated artifacts, handoff context, and Delivery Pack construction. It is not yet a standalone lineage table.

Handoff ledger persistence remains draft and org-scoped. It needs a later workspace-aware lineage or handoff slice before it can anchor RLS.

Export and storage authority remains deferred to the export-controls milestone because browser exports are not durable server-mediated export records.

## Proposed Ownership Model

Studio/Docs should use a standard ownership vocabulary, applied per table only when the table becomes real migration authority:

- Tenant scope: `org_id`, `workspace_id`
- Creation and change actors: `created_by`, `updated_by`
- Business ownership: `owner_id`
- Governance actors: `reviewer_id`, `approver_id`, `approved_by`, `rejected_by`
- Source linkage: `source_assessment_id`, `source_process_id`, `source_project_id`, `source_document_generation_id`
- Lifecycle: `status`, `archived_at`, `deleted_at`
- Audit linkage: `correlation_id`, `audit_correlation_id`

Future `document_generations` promotion should add nullable-first workspace and actor columns, source Assess references, status/lifecycle fields, and correlation fields. The generated artifact payload should not be treated as a secret store, authority record, or cross-tenant reference container.

Future document repository tables, if approved, should separate document metadata, document versions, review events, export records, and storage object references rather than forcing all behavior into `document_generations`.

Docs templates should stay separate from provider prompt templates. Any provider prompt or model execution authority must remain server-side and not browser-secret-bearing.

## FK And Backfill Strategy

Future implementation should add nullable ownership columns first. It should avoid `NOT NULL` ownership requirements until AP approves workspace mapping and backfill.

Where tables are applied migration-chain authority, future foreign keys should reference `organizations(id)`, `workspaces(id)`, and preferably the composite workspace key `(workspace_id, org_id)` to prevent cross-organization workspace references.

Source Assess references should use same-tenant constraints after both sides are real migration authority. A generated document linked to an assessment or process must not be able to point across organization or workspace boundaries.

For broad or potentially pre-existing rows, future migrations should use `NOT VALID` foreign keys and validate them in a later AP-approved milestone after backfill is complete.

M5.2d does not backfill. Future backfill must avoid committed hosted IDs, personal organization IDs, secret-derived values, or destructive updates. Canonical AP demo workspace mapping remains an AP decision.

## Migration Safety Strategy

M5.2d is docs-only. It creates no migration and changes no schema.

Future Studio/Docs migration work should be additive only. It should not drop columns, rewrite existing payloads, force runtime adapter dependency, change buyer-demo behavior, create seed data, create RLS policies, or validate broad constraints before backfill.

Local-only migration validation may be used in a future implementation only when tracked Supabase configuration and safe local tooling are available. Remote Supabase validation is out of scope unless AP separately approves it for a later milestone.

## Buyer-Demo Compatibility Strategy

The current buyer-demo flow remains unchanged. M5.2d does not affect document generation, Decision Pack rendering, Govern copy, Assess-to-Studio handoff, Docs-to-Delivery lineage, export behavior, local storage, mock data, scoring, routing, or UI.

Future ownership columns should stay nullable until runtime adapters are intentionally updated. Studio and Docs should continue to function in local/demo mode while hosted-pilot persistence is built behind explicit milestones.

## RLS Readiness Mapping

Studio/Docs ownership columns will support M5.3a and later policy work by enabling:

- workspace-scoped generated artifact reads for active workspace members;
- cross-organization and cross-workspace denial;
- Buyer Viewer read-only behavior where approved;
- contributor generation rights separate from reviewer and approver rights;
- draft, review, approved, rejected, archived, and deleted lifecycle checks;
- same-tenant source assessment and process enforcement;
- evidence and source-lineage boundaries;
- Docs-to-Delivery handoff boundaries;
- server-mediated export and storage access in the later export-controls milestone.

These are readiness directions only. Tenant isolation is not implemented or proven by this plan.

## AP Decisions Needed

- Decide whether Studio artifacts are primarily workspace-owned, project-owned, assessment-owned, or mixed.
- Decide whether Delivery/project ownership must be implemented before Studio/Docs ownership.
- Approve whether `document_generations` should be promoted as-is or replaced by separate document repository tables.
- Approve nullable-first ownership columns for future Studio/Docs migrations.
- Decide whether generated artifacts need separate metadata/version tables.
- Decide whether templates are organization-scoped, workspace-scoped, system-scoped, or mixed.
- Approve how source Assess records map to Studio/Docs workspace scope.
- Approve whether Docs-to-Delivery lineage becomes a standalone table or remains task metadata until Delivery ownership is complete.
- Decide whether export records and storage object references stay deferred to M5.5.
- Approve the timing for backfill, constraint validation, and future RLS tests.

## Risks

- Implementing Studio/Docs ownership before Delivery/project ownership could create unstable foreign-key and workspace mapping decisions.
- Draft schema can be mistaken for applied migration authority.
- Embedded generated artifact payloads can create false confidence if treated as durable evidence or lineage authority.
- Runtime browser persistence remains demo-oriented and is not a hosted-pilot persistence boundary.
- RLS readiness columns alone do not enforce tenant isolation.
- Broad FKs and backfills can fail without approved workspace mapping.
- Provider prompt/job concepts can be confused with Docs template authority.
- Export and storage records need later server-mediated controls and should not be folded into this slice.

## Recommended Next Milestone

M5.2e Delivery Ownership Columns Migration Plan.

Delivery/project workspace authority should be planned before Studio/Docs ownership is implemented, because current Studio/Docs draft persistence points at project authority and future RLS needs a stable workspace boundary.
