# M4.1d-2 App.tsx Central Guard Integration

## Purpose

M4.1d-2 activates the M4.1d-1 central View/RBAC guard in the app shell. The milestone routes App-level persisted and direct view transitions through `resolveViewAccess` so stale, disabled, invalid-scope, and missing-permission view states resolve through one deterministic guard contract.

This is the App integration step only. Sidebar and ModuleJourney metadata alignment remain future work.

## Root Cause

Before M4.1d-2, `App.tsx` still applied view transitions through local `setCurrentView` calls and module-only checks. This left several paths outside the central guard helper:

- persisted `currentView` / `currentScope` restore
- default user scope/view application
- `handleScopeChange`
- `handleViewChange`
- `handlePrimaryAction`
- project selector / Docs Forge entry
- Assess-to-Studio handoff navigation
- Docs-to-Delivery import backlog navigation
- Process Detail and Guided Assessment callbacks
- Docs workspace callbacks
- local fallback effects for `TEAMS`, `REPORTS`, and disabled modules

## App Integration Summary

`App.tsx` now uses local App-shell helpers:

- `resolveAppViewAccess(view, scope)` wraps `resolveViewAccess` with the current user, auth/org loading state, organization, enabled modules, requested view, and requested scope.
- `applyGuardedView(view, requestedScope)` applies the requested view only when allowed, otherwise applies the resolver fallback view and fallback scope.
- `setScopeIfChanged(scope)` prevents redundant scope writes and helps avoid fallback loops.

Guarding waits while auth or organization state is loading. The persisted/default view effect and stale-state fallback effect do not redirect during loading.

## Guarded Entry Points

M4.1d-2 guards these App-level transitions:

- persisted/default user scope and view application
- Sidebar and ModuleJourney `onViewChange` path through `handleViewChange`
- Header and scope switcher path through `handleScopeChange`
- `handlePrimaryAction`
- project selector / Docs Forge project entry
- Assess-to-Studio handoff into Docs Forge
- dashboard stat click into Work List
- Docs-to-Delivery import transition into Backlog
- generated-document epic creation transition into Backlog
- Process Detail back/start callbacks
- Guided Assessment exit callback
- Process Catalog detail callback
- Docs Forge cancel and complete callbacks
- Workspace done callback
- Docs Vault generation open callback
- Project view generation open callback
- stale persisted view/scope fallback effect for deferred, disabled, invalid-scope, and missing-permission states

## Fallback Behavior

The App shell now applies the fallback returned by `resolveViewAccess`:

- `auth_loading`: wait and avoid view/scope mutation.
- `deferred_view`: fall back according to the helper output, preserving Reports and Teams as deferred/decision areas.
- `disabled_module`: fall back to an accessible enabled module path where available.
- `invalid_scope`: apply the helper fallback scope and view.
- `missing_permission`: apply the helper fallback view and scope.
- `stale_persisted_view`: correct persisted state after auth/org loading resolves.

The existing organization-scope Admin workspace path is preserved because Workspace/Admin route semantics remain an AP decision area.
