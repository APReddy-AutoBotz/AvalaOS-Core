# M3.2b Provider Config Schema/RLS Proposal

Status: Planning/proposal only

Branch: `milestone/m3.2b-provider-config-schema-rls-proposal`

## Goal

Create a concrete future schema and RLS migration proposal for server-side provider config, provider key references, workspace/tenant provider policy, and AI audit/evidence events.

This milestone does not implement schema/RLS, create or edit migrations, change Supabase Functions, change runtime behavior, change product code, change tests, change package files, change Admin UI, touch Health, change scoring, add runtime agents, add MCP/A2A behavior, add external execution, add AI features, remove local-demo/internal-dev fallback, or make unsupported compliance claims.

Candidate schema discussion is not approval to migrate, alter, depend on, or implement these structures in this milestone.

## Source-Of-Truth Alignment

This proposal follows `docs/00_SOURCE_OF_TRUTH.md`:

- Evaluate before you automate. Govern before you execute.
- Deterministic scoring remains deterministic.
- AI must not decide final scores, gates, risk tiers, recommendations, approvals, or compliance status.
- Browser-stored provider keys and browser-side AI are not acceptable for pilot or production behavior.
- Server-side AI through Supabase Edge Functions remains the pilot and production direction.
- Evidence proves every governed decision.

## Current Context

M3.1a established the runtime mode contract and fail-closed orchestrator. M3.1b removed the active browser `userApiKey` flow and retired `setup-api-key.js`. M3.1c added static AI boundary scan automation. M3.2 planned server-side provider config/key-reference architecture. M3.2a proposed the provider config/key-reference contract.

`supabase/functions/_shared/ai.ts` currently reads direct server environment keys through `Deno.env.get('GROQ_API_KEY')` and `Deno.env.get('GEMINI_API_KEY')`. M3.2b does not change that behavior.

## RLS Proposal

This proposal is plan-only.

- Browser/client access: browser code must never receive raw provider secrets, resolved key values, secret manager payloads, service-role-only resolver fields, or raw key-reference internals. Browser access should be limited to safe provider/policy metadata needed for human-readable readiness views after AP approval.
- Authenticated user access: active tenant/workspace users may read only safe metadata for provider configs, policies, prompts, jobs, and usage when membership permits it. Authenticated users should not create provider configs, key refs, provider policies, provider audit events, or usage records for runtime execution.
- Admin/control-plane access: approved admin/control actors may write provider config metadata, key-reference metadata, policies, and prompt/version metadata through server-side paths only. Direct browser writes to sensitive schema should not be enabled.
- Service-role/Edge access: Edge/service-role may read provider config, key-reference metadata, policy, prompt/version, job, usage, and audit objects only as part of the approved resolver workflow. Service-role must still follow explicit resolver checks before provider calls.
- Tenant/workspace isolation: every candidate object must have clear org ownership and workspace ownership where needed. Missing ownership, ambiguous ownership, or cross-tenant lookup must fail closed.
- Metadata visibility: safe metadata may include provider type, display name, status, model policy summary, key-ref label/status, operation policy, readiness state, and sanitized error class. Safe metadata must not include secret values.
- Raw-secret non-access: raw provider secrets must never be stored in browser env, browser state, logs, evidence, product copy, ordinary application DB rows, or responses. Raw provider secrets must not be stored even encrypted in ordinary application DB rows unless AP separately approves a dedicated secret-storage model.
- Policy enforcement sequence: authenticate user, resolve tenant/workspace membership, resolve operation/mode, resolve provider config, check provider status, check workspace policy, resolve key reference through service-side access, bind prompt/version, prepare audit/evidence, then call provider only if all checks pass.
- Fail-closed cases: missing membership, missing config, disabled config, ambiguous default, missing policy, denied policy, missing key reference, disabled key reference, invalid key reference, missing prompt/version, unavailable audit path, unclear RLS ownership, or scanner classification failure.

## Secret And Key-Reference Proposal

This proposal is plan-only.

- Raw provider secrets never live in browser env, browser state, logs, evidence, product copy, ordinary application DB rows, or responses.
- Raw provider secrets must not be stored even encrypted in ordinary application DB rows unless AP separately approves a dedicated secret-storage model.
- Only server-side key references may be stored in ordinary application DB rows.
- Key references point to server-side secret material or secret manager identifiers.
- Key references are never returned to browser/client code in a form that could resolve or expose secret material.
- Edge resolver uses service-side access only after auth, tenant/workspace, provider config, provider policy, provider status, operation/mode, prompt/version, and evidence checks pass.
- Logs and evidence may store key-reference ID, safe label, status, or safe fingerprint only if AP approves the field as non-secret.
- Missing, disabled, unauthorized, malformed, unsupported, expired, or invalid key references fail closed before provider calls.

## Migration Sequencing Proposal

Future migration work must be AP-gated and should follow this order:

1. Validate existing objects: confirm whether `ai_provider_configs`, `prompt_templates`, `ai_generation_jobs`, and `ai_usage_events` can be safely reused or altered.
2. Add/alter provider config metadata if approved: add only safe metadata needed for allowed modes, operations, workspace scope, status, model policy, and key-reference linkage.
3. Add key-reference records if approved: introduce `ai_provider_key_refs` only as reference metadata to server-side secret material or secret manager identifiers.
4. Add workspace policy if approved: introduce `ai_workspace_provider_policies` if provider config cannot safely express workspace/mode/operation policy.
5. Add provider audit events if approved: introduce `ai_provider_audit_events` if usage/jobs are insufficient for config, key-reference, policy, and resolver evidence.
6. Update scanner evidence after runtime migration: update M3.1c classifications only after approved runtime files change and evidence proves the new server-side resolver boundaries.

Rollback considerations:

- Rollback must preserve fail-closed behavior when config, policy, or key-reference rows are absent.
- Removing a policy or key-reference migration must not re-enable direct browser provider execution.
- Transitional direct server env references in `supabase/functions/_shared/ai.ts` remain unchanged until a separately approved runtime slice replaces them.

Seed/demo/local-dev considerations:

- Any seed data must use safe metadata and key references only, never raw provider secrets.
- Local-dev examples should use clearly labeled non-secret references such as server env names or secret-manager placeholders.
- Local-demo/internal-dev browser fallback must remain governed by existing mode policy and must not be widened by this proposal.

## M3.1c Scanner Interaction

This proposal is plan-only.

Future scanner updates should include:

- Exact allowlist classifications for approved server-side resolver/key-reference paths, with path, pattern, reason, and future action.
- Reclassification or removal of direct `GEMINI_API_KEY` and `GROQ_API_KEY` hits after approved runtime migration moves provider secret lookup behind a resolver.
- Continued prohibition of browser `VITE_GEMINI_API_KEY`, `VITE_GROQ_API_KEY`, and other browser-exposed provider secrets outside approved `local-demo` / `internal-dev` fallback.
- Evidence for every allowlist update showing why the path is server-side, why the field is reference-only, and how browser access remains blocked.
- Stale allowlist entries should fail the scanner and be removed or updated after migration changes land.

Future implementation must stop if M3.1c cannot classify new hits safely.

## Stop Conditions

Future implementation must stop if:

- raw secrets would be stored in DB rows
- raw secrets would be returned to browser
- raw secrets would be logged
- raw secrets would be stored encrypted in ordinary application DB rows without separate AP approval for a dedicated secret-storage model
- RLS cannot enforce tenant/workspace isolation
- service-role boundary is unclear
- Edge resolver contract is unclear
- Admin UI is needed before backend contract stabilizes
- M3.1c scanner cannot classify new hits safely
- package/dependency changes are needed without AP approval
- Health, scoring, runtime-agent, MCP/A2A, or external-execution changes appear
- compliance claims are introduced
