# M3.2m Resolver Edge Function Integration Plan

## Milestone Purpose

M3.2m is a planning-only milestone for future integration of the M3.2l provider resolver decision logic into Supabase Edge Functions. It defines the required runtime sequence, stop conditions, evidence requirements, and AP approval gates for a later implementation milestone.

M3.2m does not implement runtime integration and does not authorize M3.2n implementation by itself. M3.2n may begin only after AP/ChatGPT reviews and approves the M3.2m PR.

## Dependency Chain Through M3.2l

1. M3.2h created the provider configuration governance migration for provider key references, provider configs, provider policies, and provider audit events.
2. M3.2i verified provider config RLS evidence and resolver readiness.
3. M3.2j planned the provider resolver contract.
4. M3.2k planned the resolver implementation.
5. M3.2l implemented resolver decision logic only and was post-merge verified and tagged as `avalaos-core-m3.2l-provider-resolver-decision-logic`.
6. M3.2m plans Edge Function integration and leaves all runtime changes for a later AP-approved implementation milestone.

## Current Resolver Decision-Logic Summary

M3.2l added server-side shared modules that:

- resolve allow/block decisions for `gemini` and `groq`
- support `pilot` and `production` resolver modes
- support `generate_document`, `refine_section`, and `test_provider_connection`
- enforce `org_id` as the tenant boundary
- keep `workspace_id` as optional evidence metadata only
- fail closed on missing auth, missing org, inactive membership, missing roles, unsupported provider/operation/mode, missing or ambiguous policy, ineligible provider config, ineligible key reference, unsafe audit metadata, or missing scanner classification
- create sanitized resolver audit event shells without DB writes
- validate future secret lookup eligibility without reading or returning secret material

M3.2l did not edit Edge Function entrypoints, `_shared/ai.ts`, real secret lookup, provider calls, or audit persistence.

## Target Edge Function Integration Sequence

Future integration must run in this order:

1. Handle CORS preflight.
2. Authenticate the caller with the existing server-side auth helper.
3. Parse request body.
4. Resolve trusted `org_id` with existing membership-aware org resolution.
5. Resolve server-owned AI runtime mode as `pilot` or `production`; client-provided mode may not weaken server controls.
6. Normalize requested provider and optional provider config selector.
7. Call `resolveProviderForOperation` with operation, provider, org, actor, correlation, evidence, and scanner-classification context.
8. Build the sanitized resolver audit event shell from the decision.
9. Persist resolver audit evidence only after AP approves DB audit insert implementation.
10. If blocked, return a sanitized fail-closed response before any secret lookup or provider call.
11. If allowed, evaluate future secret lookup eligibility.
12. Resolve real secret material only after AP approves secret lookup implementation.
13. Hand off to provider calls only after resolver allow, audit persistence, secret lookup, and scanner classification all pass.
14. Continue existing AI job and usage logging only after the approved provider call path succeeds.

Any missing, blocked, unsafe, unclassified, or unavailable prerequisite must stop before provider-call handoff.

## Per-Function Integration Plan

`ai-generate-document` future plan:

- Map to resolver operation `generate_document`.
- Resolve provider decision before `createAiJob` and before `generateDocumentWithProvider`.
- On blocked decision, return sanitized error fields and do not create a generation job.
- On allowed decision, future implementation may create the AI job, perform approved secret lookup, call the provider, record usage, complete the job, and return the current generated-document response shape.

`ai-refine-section` future plan:

- Map to resolver operation `refine_section`.
- Resolve provider decision before `createAiJob` and before `refineSectionWithProvider`.
- Preserve current validation for required refinement instructions.
- On blocked decision, return sanitized error fields and do not create a generation job.
- On allowed decision, future implementation may create the AI job, perform approved secret lookup, call the provider, record usage, complete the job, and return the current refined-section response shape.

`ai-provider-test-connection` future plan:

- Map to resolver operation `test_provider_connection`.
- Resolve provider decision before any provider test call.
- On blocked decision, return the existing connection-test style response with `connected: false` and a sanitized message.
- On allowed decision, future implementation may perform approved audit persistence, secret lookup, and a minimal provider validation call.
- The test path must not bypass resolver policy, key-reference, audit, or scanner gates.

## `_shared/ai.ts` Integration Plan

Future implementation should refactor `_shared/ai.ts` only if AP approves runtime integration. The preferred direction is:

- keep provider support validation for `gemini` and `groq`
- separate prompt/payload construction from secret lookup
- change high-level provider helpers so they receive approved server-only key material from a resolver-gated caller
- remove direct high-level key lookup from generation/refinement handoff after replacement is proven
- keep model environment reads classified separately from provider secret lookup

M3.2m does not edit `_shared/ai.ts`.

## Secret Lookup Plan And Stop Conditions

M3.2m only plans secret lookup. Future implementation may resolve real secret material only in a later AP-approved milestone.

Future secret lookup must stop if:

- resolver decision is not `allowed`
- decision is not future lookup eligible
- key reference is missing, inactive, expired, deleted, wrong provider, wrong org, or unsafe
- key reference is `manual_placeholder`
- key reference is `external_secret_reference` unless AP later approves external secret handling
- lookup would expose `secret_ref`, raw keys, auth headers, service-role keys, or provider payloads to the response, logs, audit metadata, or browser
- non-production validation has not proven the lookup path unless AP explicitly accepts that risk

Secret material must remain ephemeral inside the Edge Function call path and must never be returned by resolver decisions or evidence objects.

## Audit DB Insert Plan And Stop Conditions

M3.2m only plans audit persistence. Future implementation may write resolver audit rows only in a later AP-approved milestone.

Future audit persistence must:

- insert sanitized resolver decision events into `ai_provider_audit_events`
- use the M3.2l audit shell as the source of event fields
- preserve `org_id`, optional `workspace_id`, provider, provider config, key reference ID, operation, mode, policy result, status, failure class, actor, correlation ID, evidence ref, and safe metadata
- reject raw prompts, completions, provider payloads, raw provider errors, keys, auth headers, secret refs, service-role keys, and stack traces in metadata
- fail closed before provider-call handoff if audit persistence is required and unavailable

Future implementation must stop if audit metadata is unsafe, insert fails, target table is unavailable, RLS/service-role access is not verified, or scanner classification is missing.

## Provider-Call Handoff Rules

Provider-call handoff is allowed only after:

- authenticated actor and trusted org are resolved
- resolver decision is `allowed`
- sanitized audit shell is created
- audit DB persistence is implemented and succeeds, if AP requires it for the implementation milestone
- secret lookup is implemented and succeeds
- scanner classification covers the runtime path
- no raw key, secret ref, raw prompt, raw completion, provider payload, or service-role key is exposed

No provider fallback swap is allowed unless a later AP-approved policy explicitly authorizes it.

## Safe Failure Response Contract

Blocked resolver decisions should return only sanitized fields:

- `error`: stable safe message
- `correlationId`
- `failureClass`, if safe for API clients
- `safeUiMessageCategory`
- `retryCategory`

Responses must not include policy rows, provider config rows, key reference rows, `secret_ref`, raw secret material, provider payloads, raw prompts, raw completions, raw provider errors, stack traces, or service-role details.

`test_provider_connection` may keep its existing `connected: false` response shape, but the message must remain sanitized.

## Test Plan For Future Implementation

Future M3.2n implementation tests should cover:

- each Edge entrypoint blocks before provider handoff when resolver blocks
- allowed generate/refine/test flows call resolver before job creation and provider calls
- blocked generate/refine flows do not create jobs or usage rows
- blocked test-connection flow returns sanitized `connected: false`
- secret lookup is never attempted for blocked decisions
- provider calls are never attempted when audit persistence fails and audit is required
- safe response shape contains correlation and category fields only
- `_shared/ai.ts` helpers do not perform real key lookup in high-level generation/refinement handoff after replacement
- static scan catches new runtime provider paths if scanner allowlist is not updated with AP-reviewed classifications

## Scanner Classification Plan

Future implementation must update static scanner classifications only for AP-approved server-side Edge paths. Classifications should distinguish:

- server-side resolver decision calls
- server-only secret lookup
- server-only provider network calls
- sanitized audit persistence
- pre-existing demo/dev browser fallback

No browser-side provider secret or browser provider execution path may be newly classified as allowed for pilot or production behavior.

## DATABASE_URL And Non-Production Validation Plan

Non-production database validation remains required for future runtime integration. Validation should use a safe non-production `DATABASE_URL` and confirm:

- provider governance migration is applied in the target schema order
- provider policy, config, key-ref, and audit tables are present
- resolver DB query dependencies can read expected safe metadata through service-side access
- audit insert can persist sanitized events
- no production or shared customer database is touched

If non-production `DATABASE_URL` validation remains unavailable, M3.2n implementation must not proceed unless AP explicitly accepts that risk.
