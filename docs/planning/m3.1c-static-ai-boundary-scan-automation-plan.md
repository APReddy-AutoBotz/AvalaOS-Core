# M3.1c Static AI Boundary Scan Automation Plan

Status: Planning only

Branch: `milestone/m3.1c-static-ai-boundary-scan-planning`

## Goal

Plan the future M3.1c implementation for automated static scans that protect the browser AI and provider-key boundary established by M3.1a and M3.1b.

The future automation should fail closed when forbidden runtime hits appear, record every allowed hit as evidence, and avoid false positives from planning, evidence, and control documents. This plan does not implement scan automation.

## Scope

- Define forbidden runtime categories for browser AI/key boundary scans.
- Define allowed categories for documentation/control/evidence references, cleanup-only paths, explicit `local-demo` / `internal-dev` fallback paths, and server-side Supabase/Edge environment references.
- Plan scan coverage for required AI-boundary patterns.
- Plan a candidate scan command or script approach.
- Plan an allowlist structure that records path, pattern, allowed reason, and future action.
- Plan evidence output for AP review and future milestone verification.
- Plan local and CI usage for a future script-only scan command.
- Define acceptance criteria and stop conditions for future M3.1c implementation.
- Define what remains for M3.2 provider configuration and key-reference hardening.

## Out Of Scope

- Implementing scan automation.
- Changing product code, AI runtime behavior, provider behavior, scoring, Health, schema/RLS, Supabase Functions, Admin UI, runtime agents, MCP/A2A behavior, external execution, or compliance posture.
- Changing `package.json`, `package-lock.json`, dependencies, or lockfiles during this planning milestone.
- Implementing Edge provider configuration or key-reference resolution. That remains M3.2.
- Removing browser provider classes or demo/dev fallback behavior.

## M3.1a / M3.1b Boundary Assumptions

- M3.1a introduced the client/UI/orchestrator mode contract for exactly `local-demo`, `internal-dev`, `pilot`, and `production`.
- M3.1a makes `pilot` and `production` fail closed before browser fallback when Edge AI is unavailable.
- Client-side mode gating is a product/UI/orchestrator guard only. Server-side Edge enforcement remains mandatory for pilot and production.
- M3.1b removed active runtime `userApiKey` generation/refinement paths and UI prop flow.
- M3.1b deleted `setup-api-key.js`.
- M3.1b removed browser-facing `vite.config.ts` provider-secret injection for `process.env.API_KEY` and `process.env.GEMINI_API_KEY`.
- M3.1b kept `StorageKeys.API_KEY` only for deprecated cleanup of old stored API-key data.
- Browser provider factories/classes remain only for explicitly bounded `local-demo` / `internal-dev` fallback until later approved cleanup.
- M3.2 owns pilot/production server-side provider configuration and key-reference resolution.

## Required Scan Patterns

Future automation must scan for these patterns:

- `userApiKey`
- `StorageKeys.API_KEY`
- `localStorage`
- `setup-api-key`
- `autobotz_api_key`
- `VITE_GEMINI_API_KEY`
- `VITE_GROQ_API_KEY`
- `GEMINI_API_KEY`
- `GROQ_API_KEY`
- `getAiProvider`
- `getAiProviderApiKey`
- `new GeminiProvider`
- `new GroqProvider`
- `process.env.API_KEY`
- `process.env.GEMINI_API_KEY`

## 1. Forbidden Runtime Categories

Future automation should fail closed when any unapproved runtime hit appears in these categories:

- Raw browser provider key props, state, parameters, or prop-chain flows such as `userApiKey`.
- Browser localStorage provider-key setup, writes, reads, helpers, or legacy helper aliases such as `setup-api-key` or `autobotz_api_key`.
- Browser-facing Vite provider secret exposure or replacement secret paths, including `process.env.API_KEY`, `process.env.GEMINI_API_KEY`, `VITE_GEMINI_API_KEY`, or `VITE_GROQ_API_KEY` in pilot/production runtime paths.
- Direct browser provider factory or constructor use outside the explicit M3.1a demo/dev fallback boundary.
- `getAiProvider`, `getAiProviderApiKey`, `new GeminiProvider`, or `new GroqProvider` usage outside approved fallback files and tests.
- Provider secrets in browser state, logs, docs-generation runtime surfaces, exported rows, database rows, or UI copy.
- Any pattern that bypasses M3.1a mode policy or makes pilot/production depend on browser keys or browser provider execution.

## 2. Allowed Categories

Allowed hits must be classified with one of these reasons and recorded in scan evidence:

- `deprecated cleanup-only path`: for legacy cleanup such as `App.tsx` removing `StorageKeys.API_KEY`.
- `non-secret app localStorage state`: for app state that is not provider-key storage, such as dashboard layout or demo current-user state.
- `explicit local-demo/internal-dev fallback`: for browser provider fallback that is reachable only after M3.1a policy allows demo/dev fallback.
- `disabled surface`: for code paths that are present but intentionally disabled and cannot execute provider calls.
- `server-side Edge/Supabase environment reference`: for server-side environment reads inside Supabase Edge code.
- `test/control documentation`: for tests, planning docs, evidence docs, control docs, and historical migration references.
- `deferred M3.1c cleanup`: for known transitional items that M3.1c is explicitly tracking and that are not active pilot/production runtime paths.

## 3. Candidate Scan Command Or Script Approach

The future implementation should prefer a deterministic Node script over ad hoc shell-only scans. A candidate path is:

- `scripts/aiBoundaryScan.mjs`

The script should:

1. Use a fixed pattern list matching the required scan patterns above.
2. Walk repository files with explicit skip rules for generated or dependency folders such as `node_modules`, `dist`, `.git`, and build outputs.
3. Match exact strings or safe regular expressions.
4. Classify every hit through a structured allowlist.
5. Fail with a nonzero exit code when any forbidden or unclassified runtime hit appears.
6. Emit stable Markdown and/or JSON evidence.

A temporary manual command can remain the reference input while planning:

```powershell
rg -n "userApiKey|StorageKeys\.API_KEY|localStorage|setup-api-key|autobotz_api_key|VITE_GEMINI_API_KEY|VITE_GROQ_API_KEY|GEMINI_API_KEY|GROQ_API_KEY|getAiProvider|getAiProviderApiKey|new GeminiProvider|new GroqProvider|process\.env\.API_KEY|process\.env\.GEMINI_API_KEY"
```

## 4. Candidate Allowlist Structure

The future allowlist should be structured, reviewed, and intentionally narrow. Candidate fields:

```ts
type AiBoundaryAllowlistEntry = {
  pattern: string;
  path: string;
  lines?: number[];
  reason:
    | 'deprecated cleanup-only path'
    | 'non-secret app localStorage state'
    | 'explicit local-demo/internal-dev fallback'
    | 'disabled surface'
    | 'server-side Edge/Supabase environment reference'
    | 'test/control documentation'
    | 'deferred M3.1c cleanup';
  modeLimit?: 'local-demo/internal-dev' | 'server-side-only' | 'not-runtime';
  futureAction: string;
};
```

Guidance:

- Runtime allowlist entries should use exact file paths and, where stable enough, exact lines or nearby anchors.
- Broad globs may be allowed for `docs/planning/**`, `docs/quality/**`, and control docs, but the evidence output should still summarize those hits as documentation/control references.
- Runtime files should not be allowed by broad directory globs.
- Any new allowlist entry should require a future action, even when the action is "keep classified as non-secret state."
- Stale allowlist entries should fail if the referenced pattern is no longer present, unless marked as intentionally optional.

## 5. Evidence Output Format

Future automation should generate evidence that can be copied into milestone evidence docs or stored as a generated artifact. Candidate fields:

| pattern | file path | line | excerpt | category | allowed reason | future action |
| --- | --- | ---: | --- | --- | --- | --- |

The evidence should also include:

- Scan command or script version.
- Git commit SHA and branch.
- Pass/fail summary.
- Count of total hits, allowed hits, forbidden hits, unclassified hits, and stale allowlist entries.
- Explicit statement that forbidden hits fail closed.
- Explicit statement that docs/evidence/control hits are not runtime permissions.

## 6. CI / Local Usage Recommendation

Future implementation may request a script-only `package.json` change after AP approval. Candidate commands:

- `npm run test:ai-boundary-scan`
- or fold scan automation into the existing `npm run test:ai-boundary`

Recommendation:

- Run locally before opening AI-boundary PRs.
- Run in CI for every PR once the automation is stable.
- Treat unclassified runtime hits as blocking.
- Treat missing or stale allowlist entries as blocking.
- Do not add dependencies or modify `package-lock.json` unless AP separately approves.

## 7. Stop Conditions For Future Implementation

Stop future M3.1c implementation if any of these are required:

- Product behavior changes beyond scan automation.
- AI generation/refinement behavior changes.
- Scoring changes.
- Health changes.
- Schema/RLS changes.
- Supabase Function changes.
- Provider config or key-reference resolution.
- M3.2 behavior.
- Dependency or `package-lock.json` changes.
- Admin UI changes.
- Runtime agents, MCP/A2A behavior, or external execution.
- Compliance claims.
- Reintroducing browser raw keys or browser-facing provider-secret injection.
- Removing browser provider fallback in a way that breaks approved `local-demo` / `internal-dev` behavior without AP approval.

## 8. Acceptance Criteria For Future M3.1c Implementation

Future M3.1c implementation should be accepted only when:

- All required scan patterns are covered.
- Automation fails closed on forbidden runtime hits.
- Every remaining hit is classified with file path, line or anchor, allowed reason, and future action.
- Documentation/control/evidence references are separated from runtime hits.
- Deprecated cleanup-only `StorageKeys.API_KEY` handling is distinguished from active provider-key storage.
- Non-secret app `localStorage` state is distinguished from provider-key storage.
- Explicit `local-demo` / `internal-dev` fallback hits are verified as mode-gated by M3.1a.
- Server-side Supabase/Edge provider env references are classified separately from browser-exposed env references.
- No package dependency or lockfile change is introduced.
- No M3.2 provider config/key-reference resolver is introduced.
- Verification includes `npm run typecheck`, `npm run test`, `npm run test:ai-boundary`, `npm run build`, `npm audit --audit-level=moderate`, `git diff --check`, and the controlled wording scan.

## 9. What Remains For M3.2

M3.2 remains responsible for:

- Server-side Edge provider configuration.
- Tenant/workspace provider policy enforcement.
- Key-reference resolution.
- Ensuring provider secrets do not appear in client env, browser state, logs, or database rows.
- Replacing transitional demo/dev provider env references for pilot/production behavior.
- Hardening server-side missing-provider, missing-key-reference, auth, org membership, and usage-audit fail-closed behavior.

M3.1c should only protect the browser AI/key boundary with static evidence. It must not implement provider configuration or key-reference behavior.

## Risks And Open Questions

- `localStorage` is broad and includes non-secret app state; the scanner must classify provider-key risk without blocking legitimate non-secret persistence.
- Documentation and evidence intentionally mention forbidden patterns; the scanner must avoid treating control docs as runtime permissions.
- Existing browser provider fallback is still present for explicit `local-demo` / `internal-dev`; the scanner must verify it remains mode-gated without implementing runtime behavior changes.
- Line-number allowlists can become brittle. Anchored excerpts or path-plus-pattern checks may be safer than strict line matching for some entries.
- A package script may be useful for CI, but any `package.json` change must be script-only and separately approved for implementation.
