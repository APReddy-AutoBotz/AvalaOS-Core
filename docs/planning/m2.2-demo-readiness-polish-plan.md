# M2.2 Demo Readiness Polish Plan

## Goal

Make the current AvalaOS Core golden path demoable with prepared Month-End Close Evidence Pack data:

`Avala Assess -> Avala Govern Lite -> Avala Studio -> Avala Delivery Pack -> Avala Monitor`

M2.2 is a copy, demo-data, and documentation polish milestone. It must not add product capability or change behavior.

## Scope

- Prepared Month-End Close golden-path narrative.
- Presenter notes and demo script polish.
- Static UI copy polish where safe.
- Module Journey wording polish.
- Monitor/Portfolio wording polish for value, risk, blockers, and evidence-backed handoff.
- Studio guidance in demo docs that prepared data should be used unless approved server-side AI, BYOK, and provider controls are configured.
- Browser smoke attempt if local tooling is available.

## Out Of Scope

- Backend hardening.
- Server-side AI implementation.
- BYOK changes.
- Live provider or API changes.
- Schema migrations.
- Runtime agents.
- MCP/A2A behavior.
- Health repo changes.
- Scoring changes.
- Package or dependency changes.
- Compliance claims.
- Full demo automation.

## UI/Copy Changes

- Change Delivery outcome copy from `Executable backlog` to `Evidence-backed handoff`.
- Refine Monitor/Portfolio copy to describe portfolio value, risk, blockers, and evidence-backed handoff.
- Keep LandingPage copy generic and product-level only:
  - Evaluate before you automate.
  - Govern before you execute.
  - Assess -> Govern -> Studio -> Delivery Pack -> Monitor.
- Do not add Month-End Close demo-specific copy or provider-limitation wording to LandingPage.
- Keep Studio generation error copy neutral and settings-oriented.
- Do not change Studio generation behavior.

## Demo Docs Changes

- Make Month-End Close Evidence Pack the prepared demo story.
- Add presenter notes to:
  - Start from deterministic assessment and evidence.
  - Show Govern Lite owner, autonomy, risk, approval, and evidence.
  - Use prepared Studio documents instead of relying on live generation.
  - Open Delivery Pack for lineage, checklists, blockers, and exports.
  - Close in Monitor with evidence-backed portfolio visibility.
- State that live Studio generation is optional only when approved provider controls are configured.

## Test/Verification Plan

Run:

- `npm run typecheck`
- `npm run test`
- `npm run build`
- `npm audit --audit-level=moderate`
- `git diff --check`

Run the AP-approved wording scan for old branding, unsupported compliance claims, and unsafe AI-decision wording.

Allowed hits only: migration/history notes, Health separation notes, active repo identifier, no-go/control docs, and existing evidence notes.

Browser smoke:

- If local browser tooling is available, run visual smoke and record results.
- If local browser tooling is unavailable, record the attempted commands and blocker in final evidence.

## Acceptance Criteria

- Prepared Month-End Close story is explicit in demo docs.
- Presenter can show the golden path without relying on live AI generation.
- Module Journey no longer uses execution-first Delivery wording.
- Monitor/Portfolio references evidence-backed handoff without certification or compliance wording.
- LandingPage remains generic product positioning only.
- No product behavior changed.
- No scoring, Health, schema, package, dependency, package-lock, runtime, provider, generation, persistence, or workflow changes.
- Verification passes and wording scan has only allowed hits.

## Risks

- UI copy could accidentally expose provider dependency as a weakness.
- Demo docs could imply live AI generation is required for the demo.
- Monitor wording could drift into certification language.
- Browser smoke may remain blocked by local tooling availability.

## Stop Conditions

Stop immediately if M2.2 requires:

- Scoring file or scoring behavior changes.
- Health file changes.
- Schema, package, dependency, or package-lock changes.
- Adapter writes or live persistence behavior.
- Backend, provider, API, or BYOK work.
- Runtime agent, MCP/A2A, workflow engine, or Jira replacement behavior.
- Copy implying AI decides approvals, scores, risk gates, compliance status, or production execution.
- Old public branding in user-visible surfaces.
