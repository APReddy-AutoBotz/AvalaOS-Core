## Contract Proposal

Future implementation should define these server-side concepts before runtime changes:

- Provider registry: static or DB-backed server-side registry of supported provider IDs, supported operations, model metadata, request capability flags, safe display labels, and provider status semantics.
- Provider config record: tenant/workspace-scoped metadata that binds a provider type, display name, default model, model policy, status, key-reference ID, allowed modes, allowed operations, and audit metadata.
- Provider key reference: non-secret reference to server-side secret material or a secret manager identifier. It is never the raw provider key and is never returned to browser/client code.
- Workspace/tenant provider policy: server-enforced policy that determines whether a resolved user/org/workspace may use a provider config for a requested operation and mode.
- Provider enabled/disabled status: explicit status such as `active`, `disabled`, `pending_review`, or `retired`. Only `active` provider configs may be eligible for pilot/production calls.
- Allowed modes: pilot and production must use server-side Edge execution only. Local-demo/internal-dev fallback remains unchanged and must not be widened by this contract.
- Default provider behavior: a default provider may be used only when a tenant/workspace has exactly one active policy-approved default for the requested operation and mode. Missing, ambiguous, disabled, or unauthorized defaults must fail closed.
- Fail-closed response model: return a sanitized boundary error and do not call a provider when required provider config, key reference, policy, prompt/version, or evidence fields cannot be resolved.
- Sanitized error classes: classify failures without leaking secrets or raw provider payloads.
- Audit/evidence event fields: record safe provider, policy, prompt/version, status, usage, and error-class metadata.
- Prompt/template version reference: bind every provider call to a governed prompt key/version or future prompt template record before provider execution.
- Usage metadata fields: record provider-returned usage counts, model, operation type, output reference, and optional cost estimate without trusting client-submitted usage.

## Candidate Schema Proposal

Future AP-approved schema/RLS work may consider these candidate structures:

- `ai_provider_configs`: provider config metadata, tenant/workspace scope, provider type, display name, default model, model policy, status, key-reference binding, allowed modes, allowed operations, created/updated metadata.
- `ai_provider_key_refs`: reference-only records that point to server-side secret material or secret manager identifiers, with status, rotation metadata, and never any raw secret value.
- `ai_workspace_provider_policies`: tenant/workspace policy records that bind provider configs to allowed operations, modes, roles, default behavior, and enablement state.
- `ai_provider_audit_events`: append-oriented evidence for provider config changes, key-reference lifecycle events, resolver decisions, provider test results, and fail-closed decisions.
- `prompt_templates` or prompt versions: governed prompt key/version references used by Edge before provider execution.
- `ai_usage_events`: server-side usage records from provider responses with safe token/model/output metadata.

Candidate schema discussion is not approval to migrate, alter, depend on, or implement these structures in this milestone.

Future migration proposal work must answer:

- whether existing `ai_provider_configs`, `prompt_templates`, `ai_generation_jobs`, and `ai_usage_events` are sufficient as-is
- whether separate key-reference and workspace policy tables are required
- which columns and constraints enforce reference-only key storage
- which data is append-only versus mutable metadata
- how rollback preserves fail-closed behavior
- how migration validation and RLS regression will be proven before runtime use

## RLS And Access Model Proposal

Future RLS/access planning should separate metadata visibility from secret resolution:

- Tenant/workspace members may read safe provider config metadata only when their active membership allows it.
- Admin/control roles may write provider config metadata only through approved server-side paths.
- Browser/client code must never read raw secrets, resolved key values, secret manager payloads, or service-role-only resolver fields.
- Edge/service-role paths may resolve key references only after user auth, tenant/workspace membership, provider policy, provider status, allowed mode, and operation checks pass.
- Tenant/workspace policy must be enforced before provider calls and before usage/audit event creation is treated as successful.
- Missing org/workspace membership, unclear ownership, disabled status, missing policy, ambiguous default provider, missing key reference, invalid key reference, missing prompt/version, or missing evidence fields must fail closed.

## Key-Reference Rules

Future key-reference behavior should follow these rules:

- Key references must point only to server-side secret material or approved secret manager identifiers.
- Raw provider secrets must never be stored in browser env, browser state, logs, database rows, evidence rows, product copy, or generated artifacts.
- Raw provider secrets must never be returned to UI or included in API responses.
- Key-reference resolution belongs server-side only.
- Key-reference values must be allowlisted by resolver type and scope before resolution.
- Missing, disabled, unauthorized, expired, malformed, unsupported, or invalid key references must fail closed before provider calls.
- Rotation metadata can be recorded as control evidence, but raw rotated values remain outside database rows and logs.

## Edge Resolver Contract Proposal

Future Edge resolver implementation should receive only safe request inputs:

- authenticated user context from the request token
- explicit or resolved tenant/workspace identifier
- operation type such as `generate_document`, `refine_section`, or `test_provider_connection`
- requested provider type or provider config identifier when allowed
- requested mode or deployment context resolved server-side
- source/input references, not raw secrets

Before any provider call, Edge should:

1. Authenticate the user.
2. Resolve active tenant/workspace membership.
3. Resolve runtime mode/deployment context.
4. Resolve provider config by tenant/workspace, operation, mode, provider type, status, and policy.
5. Resolve the provider key reference through an allowlisted server-side resolver.
6. Bind prompt/template key/version.
7. Create or prepare required audit/evidence records.
8. Call the provider only if all checks pass.
9. Record usage and completion/failure evidence using sanitized fields.

Provider resolution order should be deterministic:

- explicit provider config ID when supplied and policy-allowed
- tenant/workspace default for the requested operation and mode
- fail closed when no active default exists or more than one default matches

Sanitized failure types should include:

- `PROVIDER_CONFIG_MISSING`
- `PROVIDER_CONFIG_DISABLED`
- `PROVIDER_CONFIG_UNAUTHORIZED`
- `PROVIDER_POLICY_MISSING`
- `PROVIDER_POLICY_DENIED`
- `PROVIDER_DEFAULT_AMBIGUOUS`
- `PROVIDER_KEY_REF_MISSING`
- `PROVIDER_KEY_REF_DISABLED`
- `PROVIDER_KEY_REF_INVALID`
- `PROVIDER_KEY_RESOLUTION_FAILED`
- `PROMPT_VERSION_MISSING`
- `AUDIT_EVIDENCE_UNAVAILABLE`
- `PROVIDER_UNAVAILABLE`
- `PROVIDER_RATE_LIMITED`
- `PROVIDER_REQUEST_FAILED`

Fields never logged:

- raw provider secrets
- resolved secret values
- authorization headers
- service-role keys
- browser/client secrets
- raw provider request bodies containing secret material
- raw provider response payloads containing secret material
- raw prompts or source document bodies
- secret manager payloads

Safe provider call metadata may include:

- provider type
- provider config ID/reference
- key-reference ID/reference status only
- tenant/workspace ID
- user ID
- operation type
- prompt/template key/version
- model
- status
- sanitized error class
- token counts from provider response
- output artifact reference
- timestamps

## M3.1c Scanner Interaction

Future scanner updates should be evidence-backed:

- Server-side key references should be classified separately from direct server environment-key reads.
- Direct `GEMINI_API_KEY` and `GROQ_API_KEY` references in `supabase/functions/_shared/ai.ts` should be reduced or reclassified after an approved resolver implementation.
- Browser `VITE_GEMINI_API_KEY` and `VITE_GROQ_API_KEY` references must remain forbidden outside approved local-demo/internal-dev fallback.
- New resolver file paths should not receive broad runtime allowlists; exact entries should include pattern, path, line or anchor, allowed reason, and future action.
- Documentation/evidence/control references should remain classified as control documentation and must not grant runtime permission.
- Any new forbidden hit, stale allowlist entry, or unclear classification must stop implementation.

## Stop Conditions

Future implementation must stop if:

- raw secrets would be exposed to browser/client
- raw secrets would be stored in database rows
- raw secrets would appear in logs or evidence
- tenant/workspace policy cannot be enforced
- missing provider config does not fail closed
- RLS ownership is unclear
- service-role boundary is unclear
- Admin UI is needed before backend contract stabilizes
- package/dependency changes are needed without AP approval
- M3.1c scanner cannot classify new hits safely
- Health, scoring, runtime-agent, MCP/A2A, or external-execution changes appear
- unsupported compliance claims are introduced
