# M4.1d Central View/RBAC Guard Design And Implementation Plan

## Purpose

M4.1d defines a central View/RBAC guard design for AvalaOS Core. It is a planning-only milestone. It does not implement guard code.

The design is framed as a View/RBAC guard rather than only a route guard because the current app shell does not use URL routing as the primary access surface. It uses `View` state, `Scope`, organization module enablement, Sidebar filtering, direct view handlers, fallback effects, and persisted app state.

## Strict Non-Scope

M4.1d does not change product code, UI, routing behavior, service logic, deterministic scoring, schema, RLS, migrations, Supabase Edge Functions, provider runtime, provider governance, Health, dependencies, deployment state, provider calls, or secrets.

M4.1d does not implement M4.1e buyer-demo safe mode, M4.1f Admin/AI Controls visibility, M4.1g export block/label policy, M4.1h route/view cleanup, or M4.1i naming/persistence cleanup.

## Current-State Diagnosis

### View And Scope State

- `App.tsx` persists `currentScope` through `StorageKeys.SCOPE`.
- `App.tsx` persists `currentView` through `StorageKeys.VIEW`.
- `currentUser.defaultScope` and `currentUser.defaultView` can override the restored state on user change.
- `ScopeType` supports `MY_WORK`, `TEAM`, `PROJECT`, and `ORGANIZATION`.
- `View` includes product module views plus deferred or overloaded views such as `REPORTS`, `TEAMS`, and `WORKSPACE`.

### Module Enablement

- `constants/moduleConfig.ts` maps views to product modules through `VIEW_MODULE_MAP`.
- `isModuleEnabled` checks only whether a product module is enabled.
- `isViewEnabled` checks only whether the view's module is enabled.
- `firstEnabledView` returns the home view for the first enabled module.
- Empty module selections are normalized back to defaults by organization setup/provider behavior.

### Sidebar Permission Behavior

- `Sidebar.tsx` owns the current navigation metadata for many user-visible views.
- Sidebar visibility checks module enablement, scope compatibility, and permissions.
- Current Sidebar permission semantics are:
  - Admin role bypasses listed permission checks.
  - Non-admin users need any one listed permission.
  - Views with no listed permissions are visible when module and scope checks pass.
- This current semantic must remain the first implementation default. Do not strengthen to all-permission enforcement unless AP separately approves it.

### App Render And Fallback Behavior

- `App.tsx` has render-time module fallback for non-organization scopes.
- `App.tsx` redirects `TEAMS` to `BOARDS` and `REPORTS` to `DASHBOARD` in a fallback effect.
- Organization scope bypasses the normal view switch and renders `OrganizationSetupView`.
- `WORKSPACE` is used for Docs generated artifact workspace and also participates in Admin/organization navigation semantics.

### Direct And Non-Sidebar Entry Points

The current app has several entry points that can set view state without using Sidebar access checks:

- `handlePrimaryAction` chooses Docs Forge, Process Catalog, Boards, or Dashboard based on enabled modules.
- Project selector / Docs Forge entry sets project scope and opens `DOCS_FORGE`.
- Assess-to-Studio handoff sets `DOCS_FORGE` after recording source context and handoff ledger entry.
- `ModuleJourney` navigates to `MODULE_HOME_VIEW[module.key]`.
- Persisted `currentView` / `currentScope` restore can rehydrate stale or no-longer-permitted states.
- Direct `setCurrentView` handlers move between Process Detail, Guided Assessment, Docs Workspace, Docs Vault, Backlog, Dashboard, and other views.
- Fallback effects in `App.tsx` redirect some states, but not through a single access decision contract.

## Root Cause

- Sidebar visibility is not central View/RBAC authorization.
- Module enablement is not permission enforcement.
- Scope compatibility is enforced in Sidebar for navigation buttons but not centrally for every view transition.
- Persisted view/scope state can create stale or confusing state after user, organization, module, or permission changes.
- Deferred views and overloaded semantics are handled through local exceptions instead of a shared guard model.

## Proposed Central Guard Concept

Future implementation should introduce one pure resolver, for example:

```ts
resolveViewAccess({
  user,
  authLoading,
  organization,
  enabledModules,
  view,
  scope,
})
```

Recommended return shape:

```ts
{
  allowed: boolean;
  fallbackView: View;
  fallbackScope?: Scope;
  reason:
    | 'auth_loading'
    | 'unauthenticated'
    | 'no_organization'
    | 'setup_required'
    | 'deferred_view'
    | 'disabled_module'
    | 'invalid_scope'
    | 'missing_permission'
    | 'stale_persisted_view'
    | 'admin_decision_pending'
    | 'allowed';
  module?: ProductModuleKey;
  requiredPermissions: string[];
  requiredScope: ScopeType[];
  guardSeverity: 'wait' | 'redirect' | 'hide' | 'disable' | 'warn' | 'allow';
}
```

The resolver must be pure and deterministic. It must not call providers, mutate state, fetch remote data, change scores, or decide business risk. It only describes whether a view state is allowed and what safe fallback should be used.

## Guard Precedence

1. Auth loading: wait/show loading; do not redirect.
2. Unauthenticated: deny to the auth flow.
3. No organization or onboarding/setup required: show onboarding/setup path.
4. Deferred or decision views: hide, disable, or fallback according to AP decision.
5. Disabled module: fallback to a safe enabled module home.
6. Invalid scope for view: fallback to a valid scope/view pair.
7. Missing permission: fallback to a safe view the user can access.
8. Allowed: render requested view.

## Initial View Metadata Policy

Future implementation should create one shared metadata source for both guard and navigation.

Minimum metadata per view:

- `view`
- `module`
- `allowedScopes`
- `requiredPermissions`
- `adminBypass`
- `status`: `active`, `deferred`, `hidden`, or `decision_pending`
- `fallbackView`
- optional `fallbackScope`
- `notes`

## Fallback Strategy

| Situation | Future fallback |
| --- | --- |
| Auth loading | Keep loading state; do not change view/scope. |
| Unauthenticated | Auth/login flow. |
| No organization | Onboarding/setup flow. |
| Disabled current module | First accessible enabled module home. |
| No accessible enabled module | Organization/setup safe state with admin guidance. |
| Missing permission | First accessible view in the same module, then first accessible enabled module home. |
| Invalid scope | Same view with a valid default scope when possible, otherwise module home. |
| Stale persisted view/scope | Replace persisted state with guard fallback only after resolver confirms denial. |
| Deferred view | AP-approved hide, disable, or fallback behavior. |

Module home defaults for future implementation:

- Assess: `PROCESS_CATALOG`.
- Docs: `DOCS_FORGE`.
- Delivery: `BOARDS`.
- Monitor: `DASHBOARD`.

## Deferred And Decision Areas

These areas must remain design-only in M4.1d:

- Reports: currently deferred/redirected; AP must decide hide vs deferred placeholder vs future implementation.
- Teams: current team-scope behavior overlaps with Boards; AP must decide route semantics.
- Workspace: currently Docs artifact workspace and organization/Admin-adjacent view state; AP must decide separation.
- Admin: visible for Admin or selected security/provider permissions; AP must decide product depth and route semantics.
- AI Controls: AP must decide hidden, static preview, or scoped implementation in M4.1f.

## Future Implementation Sequence

### M4.1d-1 Central View/RBAC Guard Helper And Tests

- Add shared view metadata.
- Add pure `resolveViewAccess` helper.
- Add focused tests for auth loading, unauthenticated, no organization, disabled module, invalid scope, missing permission, deferred view, and allowed view.
- Preserve current Sidebar permission semantics: Admin bypass OR any one listed permission.

### M4.1d-2 App Integration

- Route all App view transitions through the resolver.
- Apply guard fallback to persisted `currentView` and `currentScope`.
- Replace one-off module-only render fallback with resolver output.
- Keep UI behavior minimal and matching existing visual surfaces.

### M4.1d-3 Sidebar And ModuleJourney Alignment

- Make Sidebar and ModuleJourney consume the same metadata and access result rules.
- Keep disabled vs hidden behavior consistent with guard severity.
- Avoid separate permission logic in navigation.

### M4.1d-4 Persisted View/Scope Cleanup

- Validate restored `currentView` and `currentScope` after auth/org/module state resolves.
- Persist only allowed fallback state after guard denial.
- Avoid redirect loops while auth and org data are loading.

### M4.1d-5 Deferred Route Handling

- Apply AP decisions for Reports, Teams, Workspace/Admin, and AI Controls.
- Keep this separate from the initial guard helper if AP has not decided those product semantics.

## Future Test Requirements

- Auth loading waits and does not redirect.
- Unauthenticated user cannot render protected app views.
- No organization renders onboarding/setup flow.
- Disabled module redirects to safe enabled module home.
- Missing permission redirects or hides according to guard result.
- Invalid scope falls back to a valid scope/view pair.
- Stale persisted view/scope is corrected after auth/org load.
- Reports deferred behavior is enforced.
- Teams semantics are enforced after AP decision.
- Workspace/Admin split is enforced after AP decision.
- AI Controls visibility remains pending until AP decision.
- Sidebar and non-Sidebar entry points produce the same access result.

## Rollback Strategy

Future implementation rollback should revert App integration to the current module-only fallback while preserving the shared guard tests and design evidence as failing or pending evidence. Do not roll back unrelated M4.1a, M4.1b, or M4.1c fixes.

## AP Decisions Needed

- Final guard fallback precedence.
- Whether current Admin bypass OR any one permission remains long term.
- Reports hide/defer behavior.
- Teams route/view semantics.
- Workspace vs Admin separation.
- Admin/AI Controls visibility strategy.
- Whether first implementation may show a guard-denied explanation or should silently fallback.
