## Problem Statement

The existing redacted signal is more specific than a generic unknown startup failure, but it is still not a root-cause proof. The safe information available is that a local startup attempt timed out while the redacted metadata recorded oversized output/line-count buckets and no approved known pattern family except `unknown`.

Raw output must not be inspected, copied, summarized, or included in evidence because it may contain local paths, host/port values, target values, IDs, raw logs, environment details, or other sensitive local-machine information. Future classification must use only redacted signal fields already designed for safe evidence.

Timeout plus oversized output is useful because it gives AP a category-level direction for future remediation planning. It is insufficient for root cause because it does not identify whether the timeout came from CLI behavior, image resolution, image pull, container creation, service startup, health wait, cleanup, local environment state, or another source.

## Proposed Future Category

Future category:

- `local-startup-timeout-with-oversized-output`

This category is a redacted signal classification only. It is not a root-cause claim, not a Docker claim, not a Supabase CLI claim, not a port/config/migration claim, and not a local DB availability claim.

## Confidence Rules

- Current exact signal confidence remains `low`.
- Future implementation must not raise confidence solely because output is oversized.
- Future implementation may use `medium` confidence only if AP later approves a redacted non-unknown phase or pattern-family rule.
- Future implementation must never use `high` confidence for this category without AP approval and executable redacted evidence that still avoids raw output.

## Output Safety Gates

Future implementation must emit the timeout category only when:

- output safety passed
- scratch cleanup was removed
- no-secrets confirmation passed
- no-local-path confirmation passed
- no-raw-log confirmation passed
- emitted output contains only approved categories, counts, buckets, booleans, and confirmations
- no raw output, target value, local value, ID, secret, payload, path, or log is printed

If any gate fails, the classifier must keep `unknown-local-startup-failure` or fail closed with the existing safety failure behavior.

## Scratch Lifecycle

Future implementation may rely only on redacted metadata that survived the M5.3a-2o scratch lifecycle. It must not reopen, inspect, copy, summarize, or commit raw scratch output. If scratch cleanup fails or evidence would require copying local output, the classifier must fail closed.

## Evidence Format For Future Implementation

Future M5.3a-2q evidence should record only:

- command label

- redacted input fields used
- safety gate pass/fail status
- scratch cleanup status
- local DB availability outcome
- schema availability outcome

- tenant isolation outcome

- no-exposure confirmation

Evidence must not include raw output, paths, ports, IDs, target values, host values, DB URLs, `.env` values, provider keys, tokens, row payloads, auth headers, claim values, or local config values.

## Verification Strategy For Future Implementation

Future implementation should verify:

- classifier syntax check
- deterministic category selection from an approved redacted signal fixture or existing safe signal metadata
- safe fallback to `unknown-local-startup-failure` for incomplete or unsafe redacted signals
- no CLI interface expansion beyond AP approval
- no raw output in stdout, evidence, or tracked files
- standard typecheck, test, build, audit, AI boundary static scan, secret hygiene scan, diff checks, Supabase CLI version check, and restricted wording scans

Future verification must not require startup retry, raw output inspection, Docker diagnostics, DB commands, schema bootstrap, RLS harnesses, artifact assertions, remote validation, hosted Supabase, or provider calls unless a later AP milestone explicitly approves them.

## Rollback And Abandon Criteria

Abandon or keep `unknown-local-startup-failure` if:

- the redacted signal cannot be classified without raw output
- any safety gate fails
- scratch cleanup is not `removed`
- implementation would require local inspection
- implementation would require CLI, Docker, DB, Supabase stack, or raw log access beyond AP approval
- evidence would need to capture forbidden values
- confidence would imply root cause
