# M3.2r Non-Production Deployed Edge Smoke Mocked First

## Milestone Purpose

M3.2r is the execution milestone after M3.2q for a controlled non-production deployed Edge smoke. The goal is to prove resolver-gated deployed Edge behavior without live Gemini or Groq provider calls.

M3.2r is mocked/blocked-first:

- Run blocked deployed scenarios before any allowed scenario.
- Run allowed deployed scenarios only when provider handoff can be safely mocked or intercepted without code changes and without external provider network calls.
- Defer allowed deployed smoke when the only available path would require live provider calls, real provider keys, runtime code edits, or a new mock switch.

## Dependency Chain

M3.2r depends on:

- M3.2n Edge Function integration: resolver, audit persistence, secret lookup eligibility, and provider handoff gates are integrated into the target Edge Functions.
- M3.2n-dbval Outcome A: provider-governance schema bootstrap was validated on a fresh AP-approved non-production database.
- M3.2p mocked non-production smoke: resolver, audit, secret-reference, and provider handoff ordering passed with mocked provider behavior and transaction rollback.
- M3.2q deployed Edge smoke plan: AP/ChatGPT-reviewed plan recommended M3.2r as mock-first, with live provider calls and real key use blocked unless AP explicitly approves them.

## Deployed Smoke Scope

Target Edge Functions:

- `ai-generate-document`
- `ai-refine-section`
- `ai-provider-test-connection`

Allowed M3.2r actions:

- Confirm non-production target readiness without printing secrets or raw URLs.
- Invoke deployed Edge Functions only if the target is confirmed non-production and required credentials are available.
- Seed minimal non-production smoke rows only if deployed invocation can proceed safely.
- Clean up only M3.2q/M3.2r smoke rows.
- Use fake provider secret values only if they cannot cause live provider network calls.
- Add a smoke script only if needed and safe.

## Non-Production Target Confirmation Approach

The execution must confirm:

- `DATABASE_URL` is visible only if direct seed or verification is needed, without printing the value.
- `SUPABASE_URL` is visible and parseable, without printing the value.
- `SUPABASE_ANON_KEY` and `SUPABASE_SERVICE_ROLE_KEY` are visible, without printing values.
- `AVALA_AI_RUNTIME_MODE` is exactly `pilot` or `production`.
- Target classification does not contain production, shared, customer, demo-customer, or live markers.

If `SUPABASE_URL`, Supabase keys, or runtime mode are unavailable, M3.2r must stop before deployed invocation and record Outcome C blocked.

## Seed Strategy

If deployed invocation is safe, seed only harmless non-production smoke rows using the M3.2q identifiers and source marker:

- Actor/profile ID: `00000000-0000-4000-8000-0000000032e0`
- Smoke organization ID range: `10000000-0000-4000-8000-0000000032e0` through `10000000-0000-4000-8000-0000000032e4`
- Key-reference ID range: `20000000-0000-4000-8000-0000000032e0` through `20000000-0000-4000-8000-0000000032e4`
- Provider-config ID range: `30000000-0000-4000-8000-0000000032e0` through `30000000-0000-4000-8000-0000000032e4`
- Role ID range: `40000000-0000-4000-8000-0000000032e0` through `40000000-0000-4000-8000-0000000032e4`
- Safe source marker: `m3.2q_deployed_edge_smoke`

Seed rows may include:

- Smoke auth user/profile/membership/role rows if required by the deployed auth path.
- Smoke organizations for blocked and allowed contexts.
- Smoke provider key refs.
- Smoke provider configs.
- Smoke provider policies.

Seed rows must not include customer data, real provider keys, raw provider secrets, raw prompts, raw completions, provider payloads, or full database row dumps in evidence.

## Blocked Scenario Execution Summary

Blocked deployed scenarios must run first when deployed invocation is possible:

- Missing or invalid runtime mode blocks.
- Missing policy blocks.
- Disabled config blocks.
- `manual_placeholder` key reference blocks.
- Wrong provider-prefix `secret_ref` blocks.

Each blocked scenario must verify:

- Deployed function returns a sanitized failure.
- No provider call occurs.
- No raw provider key, raw `secret_ref`, auth header, service-role key, raw database error, or raw PostgREST error appears.
- `correlationId` exists.
- `safeUiMessageCategory`, `retryCategory`, and `failureClass` appear where expected.

If deployed invocation credentials are unavailable, blocked deployed scenarios are not attempted and the milestone records Outcome C blocked at credential readiness.

## Allowed Scenario Policy

Allowed deployed scenarios are:

- `generate_document`
- `refine_section`
- `test_provider_connection`

Allowed scenarios may run only if provider handoff can be safely mocked or intercepted without code changes and without external provider network calls. If safe mock is not possible, allowed deployed smoke must be deferred and AP must decide whether to approve live provider smoke or a controlled mock-switch milestone.

## Audit Verification Strategy

If deployed invocation proceeds, audit verification must:

- Verify audit event counts only.
- Verify sanitized metadata.
- Verify no raw prompts, completions, provider payloads, keys, auth headers, service-role keys, `DATABASE_URL`, raw `secret_ref` values, stack traces, or raw database errors.
- Avoid full database row dumps.

If deployed invocation is blocked before seed/invoke, audit verification records that no M3.2r audit rows were created.

## Cleanup Strategy

Cleanup must:

- Delete only M3.2q/M3.2r smoke rows.
- Verify zero retained smoke rows.
- Avoid deleting unrelated rows.
- Stop and report Outcome C if cleanup fails.

If execution stops before seed, cleanup records that no M3.2r smoke rows were created and no cleanup delete was required.

## Outcome Framework

Outcome A - deployed mocked/blocked smoke passed:

- Non-production target confirmed.
- Seed succeeded.
- Blocked deployed scenarios passed.
- Allowed mocked scenarios passed when a safe mock path exists.
- Audit sanitization passed.
- Cleanup succeeded.
- No live provider calls occurred.

Outcome B - partial/deferred:

- Blocked deployed scenarios passed.
- Allowed deployed smoke was deferred because safe mock was not possible without code changes or live provider calls.
- Cleanup succeeded.
- Next milestone must be an AP decision on live provider smoke or a controlled mock switch.

Outcome C - blocked:

- Target is unsafe or incomplete.
- Credentials are unavailable.
- Seed is unsafe.
- A blocked scenario fails.
- A response leaks sensitive data.
- Cleanup fails.

## Strict Non-Goals

M3.2r does not:

- Use production, shared, customer, demo-customer, or unclear databases.
- Call live Gemini or Groq APIs.
- Use real provider keys.
- Print or store `DATABASE_URL`, Supabase service-role keys, provider keys, auth headers, or Supabase secrets.
- Change schema SQL, RLS, or migrations.
- Change Admin UI or product UI.
- Touch Health.
- Change scoring.
- Change browser fallback behavior.
- Change prompt versions.
- Expand usage or cost behavior.
- Add package dependencies.
- Make unsupported claims.
- Add AI-decision product language.
- Edit Edge Function entrypoints, `_shared/ai.ts`, or runtime resolver/provider integration files.

## Risks Carried Forward

- Required deployed-target credentials may be unavailable or unsafe.
- Safe deployed provider-handoff mocking may not exist without code changes.
- Live provider smoke remains blocked unless AP explicitly approves real provider key use and network calls.
- A controlled mock switch would require a separate AP-approved milestone.
- Future deployed smoke evidence must continue to exclude secrets, raw prompts, raw completions, provider payloads, full database rows, auth headers, service-role values, and raw `secret_ref` values beyond approved naming patterns.

## Next Milestone Recommendation

If M3.2r records Outcome C because deployed-target credentials are unavailable, recommend an AP/ops readiness milestone to provide the approved non-production Supabase invocation credentials and target confirmation without exposing secrets.

If M3.2r records Outcome B because allowed deployed smoke cannot be mocked safely, recommend an AP decision milestone for either:

- Controlled live provider smoke in the approved non-production target.
- A controlled mock-switch design and implementation milestone.
