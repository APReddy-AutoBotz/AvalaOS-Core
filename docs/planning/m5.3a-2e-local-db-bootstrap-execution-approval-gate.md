## Future Execution Approval Checklist

Before a future execution milestone can run any local DB bootstrap or local artifact/RLS assertion evidence, AP must approve every item below in writing for that specific milestone:

- Local DB bootstrap execution attempt: approved or rejected.
- Supabase local stack startup: approved or rejected.
- Local migration/bootstrap command categories: approved exact categories or rejected.
- Local-only target proof approach: approved or rejected.
- Local artifact/RLS assertion execution after schema readiness: approved or rejected.
- Exact command categories allowed.
- Exact command categories still forbidden.
- Evidence output that may be captured.
- Output that must be redacted or prohibited.
- Stop behavior when local DB availability fails.
- Stop behavior when schema readiness fails.
- Stop behavior when output sanitization fails.
- Whether successful local execution may be described as local scoped evidence only.

## Local-Only Target Approval Requirements

Future local target approval must be explicit and narrow:

- Target must be local-only.
- Approved forms may include loopback-only or AP-approved local-only target labels.
- Hosted Supabase targets are forbidden.
- Remote DB targets are forbidden.
- Silent environment fallback is forbidden.
- Ignored secret-file fallback is forbidden.
- Target values must not be printed.
- Host values, port values, database names, and user names must not be printed unless AP separately approves a sanitized non-sensitive category.

If target proof cannot establish a local-only target before any DB action, future execution must stop before opening a connection.

## Schema Bootstrap Approval Requirements

Future schema bootstrap approval must specify:

- Whether schema bootstrap is allowed at all.
- Which local-only migration or bootstrap categories are allowed.
- Whether a Supabase local stack command category is allowed.
- Whether existing scripts may be used, and only in a local-only wrapper if AP approves.
- How successful schema readiness is detected without printing schema payloads, DB URLs, target values, or secrets.
- How failure is classified without dumping row payloads, target values, or claim values.
- Whether cleanup/reset behavior is allowed and how it is constrained to local state.

M5.3a-2e does not provide executable bootstrap steps. It only defines the approval gate.

## Sanitized Evidence Requirements

Future evidence may capture only approved sanitized categories:

- Command labels.
- Approved milestone id.
- Local target proof status without target value.
- Schema readiness status without target value or schema payload dump.
- Scenario counts.
- Executed count.
- Pass count.
- Fail count.
- Blocked count.
- Verification yes/no outcome.
- Sanitized failure classifications.
- Exit-code category.
- Confirmation that no secrets, target values, payloads, `.env` values, or local paths were printed or committed.

Future evidence must not capture:

- DB URLs.
- Host values.
- Port values.
- Database names.
- User names.
- `.env` values.
- Service-role values.
- Provider keys.
- Hosted identifiers.
- Fixture IDs.
- Row payloads.
- Auth headers.
- Claim values.
- JWTs or private tokens.
- Local machine paths.

## No-Go Criteria

Future execution must not proceed if any of these are true:

- AP approval is missing.
- Any required AP approval field is missing.
- Local DB execution approval is `no`.
- Local target proof is missing or rejected.
- A remote or hosted target is supplied.
- A command would open a DB connection before target proof.
- A command would rely on `.env` values or ignored secret files.
- A command would print target values, DB URLs, row payloads, auth headers, claim values, or secrets.
- Schema readiness cannot be proven without unsafe output.
- Output sanitization fails.
- Required cleanup/reset behavior is not approved.
- The proposed command category is not explicitly allowed.
- The proposed command category remains explicitly forbidden.

## Stop Conditions

Future execution must stop immediately and record a sanitized blocked result if:

- Local DB availability cannot be established.
- Schema readiness cannot be established.
- Target proof fails.
- Output sanitization fails.
- Any assertion or harness reports unsafe output.
- Any scenario fails or is blocked.
- Any required migration/schema prerequisite is missing.
- Any unapproved command category is needed to continue.
- Any secret, target value, payload, claim value, or local path would be printed.
