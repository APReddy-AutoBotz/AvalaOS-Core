## Enterprise Completion Principles

- Prefer larger grouped milestones where related work can be reviewed safely together.
- Keep high-risk proof tracks separate when execution evidence or external state is involved.
- Require explicit AP approval before any browser, DB/RLS/artifact, hosted/deployment, provider/classifier, approval-workflow, or real assertion execution.
- Require evidence before readiness claims.
- Preserve proof-boundary wording in Trust Center, Buyer Acceptance, roadmap, readiness, and ledger surfaces.
- Do not use M5.5a to start direct browser, DB, hosted, provider, classifier, or artifact execution.
- Do not change deterministic scoring formulas, weights, thresholds, hard stops, or recommendation logic without explicit score-version change and regression tests.

## Recommended Grouped Sequence After M5.5a

1. `M5.5b Evidence Surface, Approval Model, and Audit Contract Hardening`
2. `M5.5c Secure Export and Artifact Storage Design/Implementation Gate`
3. `M5.6a RLS/Tenant-Isolation Implementation Preparation Gate`
4. `M5.6b Hosted/Deployment/Operations Preparation Gate`
5. `M5.7 First AP-Approved Evidence Execution Gate`

This sequence is a proposed realignment of the prior `M5.5-M5.6` secure export, deployment, runbook, and pilot evidence bucket. It keeps the original intent but separates execution risk and proof obligations more clearly.

## Workstreams That Can Be Combined Safely

- Read-only Admin Workbench, Trust Center, Buyer Acceptance evidence surfaces, approval model documentation, and audit contract hardening can combine in M5.5b if no workflow execution or status changes occur.
- Secure export design, artifact storage design, export claim controls, and audit metadata design can combine in M5.5c if no export/PDF/download artifact is generated.
- Server-side AI static guardrails, provider governance documentation, and environment/secret boundary documentation can combine if no real provider or classifier execution occurs.

## Workstreams That Must Remain Separate

- Real DB/RLS/artifact evidence execution must stay separate from browser walkthrough execution.
- Hosted/deployment validation must stay separate from DB/RLS execution and manual browser execution.
- Real provider/classifier execution must stay separate from deterministic scoring, approval, and readiness decisions.
- Final pilot or release-candidate evidence must stay separate until prerequisite evidence exists.
- Any scoring formula, score weight, threshold, hard stop, or recommendation logic change must stay separate and include explicit score versioning and regression tests.

## Stop Conditions Before Any Future Execution

Stop before execution if any of these conditions occur:

- no separate AP-approved go/no-go gate exists for the exact execution track.
- execution scope, run count, output boundaries, prohibited artifacts, stop conditions, or proof boundaries are unclear.
- a step could expose secrets, raw logs, local paths, host/port/IP values, DB URLs, row payloads, auth headers, claim values, provider keys, private tokens, project refs, target values, container/image IDs, screenshots, browser output, or export/PDF/download artifacts outside the approved evidence contract.
- a task attempts browser, DB/RLS/artifact, hosted/deployment, provider/classifier, schema, migration, bootstrap, approval workflow, or real assertion execution without explicit approval.
- a generated artifact or output directory appears outside the approved scope.
- wording implies production, hosted, deployment, RLS, tenant-isolation, security, buyer, product, release-candidate, compliance, browser, screenshot, export, PDF, download, walkthrough, approval-workflow, or readiness proof before evidence exists.
- implementation scope expands beyond the approved milestone.
