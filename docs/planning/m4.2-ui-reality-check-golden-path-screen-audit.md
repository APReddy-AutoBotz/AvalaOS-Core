# M4.2 UI Reality Check And Golden Path Screen Audit

## Purpose

M4.2 audits the visible AvalaOS Core UI and screen-flow reality after the M4.1d guard work and before AP approves canonical demo data implementation.

The goal is to verify whether the current screens, buttons, labels, empty states, loading states, error states, and navigation paths support the intended enterprise buyer-demo flow without changing UI, product code, mock data, routing, scoring, schema, provider behavior, or Health.

## Audit Method

The audit used source inspection of App-shell navigation, guard metadata, golden-path view rendering, Assess screens, Studio screens, Docs-to-Delivery import, Delivery Pack, Monitor dashboard, and existing evidence docs.

No deployed functions were invoked. No providers were called. No secrets were opened or printed. No UI or code was changed.

## Intended Golden Path

The intended visible buyer-demo path remains:

1. Organization setup and module enablement.
2. Process Catalog.
3. Process Detail.
4. Guided Assessment.
5. Deterministic score and Decision Pack.
6. Govern Lite control summary.
7. Avala Studio / Docs Forge handoff.
8. Generated documents.
9. Docs-to-Delivery work item import.
10. Delivery work items.
11. Delivery Pack.
12. Monitor dashboard.

## Screen-Flow Audit Summary

The core screen flow is present in code. The M4.1a through M4.1d-4 fixes make the contract, lineage, and App-shell guard path substantially safer than the M4.0a baseline.

However, the visible demo path is not yet buyer-demo clean. Several current labels and empty/error states still use older workspace or AI-assistant wording, some buttons describe a different action than the code path actually performs, and the Monitor/AI Insights and Delivery export surfaces can be confusing if shown without a controlled demo script.

The next milestone should be a focused UI fix pass before canonical demo data, because canonical data will make the screens fuller but will not correct misleading labels, stale empty states, or buyer-demo risk copy.

Recommended next milestone: `B) UI fix pass first`.

Recommended order after M4.2:

1. UI fix pass for visible labels, empty states, guarded navigation copy, and buyer-demo safe disabled states.
2. Canonical demo data implementation.
3. Deferred route handling for Reports, Teams, Workspace/Admin separation, and AI Controls visibility.

## Finding Classification Key

- `Critical before demo data`: must be resolved before canonical demo data is useful.
- `Fix before buyer demo`: can be resolved after canonical data, but should not be shown to buyers as-is.
- `Fix before paid pilot`: acceptable for internal demo if controlled, but not for paid pilot readiness.
- `Can wait`: not on the core demo path or low user trust risk.
- `Needs AP product decision`: implementation depends on product direction.

## Missing UI Findings

The main product screens exist, but the following buyer-demo support surfaces are missing or incomplete:

- A single canonical demo starting point that loads a coherent process, score, generated documents, imported delivery work, Delivery Pack, and Monitor evidence trail.
- A buyer-safe explanation for disabled server-side AI insight behavior.
- A clear AP-approved route for Admin versus Workspace.
- A dedicated Admin/AI Controls visibility model.
- Modal-level lineage warning before Docs-to-Delivery import confirmation.
- Export block or incomplete-lineage label policy for Delivery Pack exports.
- Buyer-demo safe placeholder behavior for Reports and Teams if those routes appear in any script.

## Misleading UI Findings

Misleading or stale wording found in visible source:

- `AI Workspace` in the Docs repository empty state.
- `AI Assistant` in the project selector modal.
- `Run Decision Engine` as a primary assessment scoring CTA.
- `Generate BRD / PDD` as a narrower label than the actual Studio handoff set.
- Default `AI Insights` dashboard widget showing a generic load failure while the path is intentionally disabled.
- `Upgrade to Premium` and `Coming Soon` controls in the organization setup/Admin surface.

These are UI/copy issues only in M4.2. No copy was changed in this audit.

## Guard And Navigation Findings

The central guard path is materially improved:

- App-level view/scope transitions resolve through guard metadata.
- Sidebar and ModuleJourney are aligned to shared metadata.
- Persisted stale view/scope state is sanitized and safely resolved.
- Auth and organization loading wait rather than redirecting.
- Assess-to-Studio handoff normalizes Docs Forge target scope and preserves valid source context when Docs Forge can be reached.

Remaining guard/navigation risks:

- Header primary action priority can jump to Studio before Assess in an enabled-docs workspace.
- Admin remains a direct Sidebar action that switches to organization-scope Workspace.
- Reports and Teams remain deferred/decision routes.
- Invalid or unavailable routes are mostly guarded, but fallback messages should be demo-reviewed once AP decides route handling.

## Golden Path Readiness

Current status: `Not ready for buyer demo without a UI fix pass and canonical data`.

The screen architecture is sufficient to support the intended flow, but the visible demo should not proceed directly to canonical data because canonical data will amplify stale labels and incomplete disabled-state explanations.
