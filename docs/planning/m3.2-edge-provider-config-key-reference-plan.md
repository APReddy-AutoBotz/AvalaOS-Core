# M3.2 Edge Provider Config / Key-Reference Planning

Status: Planning only

Branch: `milestone/m3.2-edge-provider-config-key-reference-planning`

## Goal

Plan the future server-side provider configuration and key-reference architecture that moves pilot and production AI execution away from transitional browser demo/dev provider environment usage.

M3.2 planning does not implement provider configuration, key-reference resolution, schema changes, RLS changes, Supabase Function changes, Admin UI, runtime behavior, package changes, dependency changes, Health changes, scoring changes, or tests.

## Source-Of-Truth Alignment

This plan follows `docs/00_SOURCE_OF_TRUTH.md`:

- Evaluate before you automate. Govern before you execute.
- Deterministic scoring remains deterministic.
- AI must not determine final scores, gates, risk tiers, recommendations, approvals, or compliance status.
- Browser-stored provider keys and browser-side AI are not acceptable for pilot or production behavior.
- Server-side AI through Supabase Edge Functions remains the pilot and production direction.
- No unsupported compliance claims are made.

## Problem Statement

Current M3.1a through M3.1c work established the browser AI boundary:

- `local-demo` and `internal-dev` may still use explicit transitional browser fallback.
- `pilot` and `production` must use server-side Edge AI and fail closed when Edge AI is unavailable.
- Active browser `userApiKey` flow was removed.
- `setup-api-key.js` was retired.
- Browser-facing provider secret injection in `vite.config.ts` was removed.
- Static AI boundary scan automation now classifies remaining AI/key boundary hits.

The remaining M3.2 problem is server-side:

- `supabase/functions/_shared/ai.ts` currently reads direct server environment keys through `Deno.env.get('GROQ_API_KEY')` and `Deno.env.get('GEMINI_API_KEY')`.
- Future pilot/production behavior needs server-side provider config, tenant/workspace provider policy, and allowlisted key-reference resolution.
- Raw provider secrets must not live in browser env, browser state, logs, database rows, or product copy.
- Missing, disabled, unsupported, or policy-disallowed provider config must fail closed before provider calls.

## Target Server-Side Architecture

Future M3.2 implementation should plan and then implement only after AP approval:

- Provider registry: a server-side registry of supported provider types, display metadata, supported model policy fields, provider status values, and provider-specific request capabilities.
- Provider key reference: a non-secret reference such as an allowlisted server-side resolver token. The reference is never the raw provider key.
- Tenant/workspace provider policy: server-side policy that resolves the authenticated user's active tenant/workspace and determines which active provider config is allowed for the requested AI operation.
- Provider status: explicit states such as active, disabled, and pending review, with disabled and pending states failing closed for pilot/production execution.
- Fail-closed behavior: provider resolution must stop before provider calls when auth, tenant membership, provider config, provider status, key reference, prompt/version reference, or evidence logging requirements are missing or unclear.
- Audit/evidence event shape: server-side events should identify org/workspace, user, provider config reference, provider type, model, prompt key/version, operation type, status, sanitized error class, and output/reference metadata without storing raw secrets, raw prompts, uploaded source bodies, or raw provider payloads.
- Prompt/version reference: future provider calls should bind a governed prompt key/version, even if the first implementation uses server-side constants before DB-backed prompt administration.
- Usage metadata: usage should come from server-side provider responses and record token counts, model, provider type, operation type, output reference, and cost estimate when available without trusting client-submitted usage.
- Sanitized error classes: errors should classify missing config, disabled provider, unsupported provider, missing key reference, key resolution failure, provider request failure, provider rate limit, provider unavailable, audit persistence failure, and policy denial without leaking secrets.
- Human-readable admin/control evidence: future Admin Controls should show provider readiness, key-reference readiness, model policy status, last test status, usage/job readiness, and pilot readiness without displaying raw keys.

## Supabase / Edge Boundary Plan

Future Edge behavior should:

- Authenticate the user and resolve active org/workspace membership before provider resolution.
- Resolve provider config server-side by tenant/workspace, provider type, operation type, and active status.
- Resolve key references through an allowlisted server-side resolver only after provider config and policy pass.
- Reject client-supplied raw keys and never pass them through Edge request bodies.
- Fail closed if provider config, provider status, key reference, tenant/workspace policy, prompt/version reference, or required evidence shape is unavailable.
- Log only safe evidence fields: provider config ID/reference, provider type, key-reference status, model, prompt key/version, usage counts, operation type, status, output reference, and sanitized error class.
- Never log raw provider keys, key-reference resolved values, authorization headers, raw prompts, source document bodies, provider request payloads containing secrets, provider response payloads containing secret material, or product copy that implies unsupported compliance status.

Current direct server env reads in `supabase/functions/_shared/ai.ts` should be replaced in future implementation by server-side provider config and allowlisted key-reference resolution. This planning milestone does not make that replacement.

## Schema Planning Context

`docs/schema/ai_governance_jobs.sql` already contains prior context for:

- `ai_provider_configs`
- `prompt_templates`
- `ai_generation_jobs`

Candidate schema discussion is not approval to migrate, alter, depend on, or implement these structures in this milestone.

Future AP-approved schema or migration planning should validate whether existing structures are sufficient for:

- provider config lookup by org/workspace, provider type, status, and operation
- key-reference storage as reference-only data
- provider policy ownership and role-gated mutation
- prompt key/version reference

Stop future implementation if schema/RLS needs are unclear or if implementation requires migration, alteration, or dependency on these structures without AP approval.

## Admin / Control Planning

Future Admin Controls planning should keep the first surface read-only unless AP separately approves configuration behavior.

Future controls should define:

- Who can configure provider records and who can only view readiness.
- Editable fields such as display name, provider type, default model, model policy, status, and key-reference label or reference selector when approved.
- Never-displayed fields such as raw provider keys, resolved key values, secret manager payloads, auth headers, provider request bodies, and provider response payloads.
- Evidence shown to humans: provider status, key-reference status, policy status, model policy status, last test status, usage/job readiness, and sanitized failure reason.
- Approval gates for enabling a provider, changing model policy, rotating a key reference, or switching pilot/production readiness.

No Admin UI is implemented or approved by this planning milestone.

## Interaction With M3.1c Scanner

M3.1c static scan currently classifies:

- browser fallback provider key references in `services/geminiService.ts` as explicit `local-demo` / `internal-dev` fallback
- server-side env references in `supabase/functions/_shared/ai.ts` as server-side Edge/Supabase environment references
- planning and evidence docs as test/control documentation

M3.2 scanner interaction is planning-only. Future implementation should update:

- allowlist classifications when direct server env references move behind a key-reference resolver
- evidence docs to distinguish browser fallback classifications from server-side key-reference resolver classifications
- forbidden runtime pattern expectations so browser raw keys remain blocked and server-side direct key usage is narrowly controlled
- static scan evidence so planning/control references never become runtime permission
