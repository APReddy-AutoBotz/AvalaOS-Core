# M3.2d Provider Audit/Evidence Event Contract Planning

Status: Planning/proposal only

Branch: `milestone/m3.2d-provider-audit-evidence-contract-planning`

## Goal

Define a future audit/evidence event contract for server-side provider config, key-reference resolution, policy checks, prompt/version binding, usage metadata, and sanitized failure records.

This milestone does not implement audit events, change schema/RLS, create or edit migrations, change Supabase Functions, change runtime behavior, change product code, change tests, change package files, change dependencies, change Admin UI, touch Health, change scoring, add runtime agents, add MCP/A2A behavior, add external execution, or make compliance claims.

## Source-Of-Truth Alignment

This proposal follows `docs/00_SOURCE_OF_TRUTH.md`:

- Evaluate before you automate. Govern before you execute.
- Deterministic scoring remains deterministic.
- AI must not decide final scores, gates, risk tiers, recommendations, approvals, or compliance status.
- Browser-stored provider keys and browser-side AI are not acceptable for pilot or production behavior.
- Server-side AI through Supabase Edge Functions remains the pilot and production direction.
- Evidence proves every governed decision.

## Current Evidence / Audit Inventory

`docs/schema/ai_governance_jobs.sql` contains prior context only:

- `ai_generation_jobs` records org/user job attempts with provider config reference, prompt template reference, prompt version, job type, status, model, input/output refs, error message, timestamps, RLS, member-read policy, and member-create policy.
- `ai_usage_events` records org/user usage with job reference, provider, model, prompt template reference, prompt version, token counts, cost estimate, output artifact reference, metadata, timestamps, RLS, member-read policy, and member-create policy.

Current Edge helper flow in `supabase/functions/_shared/ai.ts`:

- `getProviderKey(provider)` reads direct server env keys with `Deno.env.get('GROQ_API_KEY')` and `Deno.env.get('GEMINI_API_KEY')`.
- `assertSupportedProvider(provider)` accepts only `groq` and `gemini`.
- `generateDocumentWithProvider` and `refineSectionWithProvider` read server env keys, call Groq or Gemini, and return generated content plus usage metadata.
- Groq usage metadata maps safe fields from `payload.usage` into `inputTokens`, `outputTokens`, `totalTokens`, and `model`.
- Gemini usage metadata maps safe fields from `payload.usageMetadata` into `inputTokens`, `outputTokens`, `totalTokens`, and `model`.
- Current direct server env-key reads and provider-call behavior remain unchanged in M3.2d.

Current gaps:

- No resolver-specific audit/evidence event exists.
- No provider config/key-reference policy event shape exists.
- No event taxonomy distinguishes provider resolution, provider call, key-reference, prompt/version, usage, and sanitized failure records.
- No AP-approved event envelope exists for provider resolver evidence.
- No future control-plane display contract exists for provider resolution evidence.

Current M3.1c scan evidence model:

- `scripts/check-ai-boundary.mjs` scans tracked and untracked text files for AI/key boundary patterns.
- `docs/quality/ai-boundary-static-scan-allowlist.json` allows planning/evidence docs as test/control documentation.
- Direct `GROQ_API_KEY` and `GEMINI_API_KEY` hits in `supabase/functions/_shared/ai.ts` are currently classified as server-side Edge/Supabase environment references.
- Browser `VITE_GEMINI_API_KEY` and `VITE_GROQ_API_KEY` hits remain limited to approved local-demo/internal-dev fallback classifications.
- M3.2d does not change the scanner or allowlist.

## Future Event Taxonomy

This taxonomy is proposal-only. It defines future event names and safe metadata expectations, not an implementation.

Evidence events must prove that a gate was checked, but must not become a storage location for sensitive payloads. Evidence should record the result, reason class, safe identifiers, and timestamps, not the underlying secret, prompt body, response body, raw provider error, or hidden chain/payload.

| Event type | Purpose | Emission point | Actor | Tenant/workspace | Safe identifiers | Prompt/version references | Operation / mode | Policy result | Failure class | Safe UI/control summary | Fields never logged |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| `provider_resolution_started` | Mark resolver start for a provider request. | Before provider policy lookup. | Authenticated actor or service actor. | Tenant/workspace from request context. | Requested provider ID and optional provider config ID. | Requested prompt/template/version refs if supplied. | Requested operation and mode. | `pending`. | None. | Provider resolution started. | Raw secrets, prompt body, response body, raw provider error, auth headers, hidden chain/payload. |
| `provider_resolution_allowed` | Prove all resolver gates passed. | After mode, auth, tenant/workspace, policy, status, operation, key-reference, and prompt/version gates pass. | Authenticated actor or service actor. | Resolved tenant/workspace. | Provider ID, provider config ID, key-reference ID or AP-approved safe fingerprint. | Prompt template/version IDs. | Resolved operation and mode. | `allowed`. | None. | Provider allowed for this operation. | Raw secrets, resolved key material, prompt body, response body, provider request body, hidden chain/payload. |
| `provider_resolution_blocked` | Prove resolver failed closed before provider call. | At first failed or unsafe resolver gate. | Authenticated actor or service actor if available. | Resolved tenant/workspace when available. | Safe provider/config/key-reference IDs when available. | Prompt/template/version refs when available. | Requested operation and mode. | `blocked`. | M3.2c failure class. | Provider request blocked by governance. | Raw secrets, prompt body, response body, raw provider error, auth headers, hidden chain/payload. |
| `provider_resolution_failed` | Record resolver internal failure using sanitized details. | When resolver cannot complete due to safe internal error. | Authenticated actor or service actor if available. | Tenant/workspace if safely resolved. | Safe provider/config/key-reference IDs when available. | Prompt/template/version refs when available. | Requested operation and mode. | `failed`. | `provider_call_blocked` or specific safe class. | Provider resolution failed safely. | Stack traces, raw exceptions, raw secrets, prompt body, response body, hidden chain/payload. |
| `provider_call_started` | Mark provider call handoff after allowed resolution. | Immediately before server-side provider call. | Service actor on behalf of authenticated actor. | Resolved tenant/workspace. | Provider ID, provider config ID, key-reference safe ID/status. | Prompt/template/version refs. | Resolved operation and mode. | `allowed`. | None. | Provider call started. | Raw key material, auth headers, provider request body, prompt body unless approved, hidden chain/payload. |
| `provider_call_succeeded` | Record sanitized successful provider call evidence. | After provider returns and response is sanitized. | Service actor on behalf of authenticated actor. | Resolved tenant/workspace. | Provider ID, provider config ID, key-reference safe ID/status. | Prompt/template/version refs. | Resolved operation and mode. | `allowed`. | None. | Provider call succeeded. | Raw response body, full model output unless approved, raw prompt/source body, raw secrets, hidden chain/payload. |
| `provider_call_failed_sanitized` | Record sanitized provider-call failure. | After provider call fails and error is mapped to safe class. | Service actor on behalf of authenticated actor. | Resolved tenant/workspace. | Provider ID, provider config ID, key-reference safe ID/status. | Prompt/template/version refs. | Resolved operation and mode. | `allowed_before_call`. | `provider_call_failed_sanitized`. | Provider call failed with sanitized reason. | Raw provider error body, auth headers, stack traces, raw secrets, prompt/response body, hidden chain/payload. |
| `provider_policy_missing` | Record missing provider policy gate. | When tenant/workspace policy lookup returns no usable policy. | Authenticated actor or service actor. | Tenant/workspace if available. | Requested provider ID/config ID if available. | Prompt/template/version refs if available. | Requested operation and mode. | `blocked`. | `provider_policy_missing`. | Provider policy not configured. | Raw secrets, raw request body, prompt body, hidden chain/payload. |
| `provider_disabled` | Record disabled or non-active provider gate. | When provider config is disabled, retired, pending, or not eligible. | Authenticated actor or service actor. | Resolved tenant/workspace. | Provider ID and provider config ID. | Prompt/template/version refs if available. | Requested operation and mode. | `blocked`. | `provider_disabled`. | Provider disabled or unavailable. | Raw secrets, key material, provider payloads, hidden chain/payload. |
| `key_reference_missing` | Record missing key-reference gate. | When provider config lacks required key-reference metadata. | Service actor after policy/status checks. | Resolved tenant/workspace. | Provider ID and provider config ID; no secret ref payload. | Prompt/template/version refs if available. | Requested operation and mode. | `blocked`. | `key_reference_missing`. | Provider key reference missing. | Raw secrets, secret-manager values, auth headers, hidden chain/payload. |
| `key_reference_disabled` | Record disabled or ineligible key-reference gate. | When key-reference metadata exists but cannot be used. | Service actor after policy/status checks. | Resolved tenant/workspace. | Provider ID, provider config ID, key-reference ID/status. | Prompt/template/version refs if available. | Requested operation and mode. | `blocked`. | `key_reference_disabled`. | Provider key reference unavailable. | Raw secrets, resolved key material, secret-manager values, hidden chain/payload. |
| `key_resolution_failed` | Record safe failure to resolve server-side key reference. | When key-reference lookup fails after allowed gates. | Service actor. | Resolved tenant/workspace. | Provider ID, provider config ID, key-reference ID/status, resolver type. | Prompt/template/version refs if available. | Requested operation and mode. | `blocked`. | `key_resolution_failed`. | Provider key resolution failed. | Raw secrets, secret-manager values, auth headers, stack traces, hidden chain/payload. |
| `prompt_version_missing` | Record missing prompt/template/version gate. | When required prompt binding cannot be resolved. | Service actor after provider/key gates where applicable. | Resolved tenant/workspace. | Provider ID, provider config ID, key-reference safe ID/status if available. | Missing prompt/template/version refs. | Requested operation and mode. | `blocked`. | `prompt_version_missing`. | Prompt version unavailable. | Prompt body, raw source body, raw provider payloads, hidden chain/payload. |
| `usage_recorded` | Record safe usage metadata after provider call. | After provider response usage is sanitized and accepted. | Service actor. | Resolved tenant/workspace. | Provider ID, provider config ID, job ID/output reference when available. | Prompt/template/version refs. | Completed operation and mode. | `allowed`. | None. | Usage recorded. | Raw prompts, raw model response, raw provider payload, raw secrets, misleading cost claims, hidden chain/payload. |

## Common Event Envelope

Candidate event/schema discussion is not approval to migrate, alter, depend on, or implement these structures in this milestone.

Future audit/evidence events should use a common envelope with safe metadata only:

| Field | Purpose | Safety rule |
| --- | --- | --- |
| `event_id` | Unique event identifier. | Safe generated ID only. |
| `event_type` | One of the proposed event taxonomy values. | Safe enum only. |
| `correlation_id` | Groups resolver/call/usage events for one request. | Safe generated or server-normalized ID. |
| `request_id` | Optional request-level ID. | Safe request reference only, not headers or tokens. |
| `actor_id` | Authenticated user or service actor ID. | Safe identifier only. |
| `tenant_id` | Tenant/org owner. | Required when available; missing ownership fails closed in implementation. |
| `workspace_id` | Workspace owner. | Required when policy needs workspace scope. |
| `mode` | Runtime mode. | Safe enum from M3.1a mode contract. |
| `operation_type` | Requested operation. | Safe enum/string such as `generate_document` or `refine_section`. |
| `provider_id` | Safe provider identifier. | No provider secret or auth data. |
| `provider_config_id` | Safe provider config reference. | Metadata reference only. |
| `key_reference_id` or safe fingerprint | Safe key-reference identifier. | Safe fingerprint only if AP separately approves it; never raw key material. |
| `prompt_template_id` | Prompt/template reference. | Reference only, not prompt body. |
| `prompt_version_id` | Prompt version reference. | Reference only, not prompt body. |
| `policy_result` | Policy result such as `pending`, `allowed`, `blocked`, or `failed`. | Safe result class only. |
| `failure_class` | Sanitized failure class when applicable. | Use M3.2c classes; no raw errors. |
| `sanitized_status` | Safe status summary. | No raw provider body or stack trace. |
| `usage_metadata` | Safe usage shell. | Token/model/output refs only when safe and non-misleading. |
| `created_at` | Event timestamp. | Server-side timestamp. |
| `schema_version` | Event contract version. | Safe integer/string version. |

Raw secrets must never be included.

## Fields Never Logged

Future audit/evidence events must not log:

- raw provider secrets
- full prompt content unless separately AP-approved
- full model response unless separately AP-approved
- raw provider error body
- provider authorization headers
- browser raw keys
- `userApiKey`
- `localStorage` values
- secret-manager values
- unredacted stack traces
- regulated/compliance claims
- hidden chain/payload or reasoning content
- resolved key material
- service-role keys
- browser session tokens, cookies, or authorization headers
- raw source document bodies
- raw provider request bodies
- raw provider response bodies

Evidence events must not become a fallback storage layer for payloads that belong in separately approved, access-controlled storage.

## Success Event Contract

Allowed provider resolution evidence should prove:

- the resolver started with a safe correlation ID
- mode was checked
- actor authentication was checked
- tenant/workspace membership was checked
- provider policy was checked
- provider status was checked
- operation authorization was checked
- key-reference eligibility was checked
- prompt/template/version binding was checked
- evidence context was prepared before provider call

Provider-call success evidence should record:

- `provider_resolution_allowed`
- `provider_call_started`
- `provider_call_succeeded`
- `usage_recorded` when safe usage metadata is available
- provider ID and provider config ID
- key-reference safe ID/status only
- prompt/template/version references
- operation type and mode
- correlation ID and timestamps
- sanitized provider status

Usage metadata shell may include:

- provider ID
- model
- input token count
- output token count
- total token count
- output artifact reference
- job ID or generation reference when available
- cost estimate only when safely available, non-misleading, and explicitly labeled as an estimate

Success events must provide human-readable control evidence such as:

- Provider was allowed by policy.
- Key reference was eligible.
- Prompt version was bound.
- Provider call completed.
- Usage was recorded from server-side provider response.

Success events must not claim compliance status, autonomous approval, deterministic score changes, or final recommendation authority.

## Failure / Block Event Contract

Fail-closed blocked events should record:

- which gate blocked the request
- sanitized M3.2c failure class
- safe reason class
- retry posture
- safe UI message category
- correlation ID
- safe tenant/workspace/provider/config/key-reference/prompt references when available
- timestamp

M3.2c sanitized failure classes that must remain supported by evidence planning:

- `mode_not_allowed`
- `unauthenticated`
- `unauthorized_workspace`
- `provider_policy_missing`
- `provider_disabled`
- `provider_not_allowed`
- `operation_not_allowed`
- `key_reference_missing`
- `key_reference_disabled`
- `key_resolution_failed`
- `prompt_version_missing`
- `provider_call_blocked`
- `provider_call_failed_sanitized`

Retry posture should be constrained:

- Auth failures may retry after sign-in.
- Missing policy, disabled provider, disallowed operation, missing key-reference, disabled key-reference, and missing prompt version require configuration changes.
- Key-resolution or provider-call failures may be retryable only if classified as transient and retry does not bypass policy.
- Blocked events must never be retried by skipping gates.

Safe UI message categories should be generic:

- AI mode unavailable
- Sign-in required
- Workspace access denied
- Provider not configured
- Provider unavailable
- Operation unavailable
- Provider key unavailable
- Prompt version unavailable
- Provider request failed

AP-review evidence fields should include only:

- event type
- failure class
- policy result
- safe identifiers
- timestamp
- correlation/request ID
- retry posture
- sanitized status

Blocked events prove governance by recording that a gate was checked and failed closed. They must not include raw secrets, prompt body, response body, raw provider error, stack trace, hidden chain/payload, or sensitive request payloads.

## Relationship To Schema Proposal

This section is planning-only.

Candidate event/schema discussion is not approval to migrate, alter, depend on, or implement these structures in this milestone.

Open schema relationship questions:

- `ai_usage_events` may be reusable for safe usage metadata, but it is not sufficient as the only evidence record for provider config, key-reference, resolver, and policy gates.
- A separate `ai_provider_audit_events` object may be needed for append-only provider resolution, policy, key-reference, prompt/version, and fail-closed evidence.
- `ai_generation_jobs` may later reference provider audit events through a candidate `provider_audit_event_id`, `resolution_event_id`, or `correlation_id`, but no foreign key is approved in this milestone.
- Candidate foreign keys may include references to provider config, key reference, prompt template/version, generation job, usage event, tenant/org, workspace, and actor records after AP approval.
- Retention should preserve governance evidence long enough for AP-approved review needs without retaining sensitive payloads.
- RLS/read visibility should separate admin/control evidence readers from normal workspace users and must never expose raw secret or sensitive payload fields.
- Service-role writer posture should prefer append-only server-side writes for resolver/provider-call evidence. Browser/client direct writes should not be approved for provider audit events.

Future schema implementation must prove that safe metadata and sensitive payloads are separated before any migration or runtime dependency.

## Stop Conditions

Future implementation must stop if:

- raw secrets would be logged
- raw secrets would be stored in DB rows
- full prompt/response logging is required without AP approval
- event schema cannot separate safe metadata from sensitive payloads
- RLS/read visibility is unclear
- service-role writer boundary is unclear
- tenant/workspace ownership is unclear
- failure classes are ambiguous
- usage/cost metadata would be misleading
- M3.1c scanner cannot classify new hits safely
- package/dependency changes are needed without AP approval
- Health, scoring, runtime-agent, MCP/A2A, or external-execution changes appear
- compliance claims are introduced
