# M5.3a-2g Local Supabase Stack Availability Remediation Plan

## Milestone Objective

M5.3a-2g is a docs-only remediation plan for the local Supabase stack availability gap discovered during M5.3a-2f.

The objective is to define how a future AP-approved remediation implementation may diagnose local stack availability safely, using sanitized status categories only, before any future local DB bootstrap, schema readiness, RLS harness, or artifact assertion execution is attempted.

M5.3a-2g does not execute remediation, rerun local stack startup, bootstrap schema, run RLS or artifact harnesses, prove local DB availability, prove schema readiness, verify artifact SELECT isolation, verify tenant isolation, or claim hosted-pilot or production readiness.

Recommended next milestone:

- `M5.3a-2h Local Supabase Stack Availability Remediation Implementation`, only after AP accepts this plan and explicitly approves implementation scope.

## Local-Only Remediation Goals

Future remediation should establish only sanitized readiness categories needed to decide whether another local execution evidence attempt is safe.

Allowed future status categories:

- Docker availability: `yes`, `no`, or `unknown`.
- Supabase CLI availability: `yes`, `no`, or `unknown`.
- Local stack startup readiness: `ready`, `blocked`, or `unknown`.
- Local port conflict category: `clear`, `blocked`, or `unknown`.
- Local config presence: `present`, `missing`, or `unknown`.
- Generated log capture: `prohibited` unless redacted and AP-approved.
- Raw local output: `prohibited`.

Future remediation must preserve local-only constraints:

- No personal hosted Supabase.
- No remote Supabase.
- No production-like credentials.
- No remote validation.
- No provider calls.
- No target values or local machine details in evidence.

## Safe Diagnostic Categories For Future Implementation

Future M5.3a-2h implementation may propose diagnostic categories, but each category must be AP-approved before execution and must produce sanitized status only.

Potential future categories:

- Tool availability category: whether required local tooling is available, reported as yes/no/unknown.
- Local stack readiness category: whether local startup can be attempted safely, reported as ready/blocked/unknown.
- Local config category: whether required generic local config is present, reported as present/missing/unknown.
- Port-conflict category: whether conflicts are detected, reported as clear/blocked/unknown.
- Cleanup-readiness category: whether safe cleanup/reset is permissible, reported as allowed/blocked/unknown.

These are planning categories only. M5.3a-2g does not run diagnostics.

## Safe Cleanup And Reset Considerations

Future remediation may plan cleanup/reset only after AP approval.

Planning constraints:

- Cleanup/reset must be local-only.
- Cleanup/reset must not affect remote, hosted, or production-like resources.
- Cleanup/reset must not rely on secret-file fallback.
- Cleanup/reset output must be sanitized status only.
- Cleanup/reset must stop before action if the target is not proven local.
- Cleanup/reset must not print local paths, target values, raw logs, container IDs, image IDs, DB URLs, or secrets.

M5.3a-2g does not execute cleanup or reset.

## Stop Conditions

Future remediation implementation must stop immediately if:

- AP approval is missing.
- A proposed diagnostic command is not explicitly allowed.
- A command would open a DB connection.
- A command would run a Supabase local stack action not approved for that milestone.
- A command would run an RLS harness or artifact assertion.
- A command would inspect `.env` or ignored local secret files.
- A command would print raw logs, local paths, target values, container IDs, image IDs, DB URLs, host values, port values, database names, user names, tokens, claims, row payloads, or secrets.
- Output sanitization cannot be proven.
- The root cause would require unsafe local detail to prove.
