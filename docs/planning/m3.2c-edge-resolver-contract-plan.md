# M3.2c Edge Resolver Contract Planning

Status: Planning/proposal only

Branch: `milestone/m3.2c-edge-resolver-contract-planning`

## Goal

Define the future Edge resolver contract for server-side provider config and key-reference resolution before any runtime implementation.

This milestone does not change Supabase Functions, runtime behavior, schema/RLS, migrations, product code, tests, package files, dependencies, Admin UI, Health, scoring, runtime agents, MCP/A2A, external execution, or compliance claims.

## Source-Of-Truth Alignment

This plan follows `docs/00_SOURCE_OF_TRUTH.md`:

- Evaluate before you automate. Govern before you execute.
- Deterministic scoring remains deterministic.
- AI must not decide final scores, gates, risk tiers, recommendations, approvals, or compliance status.
- Browser-stored provider keys and browser-side AI are not acceptable for pilot or production behavior.
- Server-side AI through Supabase Edge Functions remains the pilot and production direction.
- Evidence proves every governed decision.

## Current Edge Flow Inventory

Current direct Edge/server env-key flow:

- `supabase/functions/_shared/ai.ts` defines `getProviderKey(provider)`.
- `getProviderKey('groq')` reads `Deno.env.get('GROQ_API_KEY')`.
- `getProviderKey('gemini')` reads `Deno.env.get('GEMINI_API_KEY')`.
- `assertSupportedProvider(provider)` currently accepts only `groq` and `gemini`.
- `generateDocumentWithProvider` and `refineSectionWithProvider` call `getProviderKey` before provider calls and fail when the selected provider key is not configured server-side.
- Current Groq model selection reads `Deno.env.get('GROQ_MODEL')` with fallback to `llama-3.3-70b-versatile`.
- Current Gemini model selection reads `Deno.env.get('GEMINI_MODEL')` with fallback to `gemini-2.5-flash`.
- `services/aiEdgeClient.ts` sends `providerType` from the browser to Edge Functions for document generation and section refinement.
- `services/aiOrchestrator.ts` uses `services/aiMode.ts` to fail closed for `pilot` and `production` when Edge AI is unavailable, while retaining the approved `local-demo` / `internal-dev` browser fallback.

Current gaps:

- No tenant/workspace provider policy is resolved before Edge provider calls.
- No key-reference resolver exists between provider config and secret material.
- No provider config lifecycle check exists for status, enablement, operation policy, or allowed mode.
- No prompt/template/version binding contract exists before provider calls.
- No resolver audit/evidence event is emitted for allow, deny, or fail-closed decisions.
- No provider operation policy distinguishes `generate_document`, `refine_section`, or future operations.

M3.2c must keep existing direct server env-key reads unchanged. Replacement of direct `Deno.env.get('GROQ_API_KEY')` / `Deno.env.get('GEMINI_API_KEY')` reads requires a later AP-approved implementation slice.

## Future Edge Resolver Contract

The proposed conceptual resolver is `edgeProviderResolver`. This is a proposed module name only; M3.2c does not create or implement it.

Supported provider identifiers for the first resolver contract should be:

- `groq`
- `gemini`

Additional provider identifiers require a later AP-approved contract update and static scan evidence.

Allowed modes:

- `pilot`
- `production`
- `local-demo` and `internal-dev` remain governed by the existing M3.1a fallback policy and are not widened by this resolver proposal.

Proposed resolver input fields:

| Field | Purpose | Notes |
| --- | --- | --- |
| `mode` | Runtime mode being evaluated. | Must be validated against `local-demo`, `internal-dev`, `pilot`, and `production`; pilot/production require Edge. |
| `providerId` | Requested provider identifier. | Initially `groq` or `gemini`; unknown provider fails closed. |
| `providerConfigId` | Optional explicit provider config selector. | If supplied, must match tenant/workspace policy. |
| `operationType` | Provider operation requested. | Examples: `generate_document`, `refine_section`, `test_provider_connection`. |
| `tenantId` / `orgId` | Tenant ownership boundary. | Missing or ambiguous tenant identity fails closed. |
| `workspaceId` | Workspace policy boundary. | Missing workspace is allowed only if the approved policy explicitly supports org-level resolution. |
| `actorId` / `userId` | Authenticated actor. | Must come from the server-authenticated request context, not browser-trusted input alone. |
| `promptTemplateId` / `promptKey` | Governed prompt selector. | Must bind to an approved prompt/template reference before provider call. |
| `promptVersion` | Governed prompt version. | Missing required version fails closed. |
| `requestedModel` | Requested model metadata. | Must be constrained by provider config/model policy. |
| `correlationId` | Request/evidence correlation. | Server may create one when missing. |
| `evidenceId` | Optional prepared evidence ID. | Must not be trusted as proof of an allow decision. |
| `keyReferenceSelector` | Safe key-reference selector. | Must be reference-only and server-resolved only after policy checks pass. |

Proposed resolver output fields:

| Field | Purpose | Secret posture |
| --- | --- | --- |
| `status` | `allowed` or `blocked`. | Safe to log. |
| `providerId` | Resolved provider identifier. | Safe to log. |
| `providerConfigId` | Resolved provider config reference. | Safe to log when AP approves. |
| `providerStatus` | Sanitized provider status. | Safe values only, such as `active` or `disabled`. |
| `operationType` | Resolved operation. | Safe to log. |
| `mode` | Resolved runtime mode. | Safe to log. |
| `policyResult` | Allow/deny decision metadata. | Safe summary only. |
| `promptReference` | Prompt/template/version reference. | Safe reference only; no raw prompt body unless separately approved. |
| `usageMetadataShell` | Empty or prepared usage metadata shape. | No provider payload or raw source text. |
| `sanitizedEvidenceContext` | Fields prepared for audit/evidence. | No raw secrets, auth headers, raw provider payloads, or raw prompt/source bodies. |
| `providerCallHandoff` | Server-side-only handoff to provider-call layer. | Must not be returned to browser/client or general business logic. |

Failure output shape:

- `status`: `blocked`
- `failureClass`: one of the sanitized classes in this plan
- `safeUiMessageCategory`: generic user-facing category
- `correlationId`: safe request correlation ID
- `evidenceFields`: safe evidence subset
- `retryAllowed`: boolean or constrained retry category
- `providerCallAllowed`: always `false`

The resolver must never return raw key material to general business logic. The future provider-call layer may receive server-side-only secret material only inside the Edge/server boundary, and that material must never be included in resolver evidence, errors, logs, return payloads, or UI-facing responses.

## Fail-Closed Policy Check Order

Future implementation must follow this order and stop at the first unresolved or denied condition:

1. Validate mode.
2. Authenticate actor.
3. Verify tenant/workspace membership.
4. Resolve provider policy.
5. Verify provider enabled/status.
6. Verify operation allowed.
7. Resolve key reference server-side.
8. Verify prompt/template/version availability.
9. Prepare sanitized audit/evidence context.
10. Only then allow provider call.

If a required input, ownership boundary, policy, provider config, provider status, key reference, prompt reference, or evidence context is missing, ambiguous, disabled, denied, or unsafe, the resolver must fail closed before any provider call.

## Failure Classes

| Failure class | Meaning | Safe UI message category | Evidence fields | Fields never logged | Retry allowed |
| --- | --- | --- | --- | --- | --- |
| `mode_not_allowed` | Requested mode is invalid or not eligible for resolver use. | AI mode unavailable. | mode, operation type, correlation ID. | raw secrets, auth headers, provider payloads, raw prompt/source body. | No, unless configuration changes. |
| `unauthenticated` | No authenticated actor is available. | Sign-in required. | operation type, correlation ID, auth status. | tokens, cookies, auth headers, raw secrets. | Yes, after sign-in. |
| `unauthorized_workspace` | Actor is not authorized for the tenant/workspace. | Workspace access denied. | actor ID, tenant/workspace ID, operation type, correlation ID. | auth headers, raw secrets, cross-tenant data. | No, unless access changes. |
| `provider_policy_missing` | No provider policy exists for the requested tenant/workspace/mode/operation. | Provider not configured. | tenant/workspace ID, mode, operation type, provider ID, correlation ID. | raw secrets, key material, provider payloads. | No, unless policy is configured. |
| `provider_disabled` | Resolved provider config is disabled, retired, or pending review. | Provider unavailable. | provider ID, provider config ID, sanitized status, correlation ID. | raw secrets, key material, provider payloads. | No, unless provider is enabled. |
| `provider_not_allowed` | Provider is not allowed by policy for the tenant/workspace/mode. | Provider not allowed. | provider ID, tenant/workspace ID, mode, policy result, correlation ID. | raw secrets, key material, provider payloads. | No, unless policy changes. |
| `operation_not_allowed` | Requested operation is not allowed for the provider policy/config. | Operation unavailable. | operation type, provider ID, provider config ID, policy result, correlation ID. | raw secrets, key material, raw prompt/source body. | No, unless operation policy changes. |
| `key_reference_missing` | Required key reference is absent. | Provider key not configured. | provider ID, provider config ID, key reference status, correlation ID. | raw secrets, resolved key values, secret manager payloads. | No, unless key reference is configured. |
| `key_reference_disabled` | Key reference exists but is disabled, expired, or not eligible. | Provider key unavailable. | provider ID, provider config ID, key reference ID, sanitized key status, correlation ID. | raw secrets, resolved key values, secret manager payloads. | No, unless key reference is enabled or rotated. |
| `key_resolution_failed` | Server-side key resolution failed after policy checks passed. | Provider key unavailable. | provider ID, provider config ID, key reference ID, resolver type, sanitized error class, correlation ID. | raw secrets, resolved key values, secret manager payloads, auth headers. | Limited retry only for transient resolver failures. |
| `prompt_version_missing` | Required prompt/template/version binding cannot be resolved. | Prompt version unavailable. | prompt template ID/key, prompt version, provider ID, operation type, correlation ID. | raw prompt body unless separately approved, raw source body, raw secrets. | No, unless prompt/version is configured. |
| `provider_call_blocked` | Resolver blocked the provider call because a prerequisite was unsafe or incomplete. | Provider call blocked. | failure class, policy result, provider ID, operation type, correlation ID. | raw secrets, key material, provider payloads, raw prompt/source body. | No, unless prerequisite changes. |
| `provider_call_failed_sanitized` | Provider call was attempted after resolver allow decision but provider returned an error. | Provider request failed. | provider ID, provider config ID, operation type, sanitized provider status/error class, usage metadata if safe, correlation ID. | raw secrets, auth headers, provider request body, raw provider response body, raw prompt/source body. | Yes only for transient provider errors such as rate limit or availability. |

## Audit And Evidence Payload Proposal

Future resolver evidence should use a safe, append-oriented event shape.

Proposed event type:

- `ai_provider_resolver_decision`

Proposed payload fields:

- `eventType`
- `correlationId`
- `actorId`
- `tenantId` / `orgId`
- `workspaceId`
- `providerId`
- `providerConfigId`
- `keyReferenceId` or AP-approved safe fingerprint/label only
- `promptTemplateId` or `promptKey`
- `promptVersion`
- `operationType`
- `mode`
- `policyResult`
- `failureClass`
- `providerCallAllowed`
- `sanitizedProviderStatus`
- `sanitizedKeyReferenceStatus`
- `usageMetadata` with safe token/model/output-reference fields when available
- `createdAt`
- `completedAt` or `decidedAt`

Fields never logged:

- raw provider secrets
- resolved key material
- secret manager payloads
- authorization headers
- service-role keys
- browser/client provider secrets
- raw provider request bodies
- raw provider response bodies
- raw prompts unless separately approved
- raw source document bodies
- raw user tokens, cookies, or session material
- product copy that implies unsupported compliance status

Raw secrets must never be logged or returned.

## Provider Call Handoff

Future provider-call layer needs:

- resolved provider ID
- provider config ID
- model/config metadata constrained by policy
- operation type
- prompt/template/version binding
- correlation ID
- usage metadata shell
- sanitized evidence context
- server-bound secret access inside Edge/server only

What must stay server-side:

- key-reference resolution
- raw key material or provider authorization material
- secret manager lookups and payloads
- provider request headers
- provider raw response payloads until sanitized
- service-role access
- evidence persistence using safe fields only

Provider-specific error handling:

- Groq, Gemini, or future provider errors should be mapped to sanitized resolver or provider-call failure classes.
- Error messages returned to UI should use safe categories and correlation IDs, not provider payloads or secrets.
- Rate limit, timeout, temporary unavailable, and provider status errors may be marked retryable only when retry does not bypass policy.
- Auth, invalid key, disabled provider, missing config, missing prompt/version, or policy denial must not be retried without configuration changes.

Usage metadata handling:

- Usage should come from server-side provider responses only.
- Safe usage may include provider ID, model, token counts, operation type, output reference, status, cost estimate when available, and correlation ID.
- Usage metadata must not include raw prompts, raw source document bodies, auth headers, raw provider payloads, raw secrets, or resolved key values.

Future replacement:

- Direct `Deno.env.get('GROQ_API_KEY')` and `Deno.env.get('GEMINI_API_KEY')` should later be replaced by server-side key-reference resolution.
- That replacement is out of scope for M3.2c and requires AP-approved implementation.
- Raw provider secrets must never be logged, returned, exposed to browser/client code, or stored in ordinary application DB rows.

Out of scope until implementation:

- creating `edgeProviderResolver`
- changing `supabase/functions/_shared/ai.ts`
- adding schema/RLS or migrations
- adding provider config, key-reference, policy, or audit tables
- changing Edge Function request/response payloads
- adding Admin UI
- changing package files or dependencies
- changing runtime provider behavior or fallback behavior

## M3.1c Scanner Interaction

This section is planning-only.

Future allowlist updates for resolver files should be exact:

- Add entries only for approved server-side resolver paths.
- Use specific pattern/path/line or path-prefix entries with a reason that distinguishes server-side resolver/key-reference control from browser fallback.
- Include future action text that says direct server env-key references should be removed or reclassified after resolver implementation.
- Keep documentation and evidence references classified as control documentation, never runtime permission.

Expected future reclassification:

- Direct `GEMINI_API_KEY` and `GROQ_API_KEY` hits in `supabase/functions/_shared/ai.ts` should be removed or reclassified after approved runtime migration moves provider secret lookup behind the resolver.
- Browser `VITE_GEMINI_API_KEY`, `VITE_GROQ_API_KEY`, and other browser-exposed provider secrets remain forbidden outside the approved `local-demo` / `internal-dev` fallback.

Evidence required for allowlist updates:

- path is server-side only
- path cannot run in browser/client bundle
- key fields are references only unless inside the provider-call server boundary
- raw key material is not returned, logged, persisted, or exposed
- tests/static scan pass with 0 forbidden hits
- stale allowlist entries are removed when direct references move

Stale allowlist handling:

- Stale runtime entries must fail the scanner until removed or updated.
- Any unclear classification must stop implementation before merge.

## Stop Conditions

Future implementation must stop if:

- raw secrets would be exposed to browser/client
- raw secrets would be logged
- raw secrets would be stored in DB rows
- key-reference resolution is ambiguous
- tenant/workspace policy cannot be enforced
- provider operation policy is unclear
- prompt/template/version binding is unclear
- evidence event shape is unclear
- service-role boundary is unclear
- Admin UI is needed before backend contract stabilizes
- M3.1c scanner cannot classify new hits safely
- package/dependency changes are needed without AP approval
- Health, scoring, runtime-agent, MCP/A2A, or external-execution changes appear
- compliance claims are introduced
