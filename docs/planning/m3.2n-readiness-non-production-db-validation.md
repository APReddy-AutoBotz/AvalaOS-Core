# M3.2n-readiness Non-Production Database Validation

## Milestone Purpose

M3.2n-readiness is a readiness and validation milestone that determines whether safe non-production database validation is available before any M3.2n resolver Edge Function integration implementation may proceed.

This milestone is not Edge Function integration, resolver implementation, schema implementation, or runtime behavior. It exists to validate the non-production database prerequisite or document the exact blocker.

## Dependency Chain Through M3.2m

1. M3.2h created the provider governance migration for provider key references, provider configs, provider policies, and provider audit events.
2. M3.2l implemented resolver decision logic only.
3. M3.2m planned future resolver Edge Function integration and was post-merge verified and tagged as `avalaos-core-m3.2m-resolver-edge-function-integration-plan`.
4. M3.2m carried forward the requirement that M3.2n must not proceed if non-production `DATABASE_URL` validation remains unavailable unless AP explicitly accepts that risk.
5. M3.2n-readiness checks that prerequisite and records a go/no-go recommendation for implementation.

## Validation Objective

The objective is to determine one of two outcomes:

- Outcome A, validated: a safe non-production `DATABASE_URL` is available, the M3.2h provider governance migration can be applied or verified safely, required provider governance tables and columns exist, resolver integration dependencies are verifiable in non-production, and M3.2n may proceed after AP/ChatGPT review.
- Outcome B, blocked: a safe non-production `DATABASE_URL` is unavailable or unsuitable, no database validation is performed, and M3.2n must remain blocked unless AP explicitly accepts the risk.

## Allowed Validation Commands

Allowed readiness checks:

- Confirm the M3.2h migration file exists at `supabase/migrations/20260607150600_m3_2h_provider_config_governance.sql`.
- Check whether `DATABASE_URL` is set without printing its value.
- Run the existing migration validation command only when it will target a safe non-production database, or when `DATABASE_URL` is unset and the command is expected to stop before connecting:
  - `node scripts/applySupabaseMigrations.mjs supabase/migrations/20260607150600_m3_2h_provider_config_governance.sql`
- If a safe non-production database is available, verify required provider governance tables and columns.

Required tables for future validation:

- `ai_provider_key_refs`
- `ai_provider_configs`
- `ai_workspace_provider_policies`
- `ai_provider_audit_events`

Required columns for future validation:

- `ai_provider_key_refs.secret_ref`
- `ai_provider_key_refs.resolver_type`
- `ai_provider_configs.key_ref_id`
- `ai_workspace_provider_policies.operation`
- `ai_workspace_provider_policies.mode`
- `ai_workspace_provider_policies.allowed_roles`
- `ai_workspace_provider_policies.is_default`
- `ai_provider_audit_events.metadata`
- `ai_provider_audit_events.correlation_id`

If safely checkable, future validation should also confirm provider scope remains `gemini` and `groq`, RLS is enabled on all four provider governance tables, and no production or shared customer database is touched.

## Prohibited Actions

M3.2n-readiness does not:

- implement runtime integration
- edit `_shared/ai.ts`
- edit Edge Function entrypoints
- call provider APIs
- call `Deno.env.get`
- resolve real secrets
- return raw key material
- write DB audit rows unless using a clearly safe non-production validation database through AP-approved validation SQL
- create or edit migrations
- edit schema SQL
- change Admin UI
- change product UI
- touch Health
- change scoring
- change browser fallback behavior
- add provider secrets
- change prompt versions
- expand usage/cost behavior
- add package dependencies
- make compliance claims

## Outcome Decision Framework

Choose Outcome A only if:

- `DATABASE_URL` is set and confirmed safe for non-production validation
- migration validation applies or verifies the M3.2h provider governance migration safely
- required tables and columns are verified
- provider scope and RLS checks pass when safely checkable
- no production or shared customer database is touched

Choose Outcome B if:

- `DATABASE_URL` is absent
- `DATABASE_URL` cannot be proven non-production
- migration validation cannot run safely
- required tables or columns cannot be verified
- any validation would risk touching production or shared customer data

M3.2n must remain blocked unless:

- AP explicitly accepts the risk of proceeding without non-production database validation

## Stop Conditions

Stop before M3.2n implementation if:

- `DATABASE_URL` remains unavailable
- the available database target cannot be proven safe and non-production
- migration validation fails
- required tables or columns are missing
- provider scope cannot be verified
- RLS cannot be safely checked where required
- AP does not explicitly accept unresolved database validation risk

## Risks Carried Forward

- Safe non-production `DATABASE_URL` validation remains unavailable.
- Required provider governance tables and columns remain unverified in a live non-production database.
- RLS status remains unverified in a live non-production database.
- Provider scope remains unverified in a live non-production database.
- Real secret lookup, audit DB insert, provider-call handoff, and runtime Edge Function integration remain AP-gated and unimplemented.
- Existing `_shared/ai.ts` direct provider key reads remain pre-existing and unchanged by this milestone.
