# M3.2q Non-Production Deployed Edge Smoke Plan

## Milestone Purpose

M3.2q is a planning-only milestone for deployed non-production Supabase Edge Function smoke after M3.2p proved the resolver, audit, secret-reference, and provider-handoff sequence with mocked provider behavior against the non-production database.

M3.2q plans how to safely deploy or invoke the resolver-gated Edge Functions in a non-production Supabase environment. Deployed smoke execution is not performed in M3.2q.

Target Edge Functions:

- `ai-generate-document`
- `ai-refine-section`
- `ai-provider-test-connection`

M3.2q does not:

- Deploy Edge Functions.
- Run deployed smoke.
- Call Gemini or Groq APIs.
- Use real provider keys.
- Print or store `DATABASE_URL`, Supabase service-role keys, provider keys, auth headers, or Supabase secrets.
- Use production, shared, customer, demo-customer, or unclear databases.
- Edit Edge Function entrypoints.
- Edit `_shared/ai.ts`.
- Edit runtime resolver or provider integration files.
- Change schema SQL, RLS, or migrations.
- Change Admin UI, product UI, Health, scoring, prompt versions, browser fallback, usage, or cost behavior.
- Add package dependencies.
- Make unsupported compliance claims.

## Dependency Chain

M3.2q depends on the completed M3.2 provider-governance chain:

- M3.2n Edge integration complete: resolver, audit persistence, secret lookup eligibility, and provider handoff gates are integrated into `ai-generate-document`, `ai-refine-section`, and `ai-provider-test-connection`.
- M3.2n-dbval Outcome A complete: provider-governance schema bootstrap was validated on a fresh AP-approved non-production Supabase database with zero retained provider governance rows.
- M3.2p mocked non-production smoke complete: the resolver/audit/secret/provider-handoff sequence passed with mocked provider behavior, transaction rollback cleanup, sanitized responses, and no live provider calls.

## Non-Production Environment Requirements

M3.2r must use only a clearly isolated non-production Supabase project or local Supabase environment. Production, shared, customer, demo-customer, or unclear databases are not allowed.

Required variables must be configured without printing or committing values:

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `AVALA_AI_RUNTIME_MODE`
- `AVALA_PROVIDER_SECRET_GROQ_SMOKE` or `AVALA_PROVIDER_SECRET_GEMINI_SMOKE`
- `DATABASE_URL`, only if local seed or verification requires direct database access

Environment rules:

- Do not print variable values.
- Do not commit variable values.
- Do not paste production, shared, customer, provider, service-role, or database credentials into repository files.
- Rotate any credential pasted into chat.
- `AVALA_AI_RUNTIME_MODE` must be exactly `pilot` or `production`.
- Prefer `pilot` for non-production deployed smoke unless AP/ChatGPT explicitly approves a production-mode non-production smoke.
- Provider secret variables must use only the approved smoke reference names:
  - `AVALA_PROVIDER_SECRET_GROQ_SMOKE`
  - `AVALA_PROVIDER_SECRET_GEMINI_SMOKE`
- `DATABASE_URL` must point only to the approved non-production target and must never be printed.

## Deployed Smoke Strategy

M3.2q recommends mock-first deployed smoke for M3.2r.

The future M3.2r execution path should:

- Use fake provider secret values only.
- Use seeded non-production governance rows.
- Verify blocked responses without provider calls.
- Verify allowed mocked paths only if the deployed provider call can be safely intercepted or mocked.
- Record only sanitized operation, status, correlation, classification, and cleanup evidence.

Blocked deployed scenarios are safe to run first because the resolver should fail closed before provider secret resolution or provider handoff. They should prove that deployed Edge responses remain sanitized and do not expose resolver internals, secrets, raw prompts, provider payloads, or raw database errors.

Allowed deployed scenarios remain conditional. If deployed Edge Functions cannot mock provider calls without code changes, M3.2r must block live deployed allowed smoke until AP approves one of these paths:

- Explicit live provider smoke using a named non-production target, dedicated low-risk real provider key, and network-call approval.
- A controlled mock switch or provider-call interception mechanism approved in a separate milestone.

M3.2r must not edit Edge Function entrypoints, `_shared/ai.ts`, runtime resolver/provider integration files, schema, UI, Health, browser fallback, or package dependencies just to enable deployed mocking.

## Seed And Cleanup Plan

M3.2r should reuse the M3.2p seed design:

- One smoke auth user/profile mapping.
- Separate smoke organizations for allowed and blocked cases.
- One active membership and compatible role per smoke organization.
- Smoke `ai_provider_key_refs` rows for active, disabled-config, manual-placeholder, wrong-prefix, and missing-policy contexts.
- Smoke `ai_provider_configs` rows with the approved runtime mode and target operations where applicable.
- Smoke `ai_workspace_provider_policies` rows for allowed operations only where required.
- Smoke `ai_provider_audit_events` rows written by deployed resolver paths.

Local validation should prefer one transaction with rollback, matching M3.2p. For deployed smoke, transaction rollback may not be available across HTTP Edge Function invocations. M3.2r must decide before execution whether deployed smoke rows can be rolled back or must be temporarily retained.

If rows must be retained during deployed smoke, use only these deterministic M3.2q row identifiers and source markers:

- Actor/profile ID: `00000000-0000-4000-8000-0000000032e0`
- Allowed org ID and slug: `10000000-0000-4000-8000-0000000032e0`, `m3-2q-smoke-allowed`
- Missing-policy org ID and slug: `10000000-0000-4000-8000-0000000032e1`, `m3-2q-smoke-missing-policy`
- Disabled-config org ID and slug: `10000000-0000-4000-8000-0000000032e2`, `m3-2q-smoke-disabled-config`
- Manual-placeholder org ID and slug: `10000000-0000-4000-8000-0000000032e3`, `m3-2q-smoke-manual-key-ref`
- Wrong-prefix org ID and slug: `10000000-0000-4000-8000-0000000032e4`, `m3-2q-smoke-wrong-prefix`
- Key-ref ID range: `20000000-0000-4000-8000-0000000032e0` through `20000000-0000-4000-8000-0000000032e4`
- Provider-config ID range: `30000000-0000-4000-8000-0000000032e0` through `30000000-0000-4000-8000-0000000032e4`
- Role ID range: `40000000-0000-4000-8000-0000000032e0` through `40000000-0000-4000-8000-0000000032e4`
- Safe source marker for metadata and evidence references: `m3.2q_deployed_edge_smoke`

Cleanup steps for retained rows must run after smoke and before evidence is accepted:

1. Verify the target is still the approved non-production environment without printing connection values.
2. Count smoke audit rows by `evidence_ref` or source marker and record only the count.
3. Delete smoke `ai_provider_audit_events` rows for the M3.2q evidence reference or source marker.
4. Delete smoke `ai_workspace_provider_policies` rows linked to the M3.2q org IDs and provider-config IDs.
5. Delete smoke `ai_provider_configs` rows in the M3.2q provider-config ID range.
6. Delete smoke `ai_provider_key_refs` rows in the M3.2q key-ref ID range.
7. Delete smoke `organization_members` rows for the M3.2q actor/profile ID and org IDs.
8. Delete smoke `roles` rows in the M3.2q role ID range.
9. Delete smoke `organizations` rows with the M3.2q org IDs and slugs.
10. Delete the smoke `profiles` row for the M3.2q actor/profile ID.
11. Delete the smoke `auth.users` row for the M3.2q actor/profile ID only if M3.2r created it.
12. Recount all M3.2q smoke identifiers and record only zero-count confirmation.

Seed and cleanup constraints:

- No customer data.
- No production or shared users.
- No real source documents.
- No real provider keys in any database row.
- No raw provider secret values in `ai_provider_key_refs`.
- No `secret_ref` raw values in evidence beyond the approved naming pattern.
- No full database row dumps in evidence.
- Stop if schema changes, migrations, RLS changes, or customer data would be required.

## Smoke Scenarios

M3.2r must run blocked deployed scenarios before any allowed deployed scenario.

Blocked deployed scenarios:

- Missing or invalid runtime mode blocks.
- Missing policy blocks.
- Disabled config blocks.
- `manual_placeholder` key reference blocks.
- Wrong provider-prefix `secret_ref` blocks.

Each blocked deployed scenario must verify:

- No provider API call occurs.
- No provider secret is resolved unless the scenario explicitly requires a safe fake reference resolution after all gates pass.
- No raw key material is returned or logged.
- The response is sanitized.
- Any audit event is sanitized and written only when safe resolver context exists.

Allowed deployed scenarios:

- `generate_document` mocked allowed path.
- `refine_section` mocked allowed path.
- `test_provider_connection` mocked allowed path.

Allowed deployed scenarios may run only when provider handoff can be safely mocked or intercepted without code changes or when AP has approved a controlled mock switch. Live provider smoke is deferred unless AP explicitly approves real provider key use and network calls for the named non-production target.

## Safe Response Expectations

Blocked responses may include only:

- `error` or `message`
- `correlationId`
- `safeUiMessageCategory`
- `retryCategory`
- `failureClass`

Blocked responses must not include:

- `secret_ref`
- Raw provider key.
- Auth headers.
- Service-role key.
- Policy rows, config rows, or key-reference rows.
- Raw prompt.
- Raw completion.
- Raw provider payload.
- Stack trace.
- Database error text.
- Supabase/PostgREST raw response text.
- `DATABASE_URL`.

Allowed mocked responses must also be captured safely. Evidence may record operation name, HTTP status, sanitized classification, correlation ID presence, and whether the mocked provider boundary was reached. Evidence must not record raw completions, provider payloads, prompts, keys, headers, database URLs, service-role values, or full resolver rows.

## Audit Verification Plan

M3.2r must verify `ai_provider_audit_events` with safe row-count and sanitized metadata checks only.

Audit verification must confirm:

- Allowed decisions write sanitized audit events before provider handoff.
- Blocked decisions write sanitized audit events only when safe resolver context exists.
- No raw prompts, completions, provider payloads, auth headers, service-role keys, `DATABASE_URL`, provider keys, raw `secret_ref` values, stack traces, or raw database errors appear in audit metadata.
- Row counts and cleanup counts are safe to record.
- Correlation IDs are present and non-secret.
- Provider, operation, mode, status, failure class, and safe policy result are recorded only as sanitized evidence.

Audit verification must not dump full database rows into evidence.

## Stop Conditions

Stop future M3.2r execution if any condition is met:

- Target is not clearly non-production.
- Credentials are missing or unsafe.
- `AVALA_AI_RUNTIME_MODE` is not `pilot` or `production`.
- Deployed function cannot mock provider handoff safely.
- Live provider call would be required without AP approval.
- Seed or cleanup cannot be proven.
- Response leaks sensitive data.
- Audit metadata contains prohibited content.
- Scanner has forbidden or stale hits.
- Schema, RLS, migration, UI, Health, browser fallback, scoring, prompt-version, dependency, usage, or cost changes are required.
- Edge Function entrypoint, `_shared/ai.ts`, or runtime resolver/provider integration edits are required.

## Acceptance Criteria

M3.2q is accepted when:

- This planning doc exists and is limited to future deployed non-production Edge smoke.
- The evidence file confirms only the two requested docs changed.
- No deploy, smoke execution, provider call, real key use, secret printing, database seed, runtime, schema, migration, Admin UI, product UI, Health, browser fallback, scoring, prompt-version, usage/cost, dependency, or compliance-claim change is introduced.
- Required verification commands and scan reviews are recorded.
- M3.2r stop conditions and AP approval gates are explicit.
