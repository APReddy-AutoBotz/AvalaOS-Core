# M3.1a Mode Contract + Fail-Closed Orchestrator Plan

## Goal

Implement the first M3.1 slice by adding a client/UI/orchestrator AI runtime mode contract and making pilot/production fail closed when server-side Edge AI is unavailable.

M3.1a is a focused implementation slice. It is not full browser AI cleanup and does not implement Edge provider configuration, key-reference resolution, static scans, Admin AI Controls, or provider removal.

## Scope

- Add a pure AI runtime mode helper.
- Add focused AI mode boundary tests.
- Wire Docs generation/refinement orchestration through the mode helper.
- Fail closed for invalid mode, missing deployed/non-dev mode, and pilot/production without Edge AI.
- Preserve existing local-demo/internal-dev browser fallback only as clearly demo/dev behavior.
- Add approved Vite env typings for `VITE_AVALA_AI_MODE` and `VITE_AI_EDGE_FUNCTIONS_ENABLED`.
- Add approved package script-only `test:ai-boundary` and include it in `npm run test`.
- Record implementation evidence and verification results.

## Current Mode / Env Findings

- No AI runtime mode helper existed before M3.1a.
- Current AI Edge routing used `VITE_AI_EDGE_FUNCTIONS_ENABLED === 'true'` and Supabase configuration.
- `services/aiOrchestrator.ts` used Edge when enabled and otherwise fell back to browser provider execution.
- `services/geminiService.ts` still creates browser Gemini/Groq providers and reads Vite-exposed provider key variables; M3.1a gates pilot/production before this path.
- `vite-env.d.ts` declared `VITE_GEMINI_API_KEY` only; M3.1a adds only the approved optional mode/Edge typings.
- `package.json` had no focused AI boundary test script.

## Mode Helper Design

`services/aiMode.ts` exports:

- `AiRuntimeMode`: `local-demo | internal-dev | pilot | production`
- `AiModeBoundaryErrorCode`:
  - `AI_BOUNDARY_INVALID_MODE`
  - `AI_BOUNDARY_MODE_REQUIRED`
  - `AI_BOUNDARY_EDGE_REQUIRED`
- `AiModeBoundaryError` with stable `code` and user-safe message:
  - `Server-side AI is not configured for this workspace/environment. Contact an administrator or use an approved demo path.`
- `resolveAiMode({ configuredMode, isDev, isProd })`
- `getAiExecutionPolicy({ modeResolution, edgeEnabled })`

Rules:

- Explicit valid mode always wins.
- Local Vite/dev may default to `local-demo` only when no mode is set.
- Deployed/non-dev without explicit mode fails closed.
- Invalid mode fails closed.
- Pilot and production require Edge AI.
- `local-demo` and `internal-dev` may use browser fallback only as demo/dev behavior.
- Client-side mode gating is only a UI/orchestrator guard, not the final security boundary.
- Server-side Edge enforcement remains mandatory for pilot/production.

## Orchestrator Changes

- Resolve mode at the start of generation and refinement using:
  - `import.meta.env.VITE_AVALA_AI_MODE`
  - `import.meta.env.DEV`
  - `import.meta.env.PROD`
- Build execution policy from resolved mode and `isAiEdgeEnabled()`.
- If policy is blocked, throw `AiModeBoundaryError`.
- If policy uses Edge, call the existing Edge path.
- If policy allows browser fallback, keep the existing provider fallback path and log clear demo/dev-only context.
- Do not remove provider classes, fallback order, user API key props, provider config/key-reference behavior, or `setup-api-key.js`.

## Test Plan

- `services/aiMode.test.ts` covers:
  - valid modes resolve exactly
  - invalid mode fails closed
  - dev/no explicit mode resolves `local-demo`
  - prod/non-dev/no explicit mode fails closed
  - pilot with Edge disabled fails closed
  - production with Edge disabled fails closed
  - pilot with Edge enabled requires Edge path
  - production with Edge enabled requires Edge path
