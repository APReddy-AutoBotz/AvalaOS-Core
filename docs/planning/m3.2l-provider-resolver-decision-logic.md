# M3.2l Provider Resolver Decision Logic

## Milestone Purpose

M3.2l implements server-side provider resolver decision logic only. It creates pure resolver modules and tests that prove allow/block decisions and fail-closed gates without integrating with existing Edge Functions, resolving real secrets, writing audit rows, or calling providers.

This milestone follows AP/ChatGPT review of the earlier M3.2l plan. The approved amendment narrows M3.2l to decision logic only.

## Approved Scope

M3.2l may:

- create `supabase/functions/_shared/providerResolver.ts`
- create `supabase/functions/_shared/providerResolver.test.ts`
- create `supabase/functions/_shared/providerResolverAudit.ts` as an audit shell builder only
- create `supabase/functions/_shared/providerSecretAdapter.ts` as a future-lookup eligibility stub only
- implement allow/block resolver decisions
- implement fail-closed gates
- implement sanitized audit event shell creation
- implement tests proving fail-closed behavior
- update package scripts only as needed for tests
- create M3.2l planning and evidence docs

## Explicit Non-Scope

M3.2l must not:

- edit `supabase/functions/_shared/ai.ts`
- edit `supabase/functions/ai-generate-document/index.ts`
- edit `supabase/functions/ai-refine-section/index.ts`
- edit `supabase/functions/ai-provider-test-connection/index.ts`
- call `Deno.env.get`
- resolve real secrets
- return raw provider key material
- hand off to provider calls
- integrate into existing runtime Edge Functions
- write rows to `ai_provider_audit_events`
- call Gemini or Groq
- change schema, runtime, Admin UI, product UI, Health, scoring, browser fallback, package dependencies, prompt versions, usage/cost behavior, or compliance language

If implementation requires any excluded item, the milestone must stop instead of expanding scope.

## Implemented Decision Contract

The resolver:

- accepts injected dependencies for membership, policy, config, and key-reference lookups
- supports providers `gemini` and `groq`
- supports modes `pilot` and `production`
- supports operations `generate_document`, `refine_section`, and `test_provider_connection`
- uses `org_id` as the tenant boundary
- treats `workspace_id` as optional evidence metadata only
- matches `ai_workspace_provider_policies.allowed_roles` against existing role names or role IDs supplied by the injected membership context
- validates active policy, active config, allowed mode, allowed operation, active key reference, non-expired key reference, and scanner classification input
- blocks `manual_placeholder` and `external_secret_reference`
- never includes `secret_ref`, raw provider keys, auth headers, raw prompts, raw completions, provider payloads, or service-role keys in resolver output

The audit helper:

- builds an `ai_provider_resolver_decision` shell only
- rejects prohibited metadata keys and raw-looking secret/auth values
- does not write to the database

The secret adapter stub:

- validates that an allowed decision is eligible for future lookup
- rejects blocked decisions
- rejects non-`server_reference` resolver types
- rejects any decision object that attempts to carry a secret-reference field
- does not call `Deno.env.get`, read a secret manager, resolve secrets, or return raw key material
