# M5.3a RLS Policy Implementation And Isolation Test Plan Refresh

## Purpose

M5.3a refreshes the RLS implementation plan now that the applied migration chain has tenant authority and the first artifact authority slices. This is a docs-only planning milestone. It does not implement RLS policies, helper functions, tests, runtime adapters, or schema changes.

The refresh narrows future RLS implementation to applied authority tables only, adds nullable workspace hardening, and defines the isolation fixture and validation decisions needed before AP approves any RLS implementation slice.

## Strict Non-Scope

M5.3a does not:

- Create SQL migrations, RLS policy SQL, helper functions, tests, seed data, backfills, or constraint validation.
- Edit existing migrations or schema contracts.
- Change runtime adapters, UI, routing, view keys, Sidebar behavior, demo data, scoring, provider execution, export behavior, deployment config, CI, package dependencies, Reports/Teams/Admin behavior, or the separate Health repository.
- Connect to remote Supabase, run remote validation, run `scripts/applySupabaseMigrations.mjs`, run `npm run db:migrate`, or run `npm run test:rls:assess`.
- Read, print, summarize, copy, or expose `.env` values or secrets.
- Claim tenant isolation is implemented or hosted pilot readiness is complete.

## Explicit Exclusions

M5.3a does not secure legacy, draft, derived, or non-authority surfaces:

- `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, and `timesheet_entries`.
- Draft org-scoped policy files under `docs/schema`.
- Storage/export records, Delivery Pack records, Monitor signal records, lineage authority, and handoff authority.
- Provider runtime execution and browser/provider secret behavior.

Those surfaces remain outside the first RLS implementation scope until AP approves their authority model and migration-chain status.

## Nullable Workspace Hardening

Workspace-scoped work artifacts must fail closed when `workspace_id IS NULL` unless AP explicitly approves an org-level legacy/readiness exception. This default applies to:

- `assess_processes`
- `assessments`
- `assessment_review_events`
- `projects`
- `document_generations`
- `delivery_work_items`

The future SELECT direction is: tenant artifact rows require active org membership, active workspace membership, matching `org_id`, matching `workspace_id`, non-deleted lifecycle status, and no cross-tenant source references. Broad org-readable fallback for nullable workspace rows is intentionally excluded from the default plan.

## Recommended Implementation Split

M5.3a should split implementation instead of attempting one broad policy milestone.

| Slice | Scope | Policy stance | Reason |
| --- | --- | --- | --- |
| M5.3a-1 | Helper functions, identity/org/workspace/membership SELECT policies, and isolation fixture/test contract | SELECT-only | Establishes membership visibility and helper behavior before artifact policies depend on it. |
| M5.3a-2 | SELECT policies for `assess_processes`, `assessments`, `assessment_review_events`, `projects`, `document_generations`, and `delivery_work_items` | SELECT-only | Adds workspace-aware artifact reads after helper behavior is proven. |
| M5.3a-3 | Mutation policies for create/update/soft-delete/review/approval/export-adjacent actions | Mutations denied until AP approval | Role and permission depth must be approved before writes become policy authority. |

The default M5.3a-1 stance is SELECT-only. It should not add broad `FOR ALL` policies, hard DELETE policies, review/approval mutations, delivery mutations, or export/storage access.

## Helper Function Strategy

Future implementation should use a minimal helper set:

- Required: `is_active_org_member(p_org_id uuid)`.
- Required: `is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`.
- Optional only with AP approval: `has_org_permission(p_org_id uuid, p_permission text)`.
- Optional only with AP approval: `has_workspace_permission(p_workspace_id uuid, p_org_id uuid, p_permission text)`.

Helper implementation must account for:

- Recursive RLS risk when helpers query membership tables protected by RLS.
- `SECURITY DEFINER` risk and the need for hardened `search_path`.
- `STABLE`, not `IMMUTABLE`, because membership state can change.
- Least privilege and execute grants only to intended authenticated roles.
- No service-role exposure to browser code.
- No helper result that leaks cross-tenant membership or role existence.

Direct policy joins may be acceptable for simple reads if they do not introduce recursion or fragile policy dependencies.

## Provider Governance Handling

Provider governance tables and existing policies are review-only in this refresh. M5.3a does not modify provider governance tables, provider key-reference behavior, provider audit behavior, or provider runtime execution.

Any provider-governance RLS hardening should be a separate AP-approved slice because provider metadata boundaries, workspace specificity, audit posture, and server-only secret handling require a different review surface.

## Isolation Test Strategy

Future tests must use deterministic, non-secret fixtures:

- Two organizations.
- Two workspaces per organization.
- Active member, disabled org member, disabled workspace member, non-member.
- Buyer Viewer, contributor, Delivery Lead, reviewer/admin where needed.
- Same-org/same-workspace records, same-org/cross-workspace records, and cross-org records.

Required negative tests:

- Anonymous user is denied tenant data.
- Authenticated non-member is denied tenant data.
- Disabled org member is denied org data.
- Disabled workspace member is denied workspace data.
- Cross-org reads and mutations are denied.
- Cross-workspace reads and mutations are denied.
- Nullable workspace rows fail closed unless AP approves an exception.
- Buyer Viewer mutation is denied.
- Contributor review/approval mutation is denied.
- Delivery mutation is denied unless AP approves role-aware mutation policies.
- Cross-tenant document, work item, source Assess, source document, evidence, or handoff references are denied.
- `metadata` and `source_lineage` cannot bypass tenant checks.
- Service-role behavior remains server-only and not browser-callable.

## Local Validation Decision

The repository does not currently provide a tracked local Supabase validation path for RLS proof. Future implementation should not claim verified tenant isolation unless AP approves a repeatable local-only validation method.

If a local validation path is still unavailable, M5.3a-1 should either:

- Create implementation SQL plus a test contract only, with verification limited to static review; or
- Remain blocked until a local-only validation path is approved.

Remote Supabase validation remains prohibited for this track unless AP changes the milestone scope.

## Runtime And Hosted Readiness Posture

RLS implementation alone does not make the hosted runtime ready. Current adapters may still target legacy draft tables, may require approved adapter changes, or may fail closed when RLS is enabled. Legacy draft tables remain outside the secured authority scope.

Hosted pilot readiness still requires later adapter, export, storage, deployment, environment, operational, backup/restore, and evidence/audit work. M5.3a is a policy implementation plan refresh only.

## Risks

- RLS helper recursion or overbroad `SECURITY DEFINER` behavior could weaken tenant isolation.
- Nullable workspace rows could create accidental org-wide reads if AP approves exceptions too broadly.
- Draft org-scoped docs/schema policies could be mistaken for applied secure policy authority.
- Runtime adapters may fail closed or target legacy tables until separate adapter work is approved.
- Provider governance policy hardening has different secret-boundary concerns and should not be folded into the first artifact RLS slice.
- Static review alone cannot prove tenant isolation without repeatable local RLS tests.
