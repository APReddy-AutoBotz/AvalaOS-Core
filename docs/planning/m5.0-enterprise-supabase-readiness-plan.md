# M5.0 Enterprise Supabase Readiness Plan

Milestone: M5.0 Enterprise Supabase Readiness Plan
Branch: milestone/m5.0-enterprise-supabase-readiness-plan
Status: Planning pack for secure hosted pilot readiness

## Purpose

M5.0 defines the enterprise Supabase readiness plan for moving AvalaOS Core from buyer-demo RC readiness toward secure hosted pilot readiness. It records the current repository posture, enterprise gaps, target architecture direction, sequencing, risks, and AP decisions required before implementation milestones begin.

This milestone is documentation-only. It does not make AvalaOS Core enterprise-production ready and does not implement live Supabase persistence, schema, RLS, runtime provider execution, deployment, or operations.

## Strict Non-Scope

M5.0 does not change product UI, routing, view keys, Sidebar behavior, guard semantics, permissions, demo data, scoring, schema, RLS, migrations, Supabase config, Supabase Edge/provider/runtime code, dependencies, deployment config, CI, task-ledger content, Reports/Teams/Admin behavior, M4.1g export policy, or any separate Health repository.

M5.0 does not deploy, call providers, connect to live Supabase, create migrations, alter schema, alter RLS, add dependencies, read or expose local secret values, commit personal hosted-project values, claim compliance certification, claim enterprise-production readiness, claim live provider readiness, claim automated approval authority, or assign deterministic scoring decisions to AI.

## M4.5 Boundary

M4.5 closed AvalaOS Core as buyer-demo RC ready for the AP Invoice Exception Workflow. That means the demo journey is ready for controlled buyer demonstration and evidence review.

Buyer-demo RC ready does not mean enterprise-production ready. Enterprise-production readiness remains pending until secure hosted persistence, authenticated tenancy, RLS, audit, export control, provider-secret governance, deployment separation, and pilot operations are implemented and verified in later milestones.

## Enterprise Gaps

- Persistence gaps: Demo/browser and mock persistence remain authoritative for key flows; hosted records for evidence, lineage, handoffs, generated artifacts, delivery packs, review decisions, and export artifacts need explicit ownership and retention.
- RLS gaps: Existing schema/RLS contracts need enterprise review, role mapping, workspace authority, deny-by-default regression tests, and hosted validation before pilot use.
- Audit gaps: Current audit surfaces are partial. Export actions, provider resolution, handoff transitions, evidence changes, admin/config changes, and artifact access need consistent append-only audit coverage.
- Export-control gaps: Browser downloads and derived review packs are not enough for pilot governance. Export review, private storage, signed access, retention, revocation, and artifact metadata policy are pending.
- Provider-secret gaps: Server-side provider execution direction exists, but M5 must remove pilot reliance on browser fallback and require key references, service boundaries, safe logging, and environment verification.
- Deployment gaps: Hosted project topology, deployment target, runbooks, rollback, backup/restore, smoke checks, and pilot support ownership are not yet approved.
- Environment-separation gaps: Local, development, staging, and production separation is not yet governed with explicit project ownership, allowed values, replacement rules, and promotion checks.

## Target Architecture

- Local, development, staging, and production separation: Each environment must have distinct Supabase projects or approved isolated equivalents, separate secrets, separate storage, separate audit evidence, and explicit promotion rules.
- Supabase Auth: Hosted pilot identity should use Supabase Auth as the user authority, with profile linkage and no demo user persistence treated as pilot authority.
- Organization, workspace, user, and role model: Organizations are tenant authorities. Workspace authority must be explicitly designed before it is enforced. Users receive memberships and roles through server-governed records, not through browser-only state.
- RLS-first data access: Client reads and writes go through authenticated Supabase sessions and RLS. Service-role access is server-only for Edge/admin operations. Policies must be deny-by-default and regression tested by role, organization, workspace, and artifact type.
- Evidence and lineage persistence: Assessment evidence, review events, handoff ledger entries, document generations, delivery source-lineage metadata, delivery packs, and export artifacts need durable Supabase-backed records with retention and ownership.
- Audit event model: Enterprise audit events should cover authentication-sensitive actions, evidence changes, review transitions, handoffs, provider resolution, exports, artifact reads, admin/config changes, and policy-denied attempts where appropriate.
- Export controls: Export artifacts should use private storage, org/workspace-scoped paths, short-lived signed access, artifact metadata, review status, and audit events. Browser-only downloads remain demo behavior unless policy authorizes them for a specific local context.
- Server-side provider execution direction: Pilot and production provider calls must execute through server-side Supabase Edge functions or an approved server runtime using key references, not browser provider secrets.
- Config and environment strategy: Public client config names may be documented; secret values must live outside source control. Personal hosted values may unblock temporary local validation but must be replaceable by AP-approved environment ownership.
- Backup and restore expectations: Hosted pilot readiness requires restore objectives, backup validation, export/artifact retention expectations, and runbook evidence before pilot users depend on the system.

## Proposed M5 Sequence

1. M5.0 Enterprise Supabase Readiness Plan: Create the planning, architecture, secret hygiene, and evidence pack. No implementation.
2. M5.1 Environment Config And Secret Hygiene Gate: Approve environment separation, secret naming, no-secret verification, and personal-value replacement rules.
3. M5.2 Supabase Auth / Organization / Workspace Schema Plan: Define Auth, organization, workspace, membership, role, and tenant boundary design before schema implementation.
4. M5.3 RLS Policy Design And Test Plan: Specify deny-by-default policies, role matrices, regression tests, and hosted validation gates.
5. M5.4 Evidence, Lineage, And Audit Persistence Plan: Define durable records, append-only audit events, retention, evidence ownership, and lineage traceability.
6. M5.5 Export Controls And Review Artifact Policy Plan: Define private storage, artifact metadata, signed access, review status, retention, and export audit.
7. M5.6 Deployment Readiness And Pilot Operations Pack: Define deployment target, smoke checks, backup/restore, rollback, runbooks, support, and pilot readiness evidence.

## Risks

- Existing schema and Edge sources may not match final enterprise tenant and workspace authority without additional design.
- Demo/browser persistence can create false confidence if treated as hosted pilot evidence.
- Provider fallback code must remain strictly local-demo/internal-dev and cannot be considered pilot-ready.
- Export artifacts can expose sensitive business context if storage, review, signed access, and retention policy are incomplete.
- Audit gaps can weaken evidence trails for regulated buyer conversations.
- Personal hosted project values may help temporary validation but can create drift unless M5.1 makes replacement explicit.
- Historical naming or legacy defaults may appear in existing control files; M5 must not expand those identities or treat them as current product identity.

## AP Decisions Needed

- Approve local, development, staging, and production environment ownership.
- Decide whether pilot uses one hosted project per environment or another approved isolation strategy.
- Confirm organization and workspace authority before schema work.
- Confirm role names, permission boundaries, and escalation behavior for pilot users.
- Define evidence, lineage, audit, export, and generated-artifact retention expectations.
- Approve export review status model, signed access lifetime, and artifact revocation policy.
- Approve provider allowlist, key-reference naming convention, and server-only execution boundary.
- Approve deployment target, rollback owner, backup/restore objectives, and pilot support owner.
- Confirm whether any legacy naming or bucket defaults must be remediated before pilot.
