# M3.2h Provider Config Schema/RLS Migration

## Summary

M3.2h implements the AP-authorized provider configuration governance schema/RLS migration from M3.2g. Scope is limited to one schema/RLS migration and this planning record. It does not add provider calls, Supabase Functions, Admin UI writes, runtime behavior, product-code behavior, Health changes, scoring changes, browser fallback behavior, prompt version changes, package changes, seeds, provider keys, or unsupported compliance claims.

Actual migration file: `supabase/migrations/20260607150600_m3_2h_provider_config_governance.sql`

Hardening note: AP review rejected treating `docs/schema/provider_config_governance.sql` as the final migration artifact. Repository inspection found existing SQL schema references under `docs/schema`, including pre-existing `ai_provider_configs` in `docs/schema/ai_governance_jobs.sql`, and no committed `supabase/migrations` directory before this hardening change. This PR now uses a Supabase CLI-style migration file under `supabase/migrations` and removes `docs/schema/provider_config_governance.sql` from the PR.

Hardening note 2: AP review required the actual schema to match the approved M3.2h blueprint. The final migration provider list is narrowed to `gemini` and `groq` only. It does not include `openai`, `azure_openai`, `anthropic`, or `other`.

## Exact first migration object list

1. `ai_provider_key_refs`
2. `ai_provider_configs`
3. `ai_workspace_provider_policies`
4. `ai_provider_audit_events`

No other provider governance tables are introduced in M3.2h.

## Actual Objects Created Or Altered

Created when absent:

- `ai_provider_key_refs`
- `ai_workspace_provider_policies`
- `ai_provider_audit_events`

Created when absent or altered when pre-existing:

- `ai_provider_configs`

Pre-existing object discovered:

- `docs/schema/ai_governance_jobs.sql` already defines `ai_provider_configs` with `org_id`, `provider`, `display_name`, `key_reference`, `default_model`, `model_policy`, `status`, creator/timestamp fields, member SELECT RLS, and supporting indexes/triggers.

M3.2h does not blindly drop or recreate that prior table. The migration uses `CREATE TABLE IF NOT EXISTS` plus idempotent `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, safe constraint/policy replacement, and indexes for the missing M3.2h governance fields.

M3.2h additions to pre-existing `ai_provider_configs` where missing:

- nullable `key_ref_id` FK to `ai_provider_key_refs(id)` with `ON DELETE SET NULL`
- provider constraint narrowed to `gemini` and `groq`; when a prior table exists, this is added as `NOT VALID` so prior-context rows are not blindly dropped during migration
- `allowed_modes`
- `allowed_operations`
- `evidence_ref`
- `updated_by`
- `deleted_at`
- pending-review default for `status`
- status, display-name, allowed-mode, and allowed-operation constraints
- active non-deleted uniqueness for `(org_id, provider, display_name)`
- member SELECT policy narrowed to non-deleted rows

M3.2h also makes the prior-context `key_reference` column nullable when present and documents it as deprecated. It does not authorize raw provider keys, encrypted provider keys, auth headers, browser-readable secret values, or secret-manager payload values in that column.

## Key Implementation Decisions

- `org_id` is the enforceable tenant boundary for M3.2h because the repository has no AP-approved canonical `workspaces` authority.
- `ai_workspace_provider_policies` keeps the future-compatible workspace naming, but it does not include `workspace_id` in the final policy shape or uniqueness constraints.
- `ai_provider_audit_events.workspace_id` remains nullable and has no strict FK in this migration.
- `ai_provider_key_refs` is created before `ai_provider_configs`.
- `ai_provider_key_refs` has no FK back to `ai_provider_configs`, avoiding circular FK complexity.
- `ai_provider_configs.key_ref_id` is a nullable FK to `ai_provider_key_refs.id` with `ON DELETE SET NULL`.
- `ai_provider_configs` may already exist from `docs/schema/ai_governance_jobs.sql`; M3.2h adds only missing approved columns, constraints, indexes, comments, and RLS policy changes.
- Final provider list: `gemini`, `groq`.
- `secret_ref` means a service-only reference identifier. It must never contain raw provider keys, encrypted provider keys, auth headers, browser-readable secret values, or secret-manager payload values.
- `ai_provider_configs` must not contain raw secrets, encrypted secrets, auth headers, browser-readable secret values, or secret-manager payload values.
- The existing prior-context `ai_provider_configs.key_reference` column is treated as deprecated where present and is made nullable so future server/control workflows can use `key_ref_id`.

## Table Blueprint

### `ai_provider_key_refs`

Purpose: service-only registry for provider key reference identifiers.

Ownership model: tenant-scoped by `org_id`; service/control workflows only may manage records after AP approval.

Required columns: `id`, `org_id`, `provider`, `resolver_type`, `secret_ref`, `status`, `created_at`, `updated_at`.

Optional/deferred columns: `safe_label`, `safe_fingerprint`, `rotation_status`, `last_rotated_at`, `expires_at`, `created_by`, `updated_by`, `deleted_at`.

Sensitive fields prohibited: raw provider keys, encrypted provider keys, auth headers, bearer tokens, browser-readable secrets, secret-manager payload values, raw prompt bodies, raw completion bodies.

Expected indexes: org/provider/status lookup index and active safe-fingerprint uniqueness index.

Expected uniqueness constraints: active non-deleted uniqueness for `(org_id, provider, safe_fingerprint)` when `safe_fingerprint` is present.

Expected foreign-key relationships: `org_id` to `organizations(id)`; optional actor references to `profiles(id)`; no FK to `ai_provider_configs`.

Soft-delete or status strategy: status supports `pending_review`, `active`, `disabled`, `expired`, and `retired`; `deleted_at` supports soft-delete.

Audit/evidence role: referenced by audit events without exposing `secret_ref` to browser/client reads.

### `ai_provider_configs`

Purpose: provider configuration governance metadata and approved key-reference linkage.

Ownership model: tenant-scoped by `org_id`; readable by active org members; writes deferred to AP-approved server/control workflows.

Required columns: `id`, `org_id`, `provider`, `display_name`, `model_policy`, `allowed_modes`, `allowed_operations`, `status`, `created_at`, `updated_at`.

Optional/deferred columns: `key_ref_id`, `default_model`, `evidence_ref`, `created_by`, `updated_by`, `deleted_at`.

Sensitive fields prohibited: raw secrets, encrypted secrets, auth headers, bearer tokens, browser-readable secret values, secret-manager payload values, raw prompt bodies, raw completion bodies.

Expected indexes: key-reference lookup index and org/status lookup index.

Expected uniqueness constraints: active non-deleted uniqueness for `(org_id, provider, display_name)`.

Expected foreign-key relationships: `org_id` to `organizations(id)`; nullable `key_ref_id` to `ai_provider_key_refs(id)`; optional actor references to `profiles(id)`.

Soft-delete or status strategy: status supports `pending_review`, `active`, `disabled`, and `retired`; `deleted_at` supports soft-delete.

Audit/evidence role: config rows are the policy target and are referenced by audit events.

### `ai_workspace_provider_policies`

Purpose: future-compatible workspace-named policy table for provider use constraints.

Ownership model: tenant-scoped by `org_id`; no `workspace_id` FK or workspace-specific uniqueness is introduced in M3.2h.

Required columns: `id`, `org_id`, `provider_config_id`, `operation`, `mode`, `allowed_roles`, `is_default`, `status`, `created_at`, `updated_at`.

Optional/deferred columns: `created_by`, `updated_by`, `deleted_at`.

Sensitive fields prohibited: raw provider keys, encrypted keys, auth headers, browser-readable secret values, secret-manager payload values, raw prompt bodies, raw completion bodies.

Expected indexes: org/status lookup index and provider-config lookup index.

Expected uniqueness constraints: one active default per `(org_id, operation, mode)` where `is_default = true` and `deleted_at IS NULL`; active non-deleted uniqueness for `(org_id, provider_config_id, operation, mode)`.

Expected foreign-key relationships: `org_id` to `organizations(id)`; `provider_config_id` to `ai_provider_configs(id)`; optional actor references to `profiles(id)`.

Soft-delete or status strategy: status supports `pending_review`, `active`, `disabled`, and `retired`; `deleted_at` supports soft-delete.

Audit/evidence role: policy rows document role-allowed provider use by operation and mode for future server/control enforcement.

### `ai_provider_audit_events`

Purpose: provider governance audit and evidence event table.

Ownership model: tenant-scoped by `org_id`; browser/client member reads are deferred.

Required columns: `id`, `schema_version`, `event_type`, `org_id`, `provider`, `status`, `metadata`, `created_at`.

Optional/deferred columns: `workspace_id`, `provider_config_id`, `key_ref_id`, `operation`, `mode`, `policy_result`, `failure_class`, `actor_id`, `service_context`, `correlation_id`, `evidence_ref`.

Sensitive fields prohibited: raw provider keys, encrypted keys, auth headers, bearer tokens, browser-readable secret values, secret-manager payload values, raw prompt bodies, raw completion bodies.

Expected indexes: org/created-at lookup index, provider-config lookup index, key-reference lookup index, and correlation-id lookup index.

Expected uniqueness constraints: none.

Expected foreign-key relationships: `org_id` to `organizations(id)`; nullable `provider_config_id` to `ai_provider_configs(id)`; nullable `key_ref_id` to `ai_provider_key_refs(id)`; optional `actor_id` reference to `profiles(id)`; no strict `workspace_id` FK.

Soft-delete or status strategy: no soft-delete column; audit rows are treated as append-only evidence records by future workflow controls.

Audit/evidence role: records provider governance decisions and safety metadata without storing raw secret or prompt/completion content. `metadata` is JSONB and blocks common prohibited raw secret, prompt, and completion field names.

## RLS Blueprint

- `ai_provider_key_refs`: RLS enabled. No browser/client member SELECT policy is added because `secret_ref` is sensitive service-only reference metadata. No broad authenticated write policies are added.
- `ai_provider_configs`: RLS enabled. Active org members may SELECT non-deleted config metadata through the existing `organization_members` pattern. The prior member-read policy is replaced safely with the non-deleted M3.2h policy. No browser/client write policies are added.
- `ai_workspace_provider_policies`: RLS enabled. Active org members may SELECT non-deleted policy metadata through the existing `organization_members` pattern. No browser/client write policies are added.
- `ai_provider_audit_events`: RLS enabled. Member reads are deferred for AP-approved audit redaction and evidence-view rules. No browser/client write policies are added.

## Service-Role Boundary

Service role remains server-side only and relies on Supabase service-role behavior. M3.2h does not introduce Supabase Functions or runtime workflows that exercise the service role. Future service/control workflows must not bypass mode, auth, membership, provider policy, provider status, operation, key-reference, prompt/version, evidence, or scanner gates.

## Key-Reference Safety

`secret_ref` is a service-only reference identifier. It is not a provider key, encrypted provider key, auth header, browser-readable value, or secret payload. The migration adds comments and basic check constraints for obvious unsafe token prefixes, but final secret-safety enforcement remains a future AP-approved server/control workflow responsibility.

Browser/client code must never directly write sensitive provider/key/audit objects or resolve provider keys. Future Admin UI writes, if approved later, must be mediated by AP-approved server/control workflows, RLS, audit evidence, and secret-safety gates.

## Rollback And Reversibility Plan

Rollback should disable or drop the M3.2h provider governance objects in reverse dependency order: `ai_provider_audit_events`, `ai_workspace_provider_policies`, M3.2h-only `ai_provider_configs` extensions/constraints/indexes/policies, and `ai_provider_key_refs`. Because `ai_provider_configs` existed in prior AI governance schema context, rollback must not blindly drop that table in an environment where prior AI job schema depends on it. Reversing the `key_reference` nullability/comment or restoring the prior unique constraint/member-read policy requires AP-approved DB review and must preserve unrelated prior-context provider/job data.

## AP Approval Gate

M3.2h is acceptable only if AP confirms the schema/RLS migration matches the M3.2g authorization, contains no provider secrets, creates no runtime provider behavior, and preserves Health/scoring/browser fallback boundaries.

## Stop Conditions

Stop and request AP approval if implementation would require runtime provider execution, Supabase Function changes, Admin UI writes, browser/client provider-key access, workspace FK authority, score or Health changes, prompt/version changes, seeds with provider keys, package changes, or unsupported compliance claims.

## Final Recommendation

Proceed with M3.2h as a schema/RLS-only migration. Carry forward AP risks around service/control workflow mediation, workspace authority, audit read redaction, and secret-safety runtime enforcement to the next approved milestone.
