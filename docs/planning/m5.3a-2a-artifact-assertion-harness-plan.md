# M5.3a-2a Artifact Assertion Harness Plan

## Purpose

M5.3a-2a plans an AP-reviewable local artifact assertion harness for the artifact SELECT policies delivered in M5.3a-2.

This milestone is docs-only. It does not implement an artifact runner, SQL changes, assertion SQL changes, package scripts, package dependencies, runtime behavior, UI behavior, provider behavior, remote validation, database connections, or local configuration.

## Current M5.3a-2 Closure Baseline

M5.3a-2 is closed with post-merge verification and tag target `982235876b78a72bf1a2082e73320b3e467e6207`.

Accepted baseline:

- Artifact SELECT policies exist on `main` for `assess_processes`, `assessments`, `assessment_review_events`, `projects`, `document_generations`, and `delivery_work_items`.
- Artifact assertion SQL exists at `docs/schema/m5.3a_2_artifact_select_isolation_assertions.sql`.
- No AP-approved artifact assertion runner exists yet.
- The existing M5.3a-1 harness remains scoped to identity and membership assertions.
- The M5.3a-1 harness must not be silently repurposed for artifact assertions without AP approval.
- Artifact SELECT isolation is not verified because executable local artifact assertions have not run and passed against an AP-approved reachable local database.

## Future Runner Strategy

Recommended implementation shape for the next milestone is a separate artifact runner first, not a generalized shared runner.

Future files to plan:

- `scripts/m5.3a-2-artifact-isolation-smoke.mjs`
- `docs/quality/m5.3a-2a-artifact-assertion-harness-implementation-evidence.md`
- `package.json` only if AP approves adding `test:rls:m5.3a-2`

Reasons to prefer a separate runner first:

- It keeps the M5.3a-1 identity and membership harness boundary intact.
- It avoids changing the already accepted M5.3a-1 command contract.
- It lets artifact-specific scenario counts, metadata checks, and static SQL guards mature without weakening the membership harness.
- It gives AP a clear review surface for artifact target safety before any later generalized runner is considered.

A generalized runner may be considered later only after both membership and artifact harness behavior are stable and AP approves command and output compatibility.

## Local Target Proof Rules

The future artifact runner must prove a local target before any database action.

It must fail closed before connecting when:

- No explicit local target proof is supplied.
- A hosted database URL is supplied.
- A remote host is supplied.
- A silent environment database target is present but not explicitly accepted.
- A `DATABASE_URL`-based automatic path is attempted.
- An ignored secret-file fallback would be required.
- Unknown target arguments are supplied.

The runner must not print:

- Target values.
- Database URLs.
- Environment values.
- Service-role values.
- Provider keys.
- JWTs or private tokens.
- Hosted identifiers.
- Row payloads.
- Claim values.
- Auth headers.

## Allowed Local Target Behavior

Allowed future local target proofs:

- `localhost`
- `127.0.0.1`
- Loopback-only aliases if AP approves them.
- Docker-local targets only when they can be proven to resolve to loopback behavior, or else fail closed.

Disallowed future target behavior:

- Hosted database URLs.
- Remote hosts.
- Implicit `DATABASE_URL` execution.
- Secret-file fallback.
- Any path that prints or records the target value.

## SQL Static Guard Plan

The future runner should statically inspect the artifact assertion SQL before execution.

Allowed assertion SQL patterns:

- `BEGIN`
- `ROLLBACK`
- `SET LOCAL` for transaction-scoped timeouts and session claims.
- `CREATE TEMP TABLE`
- Transaction-scoped fixture `INSERT` statements.
- `SELECT` assertions.
- `SET LOCAL ROLE authenticated`
- Session or role reset behavior that remains local and sanitized.

Blocked assertion SQL patterns unless AP explicitly approves a later exception:

- `COMMIT`
- `CREATE POLICY`
- `ALTER POLICY`
- `DROP POLICY`
- Durable `ALTER TABLE`
- Durable `CREATE TABLE`
- Durable `DROP TABLE`
- `CREATE FUNCTION`
- `CREATE EXTENSION`
- Durable object `GRANT` or `REVOKE`
- Fixture `INSERT`, `UPDATE`, or `DELETE` outside transaction-scoped setup.
- Service-role behavior.
- Provider calls.
- Any output of target or secret values.

Temporary-table changes are acceptable only inside the local assertion transaction.

## Output Contract

Future artifact runner output must be counts-only and sanitized.

Allowed output fields:

- Artifact scenario contract present: yes/no.
- Artifact assertion SQL present: yes/no.
- Local target gate status.
- Local DB assertion execution status.
- Scenarios declared.
- Scenarios executed.
- Pass count.
- Fail count.
- Blocked count.
- Artifact SELECT isolation verified: yes/no.
- Sanitized classifications.
- Local target proof status without target value.

Disallowed output:

- Target values.
- Database URLs.
- Environment values.
- Service-role values.
- Provider keys.
- JWTs or private tokens.
- Hosted URLs.
- Bucket names.
- Personal identifiers.
- Row payloads.
- Claim values.
- Auth headers.

## Assertion Compatibility Plan

The future artifact runner should use:

- `docs/schema/m5.3a_2_artifact_select_isolation_assertions.sql`

The runner should report non-zero when:

- Local target proof fails.
- The local database is unavailable.
- The assertion SQL file is missing.
- SQL static guard checks fail.
- Any scenario fails.
- Any approved scenario remains blocked.
- Scenario metadata checks fail.
- Cleanup or rollback checks fail.
- Output sanitization checks fail.

Artifact SELECT isolation may be reported as verified only when all AP-approved artifact scenarios execute and pass with sanitized output.

The existing artifact assertion SQL declares artifact scenarios, but M5.3a-2 closure did not execute those assertions.

## Local DB Availability Decision

M5.3a-2 closure did not verify artifact SELECT isolation. No AP-approved local artifact assertion runner executed against a reachable local database.

Future implementation evidence must support two paths:

- Executable proof path: AP-approved local target is reachable, applied migrations are available locally, artifact assertions execute and pass, output is sanitized, and local artifact SELECT isolation may be claimed for the six scoped artifact tables only.
- Static and fail-closed path: no AP-approved local database is reachable, the runner proves local-target safety and fail-closed behavior only, and artifact SELECT isolation remains not verified.

## Proof Boundary

M5.3a-2a plans the harness only.

A future harness implementation can prove:

- Artifact target-safety behavior.
- Assertion SQL discovery.
- Static SQL guard behavior.
- Fail-closed behavior.
- Counts-only sanitized output behavior.

It cannot prove artifact SELECT isolation unless executable artifact assertions run and pass against an AP-approved reachable local database with the M5.3a-2 artifact policies applied.

It does not prove hosted runtime readiness, production readiness, storage/export readiness, provider readiness, mutation safety, end-user workflow readiness, backup/restore readiness, operations readiness, broader certification, or legacy draft table isolation.

## AP Decisions Needed

AP must approve:

- Separate artifact runner versus generalized runner.
- Package script name, if any.
- Accepted local target forms.
- SQL static guard allow list.
- SQL static guard block list.
- Expected artifact assertion scenario count.
- Whether implementation may proceed if no reachable local database exists.
- Conditions for claiming local artifact SELECT isolation.
- Whether local Supabase configuration remains out of scope.

## Risks

- Reusing the M5.3a-1 harness too broadly could weaken its accepted identity and membership boundary.
- A target parser that accepts remote or implicit values could create unsafe validation behavior.
- SQL assertions that contain durable changes could mutate local state or obscure rollback behavior.
- Static review alone is insufficient to prove artifact SELECT isolation.
- Local verification does not prove hosted runtime readiness.
