# M5.2e Delivery Ownership Columns Migration Plan

Milestone: M5.2e Delivery Ownership Columns Migration Plan

Branch: milestone/m5.2e-delivery-ownership-columns-migration-plan

Status: Docs-only planning pack

Recommended next milestone: M5.2f Project Authority Migration Implementation

## Purpose

M5.2e defines the Delivery ownership migration strategy needed before durable Studio/Docs ownership, Delivery Pack persistence, Monitor persistence, export controls, and workspace-aware RLS can safely depend on Delivery/project authority.

This milestone records the current Delivery schema posture, chooses project authority as the next implementation slice, and defines ownership, FK, backfill, compatibility, and RLS-readiness direction. It does not implement schema.

## Strict Non-Scope

M5.2e does not create migrations, alter schema, implement SQL, implement RLS policies, change runtime adapters, change UI, change routing, change view keys, change Sidebar behavior, change guard semantics, change permissions, change demo data, change scoring, change provider runtime, change Supabase configuration, change CI, add dependencies, change deployment configuration, change Reports/Teams/Admin behavior, change export policy, seed data, backfill data, deploy, call providers, connect to hosted Supabase, or modify the separate Health repository.

M5.2e does not create `projects`, `epics`, `sprints`, `tasks`, work-item, Delivery Pack, Monitor signal, handoff, lineage, or export ownership tables.

M5.2e does not claim tenant isolation, hosted-pilot readiness, production readiness, live-provider readiness, certification status, automated approval authority, machine-owned deterministic decisions, or current AGI capability.

No `.env` values, Supabase tokens, provider keys, service-role values, personal hosted values, organization identifiers, storage names, or secrets are needed for this milestone.

## Current Dependency Finding

The applied migration chain currently has tenant authority, Assess ownership, and provider governance only.

No `projects`, `epics`, `sprints`, `tasks`, comments, activity, timesheets, Delivery Pack, Monitor signal, handoff, lineage, or export authority exists in `supabase/migrations`.

Delivery schema exists as draft SQL under `docs/schema`, plus runtime types, mock data, UI state, and adapters. The current draft Delivery tables are organization-scoped and do not have workspace authority.

Studio/Docs durable ownership depends on project authority because draft `document_generations` references `projects`. Since `projects` is not applied migration-chain authority, Studio/Docs ownership remains blocked until Delivery/project authority is planned and implemented.

Therefore M5.2e remains docs-only and recommends project authority first.

## Current Delivery Schema Inventory Summary

Draft schema defines `projects`, `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, and `timesheet_entries` with `org_id` and stable `app_id` direction. These tables are not applied migration-chain authority.

Runtime Delivery uses `deliveryAdapter` to target draft table names when Supabase is configured. The adapter maps UI IDs through `app_id`, filters by `org_id`, and stores task assignees, reporter, dependencies, ordering, user stories, and source lineage in task metadata.

Mock/demo Delivery data has one canonical AP delivery project, epics, sprints, and tasks. The mock tasks carry source lineage to Assess, Studio document generation, handoff ledger entries, evidence refs, and Delivery Pack ID.

Delivery Pack is a derived TypeScript read model built from project, tasks, document generations, Assess records, Govern-derived state, and handoff entries. It is not a database table.

Monitor project/task signals are derived from project/task state and widgets. They are not live telemetry tables.

Portfolio is currently a view over projects and task state. It is not a separate persisted project authority.

## Project Authority Recommendation

M5.2f should make `projects` the first Delivery authority table.

The project table should stay narrow: an AvalaOS governed delivery project under organization and workspace authority. It should not expand into a generic agile platform, portfolio replacement, or live telemetry model.

Project authority should unblock:

- future Studio/Docs ownership, because document generations need a stable project target;
- future work-item ownership, because epics, sprints, tasks, comments, activity, and timesheets need a project root;
- future Delivery Pack persistence decisions;
- future Monitor project/task signal ownership decisions;
- future workspace-aware RLS tests.

Epics, sprints, tasks, comments, activity, timesheets, Delivery Pack records, Monitor signals, handoff/lineage authority, and export ownership should remain deferred until project authority exists.

## Proposed Delivery Ownership Model

Future `projects` authority should plan these fields:

- `org_id`
- `workspace_id`
- `app_id`
- `name`
- `description`
- `owner_id`
- `delivery_lead_id`
- `created_by`
- `updated_by`
- `source_process_id`
- `source_assessment_id`
- `lifecycle_stage`
- `health_status`
- `status`
- `archived_at`
- `deleted_at`
- `audit_correlation_id`

Future work-item authority should use only needed fields:

- project and workspace ownership;
- assignee and reporter actors;
- source lineage refs;
- lifecycle/status fields;
- correlation fields.

Do not force every ownership field onto every Delivery table. Do not add export storage ownership or Monitor telemetry tables as part of Delivery project authority.

## FK And Backfill Strategy

Future implementation should add nullable ownership columns first. It should not add `NOT NULL` ownership requirements until AP approves workspace/project mapping and backfill.

Future project authority should reference:

- `organizations(id)`;
- composite `workspaces(id, org_id)` where same-tenant workspace enforcement is safe;
- `profiles(id)` for actor columns;
- M5.2c `assess_processes` and `assessments` for source refs where same-organization constraints can be enforced.

Future work-item authority should reference project authority after the project table exists. It should avoid a hard FK to `document_generations` until Studio/Docs becomes applied migration authority.

Use `NOT VALID` FKs where existing/manual rows may exist. Validate constraints only after AP-approved backfill. Do not seed data, commit hosted IDs, commit personal IDs, run broad updates, or perform destructive backfill.

## Migration Safety Strategy

M5.2e creates no migration.

Future M5.2f implementation should be additive only. It should avoid drops, renames, destructive updates, RLS policies, runtime dependencies, seed data, remote Supabase validation, and `scripts/applySupabaseMigrations.mjs` unless AP separately approves a local-only path.

Static SQL review should be required before any future project authority migration is merged.

## Buyer-Demo Compatibility Strategy

Existing buyer-demo behavior remains unchanged.

The mock/local Delivery flow continues to work. Runtime adapters do not depend on future ownership columns. AP delivery work items remain local/demo-backed until future persistence adapter changes are approved.

M5.2e does not change scoring, Decision Pack rendering, Govern copy, Studio handoff, Docs-to-Delivery lineage, document export, Delivery Pack, Monitor signals, burndown widget behavior, demo data, routing, or UI.

## RLS Readiness Mapping

Delivery ownership will support future RLS work by preparing:

- workspace-scoped project and work-item reads;
- cross-organization and cross-workspace denial;
- Buyer Viewer read-only access where approved;
- Delivery Lead update gates;
- contributor own-task update gates;
- reviewer and approver gates;
- source lineage and evidence boundary checks;
- Docs-to-Delivery handoff boundaries;
- Delivery Pack and Monitor boundaries after persistence models exist;
- export boundaries later under export controls.

M5.2e does not implement RLS policies or prove tenant isolation.
