# M5.3a-2b Artifact Local DB Execution Evidence Plan

## Purpose

M5.3a-2b plans how AP can later collect local database execution evidence for the M5.3a-2 artifact SELECT policies through the already implemented M5.3a-2a artifact assertion harness.

This milestone is docs-only. It does not run local DB assertions, connect to any database, change SQL migrations, change artifact policies, change assertion SQL, change harness behavior, change package scripts, create local configuration, or claim artifact SELECT isolation.

## Local DB Execution Objective

Future local execution evidence should prove only that:

- The artifact runner connects only after AP-approved local target proof passes.
- The local database has required M5.3a-1 helper/policy foundation and M5.3a-2 artifact policies applied.
- The artifact assertion SQL executes against the local target.
- All 18 approved artifact scenarios execute and pass.
- Output remains counts-only and sanitized.
- No database targets, row payloads, claim values, or secrets are printed.

If all conditions pass, future evidence may state: local artifact SELECT isolation verified for the scoped artifact tables and scenarios only.

## Local Environment Prerequisite Plan

Future execution requires AP approval for every local environment prerequisite:

- Local Supabase/Postgres-compatible target.
- Explicit target proof through `localhost` or `127.0.0.1`, unless AP separately approves another loopback-only target.
- Required schema state with migrations through M5.3a-2 applied locally.
- M5.3a-1 helper and membership SELECT policy foundation present locally.
- M5.3a-2 artifact SELECT policies present locally.
- No remote DB target.
- No hosted database URL.
- No `.env` fallback.
- No production data.
- No durable fixture data.
- No printed service-role values, provider keys, database URLs, row payloads, claim values, or fixture identifiers.

## Migration Application Boundary

M5.3a-2b does not run migrations or database commands.

Future local execution may require a separate AP-approved local migration or bootstrap step. That future evidence must record exactly how local schema was prepared without exposing target values or secrets.

Default boundary:

- Do not use remote Supabase.
- Do not use `scripts/applySupabaseMigrations.mjs` unless AP explicitly approves a local-only invocation later.
- Do not use `npm run db:migrate` unless AP explicitly approves a local-only path later.
- Do not silently use `DATABASE_URL`.
- Do not fall back to ignored local secret files.

## Evidence Output Contract

Future local execution evidence may include only:

- Command names.
- Sanitized local target status, not target value.
- Scenario counts.
- Executed count.
- Pass, fail, and blocked counts.
- Artifact SELECT isolation verified yes/no.
- Sanitized classifications.
- Runner exit-code category.
- No-secrets, no-target-value, and no-payload confirmations.

Future evidence must not include:

- DB URLs.
- Host values.
- Port values when considered target-sensitive.
- `.env` values.
- Service-role values.
- Provider keys.
- JWT or private token values.
- Hosted identifiers.
- Row payloads.
- Claim values.
- Auth headers.
- Fixture UUIDs.
- Organization, workspace, project, or user IDs.

## Future Execution Command Plan

Future AP-approved execution may use one of these local-target commands:

- `npm run test:rls:m5.3a-2 -- --local-target localhost`
- `node scripts/m5.3a-2-artifact-isolation-smoke.mjs --local-target localhost`

Future safety checks should include:

- No-target fail-closed run.
- Hosted-target rejection run.
- Localhost run only after AP explicitly approves local DB execution.

M5.3a-2b does not run local-target commands.

## Pass/Fail Decision Rule

Future evidence can say local artifact SELECT isolation is verified for scoped artifact tables only when:

- Local target proof passes.
- Local DB is reachable.
- Required schema and policies exist.
- Artifact assertion SQL executes.
- All 18 approved scenarios execute and pass.
- Output is sanitized.
- No secrets, targets, payloads, claim values, or fixture identifiers are printed.

Future evidence must say artifact SELECT isolation is not verified when:

- No local DB is reachable.
- Required schema or policies are missing.
- Assertions are blocked.
- Any scenario fails.
- Any approved scenario remains blocked.
- Output sanitization fails.
- Target proof fails.
- Hosted or remote target input is supplied.

## Proof Boundary

Passing local artifact assertions would prove only local artifact SELECT behavior for the six scoped artifact tables and approved scenarios.

It would not prove hosted runtime readiness, production readiness, mutation safety, storage/export readiness, provider readiness, runtime adapter readiness, backup/restore readiness, operations readiness, performance readiness, certification, legacy draft table isolation, or full tenant isolation.

## AP Decisions Needed

AP must approve:

- Whether to attempt local DB execution.
- Accepted local target form.
- How local migrations and schema are prepared.
- Whether any existing migration/bootstrap script may be used in local-only mode.
- Whether port, user, and database name can be passed or must remain default.
- What evidence may be captured.
- Whether successful local execution is enough to mark local artifact SELECT isolation verified for scoped tables only.
- Confirmation that successful local execution does not imply hosted, production, mutation, export, provider, runtime, operations, or certification readiness.

## Risks

- Local schema drift could produce misleading evidence.
- A local execution path that accepts implicit targets could accidentally leave the local-only boundary.
- Capturing too much output could expose database targets, claims, row payloads, or fixture identifiers.
- Static review and fail-closed harness output do not prove artifact SELECT isolation.
- Passing local assertions would still not prove hosted or production readiness.
