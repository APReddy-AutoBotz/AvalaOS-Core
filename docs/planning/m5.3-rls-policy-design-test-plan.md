## Purpose

M5.3 defines the future row-level security policy design and tenant-isolation test strategy for Supabase-hosted pilot readiness. It translates the M5.0 enterprise readiness plan, the M5.1 environment and secret-hygiene gate, and the M5.2 Auth / organization / workspace model into an implementation-ready RLS plan.

M5.3 is design and test planning only. It does not implement RLS, create SQL, create migrations, alter schema, change runtime behavior, connect to Supabase, or complete hosted pilot readiness.

## RLS Design Principles

- Deny by default for all tenant, workspace, evidence, artifact, audit, export, and provider-governance data.
- Use `auth.uid()` as the authenticated identity anchor.
- Require active organization membership for all tenant records.
- Require active workspace membership for workspace-scoped records after workspace authority exists.
- Require explicit role and permission checks for privileged reads and every mutation class.
- Deny anonymous access to tenant data.
- Deny cross-org reads and writes.
- Deny cross-workspace reads and writes unless AP explicitly approves a role inheritance rule.
- Allow service-role access only from trusted server or Edge operations with approved audit and failure handling.
- Never expose service-role credentials to browser code.
- Keep provider secrets out of browser code and tenant-visible records.
- Govern audit and export actions through later server-side boundaries.

## Policy Input Model

Future RLS implementation should rely on these inputs:

- `auth.uid()` as the caller identity.
- User profile id linked to Supabase Auth.
- Organization membership id and membership status.
- Workspace membership id and membership status.
- Role assignment at org or workspace scope.
- Permission set from an AP-approved model.
- Status fields such as active, invited, disabled, suspended, archived, deleted, or removed.
- `org_id` and `workspace_id` ownership columns.
- Actor columns such as `created_by`, `owner_id`, `reviewer_id`, `approver_id`, `updated_by`, `requested_by`, and `assigned_to` where table behavior requires them.

## RLS Matrix Overview

The M5.3 policy matrix defines, for each class:

- Read policy direction.
- Insert policy direction.
- Update policy direction.
- Delete or soft-delete direction.
- Required policy inputs.
- Service-role exception direction.
- Negative test cases.
- Implementation dependency.

Common direction:

- Read access starts with active org membership and narrows to active workspace membership for work artifacts.
- Insert and update access require explicit role or permission authority.
- Delete should generally be soft-delete and admin-only unless AP approves a stricter immutable model.
- Audit, handoff, evidence, lineage, and export records should favor append/status-transition behavior over broad update/delete.
- Provider key-reference values and raw secrets remain server-only and must never be browser-visible.

## Negative Isolation Test Strategy

Future RLS tests must prove:

- Anonymous users cannot read tenant data.
- Authenticated non-members cannot read org data.
- Disabled org members cannot access org data.
- Workspace non-members cannot read workspace data.
- Org A users cannot read or mutate Org B records.
- Workspace A users cannot read or mutate Workspace B records.
- Buyer Viewer cannot mutate Assess, Govern, Studio, Delivery, export, or provider records.
- Contributor cannot approve Govern decisions.
- Reviewer cannot access provider secret values or service-only config.
- Service-role paths are not callable from browser code.
- Export records cannot be read without approved org/workspace scope.
- Evidence, lineage, handoff, and audit records cannot cross tenant boundaries.
- Audit events are append-only where applicable.
- Suspended, archived, or deleted statuses do not grant accidental access.

## Policy Helper Strategy

M5.3 recommends small SQL helper functions after AP approves schema implementation:

- Future `is_active_org_member(org_id)` for active organization membership.
- Future `is_active_workspace_member(workspace_id)` for active workspace membership.
- Future role and permission helpers for org and workspace scope.
- Direct policy joins only for simple profile, membership, and role reads.
- Security definer helpers only when needed to avoid recursive RLS or support tightly audited server workflows.
- Materialized permission views deferred unless performance tests prove they are needed.

Risks:

- Recursive RLS can break membership lookups if helper functions are not designed carefully.
- Complex joins duplicated across policies are harder to audit.
- Security definer helpers can become overbroad unless scoped to narrow read-only checks.

## Supabase Test Strategy

Later implementation should test RLS with:

- Supabase local CLI or an AP-approved test database.
- A migration-applied non-production test environment.
- SQL-level tests for policy truth tables and append-only behavior.
- JS integration tests for anon, authenticated, and service-role client separation.
- Sanitized test output that records pass/fail status and counts only.
- No secret values, personal hosted values, tokens, service-role values, or provider secrets printed in logs.
- CI adoption as an AP timing decision after local repeatability is proven.

## Dependencies And Sequencing

M5.3 recommends this sequence:

1. M5.2a Supabase Auth Org Workspace Schema Migration Implementation.
2. M5.3a RLS Policy Implementation And Isolation Tests.

Reason: canonical workspace tables, workspace memberships, workspace ownership columns, and final membership/role/permission table decisions must exist before robust workspace-aware RLS can be implemented and tested.

Do not jump directly to RLS implementation unless AP explicitly accepts an org-only interim scope and defers workspace isolation.

## AP Decisions Needed

- Final table naming for profiles, org memberships, workspace memberships, roles, permissions, and role assignments.
- Final org/workspace hierarchy.
- Whether workspace scope is mandatory for all hosted-pilot work artifacts.
- Whether org admins inherit workspace authority.
- Role model scope: global, org-scoped, workspace-scoped, or mixed.
- Permission model: JSONB on roles or normalized permissions / role permissions.
- Helper function strategy and whether security definer helpers are acceptable.
- Service-role exception model and required audit context.
- RLS test method: SQL-level tests, JS integration tests, or both.
- CI timing for future RLS regression execution.
- Whether implementation should split schema first and RLS second.

## Risks

- Existing draft policies are mostly org-scoped and may create false confidence if treated as workspace-ready.
- Workspace authority requires broad FK, backfill, and planning across Assess, Govern, Studio, Delivery, Monitor, evidence, lineage, handoff, audit, export, and provider records.
- Recursive RLS or complex joins can make policies fragile.
- Role and permission strings in demo/runtime guards may not map cleanly to DB permission authority.
- Service-role exceptions can become overbroad without server-only execution and audit context.
- RLS tests can leak sensitive values if scripts are not pass/fail and counts-only.
