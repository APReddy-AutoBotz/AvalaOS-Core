# M4.1d-1 Central View/RBAC Guard Helper And Tests

## Purpose

M4.1d-1 implements the first code milestone from the accepted M4.1d route/RBAC guard design: a pure central View/RBAC guard helper and focused regression tests.

This milestone does not integrate the guard into `App.tsx`, Sidebar, ModuleJourney, routing behavior, or UI. App integration remains the next milestone.

## Baseline

- Accepted design milestone: `M4.1d Route/RBAC Guard Design And Implementation Plan`.
- Baseline tag: `avalaos-core-m4.1d-route-rbac-guard-design-plan`.
- Post-merge verification commit: `303d4c10f86e178b04cc43866582666104d56b1e`.

## Strict Scope

M4.1d-1 adds only:

1. A pure View/RBAC guard helper.
2. Focused helper tests.
3. A dedicated npm regression script wired into `npm run test`.
4. Planning and evidence docs for this milestone.

M4.1d-1 does not change product code paths that render views or move users between views.

## Root Cause

M4.1d established that AvalaOS Core currently relies on several separate navigation and access surfaces:

- Persisted `currentView` and `currentScope`.
- Module enablement helpers.
- Sidebar filtering.
- Direct view handlers.
- ModuleJourney home-view navigation.
- App fallback effects.

Because those surfaces do not share one decision contract, Sidebar visibility cannot be treated as central access control. M4.1d-1 creates the pure resolver needed before any future integration work.

## Helper Summary

The new helper is `resolveViewAccess` in `services/viewAccessGuard.ts`.

The helper accepts:

```ts
{
  user,
  authLoading,
  organization,
  enabledModules,
  view,
  scope,
}
```

It returns:

```ts
{
  allowed,
  fallbackView,
  fallbackScope,
  reason,
  module,
  requiredPermissions,
  requiredScope,
  guardSeverity,
}
```

The helper is pure and deterministic. It does not mutate state, persist state, call providers, fetch data, change score behavior, or render UI.

## Guard Precedence

The implemented precedence follows the accepted M4.1d design:

1. Auth loading returns `auth_loading` with `wait`; it does not redirect.
2. Missing user after auth loading returns `unauthenticated`.
3. Missing organization returns `no_organization`.
4. Empty enabled-module configuration returns `setup_required`.
5. Unknown or stale persisted view returns `stale_persisted_view`.
6. Deferred or decision-pending views return non-allowed guard results.
7. Disabled module returns `disabled_module`.
8. Invalid scope returns `invalid_scope`.
9. Missing permission returns `missing_permission`.
10. Passing checks returns `allowed`.

## Permission Semantics

M4.1d-1 preserves the current Sidebar default:

- Admin role bypasses listed per-view permissions.
- Non-admin users are allowed when they have any one listed permission.
- Views with no listed permissions are governed by auth, organization, module, scope, and deferred-status rules.

The helper does not strengthen permission checks to all-permission enforcement.

## Metadata Summary

The helper includes initial metadata for current `View` enum values:

- View-to-module mapping.
- Allowed scope types.
- Required permissions.
- Status: `active`, `deferred`, or `decision_pending`.
- Fallback view.
- Optional fallback scope type.
- Optional decision-pending scope types.

The metadata mirrors the accepted M4.1d matrix and current Sidebar semantics where Sidebar entries exist.

## Deferred Views

Deferred and decision areas remain represented, not implemented:

- `REPORTS` remains deferred and falls back toward Dashboard.
- `TEAMS` remains decision-pending and does not become an implemented route/RBAC behavior in this milestone.
- `WORKSPACE` remains active only for Docs project workspace; organization-scope Workspace/Admin access returns `admin_decision_pending`.
- Admin and AI Controls product behavior remains deferred to AP decisions and future milestones.

## Tests

The new `test:view-access-guard` regression covers:

- Auth loading waits.
- Unauthenticated access is denied.
- No organization is setup-directed.
- Empty enabled modules are setup-directed.
- Disabled module fallback.
- Stale persisted view fallback.
- Invalid scope fallback.
- Missing permission denial.
- Any one listed permission grants access.
- Admin role bypasses per-view permissions.
- Reports deferred behavior.
- Teams decision-pending behavior.
- Workspace/Admin decision-pending behavior.
- Existing enabled module view allowed behavior.

## Intentionally Not Changed

M4.1d-1 does not:

- Import or call the helper from `App.tsx`.
- Change Sidebar behavior.
- Change ModuleJourney behavior.
- Change view transitions.
- Change UI.
- Change routing behavior.
- Change service behavior outside the pure helper.
- Change deterministic scoring.
- Change schema, RLS, or migrations.
- Change Supabase Edge Functions.
- Change provider runtime or provider governance.
- Change Health.
- Implement M4.1d-2, M4.1d-3, M4.1e, M4.1f, or M4.1g.
