- Change code, schema, RLS, migrations, Admin UI, product UI, Health, scoring, browser fallback, resolver logic, provider decision behavior, Edge Function entrypoints, `_shared/ai.ts`, dependencies, usage, or cost behavior.

## Blocked Scenario Coverage

Planned deployed blocked-only matrix:

| Scenario | Function | Expected status | Expected failure class | Live provider risk |
| --- | --- | --- | --- | --- |
| Missing provider policy | `ai-generate-document` | HTTP 403 | `provider_policy_missing` | None; stops before config/key lookup. |
| Ineligible provider config | `ai-refine-section` | HTTP 403 | `provider_config_ineligible` | None; stops before key lookup. |
| Missing key reference | `ai-provider-test-connection` | HTTP 200 blocked payload | `key_reference_missing` | None; stops before key lookup. |
| Ineligible key reference | `ai-generate-document` | HTTP 403 | `key_reference_ineligible` | None; stops before secret read. |
| Unsafe secret reference | `ai-generate-document` | HTTP 403 | `secret_reference_unsafe` | None; stops before env read and provider call. |

Unsupported safely in the current deployed contract:

- `provider_config_missing`: active provider policy cannot safely reference a missing config because the deployed schema uses a config foreign key.
- `provider_call_blocked`: not safely reachable without forcing an internal resolver exception or entering an allowed provider path.

## Acceptance Criteria

Outcome A requires:

- Approved non-production target confirmed.
- Local runtime mode is `pilot`.
- Required functions are present.
- Deployed blocked/fail-closed scenarios return expected failure classes.
- `mode_not_allowed` is absent from deployed blocked-smoke results.
- Audit sanitization passes.
- Cleanup completes.
- Retained smoke rows are zero.
- Provider execution rows are zero when available.
- No live provider call occurs.
- No real provider key is used.
- No provider billing path is exercised.
- No secret is printed or stored.
- No code/schema/UI/Health/provider behavior changes occur.

Outcome B applies if readiness is blocked before deployed scenario execution.

Outcome C applies if deployed function behavior is reachable but fail-closed behavior, audit sanitization, or cleanup does not pass.

## M3.2ab Recommendation

If Outcome A passes, recommend M3.2ab as post-merge verification for M3.2aa and then AP decision on whether any future safe mocked deployed allowed path is needed.

If Outcome B or Outcome C occurs, M3.2ab should address only the specific readiness or behavior blocker after AP review.
