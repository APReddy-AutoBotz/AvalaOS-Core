# M3.2f Provider Config AP Decision Pack

Status: Planning/decision only

Branch: `milestone/m3.2f-provider-config-ap-decision-pack`

## Milestone Purpose

Create an AP-reviewable decision pack that resolves or structures the remaining provider config decisions identified by M3.2e before any schema/RLS migration is allowed.

M3.2e concluded that the safest immediate outcome is:

- no implementation yet
- no schema/RLS migration yet
- no runtime provider calls yet
- resolve AP decision gaps before the first real implementation PR

M3.2f is that decision pack. It recommends the first safe implementation sequence after AP approval, but it does not create migrations, edit schema SQL, change Supabase Functions, change runtime behavior, change product code, change tests, change Admin UI, touch Health, change scoring, add runtime agents, add MCP/A2A behavior, add external execution, change browser fallback behavior, change package files, change dependencies, or make compliance claims.

## Source-Of-Truth Alignment

This decision pack follows `docs/00_SOURCE_OF_TRUTH.md`:

- Evaluate before you automate. Govern before you execute.
- Deterministic scoring remains deterministic.
- AI must not decide final scores, gates, risk tiers, recommendations, approvals, or compliance status.
- Browser-stored provider keys and browser-side AI are not acceptable for pilot or production behavior.
- Server-side AI through Supabase Edge Functions remains the pilot and production direction.
- Evidence proves every governed decision.

## Final Schema Shape Decision

Recommended AP decision: first implementation should introduce only the minimum safe provider-config schema/RLS subset needed to support later server-side resolver work. It should not introduce every future provider object at once.

Required-now objects for the first real schema/RLS PR after AP approval:

- `ai_provider_configs`: safe provider config metadata, provider ID, status, model policy, allowed modes, allowed operations, optional workspace scope, key-reference link, and audit metadata.
- `ai_provider_key_refs`: reference-only key metadata pointing to server-side secret material or secret manager identifiers, plus safe label/status/rotation metadata.
- `ai_workspace_provider_policies`: canonical workspace policy that binds provider configs to allowed modes, operations, roles, default selection, and enablement.
- `ai_provider_audit_events`: append-oriented provider config/resolver/policy/key-reference evidence with safe identifiers, result, reason class, timestamps, and schema version.

Deferred-later schema objects or changes:

- prompt version table creation or `prompt_templates` alteration for immutable governed prompt versions
- `ai_generation_jobs` linkage to provider audit events
- `ai_usage_events` alteration for workspace scope, service-only writes, and cost semantics
- Admin Controls tables or views
- runtime resolver artifacts
- seed/demo rows beyond non-secret placeholders

`docs/schema/ai_governance_jobs.sql` remains prior context only. Existing `ai_provider_configs`, `prompt_templates`, `ai_generation_jobs`, and `ai_usage_events` must not be treated as approved implementation dependencies until AP approves the exact migration plan.

Raw provider secrets must never be stored in ordinary application DB rows, even encrypted, unless AP separately approves a dedicated secret-storage model. The default design remains key-reference only.

## RLS Ownership Decision

Recommended AP decision:

- Provider configs are tenant-owned, with optional workspace scope when a workspace needs a narrower policy or model profile.
- Key-reference metadata is tenant/workspace-scoped but service-owned for sensitive resolution behavior.
- Workspace provider policies are workspace-owned and must reference a provider config eligible for that tenant/workspace.
- Provider audit events are service-written, append-oriented evidence records owned by tenant/workspace context.
- Prompt/template references remain prior context until prompt visibility and version immutability are approved.
- Usage records remain prior context until service-only writer and safe usage semantics are approved.

Expected access posture:

- Browser/client code may read only AP-approved safe metadata, if any, and must never read raw secret material, resolved key material, secret-manager values, service-role-only fields, or sensitive key-reference internals.
- Authenticated workspace users may read only safe provider readiness/policy metadata when active membership permits it.
- Admin/control-plane actors may write config, key-ref metadata, and policy only through approved server-side control paths.
- Edge/service-role paths may read config/policy/key-ref metadata and write audit events only as part of an approved resolver workflow.
- No actor should access raw provider secret material from ordinary application DB rows because raw secrets must not be stored there.

Fail closed when ownership, membership, actor role, workspace scope, or RLS behavior is unclear.

## Service-Role Boundary Decision

Recommended AP decision: service role is allowed only inside approved server-side workflows and must still enforce resolver gates before any provider call.

Service role may:

- read safe provider config metadata
- read key-reference metadata
- resolve workspace/provider policy
- prepare and write append-only provider audit events
- later resolve server-side secret material only after AP approves resolver implementation
- later write usage metadata only from sanitized server-side provider responses

Browser/client code may never:

- use service-role credentials
- resolve key references
- receive raw provider keys
- read secret-manager identifiers if AP classifies them as sensitive
- write provider config, key refs, provider policy, or provider audit events directly
- bypass mode, auth, membership, policy, provider status, operation, key-reference, prompt/version, or evidence gates

Stop if the future implementation cannot prove the service-role boundary is server-side, narrow, auditable, and unreachable from browser/client bundles.

## Tenant / Workspace Policy Source Decision

Recommended AP decision: `ai_workspace_provider_policies` is the canonical policy source for workspace/provider/mode/operation decisions.

Policy behavior:

- Provider config records describe provider metadata, status, allowed capability envelope, and model policy.
- Workspace policy records decide whether that provider config may be used for a specific workspace, mode, operation, role, and default-provider context.
- Workspace policy may narrow provider defaults.
- Workspace policy may override org-level provider defaults only through an explicit AP-approved precedence rule.
- Missing, conflicting, disabled, or ambiguous default policy fails closed.
- Policy evaluation must be deterministic and must write evidence before provider execution in future runtime slices.

No future provider call should be allowed based on client-submitted provider preference alone.

## Key-Reference Resolver Type Decision

Recommended AP decision: the first real implementation PR should include key-reference metadata and resolver contract fields only. It should not include a stub resolver, database-backed resolver, secret manager lookup, or provider-call handoff.

Accepted future resolver inputs at a high level:

- mode
- operation type
- tenant/org ID
- workspace ID
- actor/user ID from server-authenticated context
- provider ID or provider config ID
- key-reference selector
- prompt/template/version reference
- requested model metadata
- correlation/evidence ID

Accepted future resolver outputs at a high level:

- allowed or blocked status
- resolved provider/config safe metadata
- policy result
- key-reference safe status
- prompt/version binding reference
- sanitized evidence context
- usage metadata shell
- sanitized failure class when blocked
- server-side-only provider-call handoff only in a later AP-approved runtime slice

No runtime provider call is enabled by M3.2f or by the recommended first schema/RLS PR.

The resolver must never return raw keys to browser/client code, general business logic, logs, evidence, errors, return payloads, or UI-facing responses.

## Audit Table Strategy Decision

Recommended AP decision: use a dedicated `ai_provider_audit_events` table rather than relying only on `ai_generation_jobs` or `ai_usage_events`.

Minimum audit fields for provider config resolution attempts:

- event ID
- event type
- schema version
- correlation ID
- request ID when safe
- actor ID or service actor
- tenant/org ID
- workspace ID
- mode
- operation type
- provider ID
- provider config ID
- key-reference ID or AP-approved safe fingerprint only
- prompt/template/version references when available
- policy result
- sanitized failure class
- sanitized status
- retry posture when applicable
- safe usage metadata shell when applicable
- created timestamp

Provider audit events must be evidence-safe. They must not store raw prompt content, raw model response content, raw provider secret, resolved key material, secret-manager values, raw provider errors, provider authorization headers, browser raw keys, `userApiKey`, `localStorage` values, hidden chain/payload, or unredacted stack traces.

Existing evidence flow may reference provider audit events later by correlation ID or foreign key after AP approval.

## Prompt / Version Binding Decision

Recommended AP decision: prompt/version binding is required before future pilot/production provider calls, but it should not be implemented in the first schema/RLS PR unless AP approves only safe reference fields.

Before prompt/version binding can be implemented, AP must approve:

- whether `prompt_templates` remains the source of prompt bodies
- whether immutable `prompt_versions` are required
- who can read prompt bodies
- whether workspace-specific prompt variants are allowed
- whether runtime evidence may record only prompt references or additional prompt metadata
- how prompt body storage avoids sensitive payload leakage

M3.2f does not change prompt execution behavior.

## Usage / Cost Safety Decision

Recommended AP decision: defer usage/cost table expansion from the first schema/RLS PR unless AP explicitly approves a safe usage shell.

Minimum future safety controls:

- usage must be server-derived, not client-submitted
- token counts must map to provider response metadata only after sanitization
- cost estimates must be labeled as estimates and omitted when pricing semantics are unclear
- usage metadata must not include raw prompts, raw source text, raw model responses, raw provider payloads, auth headers, or secrets
- usage writes should be service-side only in pilot/production
- usage/cost display must not imply deterministic scoring, approval, recommendation, or compliance status

Block runtime integration until usage/cost guardrails are approved or explicitly deferred without losing fail-closed provider governance.

## Admin Controls Timing Decision

Recommended AP decision: Admin UI should come after backend contracts stabilize.

Safe sequencing:

1. Schema/RLS migration after AP approval.
2. Service-side resolver contract or implementation after schema/RLS passes.
3. Audit/evidence writer and event persistence after resolver gates are stable.
4. Admin Controls read-only evidence/readiness view after safe backend metadata exists.
5. Runtime integration only after schema, resolver, audit/evidence, scanner, and AP review are complete.

M3.2f makes no Admin UI changes.

## Future Scanner Classification Decision

Recommended AP decision: keep M3.1c fail-closed and require exact classifications for future implementation files.

Likely future allowed control-plane/server references:

- schema migration references to provider config table names and key-reference column names, classified as schema/control artifacts when no raw secrets are present
- server-side resolver references to provider IDs, key-reference metadata, and sanitized failure classes
- server-side audit/evidence references to safe event fields
- docs/planning and docs/quality references classified as test/control documentation

Likely future forbidden references:

- `VITE_GEMINI_API_KEY` or `VITE_GROQ_API_KEY` in pilot/production browser runtime paths
- raw provider key props, state, or localStorage writes
- key-reference resolver code imported into browser/client bundles
- broad runtime allowlists for services or components
- logs or evidence storing raw secrets, raw prompt bodies, raw response bodies, raw provider errors, hidden payloads, or secret-manager values

Future allowlist updates must include path, pattern, line or anchor, server-side proof, secret posture, reason, future action, static scan pass, stale allowlist check, and wording scan review.

## Explicit Stop Conditions

Stop future implementation if:

- AP has not approved final schema object list, columns, RLS posture, writer roles, and rollback plan
- schema/RLS cannot prove tenant/workspace isolation
- service-role boundary is unclear
- tenant/workspace policy source or precedence is unclear
- key-reference resolver type is unclear
- raw secrets would enter browser/client, ordinary DB rows, logs, evidence, UI responses, or general business logic
- raw secrets would be stored encrypted in ordinary DB rows without separate AP approval for a dedicated secret-storage model
- audit events would store sensitive payloads
- prompt/version body visibility is unclear and runtime depends on it
- usage/cost fields would be misleading or client-derived
- Admin UI is required before backend contract stabilizes
- M3.1c scanner cannot classify new hits safely
- package/dependency changes are needed without AP approval
- tests require changes beyond documentation-only references
- Health, scoring, runtime-agent, MCP/A2A, or external execution changes appear
- browser fallback behavior must change without separate approval
- unsupported compliance claims are introduced
