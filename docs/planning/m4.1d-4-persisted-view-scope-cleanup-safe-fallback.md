# M4.1d-4 Persisted View/Scope Cleanup And Safe Fallback

## Purpose

M4.1d-4 adds a narrow persisted view/scope cleanup layer for AvalaOS Core. It validates restored `currentView` and `currentScope` state, normalizes safe shapes, and resolves the final view/scope pair through the central View/RBAC guard before the pair remains active.

This milestone continues the M4.1d guard path after Sidebar and ModuleJourney were aligned to guard metadata.

## Root Cause

`App.tsx` persisted `currentView` under `StorageKeys.VIEW` and `currentScope` under `StorageKeys.SCOPE` through the generic `usePersistentState` hook.

That hook parses stored JSON and returns it directly as typed state. TypeScript types do not protect values restored from browser storage at runtime. A stale or malformed stored value can therefore create confusing startup state before normal navigation handlers run.

Examples of persisted-state risk:

- A stale view string from an older build.
- A scope object with an unknown `type`.
- A project scope without a usable `id` and `name`.
- A team scope without a usable `id` and `name`.
- A simple My Work or Organization scope carrying stale fields from older stored shapes.
- A valid-looking view/scope pair that is denied by the central guard because the module is disabled, the scope is invalid for the view, or the user lacks permission.

## Implementation Summary

### Pure Sanitizer

M4.1d-4 adds `services/viewStatePersistence.ts`.

The service exposes:

- `isValidView`
- `normalizePersistedView`
- `isValidScope`
- `normalizePersistedScope`
- `areScopesEqual`
- `resolvePersistedViewScopeState`

The sanitizer behavior is intentionally small:

- Unknown view values normalize to `DASHBOARD`.
- Unknown scope types normalize to My Work scope.
- Malformed project and team scopes normalize to My Work scope.
- My Work and Organization scopes normalize to clean one-field shapes.
- Valid project and team scopes retain only `type`, `id`, and `name`.
- Input objects are not mutated.

### Guard Resolution

After normalization, `resolvePersistedViewScopeState` calls `resolveViewAccess`.

If the normalized pair is allowed, the normalized pair remains active. If the normalized pair is denied after auth/org loading has completed, the resolver fallback view/scope is returned and can be written back to persisted state.

The existing organization-scope Admin Workspace behavior remains preserved as a deferred AP decision path. The helper still calls `resolveViewAccess` for the pair, but it does not force the fallback over that existing App exception.

### App Integration

`App.tsx` now keeps raw persisted `VIEW` and `SCOPE` values at the storage boundary, derives sanitized `currentView` and `currentScope` values for runtime use, and writes back normalized/fallback values after auth/org loading is ready.

The integration:

- avoids using malformed persisted scope values during render,
- avoids broad storage-hook rewrites,
- does not clear all browser storage,
- does not touch unrelated persisted keys,
- preserves existing guarded transition handlers,
- preserves Assess-to-Studio and Docs-to-Delivery behavior,
- preserves organization-scope Admin Workspace behavior pending AP decisions.

## Fallback Behavior

| Persisted condition | Result |
| --- | --- |
| Unknown view | Normalize to `DASHBOARD`, then resolve through central guard. |
| Unknown scope type | Normalize to My Work scope, then resolve through central guard. |
| Project scope missing `id` or `name` | Normalize to My Work scope, then resolve through central guard. |
| Team scope missing `id` or `name` | Normalize to My Work scope, then resolve through central guard. |
| My Work or Organization scope with stale fields | Strip stale fields and persist clean scope. |
| Valid pair denied by guard | Persist guard fallback view/scope. |
| Auth/org loading | Wait; do not redirect or mutate from loading state. |
| Organization-scope Admin Workspace | Preserve existing behavior pending AP decision. |
