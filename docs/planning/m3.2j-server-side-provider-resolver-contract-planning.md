# M3.2j Server-Side Provider Resolver Contract Planning

## Milestone Purpose

M3.2j defines the planning-only contract for a future server-side provider resolver. It records the resolver gates, trusted inputs, safe outputs, failure modes, audit event expectations, scanner expectations, and safety boundaries before any resolver implementation is authorized.

This milestone does not implement resolver code, create or edit migrations, edit schema SQL, change Supabase Functions, change runtime behavior, change Admin UI, change product code, change package or dependency files, touch Health, change scoring, change browser fallback behavior, add provider calls, add provider secrets, change prompt versions, expand usage or cost behavior, or make compliance claims.

## Dependency Chain Through M3.2i

1. M3.2h implemented the approved provider configuration governance schema/RLS migration.
2. M3.2h was post-merge verified and tagged as `avalaos-core-m3.2h-provider-config-schema-rls-migration`.
3. M3.2i reviewed M3.2h provider config RLS evidence and resolver readiness.
4. M3.2i was post-merge verified and tagged as `avalaos-core-m3.2i-provider-config-rls-evidence-resolver-readiness`.
5. M3.2i recommended proceeding to M3.2j resolver contract planning only. It did not authorize resolver implementation.

M3.2j is anchored to the M3.2h governance tables:

- `ai_provider_key_refs`
- `ai_provider_configs`
- `ai_workspace_provider_policies`
- `ai_provider_audit_events`

The contract uses `org_id` as the enforceable tenant boundary. `workspace_id`, where present in audit evidence, remains nullable reference metadata until AP approves a canonical workspace authority.

## Resolver Actors And Trust Boundaries

Future resolver actors:

- Browser/client caller: may submit a requested operation, selected provider preference, request context, and safe correlation hints. These values are untrusted until verified server-side.
- Authenticated user: must be derived from server-authenticated request context, not from browser-supplied user IDs.
- Organization member: must be checked against active membership for the resolved `org_id`.
- Server resolver: future server-side control component that evaluates policy and fails closed before any provider handoff.
- Provider-call layer: future server-side-only layer that may receive an allow decision and server-bound secret access after all resolver gates pass.
- Audit writer: future service-side workflow that appends sanitized resolver evidence to `ai_provider_audit_events`.
- AP reviewer: approves contract, implementation plan, and any future implementation milestone.

Trust boundaries:

- Browser/client input is not authoritative for actor identity, organization membership, role, provider policy, provider config status, key-reference status, or audit safety.
- Service-role access must remain inside trusted server/control workflows.
- `secret_ref` and any resolved key material must remain inside the server-side boundary.
- General business logic and browser/client responses must receive only sanitized resolver results.

## Resolver Input Contract

Future resolver inputs should be explicit, typed, and server-normalized before policy evaluation:

| Input | Source of authority | Rule |
| --- | --- | --- |
| `mode` | Server runtime context plus request metadata | Must be `pilot` or `production` for resolver-governed behavior. Other modes are outside this resolver contract and must not widen browser fallback behavior. |
| `operation` | Server-normalized request intent | Must be one of `generate_document`, `refine_section`, or `test_provider_connection`. |
| `requested_provider` | Browser/client preference or server default | Must normalize to `gemini` or `groq`; unknown providers fail closed. |
| `org_id` | Server-authenticated tenant context | Required. Missing or ambiguous tenant identity fails closed. |
| `workspace_id` | Optional request/evidence context | Evidence metadata only. Must not become authorization authority until AP approves workspace authority. |
| `actor_id` | Server-authenticated user context | Required for user-initiated operations. Browser-supplied actor IDs are not trusted. |
| `actor_roles` | Server-side membership/role lookup | Used only after active membership is verified. |
| `provider_config_id` | Optional request selector or policy default | If supplied, must belong to the same `org_id`, match policy, be active, and be non-deleted. |
| `correlation_id` | Server-generated or sanitized request hint | Server should generate one when absent or unsafe. |
| `evidence_ref` | Optional safe evidence pointer | Reference only. Must not be treated as proof of approval by itself. |
| `prompt_reference` | Existing governed prompt/template context | Reference metadata only. M3.2j does not change prompt versions. |

The resolver must not accept raw provider keys, encrypted provider keys, auth headers, secret payloads, raw prompt bodies, raw completions, raw provider responses, browser-side provider secrets, or service-role keys as input.

## Resolver Output Contract

The resolver output must be sanitized by audience.

Server-control output:

- `status`: `allowed` or `blocked`
- `policy_result`: `allowed`, `blocked`, or a constrained denial value
- `failure_class`: nullable sanitized failure class
- `provider`: `gemini` or `groq` when resolved
- `provider_config_id`: safe reference when resolved
- `key_ref_id`: safe reference when resolved
- `operation`
- `mode`
- `org_id`
- `workspace_id`: nullable evidence metadata only
- `actor_id`
- `correlation_id`
- `evidence_ref`
- `audit_event_intent`: sanitized event payload shell
- `provider_call_allowed`: boolean

Browser/client output:

- generic result category
- safe failure category when blocked
- `correlation_id`
- retry category only when safe

Server-only provider-call handoff, in a future implementation milestone only:

- resolved provider identifier
- sanitized provider config metadata
- server-side-only key-reference resolution result
- operation and mode
- correlation ID
- safe evidence shell

The resolver must never return raw key material, `secret_ref`, secret-manager payloads, auth headers, raw provider request/response bodies, raw prompts, raw completions, service-role keys, session tokens, or cookies.

## Fail-Closed Decision Gates

Future implementation must evaluate gates in a fail-closed order and stop at the first unsafe condition:

1. Confirm resolver-governed mode is `pilot` or `production`.
2. Authenticate the actor from server context.
3. Resolve `org_id` from trusted server context.
4. Verify active organization membership.
5. Resolve server-side actor roles for the organization.
6. Normalize operation and reject unsupported operations.
7. Normalize requested provider and reject unsupported providers.
8. Resolve active provider policy for `org_id`, operation, mode, and role.
9. Resolve provider config from policy/default or explicit selector.
10. Verify provider config belongs to the same `org_id`.
11. Verify provider config is active and non-deleted.
12. Verify provider config allows the requested mode and operation.
13. Verify provider config provider matches the requested or policy-selected provider.
14. Resolve key-reference eligibility through `key_ref_id`.
15. Verify key ref belongs to the same `org_id` and provider.
16. Verify key ref is active, non-deleted, non-expired, and not a `manual_placeholder`.
17. Verify `secret_ref` is reference-only and remains server-side.
18. Prepare sanitized audit event fields.
19. Verify scanner classification expectations for any future implementation path.
20. Allow provider-call handoff only after all gates pass.

Any missing, ambiguous, inactive, deleted, expired, unsupported, unsafe, or unclassified prerequisite must block the provider call.

## `secret_ref` Handling Rules

`secret_ref` is a service-only reference identifier. It is not key material.

Rules:

- Never expose `secret_ref` to browser/client code.
- Never return `secret_ref` in resolver output outside the server-only control boundary.
- Never log `secret_ref`.
- Never copy `secret_ref` into audit metadata as a raw value.
- Never treat `secret_ref` as a provider key, encrypted key, auth header, bearer token, browser-readable secret, or secret-manager payload.
- Never accept browser/client-supplied `secret_ref` as authoritative.
- Only resolve `secret_ref` after all auth, org membership, role, policy, config, operation, mode, key-reference, audit, and scanner gates pass.

Future implementation planning must define the exact secret manager or server reference adapter before any key lookup is implemented.

## Provider Config Lookup Rules

Provider config lookup must:

- use `ai_provider_configs`
- require matching `org_id`
- require `deleted_at IS NULL`
- require `status = active`
- require provider in `gemini` or `groq`
- require mode in `allowed_modes`
- require operation in `allowed_operations`
- use `key_ref_id` only as a reference to `ai_provider_key_refs`
- treat deprecated prior-context `key_reference`, when present, as ineligible for resolver use

If multiple active defaults or ambiguous configs are possible, the resolver must fail closed and require AP-approved data correction before any provider call.

## Workspace/Org Policy Lookup Rules

Policy lookup must:

- use `ai_workspace_provider_policies`
- enforce `org_id` as the tenant boundary
- require `deleted_at IS NULL`
- require `status = active`
- match requested `operation`
- match requested `mode`
- verify actor roles against `allowed_roles`
- prefer an active default only when the policy uniquely identifies one active default for `(org_id, operation, mode)`
- require the policy's `provider_config_id` to resolve to an eligible provider config

`workspace_id` must not be used as an authorization boundary in M3.2j. Workspace-specific policy authority is a carried-forward AP risk.

## Mode And Operation Authorization Rules

Resolver-governed modes:

- `pilot`
- `production`

Resolver-governed operations:

- `generate_document`
- `refine_section`
- `test_provider_connection`

Rules:

- `local-demo` and `internal-dev` browser fallback remain governed by existing M3.1a/M3.1b boundaries and are not expanded by this resolver contract.
- Unknown modes fail closed.
- Unknown operations fail closed.
- Operation authorization must pass through both policy and provider config eligibility.
- `test_provider_connection` remains a governed server-side operation and must not bypass secret-safety or audit gates.

## Key-Reference Eligibility Rules

Eligible key refs must:

- exist in `ai_provider_key_refs`
- match the resolved `org_id`
- match the resolved provider
- be active
- be non-deleted
- be non-expired
- have a resolver type eligible for provider calls
- have a non-empty `secret_ref` that remains reference-only

Ineligible key refs:

- `pending_review`
- `disabled`
- `expired`
- `retired`
- soft-deleted
- wrong provider
- wrong org
- missing `secret_ref`
- unsafe `secret_ref` shape
- `manual_placeholder`

`external_secret_reference` and `server_reference` may be contract-eligible only after M3.2k defines the implementation plan for the lookup adapter and AP approves it. M3.2j does not authorize the lookup adapter.

The resolver contract prohibits:

- raw provider keys in browser/client code
- raw provider keys in ordinary database rows
- raw provider keys in logs or audit metadata
- encrypted provider keys in browser/client-readable rows
- auth headers in logs or audit metadata
- secret-manager payload values in logs or audit metadata
- raw prompts or completions in resolver audit events

- cross-tenant provider config or key-reference leakage
- service-role key exposure outside server infrastructure

## Service-Role Boundary

Future service-role use must be server-side only and limited to the minimum workflow required to evaluate provider config, policy, key-reference eligibility, and audit append behavior.

Service-role access must not bypass:

- authentication
- active organization membership
- role authorization
- provider policy
- provider config status
- mode authorization
- operation authorization
- key-reference eligibility
- secret-safety gates
- audit redaction rules
- scanner classification

M3.2j does not introduce service-role code or service-role behavior.

## Browser/Client Boundary

Browser/client code must not:

- resolve provider keys
- receive `secret_ref`
- receive raw provider secrets
- write sensitive provider/key/audit objects directly
- decide provider policy
- decide role authorization
- decide key-reference eligibility
- decide deterministic scores, risk gates, approvals, or recommendations
- use browser-side provider execution for pilot or production

Browser/client code may receive only sanitized resolver status categories and correlation IDs after a future server-side implementation is approved.

## Error/Failure Classification

Future resolver failures should use constrained classes:

| Failure class | Meaning | Retry posture |
| --- | --- | --- |
| `mode_not_allowed` | Mode is outside resolver-governed `pilot` or `production`. | No retry without configuration change. |
| `unauthenticated` | Server-authenticated actor is missing. | Retry after sign-in. |
| `org_missing` | Trusted organization context is missing or ambiguous. | No retry without request/context correction. |
| `membership_denied` | Actor is not an active org member. | No retry without access change. |
| `role_not_allowed` | Actor role is not allowed by policy. | No retry without policy/access change. |
| `operation_not_allowed` | Operation is unsupported or not allowed by policy/config. | No retry without policy/config change. |
| `provider_not_supported` | Provider is not `gemini` or `groq`. | No retry without contract change. |
| `provider_policy_missing` | No active matching policy exists. | No retry without policy configuration. |
| `provider_policy_ambiguous` | More than one active default or matching policy creates ambiguity. | No retry until data is corrected. |
| `provider_config_missing` | Policy or selector cannot resolve config. | No retry without config correction. |
| `provider_config_ineligible` | Config is inactive, deleted, wrong org, wrong provider, or disallows mode/operation. | No retry without config change. |
| `key_reference_missing` | Config has no eligible key reference. | No retry without key-reference setup. |
| `key_reference_ineligible` | Key ref is inactive, deleted, expired, wrong org/provider, placeholder, or unsafe. | No retry without key-reference correction. |
| `secret_reference_unsafe` | `secret_ref` cannot be treated as reference-only. | No retry until corrected. |
| `audit_context_unsafe` | Safe audit event fields cannot be prepared. | No retry until evidence shape is corrected. |
| `scanner_classification_missing` | Future implementation path cannot be classified by static scan controls. | Stop implementation. |
| `provider_call_blocked` | Generic blocked status after any fail-closed gate. | Depends on underlying failure class. |

No failure response may include raw secrets, auth headers, provider payloads, raw prompts, raw completions, cookies, tokens, or `secret_ref` raw values.

## Scanner Classification Expectations

Future implementation planning must classify resolver files as server-side/control-plane only before implementation. Expected scanner requirements:

- Browser/provider secret patterns remain forbidden outside approved demo/dev fallback or server-side Edge/Supabase references.
- New resolver paths must not run in the browser bundle.
- Any new allowlist entry must be exact, justified, and include a future action.
- Documentation references remain classified as control documentation, not runtime permission.
- Stale direct provider environment references must be removed or reclassified only in a later AP-approved implementation milestone.
- Static scan must report zero forbidden hits and zero stale allowlist entries before merge.

M3.2j does not change the scanner or allowlist.

## `DATABASE_URL` / Non-Production Validation Risk

M3.2h and M3.2i carried forward a validation risk: database validation still requires a safe non-production `DATABASE_URL`.

M3.2j does not run migrations and does not apply schema changes. Future M3.2k planning must decide whether non-production database validation is required before any resolver implementation milestone, and must explicitly prohibit production or shared database validation without AP approval.
