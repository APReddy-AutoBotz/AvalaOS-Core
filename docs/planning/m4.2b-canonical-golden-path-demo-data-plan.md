# M4.2b Canonical Golden Path Demo Data Plan

## Purpose

M4.2b is a docs-only implementation plan for replacing scattered mock and demo data with one coherent buyer-demo story:

`AP Invoice Exception Handling / AP Invoice Exception Workflow`

The canonical story must connect:

Organization -> Users/Roles -> Process -> Guided Assessment -> Decision Pack -> Govern Lite -> Avala Studio -> Generated Docs -> Delivery Work Items -> Delivery Pack -> Monitor.

AP decision for this milestone: do not create full examples for every scenario currently present in the codebase. The buyer demo should show one clear end-to-end enterprise workflow, not many disconnected examples.

## Strict Non-Scope

M4.2b does not change data or runtime behavior.

- No mock/demo data changes.
- No product code changes.
- No UI changes.
- No routing changes.
- No scoring formula, score weight, threshold, gate, recommendation, or score-version changes.
- No schema, RLS, migration, or Supabase Edge Function changes.
- No provider/runtime behavior changes.
- No Health changes.
- No dependency changes.
- No deployment.
- No provider calls.
- No secrets.
- No unsupported compliance claims.
- No language implying AI decides scores, approvals, risk, recommendations, or compliance.

## Current Demo Data Problems

- Scattered scenarios compete for attention across Assess, Docs, Delivery, and Monitor.
- AP data exists but does not yet own the full golden path from organization through Monitor.
- Buyer-visible project/process names are inconsistent: AP Invoice Automation, Vendor Invoice Intake and SAP Posting, AP Automation Pilot, and generated invoice artifacts point to adjacent but not identical stories.
- Non-AP stories create stale labels and AI-first demo risk.
- Role coverage is broad but not purposeful; several personas are not necessary for the AP path.
- AP delivery tasks lack complete sourceLineage/evidenceRefs even though M4.1c and M4.2a now make lineage visible.
- Month-end close, not AP invoice, currently carries the strongest lineage fixture.
- Local demo persistence can keep stale user, view, scope, template, handoff, and module state after fixtures change.

### Process

- Name: `AP Invoice Exception Handling`

- Department: Finance Operations

- Owner: Priya Nair

### Assessment

The canonical assessment should be fully populated and deterministic-score-ready:

- Standardized enough for automation fit.
- Predictable exceptions with explicit owner review.
- Financial data sensitivity captured.
- Human approval before action required.
- SAP and invoice intake systems documented.
- Audit and financial impact risk recorded.
- Evidence quality high enough to avoid a broken demo, while still showing evidence review controls.

### Evidence Items

Recommended canonical evidence IDs:

- `ev-ap-exception-map`: current-state AP exception process map.
- `ev-ap-sop`: AP invoice exception handling SOP.
- `ev-ap-sap-sample`: representative SAP invoice screenshot summary.
- `ev-ap-exception-report`: exception volume and aging report.
- `ev-ap-po-policy`: PO/tax/duplicate exception policy.
- `ev-ap-owner-review`: process owner review note.

Evidence should use IDs and summaries only. Do not seed raw customer source content.

### Assumptions

Recommended canonical assumptions:

- Annual invoice volume and exception rate.
- Average handling time for exception review.
- SAP test/sandbox access availability.
- Duplicate/tax/vendor exception rules are stable enough for the pilot.
- Finance owner review remains required before posting or external communication.
- Monitor metrics are review/demo signals until live instrumentation is approved.

### Deterministic Score Result

Do not hard-code a fake score that contradicts the scoring engine. Seed assessment inputs so the existing deterministic engine produces a buyer-demo-safe outcome:

- Recommendation: document intelligence, workflow/HITL control, and RPA/API handoff where appropriate.
- Gate decision: preferably `Conditional Go` or `Go` with human review, depending on current scoring result from seeded inputs.
- Handoff readiness: ready for Docs, and delivery readiness only when evidence and approval conditions are represented.
- Risk tier: should reflect finance data sensitivity and owner approval, not imply ungoverned execution.

### Decision Pack

The Decision Pack should show:

- Executive summary for AP invoice exception handling.
- Deterministic score version and calculated timestamp.
- Operating model recommendation with document intelligence/workflow/HITL controls.
- Business value estimate from volume, effort, and cost assumptions.
- Governance summary with finance data controls, owner approval, audit controls, and monitoring controls.
- Evidence and assumption summaries.
- Reviewer approval history.

### Handoff Pack

The Handoff Pack should carry:

- `processId`, `assessmentId`, `organizationId`, score version, and assessment version.
- Process summary and current-state summary.
- Pain points: exception aging, duplicate invoice risk, vendor block handling, manual SAP checks.
- Systems: invoice intake mailbox/portal, SAP, vendor master, PO data, AP workflow queue.
- Data sources: invoice PDFs, PO records, vendor master, exception report, SAP posting records.
- Evidence references and assumption references.
- HITL design and blocked actions.
- Required documents: BRD/PDD/FRD or the AP-approved set.
- Suggested backlog items that become Delivery work items.

### Govern Lite Card

Govern Lite should show the AP workflow as review-ready with explicit human approval controls:

- Business owner: Priya Nair.
- Technical owner: delivery/admin user pending AP decision.
- Data sensitivity: Confidential or Internal, based on final seed design.
- Autonomy level: should not imply autonomous finance decisions.
- Approval policy: process owner approval before downstream handoff/action.
- Evidence policy: evidence review required or recommended, with no broken critical gaps in the buyer demo.
- Allowed actions: prepare evidence, draft documents, create work item candidates.
- Blocked actions: change deterministic scores, execute production actions, store raw provider keys in browser, external communications without approval.

## Golden-Path Test Script

1. Log in as the Process Analyst or Buyer persona.
2. Confirm organization is the AP-approved demo organization and modules Assess, Docs, Delivery, and Monitor are enabled.
3. Start from Header primary action and land in Process Catalog.
4. Open `AP Invoice Exception Handling`.
5. Review process details and the completed Guided Assessment.
6. Review deterministic score, recommendation, Decision Pack, and Handoff Pack.
7. Review Avala Govern Lite controls, evidence policy, approval policy, and blocked actions.
8. Open Avala Studio handoff.
9. Confirm the Studio source-context card references AP Invoice Exception Handling and the expected evidence count.
10. Generate or inspect the governed document pack.
11. Review Generated Docs and approval status.
12. Import generated work items into the AP delivery project.
13. Confirm imported work items carry source lineage and evidence refs.
14. Open Delivery work items and inspect owner/status/blocker fields.
15. Open Delivery Pack and confirm sources, audit summary, evidence checklist, and export review copy.
16. Open Monitor/Portfolio and confirm the same AP story appears in readiness, risk, and blocker signals.

## Rollback Strategy

Future implementation should keep rollback simple:

- Keep old fixture arrays recoverable from Git history.
- Change fixture IDs deliberately and document ID mapping.
- Include a local demo reset step for browser-persisted state that can mask new fixtures.
- Keep data changes isolated to fixture/data files and tests unless AP approves adapter or reset UX work.
- Avoid schema migrations for this local demo phase.
- If canonical seed breaks core flows, revert the fixture commit and clear local demo state before retesting.
