# M5.3a-1c Executable Local DB Assertion Method Plan

## Purpose

M5.3a-1c defines the AP-reviewable method for turning the M5.3a-1b local harness skeleton into an executable local DB assertion path. This milestone is docs-only. It does not implement executable DB assertions, RLS helpers, RLS policies, tenant policy SQL, SQL migrations, package changes, `supabase/config.toml`, runtime changes, DB connections, or remote validation.

M5.3a-1 helper and SELECT policy implementation remains blocked until an executable local DB assertion method is implemented, merged, verified, tagged, and accepted by AP.

## Current Harness Baseline

M5.3a-1b is closed. Its baseline is:

- `package.json` includes `test:rls:m5.3a-1`.
- The current runner is a fail-closed skeleton.
- The scenario contract declares 16 scenarios.
- The runner executes 0 scenarios.
- The runner reports tenant isolation verified: no.
- The accepted local target path still fails closed with `local-db-execution-not-implemented`.
- No RLS helper functions exist.
- No RLS policies exist.
- No executable DB assertions exist.
- No tracked `supabase/config.toml` exists.
- Remote validation remains prohibited.

## Recommended Execution Method

Use a JS runner plus SQL assertion files.

Future implementation should:

- Use the JS runner to enforce local-target proof before any DB action.
- Use SQL assertion files for executable scenario contracts.
- Run only against an AP-approved local Supabase/Postgres target.
- Use Node `pg` or equivalent local DB client only after explicit local-target proof.
- Keep `scripts/applySupabaseMigrations.mjs` out of the default assertion path.
- Never silently accept `DATABASE_URL`.
- Print only sanitized counts and classifications.

## Proof Boundary

M5.3a-1d can prove executable assertion plumbing and safe local execution behavior. It cannot prove final tenant isolation until AP-approved RLS helpers and SELECT policies exist.

Before helper and policy implementation, executable assertions can validate:

- The runner can prove a local target.
- The runner can connect only after local proof.
- The runner can set up local, generated fixtures.
- The runner can execute SQL assertions.
- The runner can fail closed and report sanitized counts.
- The current fail-closed baseline behaves as expected.

Verified local tenant isolation may only be claimed after:

- AP-approved helpers/policies exist for the scoped tables.
- AP-approved local DB target is used.
- AP-approved assertions execute.
- All expected allow/deny outcomes pass.
- Output remains sanitized.

Verified local tenant isolation does not mean production readiness, hosted pilot readiness, export/storage readiness, provider readiness, runtime adapter readiness, artifact-table coverage, or external certification.

## Auth Simulation Recommendation

Future implementation should prefer local SQL session claim simulation.

Planned actors:

- Anonymous scenario: local session with no subject claim.
- Authenticated active member: local authenticated role/session with generated local subject.
- Authenticated disabled organization member: local authenticated role/session with disabled organization membership.
- Authenticated disabled workspace member: local authenticated role/session with disabled workspace membership.
- Authenticated non-member: local authenticated role/session with no membership.
- Server-only fixture setup path: local privileged setup only, never browser-callable.

Planned mechanics:

- Use generated local UUIDs only.
- Use local session state such as `SET LOCAL ROLE authenticated` where available and compatible.
- Use local request claim state such as `SET LOCAL request.jwt.claim.sub` where compatible with the local `auth.uid()` behavior.
- Do not use real JWTs.
- Do not print tokens, claim values, row payloads, DB targets, or local fixture values.
- Consider a local claim/session setup helper only if AP approves it.
- Consider Supabase local Auth fixtures only if AP approves a generic tracked local config.

## Fixture Setup Strategy

Future fixtures should use generated local data only:

- Two local organizations.
- Two local workspaces per organization.
- Active member.
- Disabled organization member.
- Disabled workspace member.
- Non-member.
- Organization-scoped roles.
- Workspace-scoped roles.
- Same-organization and cross-organization rows.
- Same-workspace and cross-workspace rows.
- Deleted-state variants where useful.

Fixture rules:

- No personal IDs.
- No hosted URLs.
- No real project IDs.
- No provider keys.
- No service-role values.
- No tokens.
- No bucket names.
- No `.env` values.
- No row payload dumps in output.

## Local-Target Proof Strategy

Future implementation must prove the target is local before DB action.

Allowed target classes:

- `localhost`.
- `127.0.0.1`.
- Docker local network.
- Another AP-approved local token.

Rejected target classes:

- Hosted URLs.
- Remote hosts.
- Silent `DATABASE_URL`.
- Ignored secret-file fallback.
- Unknown target arguments.

The runner must not print target values. If proof is missing, the runner must fail before any DB action.

## Secret-Safe Output Strategy

Allowed output:

- Scenarios declared.
- Scenarios executed.
- Pass count.
- Fail count.
- Blocked count.
- Tenant isolation verified yes/no.
- Sanitized failure classifications.
- Local target proof status without target value.

Forbidden output:

- DB URL.
- `DATABASE_URL`.
- `.env` values.
- Service-role values.
- Provider keys.
- JWT/private tokens.
- Hosted URLs.
- Bucket names.
- Personal organization, workspace, or project IDs.
- Row payloads.
- Auth headers.
- Claim values.
