# M5.2 Supabase Auth / Organization / Workspace Schema Plan

Milestone: M5.2 Supabase Auth / Organization / Workspace Schema Plan
Branch: milestone/m5.2-supabase-auth-org-workspace-schema-plan
Status: Schema planning pack only

## Purpose

M5.2 defines the hosted pilot Auth, organization, workspace, membership, role, and permission model for AvalaOS Core before any schema implementation begins. It turns the M5.0 enterprise readiness direction and the M5.1 environment/secret hygiene gate into a concrete tenant-authority plan that later schema, RLS, evidence, audit, export, and deployment milestones can follow.

This milestone is documentation-only. It does not create migrations, alter schema, implement RLS, change runtime adapters, change product UI, connect to Supabase, deploy, call providers, or make AvalaOS Core enterprise-production ready.

## Strict Non-Scope

M5.2 does not implement Supabase schema, migrations, RLS policies, Auth UI, runtime auth changes, workspace or organization runtime behavior, deployment, provider execution, Edge Function changes, CI changes, dependencies, Health repository changes, product UI, routing, view keys, Sidebar behavior, guard semantics, permissions, demo data, scoring, Reports/Teams/Admin behavior, or export policy.

M5.2 does not deploy, connect to live Supabase, call providers, create migration files, alter schema, alter RLS, read or expose `.env*` values, print secrets, commit personal hosted-project values, claim compliance certification, claim enterprise-production readiness, claim live provider readiness, claim automated approval authority, assign deterministic scoring decisions to AI, or claim current AGI capability.

## M5.0 And M5.1 Boundary

M5.0 established that AvalaOS Core is buyer-demo RC ready for the AP Invoice Exception Workflow but not enterprise-production ready. It identified the need for Supabase Auth, organization/workspace authority, RLS-first access, evidence and lineage persistence, audit coverage, export controls, server-side provider execution, environment separation, and pilot operations.

M5.1 established the environment and secret hygiene gate. Browser config may use only approved public names, service-role credentials remain server-only, provider secrets remain server-only, `.env*` values must not be read or committed, and `test:secret-hygiene` exists as a local static gate. M5.2 assumes those boundaries remain intact.

## Target Tenant Model

The target tenant model should use Supabase Auth as identity authority, organizations as tenant roots, and workspaces as the primary hosted-pilot work boundary under organizations.

| Entity | Direction |
| --- | --- |
| `auth.users` | Supabase-owned identity authority. Browser receives authenticated session state only. |
| `profiles` or `user_profiles` | App metadata for authenticated users. Stores display name, avatar, status, timestamps, and non-secret profile attributes. It must not store role authority by itself. |
| `organizations` | Tenant root for hosted pilot data, billing/commercial settings, enabled modules, and enterprise-level configuration. |
| `organization_members` or `organization_memberships` | Active organization membership authority. Records user, organization, status, joined/invited timestamps, and org-level role assignment or link. |
| `workspaces` | Organization children and primary hosted-pilot work boundary. A workspace can represent an AP workflow context, business unit, process domain, pilot workspace, or other AP-approved boundary. |
| `workspace_memberships` | Workspace access authority. Records user, workspace, status, workspace role assignment, and optional role override rules. |
| `roles` | Named role definitions with explicit scope. AP must decide whether roles are global, org-scoped, workspace-scoped, or mixed. |
| `permissions` / `role_permissions` | Optional normalized permission model if AP decides JSONB permissions are not sufficient for RLS and audit. |
| Role assignments | Explicit org/workspace role bindings. Role assignment must be server-governed and auditable. |
| Invitations | Later table if AP approves invite lifecycle, SSO, or delegated user onboarding. |

### Tenant Boundary Rules

- Every hosted pilot record should eventually carry `org_id`.
- Most work artifacts should also carry `workspace_id`.
- `user_id` should represent actor, owner, reviewer, approver, creator, updater, assignee, or requester as appropriate.
- Membership authority comes from active organization and workspace membership records, not browser state.
- Role and permission authority comes from server-governed records, not display persona fields.
- Demo personas are local/demo presentation fixtures and must not authorize hosted pilot access.

## Role And Workspace Model Direction

Current buyer-demo roles should map into future enterprise roles without treating display personas as production identities.

| Current buyer-demo persona | Future enterprise direction |
| --- | --- |
| Platform Admin | Org admin for user management, security settings, module settings, and configuration. May receive workspace admin rights by assignment. No service-role credential exposure. |
| AP Process Owner | Workflow owner/reviewer for Assess and Govern review paths in the AP workspace. Can review and approve within assigned authority. |
| Process Analyst | Workspace contributor for process intake, assessment drafting, evidence capture, and Studio handoff preparation. |
| Delivery Lead | Workspace delivery lead for Delivery planning, work items, delivery pack review, and lineage follow-through. |
| Control Reviewer | Control and evidence reviewer for governance, documentation, evidence sufficiency, and review checkpoints. |
| Automation Contributor | Implementation contributor for delivery tasks. Does not receive provider-secret or platform security authority. |
| Buyer Viewer | Read-only or executive viewer for approved dashboards, decision packs, and review artifacts. |

M5.2 separates:

- Display persona.
- Supabase identity.
- App profile.
- Organization membership.
- Workspace membership.
- Role assignment.
- Permission set.

An organization can have multiple workspaces. A user can belong to multiple organizations if AP approves the experience and switching behavior. Workspace should become the primary data boundary for Assess, Govern, Studio, Delivery, and Monitor hosted-pilot records, while organization remains the tenant root and commercial boundary.

Current UI scopes such as `MY_WORK`, `TEAM`, `PROJECT`, and `ORGANIZATION` are product navigation context, not final database tenancy. AP must decide whether a future `WORKSPACE` scope is needed, and how that differs from the existing `View.WORKSPACE` Docs/Admin surface.

## Proposed Future Migration Sequence

M5.2 does not create migrations. A later approved implementation milestone should use additive, reversible migration steps.

1. Confirm profile authority: approve `profiles` versus `user_profiles`, required columns, Auth trigger strategy, and profile lifecycle.
2. Confirm role authority: approve role naming, role scope, permission storage, role assignment tables, and admin escalation rules.
3. Normalize organization memberships: ensure active membership status, role assignment, unique `(org_id, user_id)`, and audit fields are sufficient.
4. Add workspace authority: create `workspaces` with `id`, `org_id`, slug/name, status, timestamps, created/updated actor columns, and unique `(org_id, slug)` constraint.
5. Add workspace memberships: create `workspace_memberships` with `workspace_id`, `org_id`, `user_id`, role assignment, status, timestamps, and unique `(workspace_id, user_id)` constraint.
6. Map module ownership: add or derive `workspace_id` for Assess processes, assessments, review events, Studio document generations, Delivery projects/tasks/epics/sprints, timesheets, handoff ledger entries, delivery packs, source lineage, evidence refs, audit events, exports, and provider policy/audit records where approved.
7. Add data quality constraints: use foreign keys, indexes, status checks, soft-delete or archival strategy, timestamps, `created_by`, `updated_by`, and actor columns where needed.
8. Backfill non-production demo data: seed one non-production demo tenant and one AP Invoice Exception Workflow workspace without using personal hosted values.
9. Prepare RLS design/test inputs: hand off the approved model to M5.3 for deny-by-default policy design and regression tests.

Migration rollback expectations:

- Prefer additive migrations and reversible feature flags.
- Do not drop existing buyer-demo-compatible tables or columns during tenant-model introduction.
- Do not destructively rewrite demo/runtime data without an AP-approved migration and restore plan.
- Keep local/demo fallback behavior intact until a later implementation milestone explicitly changes it.

## RLS Readiness Requirements For M5.3

M5.3 should remain a design/test-plan milestone unless AP separately approves implementation.

M5.3 must receive:

- Final entity and naming decisions from M5.2.
- Canonical membership tables and active status rules.
- Canonical role/permission storage and assignment rules.
- A table-by-table RLS matrix for Auth, org, workspace, evidence, lineage, audit, exports, provider governance, Assess, Studio, Delivery, Monitor, and storage objects.
- Positive and negative test cases for same-org access, cross-org denial, same-workspace access, cross-workspace denial, unauthenticated denial, suspended membership denial, role denial, owner-only behavior, reviewer/admin behavior, and server-only service-role actions.
- A server-only exception boundary for service-role operations.
- No browser service-role key, provider secret, database URL, management token, or personal hosted value.

## Data Migration And Compatibility Guidance

- Buyer-demo behavior must remain intact until a later approved implementation milestone changes runtime behavior.
- Local/demo storage remains demo-only and must not become hosted pilot authority.
- Existing org-scoped records should be mapped to a default non-production AP workspace only through an AP-approved seed/backfill plan.
- Current app `Project` and `Scope` types are not enough to represent workspace tenancy.
- The current Supabase user-to-demo-persona overlay must be replaced later by DB-backed profile, membership, role, and permission loading before pilot use.
- Personal hosted values may be temporary validation aids only under the M5.1 replacement principle; architecture must remain AP-owned and environment-replaceable.

## AP Decisions Needed

- Final organization/workspace hierarchy.
- Whether workspace is mandatory for all hosted pilot records.
- `organization_members` versus `organization_memberships` naming.
- `profiles` versus `user_profiles` naming and lifecycle.
- Role model: global, org-scoped, workspace-scoped, or mixed.
- Whether permissions remain JSONB on roles or normalize into `permissions` / `role_permissions`.
- Workspace switching behavior for users with multiple organizations or workspaces.
- Whether UI `ScopeType` needs a future workspace scope.
- Demo tenant/workspace seed policy.
- Invitation and SSO timing.
- Whether existing guard permissions need canonical DB mapping before RLS.
- Whether a later M5.2a handles migration implementation before M5.3.

## Risks

- Current schema is mostly org-scoped, so workspace authority may require broad FK, backfill, adapter, and RLS planning.
- Current `View.WORKSPACE` is product navigation, not database tenancy.
- Supabase user mapping currently overlays demo personas instead of loading DB-backed profile, membership, role, and permission state.
- Rushed implementation could break buyer-demo behavior.
- Existing draft RLS policies may need redesign for workspace isolation.
- `OrgRole` and demo permission arrays are too coarse to be treated as hosted pilot authorization.
- Provider governance already has future workspace-named surfaces, but canonical workspace authority is not approved.
- Demo seed strategy can drift if personal validation projects are treated as architecture.
