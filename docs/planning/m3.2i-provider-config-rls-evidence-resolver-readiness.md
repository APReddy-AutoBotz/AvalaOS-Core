# M3.2i Provider Config RLS Evidence And Resolver Readiness Review

## Milestone Purpose

M3.2i is a planning/review-only milestone that verifies the M3.2h provider configuration schema/RLS contract and decides whether the system is ready for a future server-side provider resolver planning milestone.

This milestone does not implement a resolver. It creates no migrations, edits no schema SQL, changes no Supabase Functions, changes no runtime behavior, changes no Admin UI, changes no product code, changes no package/dependency files, touches no Health behavior, changes no scoring, changes no browser fallback behavior, adds no provider calls, stores no provider secrets, changes no prompt versions, expands no usage/cost path, and makes no compliance claims.

## M3.2h Schema/RLS Contract Summary

M3.2h introduced or hardened the approved provider governance object set:

1. `ai_provider_key_refs`
2. `ai_provider_configs`
3. `ai_workspace_provider_policies`
4. `ai_provider_audit_events`

Provider scope is limited to `gemini` and `groq`.

Tenant boundary:

- `org_id` remains the enforceable tenant boundary.
- `ai_workspace_provider_policies` keeps future-compatible naming.
- Strict `workspace_id` authority remains deferred because no AP-approved canonical workspace authority exists.

Key-reference contract:

- `ai_provider_key_refs.secret_ref` is a service-only reference identifier.
- `secret_ref` must not contain raw provider keys, encrypted provider keys, auth headers, browser-readable secret values, or secret-manager payload values.
- `ai_provider_configs.key_ref_id` is nullable and references `ai_provider_key_refs(id)`.
- `ai_provider_key_refs` has no FK back to provider configs.

Config and policy metadata contract:

- `ai_provider_configs` stores safe provider configuration metadata only.
- `ai_workspace_provider_policies` stores operation, mode, role, default, and status metadata.
- Browser/client writes to sensitive provider/key/audit objects are not authorized.
- Any future Admin UI writes must be mediated by AP-approved server/control workflows, RLS, audit evidence, and secret-safety gates.

Audit contract:

- `ai_provider_audit_events` is append-only evidence-oriented schema.
- Browser/client audit reads remain deferred pending AP-approved redaction and evidence-view rules.
- Audit metadata must not store raw keys, auth headers, browser-readable secrets, raw prompts, or raw completions.

RLS contract:

- `ai_provider_key_refs`: RLS enabled; no member read policy; no broad authenticated write policy.
- `ai_provider_configs`: RLS enabled; active org members may read non-deleted safe metadata; no browser/client write policy.
- `ai_workspace_provider_policies`: RLS enabled; active org members may read non-deleted policy metadata; no browser/client write policy.
- `ai_provider_audit_events`: RLS enabled; member read deferred; no browser/client write policy.

## `secret_ref` Safety Review

`secret_ref` is ready for resolver contract planning only as a reference identifier. It is not ready for runtime resolution in this milestone.

Required safety rules to carry forward:

- `secret_ref` must stay server-side and service-only.
- It must never contain raw provider keys, encrypted provider keys, auth headers, bearer tokens, browser-readable secret values, or secret-manager payload values.
- Browser/client code must never directly resolve provider keys.
- Browser/client code must never directly write sensitive provider/key/audit objects.
- Future resolver planning must define secret lookup boundaries, evidence requirements, and failure behavior before any implementation milestone.

## Audit Event Safety Review

`ai_provider_audit_events` is suitable for resolver contract planning as an evidence target, not as an implemented runtime writer.

Audit records must not include:

- raw provider keys
- encrypted provider keys
- auth headers
- bearer tokens
- browser-readable secret values
- secret-manager payload values
- raw prompt bodies
- raw completion bodies
- unredacted provider responses

Future resolver planning must define allowed event types, policy-result values, failure classes, correlation IDs, evidence references, actor/service context fields, redaction behavior, and audit visibility rules.

## Workspace Authority Risk

M3.2h intentionally uses `org_id` as the enforceable tenant boundary. The workspace-named policy table remains future-compatible, but M3.2i does not authorize a strict `workspace_id` FK or workspace-scoped uniqueness.

Risk carried forward:

- Resolver planning must not assume workspace authority beyond the approved `org_id` boundary.
- Any workspace-specific resolver behavior must wait for AP approval of a canonical workspace authority.
- Audit `workspace_id`, where present, remains nullable reference metadata until AP approves stronger workspace semantics.

## Admin UI Sequencing Recommendation

Admin UI writes should not precede server/control workflow design.

Recommended order:

1. M3.2j Server-Side Provider Resolver Contract Planning.
2. AP review of resolver gates, secret-safety gates, audit semantics, and failure behavior.
3. Server/control workflow implementation milestone, if AP approves it.
4. Admin UI planning for mediated write requests only after server/control workflows and evidence rules are approved.

The Admin UI must not directly write sensitive provider/key/audit objects or resolve provider keys.

## Stop Conditions Before Resolver Implementation

Stop and request AP approval if the work would require:

- provider runtime calls
- provider key resolution
- raw provider secret storage
- browser/client provider-key access
- browser/client writes to sensitive provider/key/audit objects
- Supabase Function changes
- Admin UI writes
- product-code behavior changes
- Health changes
- scoring changes
- browser fallback changes
- prompt-version changes
- package/dependency changes
- usage/cost expansion
- seeds containing provider data
- unsupported compliance claims
- score, risk-gate, approval, or regulated-decision logic changes
- workspace FK authority not approved by AP
- service-role behavior that bypasses auth, membership, provider policy, provider status, operation, mode, key-reference, prompt/version, evidence, or scanner gates

- service/control workflow mediation is not implemented
