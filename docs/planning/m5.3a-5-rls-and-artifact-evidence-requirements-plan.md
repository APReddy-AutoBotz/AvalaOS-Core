# M5.3a-5 RLS and Artifact Evidence Requirements Plan

## Purpose

M5.3a-5 defines safe future evidence requirements for RLS helper behavior and artifact SELECT isolation.

This milestone keeps the local-readiness loop closed. It does not reopen local readiness, document or run concrete command strings, run database validation, inspect schema, run RLS tests, run artifact assertions, perform hosted validation, or claim readiness.

The plan separates required future proof from the current unproven state so a later AP-approved milestone can decide whether a specific evidence run design is safe enough to authorize.

## Baseline

M5.3a-4 closed the reconciliation gate and recorded the accepted baseline:

- Local readiness remains stopped.
- Local DB remains unresolved.
- Schema readiness is not proven.
- RLS readiness is not proven.
- RLS helper behavior is not newly validated.
- Artifact SELECT isolation is not verified.
- Tenant isolation is not newly verified.
- Hosted readiness is not proven.
- Production readiness is not proven.
- Local startup success is not proven.
- No local readiness evidence exists.

M5.3a-5 does not change that baseline. It defines evidence requirements only.

## Evidence Requirement Categories

Future AP-approved evidence design must cover these categories before any execution can be considered:

- schema availability evidence;
- RLS helper behavior evidence;
- artifact SELECT isolation evidence;
- tenant boundary evidence;
- negative-path and fail-closed evidence;
- harness safety evidence;
- output sanitization evidence;
- proof-boundary evidence;
- rollback and stop-condition evidence.

Each category must define exact assertions, allowed sanitized buckets, prohibited outputs, and stop conditions before any future evidence run is approved.

## RLS Helper Behavior Evidence Requirements

A future approved evidence run must prove only the specific helper behavior assertion approved for that milestone.

Required future proof:

- helper behavior is scoped to the intended tenant, organization, and workspace boundary;
- helper behavior does not allow cross-tenant access;
- helper behavior fails closed on missing context;
- helper behavior fails closed on invalid context;
- helper behavior does not infer tenant context from unsafe sources;
- helper behavior produces only sanitized evidence;
- pass and fail criteria are explicit before execution;
- evidence distinguishes helper behavior from full RLS readiness.

Future evidence must not capture row payloads, auth headers, claim values, DB URLs, local values, secrets, provider keys, service-role values, private tokens, hosted identifiers, or machine-specific values.

Helper behavior evidence cannot be used to claim full schema readiness, full RLS readiness, artifact SELECT isolation, hosted readiness, production readiness, or local startup success.

## Artifact SELECT Isolation Evidence Requirements

A future approved evidence run must prove only the specific artifact SELECT isolation assertion approved for that milestone.

Required future proof:

- artifact SELECT access is scoped to the approved tenant, organization, or workspace context;
- unauthorized artifact SELECT attempts fail closed;
- cross-tenant artifact SELECT attempts fail closed;
- missing tenant context fails closed;
- invalid tenant context fails closed;
- no artifact row payloads are copied into evidence;
- evidence records only sanitized pass/fail buckets;
- evidence does not imply broader artifact readiness beyond the approved assertion;
- evidence does not imply hosted readiness or production readiness.

Artifact SELECT isolation evidence must distinguish the tested artifact assertion from untested artifact tables, mutation behavior, storage/export behavior, adapter behavior, hosted runtime behavior, and production readiness.

## Tenant Boundary Evidence Requirements

Future AP-approved tenant boundary evidence must define these sanitized outcomes before execution:

- same-tenant allowed path, if approved;
- cross-tenant blocked path;
- missing tenant context blocked path;
- invalid tenant context blocked path;
- artifact access blocked path;
- helper blocked path.

All results must be sanitized buckets only. Future evidence may report that an approved assertion passed, failed closed, failed open, was blocked by a safety rule, or was not run. It must not include raw output, row data, identifiers, tenant values, local values, or root-cause inference.

## Harness And Execution Safety Requirements

Any future harness or execution must require explicit AP approval before execution.

The approval must define:

- exact future milestone name;
- exact approved assertion names;
- exact run count;
- one-run limit unless AP explicitly approves otherwise;
- allowed sanitized output fields;
- prohibited output fields;
- cleanup expectations;
- stop and fail-closed conditions;
- proof-boundary wording.

Future harness evidence must not include:

- raw stdout;
- raw stderr;
- raw logs;
- row payloads;
- auth headers;
- claim values;
- DB URLs;
- local paths;
- host values;
- port values;
- IP values;
- `.env` values;
- provider keys;
- service-role values;
- private tokens;
- container IDs;
- image IDs;
- machine-specific values.

The future harness must fail closed if sanitized evidence cannot be produced safely.

## Future Allowed Evidence

Future approved evidence may include only:

- approved milestone name;
- approved assertion name;
- sanitized schema availability bucket;
- sanitized RLS helper behavior bucket;
- sanitized artifact SELECT isolation bucket;

- proof boundaries.

Any field not explicitly allowed by the future approval gate remains prohibited.
