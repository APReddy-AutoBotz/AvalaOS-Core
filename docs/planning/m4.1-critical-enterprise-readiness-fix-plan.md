# M4.1 Critical Enterprise Readiness Fix Plan

## Purpose

M4.1 converts the accepted M4.0a enterprise-readiness audit baseline into a controlled implementation roadmap. It does not fix runtime behavior. It orders the critical and high findings into small PR-sized milestones so each future change can be reviewed, tested, and approved without broad refactors or hidden scope expansion.

## Scope

- Normalize the M4.0a critical and high findings into implementation-ready planning records.
- Define the recommended M4.1a-M4.1i sequence.
- Identify dependencies, demo-readiness gates, AP decision points, and Claude/Gemini review inputs.
- Preserve the M4.0a findings without weakening, hiding, or reclassifying them as complete.

## Non-Scope

This milestone is planning-only. It does not change product code, UI, service logic, routing, scoring, schema, RLS, migrations, Supabase Edge Functions, provider runtime behavior, provider governance behavior, Health, dependencies, deployment state, or secrets.

No live provider path is validated by M4.1. No unsupported compliance claim is introduced. No future M4.1a-M4.1i fix is implemented by this milestone.

## Baseline

- Baseline milestone: `M4.0a Full Repository Inventory And Logic Map`
- Baseline tag: `avalaos-core-m4.0a-full-repo-inventory-logic-map`
- Baseline tag target: `65cd79fe92906dd5c1a9760da742b5025dadcc2c`

M4.0a established the audit baseline for repository inventory, module map, route/view map, state/data flow, handoff map, AI-boundary map, and enterprise readiness risks. M4.1 uses that baseline as evidence, not as a readiness certification.

## Why Planning Comes Before Fixing

The top M4.0a findings cross Assess, Studio, Delivery, Monitor, Admin, permissions, provider-mode positioning, and demo readiness. Fixing them without sequence control could change user-visible behavior, data lineage, route access, or buyer-demo claims in ways that need AP review. M4.1 separates contract fixes, handoff fixes, lineage fixes, route authorization planning, demo-safe product decisions, and persistence/naming cleanup into reviewable milestones.

## Prioritized Implementation Plan

### M4.1a Decision Pack Contract / Rendering Fix

- Objective: align scored Process Detail rendering with the current `DecisionPack` and `HandoffPack` contracts.
- Likely impacted files: Process Detail component, assessment fixtures or tests, focused render regression tests.
- Expected behavior change: scored process detail opens reliably and displays decision rationale, governance controls, evidence, and handoff readiness from existing deterministic results.
- Tests required: unit or component regression for a scored process with a populated `DecisionPack`; no scoring formula changes.
- Risks: accidental score/recommendation behavior changes if implementation reaches into scoring code.
- Rollback strategy: revert Process Detail rendering changes while keeping tests as failing evidence.
- Acceptance criteria: scored process detail no longer references non-contract fields; deterministic scoring output remains unchanged.
- AP decision needed before implementation: No.

### M4.1b Assess-to-Studio Full Handoff Payload Fix

- Objective: pass read-only assessment, Decision Pack, and Handoff Pack source context into Studio generation.
- Likely impacted files: app handoff state, Docs Forge props/source-context model, document generation save path, focused handoff tests.
- Expected behavior change: Studio can prefill or attach source context from the approved assessment path without re-entering the source manually.
- Tests required: Assess-to-Studio handoff test proving payload presence and preserved source refs.
- Risks: payload shape could become too broad or mutable.
- Rollback strategy: disable the source-context handoff while preserving ledger recording.
- Acceptance criteria: generated document metadata can reference its assessment/Decision Pack source.
- AP decision needed before implementation: Yes, approve minimum payload shape and buyer-visible wording.

### M4.1c Docs-to-Delivery Lineage / Evidence Refs Fix

- Objective: preserve source lineage when generated work items become Delivery epics/tasks.
- Likely impacted files: work item import path, task/epic creation helpers, Delivery Pack tests.
- Expected behavior change: imported work items carry generation, assessment, handoff, and evidence refs when available.
- Tests required: import regression proving task lineage and Delivery Pack evidence status.
- Risks: incomplete upstream payload from M4.1b could produce partial lineage.
- Rollback strategy: revert lineage attachment and keep import behavior, with Delivery Pack still warning on gaps.
- Acceptance criteria: imported tasks expose source lineage when source context exists; missing lineage remains visible.
- AP decision needed before implementation: Yes, approve minimum lineage fields and incomplete-lineage policy.

### M4.1d Route / RBAC Guard Design And Implementation Plan

- Objective: design a central route guard for view, scope, module, permissions, and org role.
- Likely impacted files in future PR: app route guard, module config, Sidebar route metadata, focused route tests.
- Expected behavior change in future PR: direct, persisted, or indirect route states follow the same access decision as navigation.
- Tests required in future PR: denied persisted route state, allowed route state, module-disabled route state, org Admin route state.
- Risks: broad route behavior changes can block valid flows if implemented too widely at once.
- Rollback strategy: restore module-only guard and Sidebar visibility while retaining design doc evidence.
- Acceptance criteria for M4.1d: design is complete enough for implementation but no implementation happens in M4.1.
- AP decision needed before implementation: Yes, approve guard precedence and denied-view fallback.

### M4.1e Buyer-Demo Safe Mode / Local-Demo AI Labeling Strategy

- Objective: decide how buyer demos should label, disable, or constrain local-demo/internal-dev AI behavior.
- Likely impacted files in future PR: demo-mode config, visible labels, AI generation surfaces, QA checks.
- Expected behavior change in future PR: buyer demo users cannot confuse local-demo behavior with pilot or production behavior.
- Tests required in future PR: mode-label visibility and no unsupported live-provider validation claim.
- Risks: too much labeling may weaken demo flow; too little labeling weakens governance trust.
- Rollback strategy: revert to controlled internal demo policy and do not show AI generation externally.
- Acceptance criteria: AP chooses one policy before implementation.
- AP decision needed before implementation: Yes.

### M4.1f Admin / AI Controls Visibility Decision

- Objective: decide whether Admin / AI Controls are hidden, static-previewed, or implemented in scoped form for demos.
- Likely impacted files in future PR: Admin navigation, organization setup view, AI Controls route/surface, provider governance read-only status.
- Expected behavior change in future PR: Admin promise matches visible UI.
- Tests required in future PR: visibility by role/permission and no incomplete-control claim.
- Risks: showing partial controls can overstate governance completeness.
- Rollback strategy: hide or label Admin / AI Controls until scoped implementation is approved.
- Acceptance criteria: AP chooses hide, static preview, or scoped implementation before code work.
- AP decision needed before implementation: Yes.

### M4.1g Delivery Pack Lineage / Export Guard

- Objective: require Delivery Pack export behavior to block or clearly label incomplete lineage/evidence refs.
- Likely impacted files in future PR: Delivery Pack builder/view/export flow and tests.
- Expected behavior change in future PR: exports cannot silently present incomplete evidence as complete.
- Tests required in future PR: complete lineage export, incomplete lineage block or label, evidence-ref warning.
- Risks: blocking export can interrupt demos if source lineage is not fixed first.
- Rollback strategy: label-only mode for controlled demo if AP approves.
- Acceptance criteria: incomplete lineage/evidence refs are blocked or explicitly labeled at export time.
- AP decision needed before implementation: Yes, choose block vs label policy.

### M4.1h Route / View Dead-State Cleanup Plan

- Objective: plan cleanup for Reports, Teams, and overloaded Workspace/Admin semantics.
- Likely impacted files in future PR: `View` enum, app render switch, Sidebar items, Reports/Teams visibility.
- Expected behavior change in future PR: route map matches visible product surfaces and does not expose placeholders as finished modules.
- Tests required in future PR: route redirect or hide behavior for Reports/Teams and Workspace/Admin separation.
- Risks: removing visible routes may affect demos that relied on placeholders.
- Rollback strategy: restore hidden route entries without exposing them in navigation.
- Acceptance criteria: each dead/overloaded view has a product decision and route behavior.
- AP decision needed before implementation: Yes.

### M4.1i Legacy Naming And Persistence Cleanup Plans

- Objective: split legacy naming cleanup from persistence cleanup because persistence changes may need separate approval.
- Legacy naming track: decide migration-safe naming for export/storage paths and buyer-visible labels without changing historical docs.
- Persistence track: classify local/demo persistence gaps and identify which require Supabase-backed, RLS-tested implementation.
- Expected behavior change in future PRs: naming cannot leak historical prototype identity into market demos; persistence behavior is disclosed or durable.
- Tests required in future PRs: naming scan for buyer-visible paths, persistence regression for affected adapters if approved.
- Risks: persistence fixes can become schema/RLS work and must not be smuggled into a cleanup PR.
- Rollback strategy: keep naming-only changes separate from any storage behavior changes.
- Acceptance criteria: two separate implementation plans exist; persistence implementation requires explicit approval.
- AP decision needed before implementation: Yes.

## Dependency Sequence

1. `M4.1a` should happen first because the core Decision Pack screen must render before downstream handoff work is meaningful.
2. `M4.1b` should happen before `M4.1c` because Delivery lineage depends on stable document source context.
3. `M4.1c` should happen before `M4.1g` if AP wants export blocking rather than labeling, because exports need lineage data to pass.
4. `M4.1d` should be designed before broad route/view cleanup so guard behavior is not duplicated in ad hoc redirects.
5. `M4.1e` and `M4.1f` require AP product decisions before buyer-facing demos and before Admin polish.
6. `M4.1h` should follow the route/RBAC guard design so dead-state cleanup and access behavior share one route contract.
7. `M4.1i` must keep naming cleanup separate from persistence cleanup; persistence implementation may require schema/RLS approval.

## Codex Implementation Guardrails For Future M4.1x PRs

- Create a branch first for every milestone.
- Keep one fix group per PR.
- Avoid broad refactors and hidden feature expansion.
- Do not change scoring formulas, weights, thresholds, hard stops, gates, or recommendation logic without explicit score-version approval and regression tests.
- Do not add provider/live AI behavior unless a later approved provider milestone authorizes it.
- Do not change schema, RLS, or migrations unless explicitly approved.
- Do not change Health.
- Do not introduce unsupported compliance claims.
- Run typecheck, tests, AI-boundary checks, static scan, build, audit, diff check, and wording scan.
- Require AP review before merge.
