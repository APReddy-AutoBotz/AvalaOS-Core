# M5.3a-7 RLS and Artifact Evidence Harness Design Plan

## Purpose

M5.3a-7 defines a docs-only future evidence harness design and assertion workflow for RLS and artifact evidence.

This plan translates the M5.3a-6 proposed assertions into a safe future harness design concept. It keeps the work at planning level only and prevents DB execution, Supabase stack work, Docker work, schema inspection, RLS tests, artifact assertions, hosted validation, local-readiness reopening, or readiness claims.

M5.3a-7 does not implement a harness, add tests, inspect schema, validate DB behavior, validate RLS behavior, validate artifact behavior, or produce readiness evidence.

## Proposed Harness Design Concept

The future harness concept is design-only:

- harness mode: `approval-gated-sanitized-evidence-harness`;
- execution mode: not implemented;
- run mode: not approved;
- DB execution: not approved;
- schema inspection: not approved;
- RLS tests: not approved;
- artifact assertions: not approved;
- output mode: sanitized buckets only;
- raw output mode: prohibited;
- local value capture: prohibited;
- proof-boundary output: required.

This plan does not add implementation code, executable pseudocode, concrete command strings, SQL, scripts, package scripts, migrations, RLS policy changes, or test files.

## Proposed Assertion Workflow

The future assertion workflow is planning-level only:

- AP selects the exact future milestone.
- AP selects exact assertion names.
- AP approves scope and tenant boundary.
- AP approves whether schema availability may be checked.
- AP approves whether RLS helper behavior may be checked.
- AP approves whether artifact SELECT isolation may be checked.
- AP approves same-tenant and negative-path evidence.
- AP approves run count.
- A future harness checks prerequisites.
- A future harness fails closed if prerequisite, scope, tenant boundary, or output safety is ambiguous.
- A future harness records only sanitized buckets.
- A future harness records proof boundaries.
- A future harness stops without retrying into broader scope.

No step in this workflow is approved for implementation or execution by M5.3a-7.

## Harness Safety Controls

Future harness design must include these safety controls before any implementation can be considered:

- one-run default unless AP approves otherwise;
- no raw stdout or stderr capture;
- no raw log capture;
- no stack traces;
- no scratch output;
- no local paths;
- no `.env` values;
- no host, port, or IP values;
- no DB URLs;
- no database, user, project, or target values;
- no row payloads;
- no auth headers;
- no claim values;
- no provider keys;
- no service-role or private tokens;
- no hosted identifiers;
- no container or image IDs;
- no machine-specific values;
- no root-cause inference;
- no readiness overclaims;
- no local-readiness reopening;
- fail closed if safe sanitized evidence cannot be produced.

Path A is a future docs-only approval gate to decide whether a synthetic or safe harness implementation may be created.

Path A must not execute DB, RLS, or artifact checks.

Path B updates source-of-truth, readiness, and task-ledger docs if tracking drift is found.

### Path C - Hosted Or Nonlocal Evidence Planning

Path C creates a separate docs-only planning track for hosted or nonlocal validation.

Path C must not imply local readiness.

### Path D - Pause

Path D stops here and waits for AP's broader M5.3 roadmap decision.

Recommend Path A only if AP wants to continue toward harness implementation planning.

Otherwise choose Path B if tracking drift is the priority.
