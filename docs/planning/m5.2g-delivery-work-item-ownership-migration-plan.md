# M5.2g Delivery Work Item Ownership Migration Plan

Milestone: M5.2g Delivery Work Item Ownership Migration Plan

Branch: milestone/m5.2g-delivery-work-item-ownership-migration-plan

Status: Docs-only planning pack

Recommended next milestone: M5.2g-a Delivery Work Item Authority Migration Implementation Plan/Implementation

## Purpose

M5.2g defines the Delivery work-item ownership strategy needed before future workspace-aware RLS, Delivery Pack persistence, Monitor persistence, export controls, and hosted Delivery runtime work can safely rely on durable work-item records.

This milestone is planning only. It chooses a narrow future Delivery work-item authority direction and records the migration, FK, RLS-readiness, runtime-compatibility, and legacy-draft-schema risks that AP must approve before implementation.

## Strict Non-Scope

M5.2g does not create SQL migrations, edit existing migrations, alter schema, implement RLS policies, validate constraints, seed data, backfill data, change runtime adapters, change UI, change routing, change view keys, change demo data, change scoring, change provider execution, change export behavior, change deployment configuration, change CI, change package dependencies, change Reports/Teams/Admin behavior, or modify the separate Health repository.

M5.2g does not connect to hosted Supabase, run remote validation, run `scripts/applySupabaseMigrations.mjs`, run `npm run db:migrate`, run `npm run test:rls:assess`, call providers, or inspect `.env` values or secrets.

M5.2g does not claim hosted Delivery runtime readiness, tenant isolation enforcement, production readiness, certification status, machine-owned approvals, or current AGI capability.

## Current Authority Chain

Applied migration-chain authority currently includes:

- M5.2a tenant and workspace authority: `profiles`, `organizations`, `workspaces`, `roles`, `organization_members`, and `workspace_memberships`.
- M5.2c Assess authority: `assess_processes`, `assessments`, and `assessment_review_events`.
- M5.2f Project authority: `projects`.
- M5.2d-a Studio/Docs generation authority: `document_generations`.

Delivery work-item authority is not implemented. No applied migration currently creates `delivery_work_items`, `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, or `timesheet_entries`.

## Option A Versus Option B

| Decision area | Option A: promote draft Agile-like tables | Option B: create one `delivery_work_items` authority table first |
| --- | --- | --- |
| Product fit | Risks expanding into generic project-management behavior. | Keeps AvalaOS focused on governed delivery work items. |
| Buyer-demo compatibility | Mirrors current adapter shape but could create false hosted-runtime confidence. | Preserves buyer-demo runtime while planning a cleaner authority table. |
| Current adapter shape | Closer to current `deliveryAdapter` targets: `tasks`, `epics`, `sprints`, comments, activity, timesheets. | Requires later AP-approved adapter migration; no runtime claim in M5.2g-a. |
| Mock data shape | Closer to current canonical demo tasks, epics, and sprints. | Can preserve key task semantics without committing the full draft Agile model. |
| RLS readiness | More tables, more cross-table policies, more legacy cleanup. | Smaller first isolation surface with project/workspace ownership. |
| Source lineage and evidence boundaries | Draft `tasks.metadata` can blur lineage authority. | Explicitly marks `source_lineage` as compatibility payload, not authority. |
| Migration complexity | Broad table promotion, cascading FK review, old policies, and backfill risk. | One additive table with staged FKs and no validation. |
| Delivery Pack needs | Still requires pack authority decision after multiple tables. | Gives pack planning a single work-item anchor first. |
| Monitor needs | Risks premature telemetry or signal persistence. | Keeps Monitor derived until AP approves persistence. |
| Legacy cleanup risk | Must confront old org-scoped policies, cascades, and single-column project FKs immediately. | Leaves legacy draft/manual tables untouched and plans cleanup separately. |
| Backfill risk | High because epics, sprints, tasks, comments, activity, and timesheets all need mapping. | Lower because one table can start nullable-first with no seed/backfill. |
| Runtime fail-closed risk | Existing runtime may appear closer to hosted-ready before RLS is approved. | Explicitly separates authority planning from runtime enablement. |

Recommendation: choose Option B unless AP identifies a hard blocker. Future M5.2g-a should create one narrow AvalaOS-specific `delivery_work_items` authority table and defer draft Agile-like table promotion.

Future M5.2g-a should explicitly defer:

- `sprints`
- `tasks`
- `task_comments`
- `task_activity_events`
- `timesheet_entries`

## App ID Hardening

If AP approves `app_id`, it must be treated as a nullable compatibility bridge only.

`app_id` is:

- not tenancy authority;
- not globally unique;
- not a Supabase identifier;
- not a replacement for `id`;
- not a secure external reference;
- not enough to authorize reads or writes.

Future uniqueness should be scoped to active, non-deleted rows by at least `(org_id, project_id, app_id)`.

If AP rejects `app_id`, later runtime adapter work must map through UUID authority instead of committing UI/demo identifiers as durable lookup authority.

## Metadata And Source Lineage Hardening

`metadata` and `source_lineage` should be compatibility payloads only.

They are not:

- durable evidence authority;
- durable approval authority;
- durable lineage authority;
- export or storage authority;
- review authority;
- cross-tenant reference authority.

They must not store:

- secrets;
- provider keys;
- service-role values;
- raw private tokens;
- personal hosted values;
- unsupported certification claims.

If metadata carries extra assignees, dependencies, subtasks, user stories, or order rank, those fields remain non-authority UI/runtime compatibility until AP approves normalized persistence.

## FK And Delete Behavior Proposal

Future M5.2g-a should use staged `NOT VALID` FKs where existing/manual rows may exist.

Tenant, source, project, and document references should use default/no-cascade delete behavior:

- `org_id -> organizations(id)`
- `(workspace_id, org_id) -> workspaces(id, org_id)`
- `(project_id, org_id) -> projects(id, org_id)`
- `(document_generation_id, org_id) -> document_generations(id, org_id)`
- `(source_process_id, org_id) -> assess_processes(id, org_id)`
- `(source_assessment_id, org_id) -> assessments(id, org_id)`

Actor/profile references should use `ON DELETE SET NULL` where appropriate:

- `created_by`
- `updated_by`
- `owner_id`
- `assigned_to`
- `reporter_id`

Future M5.2g-a should not validate constraints, seed data, backfill data, or perform destructive cleanup unless AP separately approves that work.

## Legacy Draft Schema Risk

Old draft/manual non-production Delivery tables may exist:

- `epics`
- `sprints`
- `tasks`
- `task_comments`
- `task_activity_events`
- `timesheet_entries`

Option B must not alter or drop those tables in M5.2g-a. M5.2g-a must not claim that those legacy draft Delivery tables are secured or hosted-ready.

If AP later chooses draft-table promotion, separate cleanup planning is required for:

- legacy org-scoped Delivery policies;
- old cascade FKs;
- single-column project FKs;
- task and timesheet cascade behavior;
- draft `app_id` uniqueness that is not active/non-deleted scoped;
- adapter migration away from draft Agile-like tables or into a deliberately approved promotion path.

## RLS Posture

Future M5.2g-a may enable RLS on `delivery_work_items`, but should create no RLS policies.

This is fail-closed readiness only. Tenant isolation remains unproven until M5.3a or a later AP-approved RLS implementation and isolation-test milestone.

Future RLS tests must cover:

- anonymous denial;
- authenticated non-member denial;
- disabled-member denial;
- cross-organization denial;
- cross-workspace denial;
- Buyer Viewer read-only behavior where AP approves read access;
- Delivery Lead permitted updates;
- assigned contributor own-work updates only;
- no cross-tenant source lineage references;
- no cross-tenant document, evidence, handoff, or Assess references.

## Runtime Compatibility Posture

No runtime changes are made in M5.2g.

No runtime changes should be made in future M5.2g-a unless AP separately approves them.

Current `deliveryAdapter` still targets draft Agile-like tables when Supabase is configured. A future `delivery_work_items` table does not make hosted Delivery runtime ready by itself. Supabase runtime may remain fail-closed or incompatible until adapter work, RLS policies, and isolation tests are separately approved.

Buyer-demo local/mock behavior remains the controlling compatibility boundary for M5.2g.

## Risks

- Over-scoping into generic project-management behavior.
- Unified work-item model may not map perfectly to existing draft Agile-like tables.
- Runtime adapter compatibility remains unresolved.
- `app_id` compatibility debt may persist if not migrated to UUID authority.
- `source_lineage` JSON can create false authority if mistaken for durable lineage.
- Comments, activity, timesheets, dependencies, subtasks, and user stories can expand scope.
- Delivery Pack and Monitor derived-versus-persisted boundaries remain unsettled.
- RLS false confidence remains until policies and isolation tests exist.
- Legacy draft schema cleanup or quarantine may be needed.
- Backfill and constraint validation remain future risks.
- Buyer-demo compatibility could be harmed if runtime begins depending on new authority too early.
