# M4.0a Full Repository Inventory And Logic Map

## Purpose

M4.0a is an audit-only milestone for the AvalaOS Core repository. Its purpose is to inspect the full tracked codebase and document the repository structure, active modules, route/view coverage, golden path logic, state and persistence behavior, AI boundary, and enterprise readiness risks before any M4.1 implementation work begins.

This milestone does not fix defects. It creates a shared map of what exists, what is partial, what is unreachable, and what could weaken enterprise buyer trust.

## Why This Audit Matters For Market Readiness

AvalaOS Core is positioned as a governed AI and automation delivery platform. The product promise depends on a reliable chain:

Organization -> Process Catalog -> Guided Assessment -> Deterministic Score -> Recommendation / Decision Pack -> Avala Govern Lite -> Avala Studio Handoff -> Generated Documents -> Work Items -> Avala Delivery Lite Board -> Avala Monitor Dashboard.

Before market-facing polish, the team needs evidence that this chain is understandable, defensible, and correctly bounded. The audit focuses on:

- Whether deterministic scoring remains the source of authority for scores, gates, recommendations, risk, and handoff readiness.
- Whether AI remains assistive and server-side for pilot or production behavior.
- Whether human approval and evidence are visible before handoff.
- Whether route, module, RBAC, and persistence behavior match the product story.
- Whether demo/local behavior could be confused for pilot or production readiness.

## Scope

This inspection covers tracked files in `APReddy-AutoBotz/AvalaOS-Core` only:

- Root app and configuration files.
- React/Vite app shell and route/view logic.
- Components under `components/assess`, `components/docs`, `components/delivery`, `components/auth`, and `components/shared`.
- Domain types in `types.ts`.
- Module gating in `constants/moduleConfig.ts`.
- Mock data, starter packs, templates, and question banks.
- Services, adapters, persistence helpers, deterministic scoring, governance, handoff, delivery pack, export, and AI boundary logic.
- Supabase Edge Function sources, shared provider resolver logic, schema contracts, migration files, and RLS docs.
- Scripts, tests, CI workflow, docs, `.agent` operating files, and MCP server specs.

## Non-Scope

The milestone explicitly excludes:

- Product behavior changes.
- UI code changes.
- Service logic changes.
- Route or navigation changes.
- Scoring formula, weight, threshold, hard-stop, gate, or recommendation changes.
- Schema, RLS, migration, or Supabase Edge Function changes.
- Provider governance/runtime changes.
- Deployed function invocation.
- AI provider calls.
- Provider key creation, printing, or use.
- Dependency changes.
- Health work.
- Unsupported compliance claims.
- Any claim that live provider paths are validated by this milestone.

## Inventory Method

The audit used tracked-file inventory and static inspection only. No `.env` files were opened or scanned.

Primary methods:

- Confirm repository remote, branch, and baseline tag.
- Inspect tracked files with `git ls-files`.
- Inspect typed tracked files with `git ls-files "*.ts" "*.tsx" "*.js" "*.mjs" "*.json" "*.md" "*.sql" "*.css"`.
- Inspect app shell route rendering in `App.tsx`.
- Compare `View` enum coverage, `VIEW_MODULE_MAP`, Sidebar item visibility, nested delivery/team/my-work render switches, and Admin routing.
- Inspect persistence boundaries across `StorageKeys`, `usePersistentState`, adapters, Supabase configuration checks, and mock data.
- Inspect deterministic scoring, assessment transitions, Govern Lite card creation, Docs generation, handoff recording, Delivery Pack construction, delivery policy checks, and timesheet permissions.
- Inspect AI runtime policy, browser fallback policy, Edge client invocation, Supabase Edge Function entrypoints, provider resolver, and server secret lookup rules.
- Run static scans for route/module/persistence/AI/provider/security wording patterns.
- Run required verification commands after docs were authored.

## Guardrails

Product law applied throughout:

- Assess before automation.
- Govern before execution.
- Humans approve risk.
- Evidence proves every decision.

Non-negotiable audit guardrails:

- Deterministic scoring remains deterministic.
- AI must not decide final scores, gates, risk tiers, approvals, recommendations, or regulated decisions.
- Browser-side raw provider secrets and browser-side AI execution are not acceptable for pilot or production behavior.
- Server-side AI through Supabase Edge Functions remains the pilot and production direction.
- Avala Govern Lite remains a governance surface, not runtime execution control.
- Avala Delivery Lite remains a governed handoff and work item surface, not a full Jira replacement.
- No unsupported compliance claims are added.
- Health is not modified.
- Historical prototype repositories are not used.

## No-Fix Rule

M4.0a records issues but does not correct them. Findings are intentionally deferred into future milestones because fixes may require product decisions, UX review, schema/RLS design, AI provider strategy, or staged regression testing.

Do not fix yet:

- Route/navigation changes until M4.1.
- RBAC enforcement changes until M4.2.
- Persistence and schema/RLS changes until a dedicated data milestone.
- Design system or visual trust changes until UX review.
- Provider runtime or deployed Edge changes until an approved provider milestone.
- Scoring, recommendation, gate, threshold, or weight changes without explicit score versioning and regression tests.
- Health proof-pack work inside AvalaOS Core unless explicitly requested.

## Acceptance Criteria

M4.0a is accepted when:

- The repo remote is confirmed as `APReddy-AutoBotz/AvalaOS-Core`.
- The baseline tag is recorded.
- Exactly the four allowed docs are created or updated.
- No code, UI, service, routing, scoring, schema, Supabase function, provider runtime, dependency, or Health files are changed.
- The repo inventory covers root files, app entrypoints, components, services, constants, mock data, hooks/providers, Supabase functions, schema docs, scripts/tests, docs, agent/workflow files, assets, and styles.
- Every product module and submodule is mapped with purpose, key screens, services, data sources, persistence mode, readiness, missing pieces, and risks.
- Every `View` enum value is compared against App render coverage, Sidebar visibility, module mapping, scope restrictions, permissions, and reachability.
- Golden path handoffs are mapped with source, target, trigger, data object, service/function, evidence record, approval requirement, persistence, UX clarity, and risks.
- State and data domains are classified by source of truth, read path, write path, persistence layer, tests, and enterprise risk.
- A severity-ranked enterprise readiness findings table is produced.
- The evidence doc answers whether this can be demoed to a serious enterprise buyer.
- Required verification commands are run and recorded.
