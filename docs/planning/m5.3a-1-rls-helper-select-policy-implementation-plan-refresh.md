## Helper Function Plan

Future helper functions are limited to:

- `public.is_active_org_member(p_org_id uuid)`
- `public.is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`

Helper posture:

- `LANGUAGE sql`
- `STABLE`, not `IMMUTABLE`
- boolean-only output
- hardened `search_path`
- schema-qualified references wherever possible
- `SECURITY DEFINER` only if AP approves it to avoid recursive RLS failure
- no tenant metadata output
- no role or permission existence output
- no service-role or browser exposure
- `REVOKE EXECUTE FROM PUBLIC`
- `REVOKE EXECUTE FROM anon` where applicable
- `GRANT EXECUTE` only to `authenticated`

Helper predicates must require `auth.uid()`, active non-deleted membership, active non-deleted parent organization, and active non-deleted workspace where applicable. Permission helpers are explicitly out of scope for this slice.

## Recursion And Security Pattern

Main recursion risks:

- Helper functions query membership tables while membership tables are RLS-protected.
- Membership policies depend on helper functions that query the same membership tables.
- Role policies depend on membership visibility.
- Organization and workspace policies depend on helpers that resolve memberships.

Safest future pattern:

- Use direct `profiles.id = auth.uid()` for profile visibility.
- Use the two approved helpers only for membership resolution and parent org/workspace state checks.
- Prefer a minimal `SECURITY DEFINER` design for those two helpers if local validation shows direct joins recurse or fail closed incorrectly.
- Keep helper ownership, grants, and `search_path` hardened before any policy relies on helpers.
- Do not add permission helpers, role helpers, artifact dependencies, provider dependencies, or service-role/browser behavior in this slice.
- Validate that disabled, invited, archived, suspended, removed, or deleted membership cannot bootstrap access.
- Validate that disabled, archived, suspended, or deleted parent org/workspace rows cannot grant access.

Future changes should focus on `docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql` so the current 16 blocked scenarios become executable pass/fail assertions after helpers and SELECT policies exist.

## Risks

- Recursive RLS can block legitimate membership bootstrap if helpers and policies depend on each other incorrectly.
- Overbroad definer helpers can bypass tenant boundaries if grants or `search_path` are weak.
- Role metadata or membership metadata can leak cross-tenant existence if policies are too broad.

- Runtime may remain fail-closed or incompatible until later adapter, artifact policy, export, storage, deployment, and operations work.
