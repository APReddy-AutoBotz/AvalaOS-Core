# M5.3a-1 Local Isolation Validation Plan

## Purpose

This architecture plan defines the local validation gate for the future M5.3a-1 RLS helper and SELECT policy implementation. It is docs-only. It creates no migrations, helper functions, policies, fixtures, tests, seeds, backfills, runtime changes, or validation scripts.

## Current Validation Finding

Repository inspection found:

- No tracked `supabase/config.toml`.
- Existing `test:rls:assess` and `db:migrate` scripts call `scripts/applySupabaseMigrations.mjs`.
- `scripts/applySupabaseMigrations.mjs` requires `DATABASE_URL`.
- Remote Supabase validation remains prohibited.

Therefore, future RLS implementation should not claim verified tenant isolation unless AP approves a repeatable local-only validation harness.

## Validation Gate Decision

Default posture:

- M5.3a-1 implementation is blocked until AP approves local validation.
- Static-review-only implementation is allowed only as an explicit AP override.
- Static-review-only implementation must not claim verified tenant isolation.
- No `.env` values, secrets, provider keys, service-role values, personal hosted values, hosted URLs, bucket names, or remote project identifiers may be used or printed.

Recommended next milestone:

`M5.3a-1a Local RLS Validation Harness Plan`

## Future Harness Requirements

A future local validation harness should provide:

- A repeatable local-only database target.
- Non-secret generated fixture IDs.
- Clear separation of anonymous, authenticated, and server-only setup behavior.
- Pass/fail and count output only.
- No secret value output.
- No remote Supabase connection.
- No dependence on ignored local secret files.
- Cleanup or isolated local state for repeated runs.

## Fixture Model

Future M5.3a-1 fixtures should be limited to identity and membership authority:

| Fixture group | Required shape |
| --- | --- |
| Organizations | Two local organizations. |
| Workspaces | Two workspaces per organization. |
| Users | Active member, disabled organization member, disabled workspace member, non-member. |
| Roles | Scoped organization roles and scoped workspace roles. |
| Membership records | Same-org, cross-org, same-workspace, cross-workspace, disabled, and deleted-state variants. |

Fixtures must not include personal values, hosted values, provider keys, service-role values, tokens, bucket names, or `.env` values.

## Isolation Test Contract

Future M5.3a-1 negative tests:

- Anonymous user is denied.
- Authenticated non-member is denied.
- Disabled organization member is denied.
- Disabled workspace member is denied.
- Org A user cannot read Org B organization metadata.
- Org A user cannot read Org B membership metadata.
- Workspace A user cannot read Workspace B workspace membership metadata.
- Workspace A user cannot read Workspace B scoped role metadata.
- Service-role behavior remains server-only and is not browser-callable.

Future M5.3a-1 positive tests:

- Own active, non-deleted profile is readable.
- Own active, non-deleted organization memberships are readable.
- Own active, non-deleted workspace memberships are readable.
- Organization metadata is readable only through active membership.
- Workspace metadata is readable only through active workspace membership.
- Role metadata is scoped to active organization or workspace membership.

Artifact-table tests are deferred to M5.3a-2 or later.

## Local Client Separation

Future tests should separate:

- Anonymous local client.
- Authenticated local member client.
- Authenticated local disabled-member client.
- Authenticated local non-member client.
- Server-only fixture setup path, if AP approves a local service role pattern.

The server-only setup path must never be represented as browser-callable behavior.

## Static Review Gates Before Local Execution

Before any local validation run, static review should confirm:

- Helper names match the approved plan.
- Helpers are boolean-returning only.
- Helpers are `STABLE`, not `IMMUTABLE`.
- `SECURITY DEFINER` is used only with AP-approved justification.
- `search_path` is hardened.
- Execute grants are restricted to `authenticated`.
- No broad `FOR ALL` policies exist.
- No mutation or DELETE policies exist.
- No artifact, provider governance, storage/export, legacy draft, Delivery Pack, Monitor, lineage, or handoff policies exist.
- No service-role behavior is exposed to browser contexts.
- No secret-looking literals or personal hosted identifiers are committed.

- `docs/schema/m5.3a_1_rls_membership_isolation_regression.sql`

## Runtime And Hosted Readiness

Local isolation validation for M5.3a-1 would prove only the identity, organization, workspace, role, and membership SELECT policy surface covered by the harness. It would not prove hosted runtime readiness.

Hosted pilot readiness still requires later artifact policies, runtime adapter approval, export and storage controls, deployment evidence, operations runbooks, backup/restore expectations, audit/evidence persistence, and AP acceptance.

## AP Decisions Needed

- Approve the local-only validation method.
- Decide whether M5.3a-1 implementation is blocked until the local harness exists.
- Decide whether static review alone is acceptable as an explicit override.
- Approve fixture setup method and client separation.
- Approve whether a local server-only setup path may be used.
- Approve output rules for pass/fail counts and no secret values.

## Risks

- No tracked local Supabase config means implementation can drift into static-review-only evidence unless AP approves a harness.
- `DATABASE_URL`-based scripts can create accidental secret exposure if not wrapped by no-value logging rules.
- Remote validation would violate the current milestone boundary.
- Static review cannot prove cross-tenant denial behavior.
