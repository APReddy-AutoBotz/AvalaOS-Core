# M5.2g Delivery Work Item Schema Inventory

Milestone: M5.2g Delivery Work Item Ownership Migration Plan

Branch: milestone/m5.2g-delivery-work-item-ownership-migration-plan

Status: Docs-only architecture inventory

## Purpose

This inventory separates applied authority from draft schema, runtime adapter targets, mock/demo data, derived read models, browser/local persistence, and deferred authority for Delivery work-item planning.

## Applied Authority Baseline

Applied migration-chain authority currently includes:

- tenant and workspace tables: `profiles`, `organizations`, `workspaces`, `roles`, `organization_members`, `workspace_memberships`;
- Assess tables: `assess_processes`, `assessments`, `assessment_review_events`;
- Delivery project table: `projects`;
- Studio/Docs generation table: `document_generations`;
- provider governance tables from earlier applied migrations.

No applied migration currently creates `delivery_work_items`, `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, `timesheet_entries`, Delivery Pack records, Monitor signal records, lineage authority, handoff authority, or export records.

## Draft Schema Findings

`docs/schema/initial_schema.sql` defines draft `projects`, `epics`, `sprints`, `tasks`, `document_generations`, and `timesheet_entries`. These draft tables are useful design references but are not fully aligned with M5 workspace-aware authority.

Draft Delivery risks include:

- `epics.project_id`, `sprints.project_id`, and `tasks.project_id` use single-column project references.
- Several tenant artifact references use `ON DELETE CASCADE`.
- `tasks.parent_id` uses cascade behavior.
- `task_comments`, `task_activity_events`, and `timesheet_entries` depend on draft `tasks`.
- Draft policies are mostly organization-scoped.
- Draft `app_id` indexes are not active/non-deleted scoped.

## Runtime Adapter Findings

`services/adapters/deliveryAdapter.ts` currently targets draft Delivery tables when Supabase is configured:

- `projects`
- `tasks`
- `epics`
- `sprints`
- `task_comments`
- `task_activity_events`

The adapter uses `app_id` as a UI/demo bridge and persists task compatibility payloads in `tasks.metadata`, including assignees, reporter, parent/subtask refs, dependencies, order rank, comments, user stories, activity log, and `sourceLineage`.

`services/adapters/timesheetAdapter.ts` currently targets `timesheet_entries` and resolves task/project `app_id` through draft table relationships.

These adapter paths are not changed in M5.2g. Future `delivery_work_items` authority does not make these Supabase runtime paths hosted-ready by itself.

## Option B Target Architecture

Option B creates one AvalaOS-specific authority table first:

- `delivery_work_items`

The target table should represent governed Delivery work items under organization, workspace, and project authority. It should not attempt to replace a full agile project-management system.

The table should be able to link, where available, to:

- `projects(id, org_id)`;
- `document_generations(id, org_id)`;
- `assess_processes(id, org_id)`;
- `assessments(id, org_id)`;
- `profiles(id)` for actors.

The table may preserve compatibility payloads in `metadata` and `source_lineage`, but those payloads remain non-authority until AP approves normalized persistence.

## Explicit Deferrals

M5.2g and future M5.2g-a should defer:

- `epics`;
- `sprints`;
- `tasks`;
- `task_comments`;
- `task_activity_events`;
- `timesheet_entries`;
- Delivery Pack records and items;
- Monitor project/task signal tables;
- burndown or live telemetry tables;
- standalone lineage authority;
- standalone handoff authority;
- export and storage object records;
- runtime adapter changes;
- RLS policies.

## Dependency Direction

M5.2g-a can create the work-item authority table after AP approves Option B. RLS implementation should wait until AP confirms enough ownership boundaries exist for meaningful project, work-item, Studio/Docs, Assess, lineage, handoff, export, and provider isolation tests.

M5.3a should not be treated as unblocked solely by this planning milestone.
