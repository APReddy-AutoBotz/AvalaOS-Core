## Permission Semantics

First implementation default:

- Admin role bypasses listed permissions.
- Non-admin users are allowed if they have any one listed permission.
- Views with no listed permissions are allowed when module and scope checks pass.

Do not change to all-permission enforcement without AP approval.

## Module Enablement Dependency Map

| Input | Current source | Current behavior | Future guard role |
| --- | --- | --- | --- |
| Enabled modules | `currentOrganization.enabledModules` | Defaults to all modules when missing or empty. | Determines whether module view can render. |
| View module | `VIEW_MODULE_MAP` | Maps most `View` values to product modules. | Central metadata source should replace duplicate local checks. |
| Module home | `MODULE_HOME_VIEW` | Used by `firstEnabledView` and ModuleJourney. | Used as first fallback candidate after access filtering. |
| Organization setup | Organization provider/setup view | Empty module set is prevented or normalized. | No accessible module should use setup/admin-safe fallback. |

## Fallback Map

| Denial reason | Guard severity | Fallback |
| --- | --- | --- |
| `auth_loading` | `wait` | Keep current loading shell; do not redirect. |
| `unauthenticated` | `redirect` | Auth/login flow. |
| `no_organization` | `redirect` | Onboarding wizard. |
| `setup_required` | `redirect` | Organization setup/Admin-safe view. |
| `deferred_view` | `hide` or `redirect` | AP-approved hide/defer behavior. |
| `disabled_module` | `redirect` | First accessible enabled module home. |
| `invalid_scope` | `redirect` | Same module home under a valid scope. |
| `missing_permission` | `hide` or `redirect` | First accessible view in same module, else first accessible enabled module home. |
| `stale_persisted_view` | `redirect` | Resolver fallback after auth/org state resolves. |
| `admin_decision_pending` | `hide` or `warn` | AP-approved Admin/AI Controls behavior. |

## Guard Decision Table

| Scenario | Expected future result |
| --- | --- |
| Auth is still loading | Return `allowed: false`, `reason: auth_loading`, `guardSeverity: wait`; no state mutation. |
| User is absent after loading | Deny to auth flow. |
| User exists but no organization exists | Deny to onboarding/setup flow. |
| Organization scope is active | Render organization setup/Admin-safe surface only after Admin/permission checks pass. |
| View module is disabled | Deny and fallback to accessible enabled module home. |
| Scope is not in view metadata | Deny and fallback to valid scope/view pair. |
| User lacks all listed permissions and is not Admin | Deny by `missing_permission`. |
| View status is `deferred` or `decision_pending` | Follow AP-approved hide/defer/fallback behavior. |
| Persisted view is stale | Resolve after auth/org load and replace with fallback. |
| All checks pass | Return `allowed: true`. |

## Non-Sidebar Entry Point Risk Map

| Entry point | Current risk | Future guard requirement |
| --- | --- | --- |
| `handlePrimaryAction` | Selects the first enabled module path without permission/scope validation. | Resolve target view before setting it. |
| Project selector / Docs Forge entry | Sets project scope and `DOCS_FORGE` outside Sidebar. | Validate Docs module, project scope, and permission. |
| Assess-to-Studio handoff | Opens `DOCS_FORGE` after source-context handoff. | Validate Docs access before navigation while preserving source context behavior. |
| `ModuleJourney` | Navigates to module home views from enabled module list only. | Use shared guard metadata and access result. |
| Persisted `currentView` / `currentScope` restore | Can rehydrate stale view/scope after user, module, or org changes. | Resolve after auth/org loading completes and replace only denied state. |
| Direct `setCurrentView` handlers | Many flows set Process Detail, Guided Assessment, Workspace, Backlog, Docs, and Dashboard directly. | Centralize navigation through a guarded setter or access resolver wrapper. |
| Fallback effects in `App.tsx` | Redirects Reports, Teams, and disabled modules through local exceptions. | Replace with resolver reasons and AP-approved fallback rules. |

## Risks If Not Implemented

- Sidebar can hide a view while direct or persisted view state can still attempt to render it.
- Module enablement can be mistaken for permission authorization.
- Stale persisted state can undermine demo trust after org/module/user changes.
- Reports, Teams, Workspace, Admin, and AI Controls can continue to rely on one-off behavior.
- Future route/view cleanup can duplicate guard logic unless the central model lands first.
