# M5.0 Supabase Persistence, RLS, And Audit Architecture Map

Milestone: M5.0 Enterprise Supabase Readiness Plan
Status: Architecture planning map only

## Purpose

This map records the current repository architecture relevant to Supabase persistence, RLS, evidence, lineage, audit, exports, provider governance, and environment separation. It defines the target direction for later M5 implementation plans without changing runtime behavior.

## Target Supabase-Backed Architecture

The target hosted pilot architecture is Supabase-backed and RLS-first. The browser uses public Supabase client config and authenticated sessions. Application data flows through Supabase tables protected by policy. Privileged actions run only in server-side Edge functions or another AP-approved server runtime.

Local and demo flows remain useful for development and rehearsals, but hosted pilot authority comes from authenticated Supabase records, private storage, server-side provider execution, durable audit events, and controlled export artifacts.

## Auth, Org, Workspace, User, And Role Model Direction

- Supabase Auth is the identity authority for hosted pilot users.
- Profiles link authenticated users to application identity metadata.
- Organizations are tenant authorities.
- Workspace authority must be explicitly designed before enforcement. Current UI scope is not enough for enterprise tenant policy.
- Membership records assign users to organizations and, after design approval, workspaces.
- Roles should be server-governed and mapped to RLS policy, route guard semantics, export permissions, and admin/config actions.
- Demo users and browser-only state must not authorize hosted pilot actions.

## RLS-First Access Model

- Every hosted pilot table that contains tenant, evidence, artifact, or audit data must have RLS enabled and deny by default.
- Browser access uses authenticated user claims and membership records.
- Role checks should use stable database helpers or policy patterns, not duplicated client-only logic.
- Service-role access is server-only and limited to Edge/admin operations that require elevated persistence, audit, or storage actions.
- Org and workspace boundaries must be tested with positive and negative cases.
- Policy tests should cover owner, contributor, reviewer, admin, cross-tenant denial, unauthenticated denial, export access, handoff access, and artifact reads.

## Evidence, Lineage, And Audit Persistence Direction

- Assessment inputs, deterministic scoring outputs, reviewer actions, evidence references, and decision-pack context should be durable records.
- Handoff ledger entries should be append-oriented and linked to source evidence, generated artifacts, and Delivery work.
- Document-generation records should preserve enough metadata for lineage without storing secrets or unsupported provider payloads.
- Delivery task source-lineage metadata should link back to Assess and Studio authorities, not become an isolated note field.
- Audit events should cover review transitions, evidence changes, handoffs, provider resolution, export creation, artifact access, admin/config changes, and denied sensitive actions where policy permits.
- Audit records should be append-only or tamper-evident according to the approved M5.4 design.

## Export-Control Direction

- Hosted pilot exports should use private storage and org/workspace-scoped object paths.
- Export records should include artifact type, source authority, review status, creator, created time, retention metadata, and audit references.
- Access should use short-lived signed links or an approved server-mediated delivery pattern.
- Browser-only download behavior remains demo/local behavior unless a later policy explicitly authorizes it.
- Delivery packs should remain review artifacts with omission policy and must not include raw source files, provider payloads, prompts, secrets, or tenant-confidential content outside approved fields.

## Provider Governance Direction

- Pilot and production provider execution runs server-side.
- Provider selection uses approved provider configuration and key-reference records.
- Secret material lives only in server-side environment storage or an approved secret manager.
- Logs, audit records, database rows, evidence bundles, and exported artifacts store references and outcomes, not secret values.
- Provider test and generation events should be audited with provider, model, policy, requester, and status metadata.
- Browser fallback remains limited to local-demo/internal-dev and must fail closed for pilot/production modes.

## Environment Separation Model

- Local: developer-only values and demo fallback allowed; no claim of pilot authority.
- Development: shared engineering validation with non-production data and replaceable secrets.
- Staging: production-like schema, RLS, storage, provider, and deployment checks using approved non-production values.
- Production: AP-approved tenant, secrets, backup/restore, audit retention, provider allowlist, export policy, and operations owner.

Each environment needs distinct config, secret ownership, storage, audit evidence, and verification. Personal project values may be temporary validation aids, but the architecture must be replaceable and governed.

## Backup, Restore, And Pilot Operations Considerations

- Define restore objectives before pilot data is accepted.
- Test backup/restore for application tables, private artifacts, audit records, and configuration records.
- Define rollback steps for schema, Edge function deployment, storage policy, and app deployment.
- Define smoke tests for Auth, tenant isolation, RLS denial, evidence persistence, export creation, provider resolution, and audit writes.
- Define support ownership, incident escalation, retention windows, and pilot data disposal.
