# M5.3a-2b Local Execution Evidence Flow

## Purpose

This document plans the future evidence flow for local artifact SELECT assertion execution. It does not implement execution, change the harness, run a local DB, or modify any SQL.

## Current Components

Accepted components on `main`:

- Artifact SELECT policy migration: `supabase/migrations/20260607153500_m5_3a_2_artifact_select_policies.sql`
- Artifact assertion SQL: `docs/schema/m5.3a_2_artifact_select_isolation_assertions.sql`
- Artifact runner: `scripts/m5.3a-2-artifact-isolation-smoke.mjs`
- Package script: `test:rls:m5.3a-2`
- Membership runner retained for baseline checks: `scripts/m5.3a-1-rls-isolation-smoke.mjs`
- Package script: `test:rls:m5.3a-1`

## Future Evidence Flow

Planned AP-approved local execution flow:

1. Confirm no-target artifact harness fails closed.
2. Confirm hosted target input rejects before DB action.
3. Confirm AP-approved local target proof is supplied.
4. Confirm local DB is reachable.
5. Confirm required schema and policies are present locally.
6. Execute artifact assertion SQL through the M5.3a-2 runner.
7. Read only the sanitized summary output.
8. Confirm 18 scenarios declared.
9. Confirm 18 scenarios executed.
10. Confirm 18 scenarios passed.
11. Confirm fail and blocked counts are zero.
12. Confirm artifact SELECT isolation verified is `yes`.
13. Confirm output contains no target values, secrets, row payloads, claim values, or fixture identifiers.

If any step fails, evidence must record artifact SELECT isolation as not verified.

## Local Schema State

Future local execution requires local schema state equivalent to the approved M5.3a-2 path:

- M5.3a-1 helper and SELECT policy foundation applied.
- M5.3a-2 artifact SELECT policies applied.
- Six scoped artifact authority tables available.
- Artifact assertion SQL compatible with the local schema.

M5.3a-2b does not apply migrations. Future evidence must record the AP-approved method used to prepare local schema.

## Command Flow

Planning-only command model:

- Baseline no-target artifact run: `npm run test:rls:m5.3a-2`
- Baseline no-target membership run: `npm run test:rls:m5.3a-1`
- Future local artifact run, only after AP approval: `npm run test:rls:m5.3a-2 -- --local-target localhost`
- Alternative direct future local run, only after AP approval: `node scripts/m5.3a-2-artifact-isolation-smoke.mjs --local-target localhost`

M5.3a-2b runs only no-target fail-closed harness checks. It does not run local-target commands.

## Evidence Data Model

Future evidence record should include:

- Command name.
- Sanitized target status.
- Local DB execution status.
- Scenario contract present yes/no.
- Assertion SQL present yes/no.
- Scenarios declared.
- Scenarios executed.
- Pass count.
- Fail count.
- Blocked count.
- Artifact SELECT isolation verified yes/no.
- Sanitized classifications.
- Exit-code category.
- No-secrets/no-target/no-payload confirmation.

Future evidence record must not include:

- Target values.
- DB URLs.
- Environment values.
- Host or port values when target-sensitive.
- Service-role values.
- Provider keys.
- Token values.
- Hosted identifiers.
- Row payloads.
- Claim values.
- Fixture IDs.
- Auth headers.

## Failure Modes

Artifact SELECT isolation remains not verified when:

- No local target proof is supplied.
- Local target proof is rejected.
- Local DB is unreachable.
- Required schema or policies are missing.
- Assertion SQL cannot execute.
- Scenario metadata is incomplete.
- Any scenario fails.
- Any approved scenario is blocked.
- Output sanitization fails.
- Cleanup or rollback fails.

## Proof Boundary

Successful future local execution can prove only local artifact SELECT behavior for the six scoped artifact tables and the approved 18 scenarios.

It cannot prove hosted runtime readiness, production readiness, mutation safety, storage/export readiness, provider readiness, runtime adapter readiness, operations readiness, performance readiness, certification, legacy draft table isolation, or full tenant isolation.
