# M5.3a-2 Artifact Isolation Validation Plan

## Purpose

This plan defines how future M5.3a-2 implementation should validate SELECT-only artifact isolation for the six approved artifact tables.

This document is planning only. It does not add assertion SQL, harness commands, policies, migrations, scripts, or database execution.

## Current Harness Baseline

The M5.3a-1 harness exists as:

- `npm run test:rls:m5.3a-1`
- `scripts/m5.3a-1-rls-isolation-smoke.mjs`
- `docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql`

The current harness is scoped to identity, organization, workspace, role, and membership tables. It statically rejects artifact-table references. Therefore M5.3a-2 should not reuse the M5.3a-1 command unchanged for artifact assertions.

## Future Assertion Strategy

Future M5.3a-2 implementation should add a separate artifact assertion contract:

- `docs/schema/m5.3a_2_artifact_select_isolation_assertions.sql`

An optional new or generalized runner command may be added only if AP approves:

- local-target proof behavior
- sanitized output shape
- expected scenario counts
- non-zero fail-closed behavior
- no target or secret output

Do not change the M5.3a-1 harness command in this planning milestone.

## Local Target Safety

Future artifact assertion execution must preserve the M5.3a-1 local target posture:

- No remote database target.
- No unguarded database target command.
- No silent environment database target acceptance.
- No fallback to ignored local secret files.
- No target values printed.
- No row payloads printed.
- No claim values printed.
- No DB URLs printed.
- No secrets printed.
- Counts-only sanitized output.

## Fixture Shape

Future artifact assertions should use generated local transaction-scoped fixtures only:

- Two local organizations.
- Two local workspaces per organization.
- Active workspace member.
- Disabled organization member.
- Disabled workspace member.
- Authenticated non-member.
- Same-workspace artifact rows for all six tables.
- Cross-workspace artifact rows.
- Cross-org artifact rows.
- Nullable `org_id` and nullable `workspace_id` rows where constraints permit local insertion.
- Deleted rows where `deleted_at` exists.
- Orphan parent/source rows where constraints permit local insertion.
- Cross-tenant parent/source references where constraints permit local insertion.

All fixture rows must roll back. No durable demo, production, runtime, seed, or backfill data may remain.

## Planned Positive Scenarios

Future assertions should prove that an active workspace member can read same-workspace rows for:

- `assess_processes`
- `assessments`
- `assessment_review_events`
- `projects`
- `document_generations`
- `delivery_work_items`

The positive path should require the artifact row and relevant parent/source rows to share the same `org_id` and `workspace_id`.

## Planned Negative Scenarios

Future assertions should deny:

- Anonymous access.
- Authenticated non-member access.
- Disabled organization member access.
- Disabled workspace member access.
- Cross-org access.
- Cross-workspace access.
- Nullable `org_id` rows.
- Nullable `workspace_id` rows.
- Deleted artifact rows where `deleted_at` exists.
- Orphan parent references.
- Cross-tenant parent references.
- Cross-tenant source references.
- Access based only on source references.
- Access based on JSON payload fields.
- Access based on `source_lineage`.

## Per-Table Assertion Notes

`assess_processes`:

- Positive: same-workspace process readable.
- Negative: null org, null workspace, deleted, cross-org, cross-workspace denied.

`assessments`:

- Positive: same-workspace assessment with same-workspace non-deleted process readable.
- Negative: deleted assessment, orphan process, deleted process, cross-org process, cross-workspace process denied.

`assessment_review_events`:

- Positive: same-workspace review event with same-workspace non-deleted assessment and process readable.
- Negative: orphan assessment, orphan process, cross-org or cross-workspace owning rows denied.
- No row-level `deleted_at` assertion because the table has no `deleted_at`.

`projects`:

- Positive: same-workspace non-deleted project readable.
- Negative: deleted project, null workspace, cross-workspace source references denied.

`document_generations`:

- Positive: same-workspace document generation with same-workspace non-deleted project readable.
- Negative: deleted document generation, deleted project, orphan project, cross-workspace project, cross-tenant source references denied.

`delivery_work_items`:

- Positive: same-workspace work item with same-workspace non-deleted project readable.
- Negative: deleted work item, deleted project, orphan project, cross-workspace project, cross-tenant document generation reference, cross-tenant source references denied.

## Output Contract

Future artifact assertion output should report only:

- scenarios declared
- scenarios executed
- pass count
- fail count
- blocked count
- artifact SELECT isolation verified: yes or no
- sanitized classifications
- local target proof status without target value

It must not output:

- DB URLs
- target values
- `.env` values
- service-role values
- provider keys
- JWT or private token values
- row payloads
- claim values
- hosted identifiers
- personal local paths

## Local DB Availability Decision

M5.3a-1 closure did not verify local tenant isolation because no reachable AP-approved local DB ran assertions.

Future M5.3a-2 implementation evidence must record one of two outcomes:

- Local artifact SELECT isolation verified for scoped tables only, if executable assertions run and pass against an AP-approved local DB.
- Artifact isolation not verified, if no reachable AP-approved local DB exists or assertions cannot execute.

Static review alone is not sufficient for an artifact isolation claim.

## Proof Boundary

Future M5.3a-2 validation can prove only local artifact SELECT isolation for the six approved tables.

It does not prove:

- Hosted runtime readiness.
- Production readiness.
- Storage/export readiness.
- Provider readiness.
- Mutation safety.
- End-user workflow readiness.
- Legacy draft table isolation.
