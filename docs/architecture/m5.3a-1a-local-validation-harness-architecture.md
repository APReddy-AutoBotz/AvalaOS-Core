# M5.3a-1a Local Validation Harness Architecture

## Purpose

This architecture plan defines the future local-only validation harness shape for M5.3a-1 helper and SELECT policy implementation. It is docs-only and creates no config, scripts, SQL, tests, helpers, policies, or runtime behavior.

## Current Validation Architecture Inventory

| Surface | Current finding | Harness implication |
| --- | --- | --- |
| Tracked Supabase config | The tracked Supabase file inventory does not include `supabase/config.toml`. | Local Supabase startup cannot be assumed from tracked config. |
| `package.json` DB scripts | `db:migrate` and `test:rls:assess` call `scripts/applySupabaseMigrations.mjs`. | These scripts are not approved harness commands for M5.3a-1a. |
| Migration runner | `scripts/applySupabaseMigrations.mjs` requires `DATABASE_URL`. | Unsafe as default because the environment can point to remote or secret-bearing targets. |
| Existing Assess regression SQL | `docs/schema/rls_regression_assess_handoff.sql` covers Assess review events and handoff ledger behavior. | Historical/draft support only; not the M5.3a-1 identity/membership harness. |
| Draft role policy SQL | `docs/schema/delivery_role_policies.sql` and `docs/schema/timesheet_role_policies.sql` include role-aware draft policies. | Out of scope; M5.3a-1a validates neither legacy draft tables nor mutation policies. |
| Applied authority migrations | M5.2a authority tables are the future M5.3a-1 target; artifact authority tables remain deferred. | Harness should focus on identity/org/workspace/role/membership SELECT behavior. |
| Provider governance migration | Existing provider governance has its own tables and policies. | Leave untouched; do not include in M5.3a-1 harness. |

## Target Harness Architecture

The future harness should have four layers:

1. Static safety gate.
2. Local target gate.
3. Fixture and identity simulation layer.
4. Scenario execution and sanitized evidence layer.

### Static Safety Gate

Before connecting to any database, future implementation should verify:

- No tracked `.env*` files.
- No remote Supabase URL in committed harness files.
- No personal hosted identifiers in committed harness files.
- No service-role, provider key, JWT/private token, or bucket-name output path.
- No broad `FOR ALL`, mutation, DELETE, artifact, provider governance, or legacy draft-table policy validation in the M5.3a-1 harness.

### Local Target Gate

The future runner must prove the target is local before any database action:

- Allowed target hosts: `localhost`, `127.0.0.1`, Docker local network, or another AP-approved local-only target.
- Rejected targets: hosted Supabase URLs, personal project refs, remote database hosts, or unapproved environment-provided values.
- The runner must not print the target value.
- The runner must not silently accept `DATABASE_URL`.
- Missing local target proof must fail closed.

### Fixture And Identity Simulation Layer

The fixture layer should use generated local UUIDs and local labels only. It should support:

- Two local organizations.
- Two local workspaces per organization.
- Active member.
- Disabled organization member.
- Disabled workspace member.
- Non-member.
- Org-scoped and workspace-scoped roles.
- Same-org/cross-org membership rows.
- Same-workspace/cross-workspace membership rows.
- Deleted-state variants where useful.

The identity simulation layer should evaluate:

- SQL role/session claim simulation for `auth.uid()`.
- Local Supabase Auth fixtures if AP approves tracked local config.
- Local anon/authenticated client simulation if AP approves a local stack.
- Server-only fixture setup path only if AP approves local privileged setup.

## M5.3a-1 Harness Scope

Future validation is limited to:

- `profiles`
- `organizations`
- `workspaces`
- `roles`
- `organization_members`
- `workspace_memberships`

The harness should validate SELECT policy behavior only.

Deferred:

- `assess_processes`
- `assessments`
- `assessment_review_events`
- `projects`
- `document_generations`
- `delivery_work_items`
- provider governance tables
- legacy draft tables
- storage/export records
- Delivery Pack records
- Monitor signal records
- lineage/handoff records
- INSERT/UPDATE/DELETE policies
- mutation, approval, review, export, and delivery policies

## Scenario Execution Model

Future scenario output should be counts-only and sanitized:

- Scenario label.
- Expected result.
- Actual pass/fail.
- Aggregate pass/fail counts.
- Sanitized failure classification.

The output must not include:

- Database URLs.
- Secret values.
- Token values.
- Hosted project values.
- Row payload dumps.
- Personal IDs.
- Provider keys.
- Bucket names.

## Future Runbook

Recommended future runbook sequence:

1. Confirm clean working tree.
2. Confirm AP-approved local harness files are present.
3. Confirm no tracked `.env*` files.
4. Confirm local target proof without printing the target.
5. Confirm local migration setup path.
6. Apply only approved migration set to local state.
7. Apply future M5.3a-1 helper/policy SQL only when AP-approved.
8. Set up generated local fixtures.
9. Simulate anonymous and authenticated local identities.
10. Run negative and positive scenarios.
11. Print counts-only sanitized results.
12. Roll back, reset, or clean local state.
13. Record evidence without secrets.

## Dependency On M5.3a-1

M5.3a-1 helper/policy implementation should remain blocked until this future harness is implemented and AP-approved. The harness milestone should not implement helper functions or policies itself.

## Failure Handling

The future harness should fail closed when:

- Local target proof is missing.
- A target appears remote.
- A required local config is missing.
- A command would require `DATABASE_URL` without an AP-approved local wrapper.
- Secret-like output is detected.
- Scenario output would require row payload dumps.
- Cleanup/reset cannot be confirmed.

Failure evidence should record only sanitized labels and counts.

## Runtime And Hosted Readiness

The future harness validates local RLS behavior only for the M5.3a-1 scope. It does not prove hosted runtime readiness, artifact policy readiness, adapter readiness, export/storage readiness, deployment readiness, provider readiness, audit readiness, or pilot readiness.
