# M5.2b Artifact Ownership Domain Inventory

Milestone: M5.2b Artifact Ownership Columns Migration Plan

Status: Architecture inventory and migration sequencing map

## Purpose

This document records the current artifact persistence surfaces that need ownership planning before workspace-aware RLS can be implemented. It distinguishes applied migration tables from draft SQL contracts, browser/demo persistence, and runtime-derived artifacts.

M5.2b is docs-only. It does not create migrations or change schema.

## Inventory Method

The inventory reviews:

- M5.2a authority migration.
- Provider-governance migration.
- Draft SQL contracts under `docs/schema`.
- Supabase-facing adapters.
- Browser/demo storage surfaces.
- Export, delivery pack, evidence, lineage, and handoff services.
- M5.2 and M5.3 planning/security docs.

The review used tracked files only and did not inspect `.env*` values or secret files.

## Runtime And Demo Surfaces

| Surface | Current behavior | Ownership implication |
| --- | --- | --- |
| `services/adapters/assessAdapter.ts` | Reads and writes org-scoped Assess tables when Supabase is configured; falls back to demo data otherwise. | Future M5.2c must avoid breaking current fallback and should not require runtime to send `workspace_id` until adapter changes are approved. |
| `services/adapters/deliveryAdapter.ts` | Uses org-scoped projects, epics, sprints, tasks, comments, and activity rows with app-id mapping. | Delivery ownership needs careful project/workspace mapping and app-id compatibility. |
| `services/adapters/docsAdapter.ts` | Uses org-scoped `document_generations` related to projects. | Studio/Docs ownership likely derives workspace from project or source assessment. |
| `services/adapters/handoffLedgerAdapter.ts` | Uses org-scoped handoff ledger rows and local fallback. | Handoff workspace ownership needs source/target mapping and append-oriented policy. |
| `services/adapters/timesheetAdapter.ts` | Uses org-scoped timesheet rows related to tasks. | Workspace should derive from task/project to avoid mismatch. |
| `services/storage.ts` | Stores demo-only user, scope, view, Assess, Docs, Delivery, audit-adjacent logs, and handoff ledger data. | Browser storage is not hosted pilot authority and should not receive ownership migration logic in M5.2b. |
| Export services | Create browser downloads or rendered artifacts. | Export records and private storage are deferred to M5.5. |
| Delivery Pack services | Build runtime review packs from Assess, Studio, Delivery, lineage, and handoff data. | Durable Delivery Pack persistence should wait for parent domain ownership. |

## Proposed Implementation Slices

### M5.2c Assess Govern Ownership Columns Migration

Target first because Assess/Govern records anchor the product law and RLS test design.

Likely migration target:

- `supabase/migrations/20260607151000_m5_2c_assess_govern_ownership_columns.sql`

Candidate table classes:

- `assess_processes`
- `assessments`
- `assessment_review_events`
- General audit columns only if AP approves a narrow Assess-linked audit slice

Candidate columns:

- Nullable `workspace_id`
- `created_by`
- `updated_by`
- `reviewer_id`
- `approver_id`
- `correlation_id`
- `deleted_at` or `archived_at` only where lifecycle requires it

Do not include RLS policies or seed data.

### Later Studio / Docs Slice

Candidate table classes:

- `document_generations`
- Future document vault records if AP approves persistence

Candidate ownership:

- `org_id`
- `workspace_id`
- `created_by`
- `reviewer_id`
- `status`
- Source process, assessment, project, and generation refs

### Later Delivery / Delivery Pack Slice

Candidate table classes:

- `projects`
- `epics`
- `sprints`
- `tasks`
- `task_comments`
- `task_activity_events`
- `timesheet_entries`
- Future delivery pack records

Candidate ownership:

- `workspace_id` on projects and derived or validated workspace ownership for child rows
- Actor fields where mutation policy needs them
- Lifecycle fields for soft-delete/archive as AP approves

### Later Evidence / Lineage / Handoff / Audit / Export Slice

Candidate table classes:

- Evidence refs
- Source lineage
- Handoff ledger
- Audit events
- Export records
- Storage object path policy direction

Candidate ownership:

- `org_id`
- `workspace_id`
- Actor and correlation fields
- Retention and sensitivity metadata where AP approves

### Later Provider Governance Slice

Candidate table classes:

- Provider policy metadata
- Provider audit events
- Provider config/key refs only if AP approves workspace-specific provider policy authority

Candidate ownership:

- Keep config/key refs org-scoped unless workspace-specific provider control is approved.
- Add strict FK to provider audit `workspace_id` only after workspace policy and audit requirements are approved.

## Backfill Map

| Data class | Backfill direction |
| --- | --- |
| Existing fresh local databases | No broad backfill in M5.2b; future migrations add nullable columns. |
| Non-production demo rows | Map to AP-approved demo workspace in a separate backfill milestone. |
| Hosted rows if any exist | Require AP-approved migration-history and data-mapping plan before updates. |
| Runtime local storage | No backfill; browser storage remains demo-only. |
| Provider-governance records | Keep org-scoped until AP decides workspace provider authority. |

## RLS Readiness Dependency

M5.3a should not assume tenant isolation from ownership columns alone. RLS readiness needs:

- Ownership columns on real artifact tables.
- Approved workspace mapping.
- Active org and workspace membership helpers.
- Role/permission helper strategy.
- Positive and negative fixture data.
- SQL and JS isolation tests.
- Server-only service-role and provider-secret boundaries from M5.1.

## AP Decisions Needed

- Whether M5.2c implements only Assess/Govern ownership.
- Whether `workspace_id` is mandatory or nullable-first for each artifact class.
- Whether broad artifact FKs should use `NOT VALID`.
- Whether project should be the workspace-derived root for Delivery child rows.
- Whether Decision Pack, Delivery Pack, evidence refs, source lineage, audit events, and export records get standalone tables before or after RLS.
- Whether provider-governance workspace scope is needed before provider policy implementation.
