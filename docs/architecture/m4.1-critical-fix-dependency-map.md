## Implementation Risk Map

| Risk | Severity | Mitigation |
| --- | --- | --- |
| Fixing Decision Pack display accidentally changes scoring behavior. | High | Keep `M4.1a` to rendering only and require scoring regressions. |
| Handoff payload becomes a mutable cross-module object. | High | Make `M4.1b` payload read-only source context with minimum fields approved by AP. |
| Delivery lineage is partial until upstream payload exists. | High | Keep `M4.1b` before `M4.1c`; label missing refs. |
| Route guard blocks legitimate flows. | High | Make `M4.1d` a design/plan first, then test direct, persisted, and flow-driven routes in implementation PR. |
| Buyer sees local-demo AI behavior as pilot-ready behavior. | High | Require `M4.1e` AP decision before serious buyer demos. |
| Admin / AI Controls screen overpromises governance completeness. | High | Require `M4.1f` AP decision before implementation or polish. |
| Delivery Pack export overstates evidence. | High | Require `M4.1g` block-or-label export behavior for incomplete lineage/evidence refs. |
| Persistence cleanup expands into schema/RLS work. | High | Split `M4.1i` persistence cleanup from legacy naming cleanup and require separate approval. |

## Recommended Dependency Decision

Start implementation with `M4.1a Decision Pack contract/rendering fix`. Keep `M4.1b` and `M4.1c` separate. Treat `M4.1d` as a route/RBAC guard design and implementation plan before code work. Do not implement `M4.1e` or `M4.1f` before AP records product decisions.
