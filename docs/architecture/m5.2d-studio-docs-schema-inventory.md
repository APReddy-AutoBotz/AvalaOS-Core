# M5.2d Studio Docs Schema Inventory

Milestone: M5.2d Studio Docs Ownership Columns Migration Plan

Branch: milestone/m5.2d-studio-docs-ownership-columns-migration-plan

Status: Docs-only inventory

## Purpose

This map records the current Studio and Docs persistence posture, separates applied migration authority from draft/schema/runtime concepts, and identifies the ownership dependencies that must be resolved before a Studio/Docs migration is implemented.

## Current Architecture Finding

Studio/Docs persistence is not ready for an ownership-column migration because its primary draft table, `document_generations`, depends on `projects`. Neither `document_generations` nor `projects` is applied migration-chain authority today.

The current runtime remains browser/demo-first for many Docs surfaces. Generated artifacts, source context, document exports, and Docs-to-Delivery lineage are assembled in services and app state rather than server-enforced tenant tables.

## Current File And Module Inventory

`docs/schema/initial_schema.sql` defines draft `document_generations` and `projects` contracts. This is not the applied migration chain.

`docs/schema/docs_role_policies.sql` defines draft role-oriented policy concepts for document generation. The posture is organization-scoped and not workspace-ready.

`services/adapters/docsAdapter.ts` reads and writes `document_generations`, filters by `org_id`, and resolves project IDs through `projects`. It has no workspace ownership dependency today.

`components/docs/DocsProvider.tsx` loads and stores document generations through the Docs adapter using the active organization context.

`services/documentExportService.ts` produces browser downloads from generated artifacts. It does not create durable export records.

`services/assessToStudioHandoff.ts` builds runtime source context from Assess and Govern-derived data for Studio-generated artifacts.

`services/docsToDeliveryLineage.ts` builds runtime lineage metadata that later travels into Delivery task metadata and Delivery Pack construction.

`services/deliveryPackService.ts` and `services/deliveryPackExportService.ts` assemble and export delivery packs from runtime data and browser state. They are not Studio/Docs persistence authority.

`services/storage.ts` includes local browser persistence keys for Docs templates, document generations, handoff ledger, and other demo surfaces. This remains demo persistence, not hosted authority.

`docs/schema/ai_governance_jobs.sql` defines draft provider/job tables. These should remain separate from Docs template or artifact ownership until server-side provider execution is approved.

`docs/schema/storage_buckets.sql` defines draft storage controls. Export and storage object authority is deferred.

## Target Studio/Docs Ownership Direction

Future Studio/Docs persistence should make generated artifacts workspace-scoped and same-tenant-source-aware. A future `document_generations` promotion should include nullable-first `workspace_id`, creator/updater fields, reviewer/approver fields where needed, source Assess references, lifecycle fields, and audit correlation fields.

If AP approves a richer document repository model, `document_generations` should not absorb every document concern. Repository metadata, document versions, review events, exports, storage object references, and lineage records should be modeled separately.

## Dependency Map

M5.2a provides tenant authority through profiles, organizations, workspaces, roles, organization memberships, and workspace memberships.

M5.2c provides Assess ownership readiness through Assess process, assessment, and review-event tables.

M5.2d records that Studio/Docs should wait for Delivery/project ownership because current draft document generation rows reference projects.

M5.2e should plan Delivery/project ownership and decide whether project is the work boundary, a child of workspace, or an artifact under workspace.

A later Studio/Docs implementation can then add ownership columns with the right project/workspace relationship.

M5.3a RLS implementation should wait until the necessary artifact ownership columns exist and tests can exercise workspace-aware isolation.

## Implementation Readiness

Studio/Docs implementation is not recommended in M5.2d. The migration would be premature because project authority, workspace mapping, source Assess linkage, repository modeling, export ownership, and backfill strategy are not yet approved.

The safe implementation path is:

1. Plan Delivery/project ownership in M5.2e.
2. Implement the first Delivery/project ownership migration slice after AP approval.
3. Implement Studio/Docs ownership using the approved project/workspace boundary.
4. Implement RLS policies and isolation tests after ownership columns exist.
