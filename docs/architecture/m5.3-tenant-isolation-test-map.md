# M5.3 Tenant Isolation Test Map

Milestone: M5.3 RLS Policy Design And Test Plan
Status: Tenant-isolation test planning only

## Purpose

This map defines the future fixture and test strategy for proving Supabase tenant isolation after schema and RLS implementation are approved. It does not create fixtures, scripts, SQL, migrations, runtime code, or live Supabase validation.

## Fixture Strategy

Future RLS implementation should use a non-production fixture set with:

- Two organizations: Org A and Org B.
- Two workspaces per organization: Workspace A1, Workspace A2, Workspace B1, and Workspace B2.
- Active member.
- Disabled member.
- Non-member.
- Platform Admin.
- Process Owner.
- Control Reviewer.
- Delivery Lead.
- Automation Contributor.
- Buyer Viewer.
- Same org and same workspace records.
- Same org and different workspace records.
- Different org records.
- Assess records.
- Govern and Decision Pack records.
- Studio/generated artifact records.
- Delivery records.
- Evidence refs.
- Source lineage.
- Handoff ledger entries.
- Export records.
- Audit events.
- Provider governance metadata and audit records.

Fixture values must be non-secret, non-personal, deterministic, and replaceable across local, development, staging, and CI-like validation.

## Role Coverage Direction

| Role or persona | Test purpose |
| --- | --- |
| Platform Admin | Org administration, member management, workspace administration if AP approves inheritance, and provider metadata governance. No service-role credential exposure. |
| Process Owner | Assess and Govern review/approval authority inside assigned workspace. |
| Control Reviewer | Review and control/evidence approval authority where assigned. |

| Non-member | No org or workspace tenant data access. |

## Positive Test Strategy

| Scenario | Expected result |
| --- | --- |
| Active workspace member reads authorized workspace records | Allowed. |
| Process Owner performs approved Assess review operation | Allowed inside assigned workspace. |
| Process Owner performs approved Govern operation | Allowed inside assigned workspace and policy scope. |
| Control Reviewer reviews assigned evidence or artifact | Allowed where role and assignment match. |
| Delivery Lead updates authorized Delivery records | Allowed inside assigned workspace. |
| Automation Contributor updates own assigned Delivery work | Allowed only for permitted fields and assigned scope. |
| Buyer Viewer reads approved Decision Pack or dashboard artifact | Allowed read-only. |
| Platform Admin reads org administration metadata | Allowed within approved org scope. |
| Server-side export path creates export metadata with audit context | Allowed only through approved server path. |
| Server-side provider resolver writes safe provider audit metadata | Allowed only through approved server path. |

## Test Method Recommendation

M5.3 recommends future implementation use both SQL-level and JS integration tests:

- SQL-level tests for direct policy truth tables, append-only behavior, status denial, and cross-tenant denial.
- JS integration tests for anon, authenticated user, and service-role client separation.
- Separate client fixtures for anonymous, active member, disabled member, non-member, viewer, contributor, reviewer, owner, delivery lead, platform admin, and server role.
- Tests should record pass/fail results and counts only.
- Tests must not print secrets, tokens, service-role values, provider secrets, full hosted-project values, or personal values.
- CI timing remains an AP decision after local repeatability and non-production environment ownership are approved.

## Client Separation Model

| Client class | Future test use |
| --- | --- |
| Anonymous client | Prove tenant data denial. |
| Authenticated user client | Prove membership, workspace, role, status, and actor-scoped behavior. |
| Admin-like authenticated user | Prove admin permissions without service-role leakage. |
| Service-role server client | Prove server-only exceptions, audit context requirements, and browser inaccessibility. |

## Fixture Data Classes

Each org/workspace pair should include:

- Assess process and assessment.
- Govern or Decision Pack review record.
- Studio/generated artifact record.
- Delivery project, task, comment, activity, and timesheet-like record as approved by schema.
- Delivery Pack record.
- Monitor signal or project/task signal.
- Evidence ref.
- Source lineage row.
- Handoff ledger row.
- Audit event row.
- Export metadata row and private storage path reference.
- Provider config metadata, key-reference metadata, provider policy metadata, and provider audit event.

## Sequencing Dependency

This test map depends on a later schema implementation milestone. Until canonical workspace tables, workspace memberships, ownership columns, role assignment shape, and permission model are implemented, M5.3 tests remain a plan, not executable tenant isolation proof.

Recommended implementation order:

1. M5.2a Supabase Auth Org Workspace Schema Migration Implementation.
2. M5.3a RLS Policy Implementation And Isolation Tests.

## Acceptance Criteria For Future M5.3a

- Every tenant table class has at least one same-scope positive test and one cross-scope negative test.
- Every privileged mutation has at least one allowed role test and one denied lower-role test.
- Every workspace-scoped table denies org-only members without workspace membership unless AP approves inheritance.
- Audit and handoff records prove append/status-transition behavior.
- Export and storage tests prove private, scoped, and revocable access.
- Provider governance tests prove metadata-only browser access and server-only secret handling.
