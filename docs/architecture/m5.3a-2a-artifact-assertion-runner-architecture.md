# M5.3a-2a Artifact Assertion Runner Architecture

## Purpose

This document plans the future artifact assertion runner architecture. It does not create scripts, SQL, package entries, local configuration, database connections, or runtime behavior.

## Current Architecture Baseline

Existing accepted components:

- M5.3a-1 identity and membership harness command: `npm run test:rls:m5.3a-1`.
- M5.3a-1 runner: `scripts/m5.3a-1-rls-isolation-smoke.mjs`.
- M5.3a-1 assertion SQL: `docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql`.
- M5.3a-2 artifact assertion SQL: `docs/schema/m5.3a_2_artifact_select_isolation_assertions.sql`.

The current M5.3a-1 runner is accepted for identity and membership assertions only. M5.3a-2 artifact assertions have no AP-approved executable runner yet.

## Recommended Future Runner Topology

Recommended first implementation:

- Add a separate artifact runner at `scripts/m5.3a-2-artifact-isolation-smoke.mjs`.
- Optionally add `test:rls:m5.3a-2` only after AP approves package-script impact.
- Keep M5.3a-1 runner behavior unchanged.
- Use `docs/schema/m5.3a_2_artifact_select_isolation_assertions.sql` as the artifact assertion contract.
- Capture evidence in `docs/quality/m5.3a-2a-artifact-assertion-harness-implementation-evidence.md`.

This keeps identity/membership and artifact boundaries separate while artifact harness behavior is reviewed.

## Runner Flow

Planned future flow:

1. Parse arguments.
2. Reject unknown arguments.
3. Prove explicit local target without printing it.
4. Reject hosted, remote, implicit, or secret-file targets before database action.
5. Confirm assertion SQL file exists.
6. Run SQL static guard checks.
7. Connect only after local target proof passes.
8. Execute assertion SQL in a local transaction flow.
9. Read only sanitized summary output.
10. Confirm cleanup or rollback behavior.
11. Emit counts-only output.
12. Return non-zero unless all approved scenarios execute and pass.

## Scenario Contract

The artifact assertion contract should cover:

- Positive same-workspace artifact reads for six artifact tables.
- Anonymous denial.
- Non-member denial.
- Disabled membership denial.
- Cross-organization denial.
- Cross-workspace denial.
- Nullable organization denial.
- Nullable workspace denial.
- Deleted artifact denial where supported.
- Orphan parent denial.
- Cross-tenant parent/source denial.
- JSON payload fields do not bypass policy authority.
- Source and lineage references do not grant visibility.

M5.3a-2a does not change the assertion SQL. Future implementation should validate scenario metadata before execution.

## Local Database Availability Paths

Executable proof path:

- AP-approved local target is provided.
- Local database is reachable.
- Applied migrations and artifact policies are available.
- Artifact assertion SQL executes.
- All approved scenarios pass.
- Output remains sanitized.

Static and fail-closed path:

- No AP-approved local target exists or local database is unavailable.
- The runner proves target-safety behavior and fails closed.
- Artifact SELECT isolation remains not verified.

## Output Model

Planned sanitized output fields:

- Artifact scenario contract present.
- Assertion SQL present.
- Local target gate status.
- Local DB assertion execution status.
- Scenarios declared.
- Scenarios executed.
- Pass count.
- Fail count.
- Blocked count.
- Artifact SELECT isolation verified.
- Sanitized classifications.

The output model must not include target values, URLs, environment values, service-role values, provider keys, JWTs, hosted identifiers, row payloads, claim values, or auth headers.

## Package Script Decision

Recommended future script name:

- `test:rls:m5.3a-2`

This should be added only in the implementation milestone after AP approves package-script impact. M5.3a-2a does not edit `package.json`.

## Relationship To Existing Harness

The future artifact runner may reuse safe design ideas from the M5.3a-1 runner:

- Explicit local-target proof.
- Counts-only output.
- Secret-safe classifications.
- Non-zero fail-closed behavior.

It should not modify or weaken M5.3a-1 runner behavior in the first artifact harness implementation.

## Future Implementation Files

Recommended future files:

- `scripts/m5.3a-2-artifact-isolation-smoke.mjs`
- `docs/quality/m5.3a-2a-artifact-assertion-harness-implementation-evidence.md`
- Optional `package.json` script entry for `test:rls:m5.3a-2` if AP approves.

No future implementation should require remote validation, provider calls, or secret-file fallback.

## Proof Boundary

The runner architecture can prove local assertion execution behavior. It cannot prove hosted readiness or broader runtime readiness. Artifact SELECT isolation may be claimed only when assertions run and pass against an AP-approved reachable local database.
