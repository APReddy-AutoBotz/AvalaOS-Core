# M5.3a Tenant Isolation Test Plan

## Purpose

This document defines the future tenant isolation test plan for applied authority tables. It is planning only. It does not add tests, SQL, migrations, helper functions, seed data, runtime code, or Supabase validation.

## Applied Authority Test Surface

The first isolation test surface is limited to applied authority:

- Tenant authority: `profiles`, `organizations`, `workspaces`, `roles`, `organization_members`, `workspace_memberships`.
- Assess authority: `assess_processes`, `assessments`, `assessment_review_events`.
- Project authority: `projects`.
- Studio/Docs authority: `document_generations`.
- Delivery work item authority: `delivery_work_items`.

The plan intentionally excludes draft or non-authority tables: `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, `timesheet_entries`, storage/export tables, Delivery Pack records, Monitor signal records, lineage authority, and handoff authority.

## Fixture Strategy

Future tests should use deterministic, non-secret fixtures with generated local UUID values only:

| Fixture group | Required shape |
| --- | --- |
| Organizations | Two organizations, `Org A` and `Org B`. |
| Workspaces | Two workspaces per organization. |
| Users | Active member, disabled org member, disabled workspace member, non-member. |
| Roles | Buyer Viewer, contributor, Delivery Lead, reviewer/admin as needed. |
| Records | Same-org/same-workspace, same-org/cross-workspace, and cross-org rows for each applied artifact table. |
| References | Same-scope and cross-scope source Assess, document generation, project, and work item references. |

Fixtures must not include personal organization IDs, hosted project IDs, Supabase project values, live URLs, provider keys, service-role values, or `.env` values.

## Negative Isolation Matrix

| Scenario | Expected result |
| --- | --- |
| Anonymous user reads tenant authority or artifact rows | Denied. |
| Authenticated non-member reads org data | Denied. |
| Disabled org member reads org data | Denied. |
| Disabled workspace member reads workspace data | Denied. |
| Org A user reads Org B authority or artifact rows | Denied. |
| Workspace A user reads Workspace B artifact rows in same org | Denied. |
| Workspace-scoped artifact row has `workspace_id IS NULL` | Denied by default unless AP approves a narrow exception. |
| Buyer Viewer inserts, updates, deletes, reviews, approves, exports, or mutates delivery records | Denied. |
| Contributor attempts review or approval mutation | Denied. |
| Delivery mutation occurs without AP-approved role-aware mutation policy | Denied. |
| Artifact row references cross-tenant project, document, Assess, evidence, lineage, or handoff source | Denied. |
| `metadata` or `source_lineage` contains cross-tenant identifiers | Does not grant access or bypass policy predicates. |
| Browser attempts service-role behavior | Denied by architecture; service-role remains server-only. |

## Positive Read Strategy

Positive tests for the first SELECT slices should stay narrow:

- Active org member can read own org metadata after membership SELECT policies exist.
- Active workspace member can read own workspace metadata after workspace membership policies exist.
- Active workspace member can read same-org, same-workspace artifact rows after artifact SELECT policies exist.
- Buyer Viewer can read approved read-only surfaces only after AP approves read semantics.

Positive mutation tests are deferred until AP approves role/permission depth and mutation policy slices.

## Client Separation

Future tests should separate client behavior:

- Anonymous client for public/no-session checks.
- Authenticated user clients for member, disabled member, and non-member checks.
- Service-role or elevated client only for local fixture setup and server-only behavior checks.

Service-role values must never be printed, committed, exposed to browser code, or used in remote validation. Tests should report pass/fail and counts, not secret values.

## Nullable Workspace Test Requirements

For each workspace-scoped artifact table, future tests must include non-null workspace rows and nullable workspace rows:

- `assess_processes`
- `assessments`
- `assessment_review_events`
- `projects`
- `document_generations`
- `delivery_work_items`

Default expected behavior is fail-closed for nullable workspace rows. If AP approves a temporary org-level exception, tests must identify it by table, role, scope, and removal condition.

## Helper Validation Strategy

Future helper tests should verify:

- `is_active_org_member(p_org_id uuid)` returns true only for active, non-deleted memberships matching `auth.uid()`.
- `is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)` returns true only for active, non-deleted workspace memberships in the same organization.
- Disabled, invited, archived, deleted, cross-org, and cross-workspace states do not grant access.
- Helpers do not expose role or membership existence across tenants.
- Helper execution grants are limited to intended authenticated roles.

Permission helpers remain optional and should not be tested until AP approves role/permission depth.

## Local Validation Decision Point

The current repository state does not provide a tracked local Supabase validation configuration for repeatable RLS proof. Before implementation claims verified isolation, AP must approve a local-only validation path.

If local validation remains unavailable, M5.3a-1 should either:

- Create helper/policy implementation SQL plus a test contract with static review only; or
- Remain blocked until local validation is available.

Remote Supabase validation remains out of scope.

## Future Test Artifacts

Future implementation may add test assets only after AP approval. Candidate assets:

- SQL-level isolation test fixtures for local Supabase.
- JavaScript integration tests that use separate anon/authenticated/server-only local clients.
- Static policy checks that confirm no broad `FOR ALL`, no hard DELETE, no browser service-role, and no nullable workspace org-wide fallback.

M5.3a refresh does not create those assets.

## Runtime Caveats

RLS tests do not prove hosted runtime readiness. Runtime adapters may still:

- Target legacy draft Delivery tables.
- Depend on read/write shapes that are not approved for hosted RLS.
- Fail closed after SELECT policies are added.
- Require separate export, storage, evidence, audit, deployment, and operations work.

The tests should prove tenant isolation properties for applied authority only, not full hosted pilot readiness.
