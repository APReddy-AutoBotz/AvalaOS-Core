# M5.2e Delivery Schema Inventory

Milestone: M5.2e Delivery Ownership Columns Migration Plan

Branch: milestone/m5.2e-delivery-ownership-columns-migration-plan

Status: Docs-only architecture inventory

## Purpose

This inventory separates applied migration authority from draft schema, runtime types, mock data, adapters, and UI concepts for Delivery/project ownership planning.

## Current Architecture Finding

No Delivery/project tables exist in the applied migration chain. Delivery persistence is draft-only under `docs/schema` plus runtime/mock/adapters.

Studio/Docs ownership remains blocked because draft `document_generations` references `projects`, and `projects` is not applied migration-chain authority.

## Applied Migration Chain

Applied migrations currently include:

- tenant authority tables from M5.2a;
- provider governance tables;
- Assess process, assessment, and assessment review event ownership from M5.2c.

They do not include Delivery/project authority.

## Draft Schema Posture

`docs/schema/initial_schema.sql` defines draft `projects`, `epics`, `sprints`, `tasks`, `document_generations`, and `timesheet_entries`.

`docs/schema/delivery_app_id_mapping.sql` adds stable `app_id` columns and unique `(org_id, app_id)` indexes for draft Delivery tables.

`docs/schema/delivery_comments_activity.sql` defines draft task comments and activity tables.

`docs/schema/delivery_role_policies.sql` defines org-scoped helper functions, task mutation guards, and role-aware policy concepts.

These drafts are useful design references, but they are not applied migration-chain authority and are not workspace-ready.

## Runtime And Mock Posture

`services/adapters/deliveryAdapter.ts` targets draft Delivery table names when Supabase is configured, maps stable app IDs, filters by organization, and stores task lineage/assignee/reporter details in metadata.

`components/delivery/DeliveryProvider.tsx` enforces app-side Delivery policy checks and coordinates projects, epics, sprints, and tasks for the current organization.

`data/mockData.ts` contains one canonical AP project, epics, sprints, tasks, source lineage, handoff entries, and derived Delivery Pack inputs.

`services/deliveryPackService.ts` builds a derived Delivery Pack from runtime objects. It does not persist pack records.

Monitor widgets derive project health, burndown, task counts, and readiness signals from runtime project/task state.

## Target Dependency Direction

M5.2f should create project authority first. Later slices can add work-item authority, Delivery Pack persistence, Monitor persistence, handoff/lineage authority, and export ownership after AP approves their boundaries.

M5.3a RLS should wait until enough ownership schema exists to test workspace-aware project, work-item, Studio/Docs, lineage, handoff, export, and provider boundaries.
