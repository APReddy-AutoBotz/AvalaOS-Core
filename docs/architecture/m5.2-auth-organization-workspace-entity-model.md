# M5.2 Auth, Organization, And Workspace Entity Model

Milestone: M5.2 Supabase Auth / Organization / Workspace Schema Plan
Status: Architecture entity model only

## Purpose

This map records the current Auth, organization, workspace, role, permission, and ownership surfaces that affect hosted pilot tenant design. It defines the target entity model for later implementation without creating migrations or changing runtime behavior.

## Target Entities

### `auth.users`

Supabase Auth should be the identity authority for hosted pilot users. Application code should treat `auth.uid()` as the policy input for RLS and server-side operations. Auth users must not be confused with demo personas.

### `profiles` Or `user_profiles`

Application profile metadata should be stored in one profile table linked one-to-one with `auth.users`. Required direction:

- Primary key references `auth.users(id)`.
- Store email, display name, avatar, status, timestamps, and last-login metadata as needed.
- Do not store raw secrets.
- Do not treat profile fields as role authority.
- AP must approve whether to keep `profiles` or rename/introduce `user_profiles`.

### `organizations`

Organizations are tenant roots. Direction:

- UUID primary key.
- Unique slug or equivalent tenant slug.
- Name, status, settings, enabled modules, subscription/commercial fields, timestamps, and optional soft-delete/archive field.
- Organization-level settings must not override RLS.

### `organization_members` Or `organization_memberships`

Organization membership is the authority for org access.

- UUID primary key.
- `org_id` FK to organizations.
- `user_id` FK to profiles.
- Active/invited/suspended/removed status.
- Org role assignment or FK to role assignment.
- Joined/invited timestamps.
- `created_by` / `updated_by` where useful.
- Unique `(org_id, user_id)`.

AP must approve whether the existing table name remains `organization_members` or whether future docs/migrations normalize on `organization_memberships`.

### `workspaces`

Workspaces are organization children and should become the primary hosted-pilot work boundary.

- UUID primary key.
- `org_id` FK to organizations.
- Name and slug.
- Workspace type or purpose if AP needs process-domain, business-unit, pilot, or project-like grouping.
- Status such as active, archived, or suspended.
- Timestamps.
- `created_by` and `updated_by`.
- Unique `(org_id, slug)`.

For the canonical non-production demo, `Avala Demo Enterprise` should have one AP Invoice Exception Workflow workspace/process context.

### `workspace_memberships`

Workspace membership is the authority for workspace-level access.

- UUID primary key.
- `org_id` FK to organizations for efficient policy checks.
- `workspace_id` FK to workspaces.
- `user_id` FK to profiles.
- Workspace role assignment or FK to role assignment.
- Active/invited/suspended/removed status.
- Timestamps and actor columns.
- Unique `(workspace_id, user_id)`.

### `roles`

Roles define named authority. AP must decide whether roles are:

- Global product roles.
- Organization-scoped roles.
- Workspace-scoped roles.
- A mixed model where standard system roles exist globally and tenant custom roles exist per org/workspace.

Role records should include status, scope, timestamps, and ownership. Existing buyer-demo roles should map to the future model without preserving demo persona as authorization.

### `permissions`

Permissions should be canonical and stable if AP wants auditable DB-backed authorization. Options:

- Keep JSONB permissions on role records for a smaller initial implementation.
- Normalize into `permissions` and `role_permissions` for stronger audit, RLS helper functions, and policy testability.

M5.2 recommends AP decide this before migrations.

### Role Assignments

Role assignments should bind user, role, and scope explicitly:

- Org-scoped assignment: user has a role within an organization.
- Workspace-scoped assignment: user has a role within a workspace.
- Optional inherited permissions: AP must approve whether org admins inherit workspace admin rights by default.

### Invitations

Invitations are deferred unless AP approves invite lifecycle or SSO timing before migration implementation. A future invitation table should avoid secrets in rows and store invite status, invited email, org/workspace target, role target, inviter, expiry, accepted timestamp, and audit references.

### M5.4 Placeholders

Evidence, lineage, handoff, generated artifacts, and audit records should reserve or plan for:

- `org_id`.
- `workspace_id`.
- Source entity type/id.
- Actor fields.
- Evidence refs.
- Correlation IDs.
- Retention class.
- Append-oriented audit behavior.

### M5.5 Placeholders

Export records should reserve or plan for:

- `org_id`.
- `workspace_id`.
- Source artifact/entity.
- Export type.
- Creator.
- Review status.
- Private storage path reference.
- Signed access metadata.
- Retention and revocation metadata.
- Audit event references.

## Tenant Boundary Model

| Boundary input | Direction |
| --- | --- |
| `org_id` | Required for every hosted pilot tenant record. It is the root tenant boundary. |
| `workspace_id` | Required or derived for most work artifacts after AP approves workspace authority. It is the primary work boundary. |
| `user_id` | Used for identity, actor, owner, reviewer, approver, creator, updater, assignee, requester, and membership records. |
| Membership authority | Active org membership plus active workspace membership for workspace-scoped data. |
| Role/permission authority | Server-governed role assignment and permission mapping, not browser state or demo persona. |
| Service authority | Service-role access remains server-only and limited to Edge/admin operations that have explicit audit and failure rules. |

## Hosted Pilot Tenant Mapping

Non-production demo mapping:

- Tenant: `Avala Demo Enterprise`.
- Workspace/process context: AP Invoice Exception Workflow.
- Initial scope: one organization, one canonical AP workspace, one AP process context, one canonical project/delivery context.
- Seed values must remain non-secret, non-personal, and environment-replaceable.

The demo tenant/workspace seed should be limited to non-production. Production pilot tenants need AP-approved environment ownership, invite/SSO decision, retention policy, and support ownership.

## Compatibility With Current Buyer-Demo Flow

- Do not change demo users, canonical AP seed, view keys, Sidebar behavior, guard semantics, route behavior, scoring, or local storage behavior in M5.2.
- Existing local/demo fallback remains valid for buyer-demo RC.
- Future schema changes must preserve deterministic scoring and human review boundaries.
- `View.WORKSPACE` should not be reinterpreted as a DB workspace until AP approves UI and routing implications.
- Existing `ScopeType` remains navigation context until a future approved implementation changes it.
