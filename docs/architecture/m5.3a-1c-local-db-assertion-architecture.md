# M5.3a-1c Local DB Assertion Architecture

## Purpose

This architecture plan defines how a future executable local DB assertion method should be structured for M5.3a-1 identity, organization, workspace, role, and membership RLS validation. M5.3a-1c is docs-only and implements no runner changes, SQL assertions, helper functions, RLS policies, migrations, package changes, local config, or DB connections.

## Current Architecture Baseline

Current baseline from M5.3a-1b:

- `test:rls:m5.3a-1` exists.
- `scripts/m5.3a-1-rls-isolation-smoke.mjs` is a fail-closed harness skeleton.
- `docs/schema/m5.3a_1_rls_membership_isolation_regression.sql` declares 16 scenarios.
- The harness executes 0 DB assertions.
- Accepted local target mode still fails closed because executable DB assertions are not implemented.
- No tracked `supabase/config.toml` exists.
- `scripts/applySupabaseMigrations.mjs` requires `DATABASE_URL` and is not safe as the default assertion runner.

## Target Architecture

Future M5.3a-1d should use a layered architecture:

1. Static safety gate.
2. Local-target proof gate.
3. Local DB connection setup.
4. Fixture setup and session/auth simulation.
5. SQL assertion execution.
6. Sanitized result aggregation.
7. Rollback or cleanup.

The JS runner owns orchestration and output. SQL files own executable assertions.

## Execution Flow

Future execution sequence:

1. Parse command arguments.
2. Reject unknown or unsafe arguments.
3. Check that executable assertion mode is implemented.
4. Prove local target without printing the target value.
5. Reject remote-looking targets and environment-only DB targets.
6. Establish local DB connection only after proof.
7. Start transaction or isolated local state.
8. Apply generated local fixtures.
9. Simulate anonymous/authenticated local sessions.
10. Execute scenario assertions.
11. Aggregate expected versus actual outcomes.
12. Roll back or clean local state.
13. Print counts and sanitized classifications only.

## Execution Option Architecture Comparison

| Option | Architecture fit | Main gap |
| --- | --- | --- |
| SQL-only local execution | Minimal execution path. | Lacks built-in local-target proof and sanitized output controls. |
| JS runner plus SQL assertions | Best fit. JS gates target and output; SQL exercises DB behavior. | Requires careful local connection and auth simulation design. |
| Supabase local stack with generic config | Best local substrate if AP approves config. | Requires new tracked generic config and local runbook. |
| Direct local Postgres/session claims | Viable fallback. | Must prove `auth.uid()` compatibility. |
| Mocked/no-DB | Useful for runner unit checks only. | Cannot prove RLS behavior. |

## Recommended Architecture

Use JS plus SQL assertions against an AP-approved local Supabase/Postgres stack.

Recommended components:

- JS runner: `scripts/m5.3a-1-rls-isolation-smoke.mjs`.
- Scenario contract: keep `docs/schema/m5.3a_1_rls_membership_isolation_regression.sql`.
- Executable assertions: add `docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql` in future implementation.
- Local DB client: Node `pg`, after local-target proof only.
- Optional local config: `supabase/config.toml`, only if AP approves a generic local-only file.

## Target-Proof Architecture

The local-target gate must run before any DB connection.

Allowed:

- `localhost`.
- `127.0.0.1`.
- Docker local network.
- AP-approved local token.

Rejected:

- Hosted URLs.
- Remote hosts.
- Silent `DATABASE_URL`.
- Ignored secret-file fallback.
- Unknown target arguments.

The target value must never be printed.

## Fixture Architecture

Fixtures should be generated inside transaction-scoped execution or isolated local state:

- Local organization A and B.
- Two local workspaces per organization.
- Active member.
- Disabled organization member.
- Disabled workspace member.
- Non-member.
- Organization roles.
- Workspace roles.
- Deleted-state variants where useful.

Fixture setup is server-only local setup. It must not represent browser-callable behavior and must not emit row payloads.

## Auth Simulation Architecture

Future assertion execution should prefer local SQL session claim simulation:

- Anonymous scenario: no local subject claim.
- Authenticated scenario: local authenticated session role plus local subject claim.
- Disabled scenarios: same local authenticated posture with disabled/deleted membership state.
- Non-member scenario: local authenticated posture with no membership rows.

Candidate mechanisms:

- `SET LOCAL ROLE authenticated`, if available and compatible.
- `SET LOCAL request.jwt.claim.sub`, if compatible with local `auth.uid()`.
- AP-approved local claim setup helper, only if required.
- Supabase local Auth fixtures, only if generic local config is approved.

No real JWTs are required for this method.

## SQL Assertion Architecture

Executable assertion SQL should:

- Be transaction-scoped.
- Use generated local UUIDs only.
- Set up local fixtures.
- Execute each scenario.
- Record only scenario id, expected result, actual result, and sanitized classification.
- Roll back or clean up.

Executable assertion SQL must not:

- Define helper functions.
- Define RLS policies.
- Alter applied schema.
- Create migrations.
- Insert durable production/demo/runtime data.
- Validate constraints.
- Assert artifact, provider, legacy, storage, export, Delivery Pack, Monitor, lineage, or handoff policies.
- Assert mutation or DELETE policies.

## Result Aggregation

The runner should map SQL assertion output into sanitized counts:

- Scenarios declared.
- Scenarios executed.
- Pass count.
- Fail count.
- Blocked count.
- Tenant isolation verified yes/no.
- Sanitized failure classifications.

The runner must not print SQL row payloads, target values, claim values, fixture IDs, or DB errors containing target values.

## Failure Modes

Future runner must fail closed when:

- Executable assertions are not implemented.
- Local-target proof is missing.
- Target appears remote.
- Required local config is missing.
- DB connection cannot be proven local.
- Fixture setup fails.
- Any scenario fails.
- Cleanup or rollback fails.
- Output sanitization detects a forbidden value class.

## Runtime And Hosted Readiness

This architecture can support verified local assertion execution for the approved M5.3a-1 scope. It does not prove hosted runtime readiness, adapter readiness, artifact policy readiness, export/storage readiness, provider readiness, deployment readiness, or external certification.
