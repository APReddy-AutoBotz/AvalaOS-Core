# M5.1 Environment Config Boundary Map

Milestone: M5.1 Environment Config And Secret Hygiene Gate
Status: Architecture boundary map

## Purpose

This map records the current environment and configuration boundaries that M5.1 protects. It defines the target local/development/staging/production separation and explains how the M5.1 gate supports later M5 Supabase readiness milestones without changing runtime behavior.

## Current Environment And Config Inventory

| Area | Current posture |
| --- | --- |
| Vite public environment usage | Browser code reads public client config for Supabase URL, Supabase anon key, AI Edge enablement, and Avala AI runtime mode. These names are public-config names, not server credentials. |
| Supabase URL and anon key usage | `services/supabaseClient.ts` initializes the browser Supabase client with public Vite config and falls back to placeholder/mock behavior when unconfigured. CI smoke scripts can receive public client config through CI secret storage. |
| Service-role and server-only expectations | Supabase Edge shared helpers require server-only URL, anon key for auth checks, and service-role credentials for privileged REST operations. Migration and smoke scripts use process environment names and must not print values. |
| Edge Function environment expectations | Edge sources use server environment access for Supabase, provider execution, provider resolution, storage bucket selection, and usage/audit writes. M5.1 documents the boundary only; it does not deploy or call Edge functions. |
| Provider environment and config usage | Server-side provider helpers include direct server environment names and key-reference governance. Browser provider fallback remains local-demo/internal-dev debt and is not pilot or production approved. |
| Browser storage and demo persistence surfaces | `services/storage.ts` and demo adapters retain non-pilot browser state for theme, current user, scope, view, Assess, Docs, Delivery, audit-adjacent logs, handoff ledger, and governance config. This is demo persistence, not hosted pilot authority. |
| CI environment posture | `.github/workflows/ci.yml` runs typecheck/build/scoring checks and an optional Supabase smoke job controlled by repository variables and GitHub secrets. M5.1 does not change CI. |
| Deployment environment posture | Hosted deployment, environment ownership, promotion gates, backup/restore, and pilot operations remain pending M5.6 work. M5.1 provides the secret-hygiene boundary that later deployment work must honor. |

## Target Local, Development, Staging, And Production Separation

| Environment | Target boundary |
| --- | --- |
| Local | Developer-only values and demo fallback allowed. No claim of hosted pilot authority. Ignored environment files may exist locally but must not be staged or inspected for values. |
| Development | Shared engineering validation with non-production data, governed replacement of personal values, and sanitized verification. |
| Staging | Production-like schema, RLS, storage, provider, export, and deployment checks using AP-approved non-production values. |
| Production | AP-approved tenant, secrets, backup/restore, audit retention, provider allowlist, export policy, incident owner, and pilot operations evidence. |

Each environment needs distinct config, secret ownership, storage, audit evidence, promotion rules, and rollback expectations.

## Browser, Edge, Scripts, And CI Variable Flow

- Browser: Receives only intentionally public Vite config and authenticated user session state. It must not receive service-role credentials, database connection strings, management tokens, raw provider secrets, storage-admin credentials, or secret reference values unless a later AP decision explicitly changes the policy.
- Edge/server runtime: Owns service-role access, server-side provider execution, provider key-reference resolution, privileged storage/export operations, and audit writes.
- Scripts: May use server-only names for migration or smoke validation, but must avoid value output. Scripts that parse config must report structural status only.
- CI: Stores values in GitHub secret storage or approved CI secret storage. M5.1 adds a local package script only; CI adoption is deferred.

## Client-Public Versus Server-Only Boundaries

- Client-public variables can include public Supabase URL, public anon key, AI Edge enablement, and runtime mode names.
- The anon key is public only when protected by an RLS-first architecture and must not be confused with service-role authority.
- Server-only variables include service-role credentials, database URLs, access tokens, management tokens, database passwords, direct provider secrets, storage-admin credentials, and raw private tokens.
- Browser storage and demo adapters must not authorize hosted pilot access, role membership, exports, provider execution, or audit outcomes.

## Provider Key-Reference Future Direction

M5.1 keeps provider secret values server-only and records the future key-reference naming direction. Later milestones should replace pilot behavior with server-side provider execution through approved key references, provider allowlists, safe logging, and audit coverage.

The target shape is:

- Browser requests generation or export through authenticated Supabase calls.
- Edge/server resolves provider policy and key-reference eligibility.
- Edge/server reads secret material from approved environment storage or a secret manager.
- Database, audit, logs, evidence, and exports store provider metadata, model, status, correlation IDs, and key-reference names only.
