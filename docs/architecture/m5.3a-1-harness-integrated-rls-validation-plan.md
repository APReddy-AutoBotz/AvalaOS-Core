# M5.3a-1 Harness Integrated RLS Validation Plan

## Purpose

This architecture refresh defines how future M5.3a-1 helper and SELECT policy implementation should use the M5.3a-1d executable local assertion method. It is docs-only. It does not change harness behavior, assertion SQL, package scripts, SQL migrations, helper functions, RLS policies, tests, runtime behavior, or local DB configuration.

## Current Harness Baseline

M5.3a-1d closed with executable assertion plumbing:

- `npm run test:rls:m5.3a-1` exists.
- The runner validates the scenario contract and assertion SQL file.
- The runner requires explicit local-target proof before DB action.
- The runner rejects hosted URL targets before DB action.
- The runner never prints target values or secret values.
- The runner loads `docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql` after target proof.
- The current assertion SQL declares 16 scoped scenarios, executes 0, blocks 16, and reports tenant isolation verified: no.
- A `localhost` target is accepted as local proof, but M5.3a-1d did not prove a reachable local DB was available.

## Validation Scope

Future M5.3a-1 validation remains limited to:

- `profiles`
- `organizations`
- `workspaces`
- `roles`
- `organization_members`
- `workspace_memberships`

Deferred from this validation slice:

- artifact tables
- provider governance tables and policies
- legacy draft tables
- storage/export
- Delivery Pack
- Monitor
- lineage/handoff
- all mutation policies

## Scenario Transition Plan

The existing 16 scenarios should become executable after helpers and SELECT policies exist.

Negative scenarios:

- `anonymous_denied`
- `authenticated_non_member_denied`
- `disabled_org_member_denied`
- `disabled_workspace_member_denied`
- `cross_org_organization_metadata_denied`
- `cross_org_membership_metadata_denied`
- `cross_workspace_membership_metadata_denied`
- `cross_workspace_role_metadata_denied`
- `system_role_visibility_denied`
- `service_role_not_browser_callable`

Positive scenarios:

- `own_active_profile_readable`
- `own_active_org_membership_readable`
- `own_active_workspace_membership_readable`
- `active_org_metadata_readable`
- `active_workspace_metadata_readable`
- `scoped_role_metadata_readable`

Expected future behavior after implementation:

- no target: 16 declared, 0 executed, non-zero, tenant isolation verified: no
- rejected target: 16 declared, 0 executed, non-zero, tenant isolation verified: no
- local target unavailable: 16 declared, 0 executed, non-zero, tenant isolation verified: no
- local target reachable but metadata missing: non-zero, tenant isolation verified: no
- local target reachable with helpers/policies but any scenario fails: 16 declared, 16 executed, non-zero, tenant isolation verified: no
- local target reachable with helpers/policies and all scenarios pass: 16 declared, 16 executed, zero, tenant isolation verified: yes for the scoped tables only

## Assertion SQL Integration

Future implementation should update `docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql` to create transaction-scoped generated local fixtures, simulate anonymous and authenticated local session claims, execute SELECT assertions for the six scoped tables, record pass/fail/blocked status per scenario, return a counts-only summary row, and roll back or clean up all local fixture data.

The assertion file must not define helper functions, define RLS policies, alter applied schema, create durable seed data, validate constraints, test artifact tables, test provider governance tables, test legacy tables, test storage/export, Delivery Pack, Monitor, lineage, or handoff surfaces, run mutation or DELETE checks, or return row payloads, claim values, target values, or secret values.

The existing scenario contract file may remain as a scenario declaration record. It should only be updated if assertion semantics require contract wording to stay aligned.

## Fixture Architecture

Future assertions should create generated local fixtures inside a transaction:

- two local organizations
- two local workspaces per organization
- active member
- disabled organization member
- disabled workspace member
- non-member
- organization-scoped roles
- workspace-scoped roles
- same-org and cross-org rows
- same-workspace and cross-workspace rows
- deleted-state variants

All identifiers must be generated local values. No production/demo/runtime rows, personal IDs, hosted IDs, provider keys, target values, tokens, bucket names, or `.env` values are allowed.

## Auth Simulation Architecture

Future assertions should simulate:

- anonymous client with no subject claim
- authenticated active member with generated local UUID subject claim
- authenticated disabled organization member with generated local UUID subject claim
- authenticated disabled workspace member with generated local UUID subject claim
- authenticated non-member with generated local UUID subject claim
- server-only local fixture setup path only if AP approves it

No real JWTs, token printing, service keys, browser-callable fixture setup, row dumps, or claim-value output.

## Local DB Availability Decision

M5.3a-1d did not prove reachable local DB availability. Future implementation evidence must distinguish:

- executable proof path: AP-approved local target reachable, assertions execute, scoped scenarios pass
- static/fail-closed path: no reachable AP-approved local target, assertions do not execute, tenant isolation not verified

Future implementation must never use remote DB, silently use environment DB targets, fall back to ignored secret files, print target values, or rely on `.env` values.

## Output Contract

The runner output must remain sanitized and counts-only:

- local target gate status
- local DB assertion execution status
- environment target variable count by count only
- scenario contract present yes/no
- assertion contract present yes/no
- scenarios declared
- scenarios executed
- pass count
- fail count
- blocked count
- tenant isolation verified yes/no
- harness status
- sanitized classifications

The output must never include DB URLs, target values, `.env` values, service values, provider keys, JWT/private tokens, hosted URLs, bucket names, personal IDs, row payloads, auth headers, or claim values.

## Future Implementation File Strategy

Recommended files for the future implementation milestone:

- `supabase/migrations/<actual_next_timestamp>_m5_3a_1_rls_helpers_membership_select_policies.sql`
- update `docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql`
- update `scripts/m5.3a-1-rls-isolation-smoke.mjs` only if assertion output compatibility requires it
- `docs/quality/m5.3a-1-rls-helper-select-policy-implementation-evidence.md`

No `supabase/config.toml` should be added unless AP separately approves a generic local config milestone.

## Verification Gates

Future implementation should run standard typecheck, test, focused tests, build, audit, static scans, `npm run test:rls:m5.3a-1` with no target to confirm fail-closed behavior, `npm run test:rls:m5.3a-1` with an explicit AP-approved local target if a reachable local DB exists, `git diff --check`, `git diff --cached --check`, `supabase --version`, and restricted wording scan.

No remote Supabase validation, provider calls, `scripts/applySupabaseMigrations.mjs`, `npm run db:migrate`, `npm run test:rls:assess`, or unguarded database-target commands.

## Proof Boundary

Local tenant isolation can be recorded as verified only when an AP-approved local target is used, helper and SELECT policy SQL exists, executable assertions run, all 16 scoped scenarios pass, output is sanitized, and no remote validation is used.

That proof is scoped only to identity, organization, workspace, role, and membership SELECT behavior. It does not prove hosted runtime readiness, production readiness, artifact-table coverage, export/storage readiness, provider readiness, or certification.

## Risks

- Assertion SQL can drift from policy semantics if not updated with the helper/policy migration.
- Local DB unavailability can leave the implementation static-reviewed only.
- Auth simulation may need AP hardening once real helper and policy SQL exists.
- Adding runner behavior changes without AP approval can weaken the M5.3a-1d proof boundary.
