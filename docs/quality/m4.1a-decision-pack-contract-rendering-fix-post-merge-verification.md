## Root Cause Summary

M4.0a identified critical finding `M4A-C01`: scored Process Detail could break or fail to show Decision Pack rationale because it referenced stale flat fields outside the current score contract:

- `decisionPack.whyThisRecommendation`
- `decisionPack.governanceControls`

The current deterministic score contract stores the rationale under `decisionPack.recommendedOperatingModel.whyThis`, nested governance control groups under `decisionPack.governance.*Controls`, and the flattened handoff governance controls under `handoffPack.governanceControls`.

## Minimal Fix Summary

M4.1a accepted implementation:

- Added `components/assess/decisionPackRenderModel.ts` as a small Process Detail display adapter.
- Updated `ProcessDetailStubView` to render Decision Pack rationale, governance controls, required document types, and backlog seeds through contract-safe helper functions.
- Kept `DecisionPack` and `HandoffPack` contracts unchanged.
- Kept deterministic scoring outputs unchanged.
- Wired focused render-model regression coverage into `npm run test`.

## Readiness Hardening Summary

AP requested one hardening follow-up before approval: Process Detail still directly read `scores?.supportingScores.handoffReadiness`, which could throw if `scores` existed but `supportingScores` was missing or partial.

The final M4.1a PR added `getReadinessValue` to the render-model adapter and changed Process Detail to read the readiness card value through that helper. The helper uses `scores?.supportingScores?.handoffReadiness` when present and falls back to assessment completion quality, so partial score records no longer dereference a missing `supportingScores` object.
