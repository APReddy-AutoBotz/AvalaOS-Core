## Risks Found

- `userApiKey` still exists in `App.tsx` state and is passed through multiple AI-adjacent UI surfaces.
- `aiOrchestrator` still accepts `userApiKey`, although current calls pass `allowUserKey=false` to the provider factory.
- `setup-api-key.js` remains an active-looking helper and can write `autobotz_api_key` to localStorage if populated.
- `StorageKeys.API_KEY` remains in `services/storage.ts`; `App.tsx` still removes that old stored key on mount.
- `services/geminiService.ts` reads `VITE_GROQ_API_KEY` and `VITE_GEMINI_API_KEY`.
- `vite-env.d.ts` still types `VITE_GEMINI_API_KEY`; `VITE_GROQ_API_KEY` is not typed.
- `vite.config.ts` injects `GEMINI_API_KEY` into browser-facing `process.env.API_KEY` and `process.env.GEMINI_API_KEY`.
- `GeminiProvider` and `GroqProvider` still perform direct browser provider calls and must remain demo/dev-only until later cleanup.
- Broad prop-chain cleanup could create unrelated UI churn if not handled carefully.
- Static scan automation is useful but remains deferred to M3.1c unless separately approved.

## AP Amendments Applied

- `StorageKeys.API_KEY` should remain temporarily as deprecated cleanup-only if `App.tsx` still removes old stored keys.
- The cleanup path should not be removed until scan evidence proves it is safe.
- No active localStorage API-key setup helper may remain after future M3.1b implementation.
- Provider factory files should not be moved or renamed if doing so creates broad import churn.
- Browser provider paths may be marked and isolated through comments, naming, and evidence.
- If provider factory movement is required, future implementation must stop and request AP approval.
- Future implementation should remove `vite.config.ts` browser-facing `process.env.API_KEY` and `process.env.GEMINI_API_KEY` injection.
- Future implementation must not replace removed Vite injection with another client-exposed provider-secret path.
- If `vite.config.ts` injection removal breaks explicit `local-demo` or `internal-dev` fallback, future implementation must stop and report.
- Preferred future action for `setup-api-key.js` is deletion.
- If `setup-api-key.js` is not deleted, it must become non-executable documentation only.
- `userApiKey` should be removed from active AI generation/refinement paths where safe.
- If broad prop-chain removal creates large unrelated UI churn, remaining hits must be isolated or neutralized and recorded for M3.1c.
- Future acceptance is that pilot/production do not depend on `userApiKey`.
- Remaining `userApiKey` hits must be documented as inactive, demo/dev-only, disabled-surface, or deferred cleanup.
- Future M3.1b implementation must manually scan the approved key/provider patterns and record every remaining hit with pattern, file path, allowed reason, and future action.
- M3.1b must not implement Edge provider config/key-reference resolution. That remains M3.2.
- M3.1b must not remove `GeminiProvider` or `GroqProvider` if doing so breaks explicit local-demo/internal-dev fallback.
- M3.1b must not implement static scan automation unless separately approved; that remains M3.1c.
