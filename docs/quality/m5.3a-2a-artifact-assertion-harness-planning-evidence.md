## Target Proof Plan

Future artifact runner must:

- Require explicit local target proof before database action.
- Allow only loopback or AP-approved local-only targets.
- Reject hosted URLs and remote hosts before database action.
- Reject silent environment database targets.
- Reject automatic `DATABASE_URL` execution.
- Reject secret-file fallback.
- Reject unknown arguments.
- Never print target values.

## SQL Static Guard Plan

Future runner should allow only transaction-scoped assertion behavior such as `BEGIN`, `ROLLBACK`, transaction-scoped `SET LOCAL`, temporary tables, local fixture inserts, and SELECT assertions.

Future runner should block durable or unsafe behavior such as commits, policy changes, durable table changes, function creation, extension creation, durable grants, fixture mutation outside transaction scope, service-role behavior, provider calls, and target or secret output.

## Output Contract

Future artifact runner output should include only:

- Artifact scenario contract present.
- Assertion SQL present.
- Local target gate status.
- Local DB assertion execution status.
- Scenarios declared.
- Scenarios executed.
- Pass, fail, and blocked counts.
- Artifact SELECT isolation verified.
- Sanitized classifications.
- Local target proof status without target value.

Future output must not include database targets, URLs, environment values, service-role values, provider keys, JWTs, private tokens, hosted identifiers, bucket names, personal identifiers, row payloads, claim values, or auth headers.

## Assertion Compatibility Plan

The future runner should use `docs/schema/m5.3a_2_artifact_select_isolation_assertions.sql`.

Artifact SELECT isolation may be verified only when all AP-approved artifact scenarios execute and pass. The runner should return non-zero for target gate failure, local database unavailability, missing assertion SQL, static guard failure, failed or blocked scenarios, metadata failure, cleanup failure, or output-sanitization failure.
