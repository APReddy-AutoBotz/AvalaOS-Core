## Implementation Summary

M5.3a-1d implements executable local assertion method plumbing only. It does not implement RLS helper functions, RLS policies, tenant policy SQL, SQL migrations, schema changes, production/demo/runtime seed data, backfill, constraint validation, runtime adapters, UI, routing, view keys, Sidebar behavior, demo data, scoring, provider execution, export behavior, deployment config, CI workflows, Reports/Teams/Admin behavior, or Health repository changes.

The harness now:

- Requires explicit local-target proof before any DB action.
- Rejects hosted URLs, remote hosts, silent environment DB targets, ignored secret-file fallback, and unknown target args.
- Never prints target values.
- Never prints DB URLs, `.env` values, service-role values, provider keys, JWT/private tokens, hosted URLs, bucket names, personal IDs, claim values, auth headers, or row payloads.
- Outputs only scenario counts, pass/fail/blocked totals, tenant isolation verified yes/no, sanitized classifications, and local target proof status without target value.
- Returns non-zero when target proof is missing, target is rejected, local DB connection is unavailable, assertions cannot run, assertions fail, cleanup fails, or output sanitization would fail.
- Returns zero only when AP-approved executable local DB assertions run and pass.

## Executable Assertion Method Summary

The implementation preserves the existing M5.3a-1b regression contract and adds a split assertion file at `docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql`.

The assertion SQL is transaction-scoped and ends with rollback. It uses temp tables only, generated local UUIDs only, local session claim simulation checks, metadata checks for the six identity/membership tables, and current baseline scenario rows. It does not create helpers, policies, migrations, durable seed data, schema changes, constraint validation, mutation/DELETE checks, artifact/provider/legacy/storage/export/Delivery Pack/Monitor/lineage/handoff assertions, or row payload output.

## Local-Target Proof Behavior

- Default command without `--local-target` fails closed before DB action.
- `--local-target localhost` is accepted as local proof, then attempts only a local DB connection with sanitized output.
- If no local DB/config is available, the local target command fails closed with `local-db-connection-unavailable` and no target values printed.
- Hosted URL target input is rejected before DB action. The rejected target value is not printed.
- Unknown args, invalid ports, invalid local database names, and invalid local user names fail closed before DB action.

## Auth Simulation Behavior

The SQL assertion file implements local session-claim plumbing checks without real JWTs. It uses generated local UUIDs and local `request.jwt.claim.sub` session state only. It does not print claim values and does not create a browser-callable fixture setup path.

The current milestone does not implement final role switching or policy-level access assertions. Those remain blocked until AP-approved helpers and SELECT policies exist.

## SQL Assertion File Behavior

The SQL assertion file:

- Runs inside a transaction.
- Uses temp tables with rollback.
- Generates local UUIDs with `gen_random_uuid()`.
- Checks local session claim set/clear plumbing.
- Checks availability of `auth.uid()` and scoped table/RLS metadata when a local DB is reachable.
- Returns a counts-only summary row for the runner.
- Marks all 16 scoped tenant-isolation scenarios as blocked under the current no-helper/no-policy baseline.
- Reports tenant isolation verified: no.

## Tenant Isolation Boundary

M5.3a-1d does not implement or prove tenant isolation. It proves only that the harness can enforce local-target proof, reject unsafe targets, load assertion SQL, attempt executable local assertions after proof, fail closed safely when local DB/config is unavailable, and produce sanitized counts-only output.

Verified local tenant isolation may only be claimed after AP-approved RLS helpers and SELECT policies exist, an AP-approved local DB target is used, AP-approved assertions execute, all expected allow/deny outcomes pass, and output remains sanitized.
