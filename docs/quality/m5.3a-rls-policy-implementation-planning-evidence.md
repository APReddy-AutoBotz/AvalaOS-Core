## Recommended First RLS Scope

Recommended implementation split:

- `M5.3a-1`: helper functions plus identity/org/workspace/membership SELECT policies and isolation fixture/test contract.
- `M5.3a-2`: SELECT policies for `assess_processes`, `assessments`, `assessment_review_events`, `projects`, `document_generations`, and `delivery_work_items`.
- `M5.3a-3`: mutation policies only after AP approves role/permission depth.

The first implementation slice should be SELECT-only, with no broad `FOR ALL` policies, no hard DELETE policies, and no role-aware mutations.

## Helper Strategy

Future required helpers:

- `is_active_org_member(p_org_id uuid)`
- `is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`

Optional only with AP approval:

- `has_org_permission(p_org_id uuid, p_permission text)`
- `has_workspace_permission(p_workspace_id uuid, p_org_id uuid, p_permission text)`

The refresh records recursion risk, `SECURITY DEFINER` risk, `search_path` hardening, `STABLE` function classification, least privilege, authenticated-only execute grants, no browser service-role exposure, and no cross-tenant leakage.

## Policy Matrix Summary

The policy matrix covers each applied authority table and recommends:

- SELECT policies only in the first slices.
- Active org membership for org-scoped reads.
- Active workspace membership for workspace-scoped artifact reads.
- Mutation denial until AP approves role/permission depth.
- No hard DELETE policies.
- Server-only service-role posture with actor/correlation context where future workflows require elevated operations.

## Nullable Workspace Hardening

Workspace-scoped artifact rows should fail closed when `workspace_id IS NULL` unless AP explicitly approves a narrow exception. The planning refresh applies that default to:

- `assess_processes`
- `assessments`
- `assessment_review_events`
- `projects`
- `document_generations`
- `delivery_work_items`

## Provider Governance Handling

Provider governance migrations and existing provider metadata policies were inventoried at planning level only. M5.3a leaves provider governance untouched and defers any provider-governance RLS hardening to a separate AP-approved slice.

## Negative Isolation Test Plan

Future tests should cover:

- Anonymous denial.
- Authenticated non-member denial.
- Disabled org member denial.
- Disabled workspace member denial.
- Cross-org denial.
- Cross-workspace denial.
- Nullable workspace rows fail closed by default.
- Buyer Viewer mutation denial.
- Contributor review/approval mutation denial.
- Delivery mutation denial until role-aware mutation policies are approved.
- Cross-tenant document, work item, source, evidence, lineage, and handoff reference denial.
- `metadata` and `source_lineage` cannot bypass tenant checks.
- Service-role behavior remains server-only and not browser-callable.

## Fixture Strategy

Future fixtures should be non-secret and deterministic:

- Two organizations.
- Two workspaces per organization.
- Active member, disabled org member, disabled workspace member, non-member.
- Buyer Viewer, contributor, Delivery Lead, and reviewer/admin where needed.
- Same-org/same-workspace, same-org/cross-workspace, and cross-org records.

No personal IDs, hosted URLs, project values, provider keys, service-role values, bucket names, tokens, or `.env` values may be used.

If no local validation path exists, M5.3a-1 should either produce implementation SQL plus a test contract with static review only, or remain blocked until local validation is available. Remote Supabase validation remains prohibited.

## Runtime And Hosted Readiness Posture

The refresh records that RLS implementation alone does not make hosted runtime ready. Current adapters may still target legacy draft tables, require approved adapter changes, or fail closed. Legacy draft tables remain outside the secured authority scope. Hosted pilot readiness requires later adapter, export, storage, deployment, operations, backup/restore, and evidence/audit work.

- Recursive helper behavior or overbroad definer functions could weaken isolation.
- Nullable workspace exceptions could accidentally create org-wide artifact visibility.
- Draft org-scoped policy files could be mistaken for applied secure policy authority.

- Provider governance has separate secret-boundary risks and should not be folded into the first artifact RLS slice.
- Static review alone cannot prove tenant isolation.
