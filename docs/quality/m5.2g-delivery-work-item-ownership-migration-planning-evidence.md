## Inventory Summary

Applied migration-chain authority currently includes tenant/workspace authority, Assess authority, `projects`, and `document_generations`.

Delivery work-item authority is not implemented. The applied migration chain does not include `delivery_work_items`, `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, `timesheet_entries`, Delivery Pack records, Monitor signal records, lineage authority, handoff authority, or export records.

Draft schema and runtime adapters still reference Agile-like tables and `app_id` mapping. Mock/demo data includes delivery tasks, assignees, reporters, dependencies, user stories, source lineage, Delivery Pack refs, handoff refs, and evidence refs.

## Option A Versus Option B Comparison

Option A promotes draft Agile-like tables: `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, and `timesheet_entries`. It is closer to current adapter and mock shapes but creates broader migration scope, cleanup needs for old cascade FKs and org-scoped policies, more backfill risk, more RLS complexity, and greater risk of expanding into generic project-management behavior.

Option B creates one narrow AvalaOS-specific `delivery_work_items` authority table first. It keeps product scope aligned to governed delivery, improves first-slice RLS readiness, avoids immediate legacy draft-table cleanup, and preserves buyer-demo runtime behavior while deferring adapter work.

Recommendation: Option B.

## FK And Delete Proposal

Future tenant/source/project/document FKs should use default/no-cascade delete behavior:

- `org_id -> organizations(id)`
- `(workspace_id, org_id) -> workspaces(id, org_id)`
- `(project_id, org_id) -> projects(id, org_id)`
- `(document_generation_id, org_id) -> document_generations(id, org_id)`
- `(source_process_id, org_id) -> assess_processes(id, org_id)`
- `(source_assessment_id, org_id) -> assessments(id, org_id)`

Actor/profile FKs should use `ON DELETE SET NULL` where appropriate:

- `created_by`
- `updated_by`
- `owner_id`
- `assigned_to`
- `reporter_id`

Future M5.2g-a should use staged `NOT VALID` FKs, avoid constraint validation, avoid seed/backfill, and avoid destructive cleanup unless AP separately approves it.

## App ID Hardening

If AP approves `app_id`, it must be nullable compatibility bridge only.

`app_id` is not tenancy authority, not globally unique, not a Supabase identifier, not a replacement for `id`, and not sufficient for authorization.

Future uniqueness should be scoped to active, non-deleted rows by at least `(org_id, project_id, app_id)`.

If AP rejects `app_id`, future runtime adapter work must map through UUID authority instead.

## Metadata And Source Lineage Hardening

`metadata` and `source_lineage` are compatibility payloads only. They are not durable evidence, approval, lineage, export/storage, review, or cross-tenant reference authority.

They must not store secrets, provider keys, service-role values, raw private tokens, personal hosted values, or unsupported certification claims.

Extra assignees, dependencies, subtasks, user stories, and order rank remain non-authority UI/runtime compatibility until AP approves normalized persistence.

## Legacy Draft Schema Risk

Legacy draft/manual non-production tables may exist: `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, and `timesheet_entries`.

Option B must not alter or drop those tables in M5.2g-a. It must not claim that those legacy draft tables are secured or hosted-ready.

Later cleanup or quarantine may be needed for legacy org-scoped policies, cascade FKs, single-column project FKs, timesheet/task cascade behavior, draft `app_id` indexes, and adapter migration.

## RLS Posture

M5.2g does not implement RLS. Future M5.2g-a may enable RLS on `delivery_work_items` with no policies, but that is fail-closed readiness only.

Tenant isolation remains unproven until M5.3a or a later AP-approved RLS implementation and isolation-test milestone.

Future tests should cover anonymous denial, non-member denial, disabled-member denial, cross-organization denial, cross-workspace denial, Buyer Viewer read-only behavior, Delivery Lead updates, assigned contributor own-work updates, and no cross-tenant source/document/evidence/handoff refs.
