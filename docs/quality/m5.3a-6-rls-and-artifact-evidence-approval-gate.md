# M5.3a-6 RLS and Artifact Evidence Approval Gate

## Purpose

M5.3a-6 defines the approval boundary before any future RLS or artifact evidence run design can begin.

This gate converts the M5.3a-5 evidence requirements into a concrete approval checklist. It prevents accidental DB execution, Supabase stack work, Docker work, schema inspection, RLS tests, artifact assertions, hosted validation, local-readiness reopening, or readiness claims.

This milestone requires explicit AP approval before any future harness design or evidence execution. It is docs-only and does not build a harness, execute a run, validate DB behavior, validate RLS behavior, validate artifact behavior, or prove readiness.

## Approval Checklist For Any Future Evidence Run Design

Before any future evidence run design can be started, AP must explicitly approve:

- exact future milestone name;
- exact assertion names;
- exact evidence scope;
- exact tenant boundary scope;
- whether schema availability may be checked;
- whether RLS helper behavior may be checked;
- whether artifact SELECT isolation may be checked;
- whether same-tenant allowed-path evidence is permitted;
- whether cross-tenant blocked-path evidence is permitted;
- whether missing or invalid tenant context blocked-path evidence is permitted;
- run count limit;
- allowed sanitized output fields;
- prohibited output fields;
- cleanup expectations;
- stop and fail-closed conditions;
- proof-boundary wording;
- whether any DB execution is allowed in a later milestone;
- confirmation that no local-readiness reopening is allowed.

No future design may proceed if any required approval item is absent or ambiguous.
