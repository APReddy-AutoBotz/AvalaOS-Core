## Current Studio/Docs Schema Inventory Summary

`document_generations` exists in draft schema and runtime adapter usage, not in the applied migration chain. It has draft `org_id`, `project_id`, `template_id`, generated timestamp, and artifact payload fields. It does not have workspace ownership, creator/updater, reviewer/approver, source Assess references, lifecycle fields, or audit correlation fields.

Generated artifacts, Assess-to-Studio source context, document exports, Docs-to-Delivery lineage, and Delivery Pack document outputs are runtime/browser constructs today. They are not durable tenant authority.

The current Document Vault is a product surface and browser workflow, not an applied database table. Templates are runtime/static/browser-managed and must remain separate from provider prompt/job authority.

The key blocker is that draft Studio/Docs persistence points at `projects`, while `projects` is not applied migration-chain authority and has no approved workspace mapping yet.

## Docs-Only Decision Summary

M5.2d remains docs-only. No migration was created because implementing Studio/Docs ownership now would require premature project/workspace decisions and could produce unstable RLS inputs. The safe sequence is to plan Delivery/project ownership first, then implement Studio/Docs ownership using the approved workspace boundary.

## Proposed Ownership Model Summary

Future Studio/Docs migration work should use nullable-first ownership columns:

- `org_id`
- `workspace_id`
- `created_by`
- `updated_by`
- `owner_id`
- `reviewer_id`
- `approver_id`
- `approved_by`
- `rejected_by`
- `source_assessment_id`
- `source_process_id`
- `source_project_id`
- `source_document_generation_id`
- `status`
- `archived_at`
- `deleted_at`
- `correlation_id`
- `audit_correlation_id`

Those columns should be applied per table only where they match approved authority. They should not all be forced onto every Studio/Docs concept.

## FK And Backfill Strategy Summary

Future implementation should add nullable columns first, use same-tenant composite foreign keys where the referenced tables are applied authority, prefer `NOT VALID` constraints for broad or potentially pre-existing rows, and validate constraints only after AP-approved backfill.

M5.2d performs no backfill, seeds no data, commits no hosted identifiers, and uses no personal environment values.

## Migration Safety Summary

M5.2d creates no migration. Future implementation should be additive only, avoid destructive updates, avoid runtime dependencies until adapters are updated, avoid broad FK validation before backfill, create no RLS policies until the RLS implementation milestone, and avoid remote Supabase validation unless AP approves it later.

## Buyer-Demo Compatibility Summary

The buyer-demo flow remains unchanged. M5.2d does not change document generation, Govern-derived rendering, Assess-to-Studio handoff, Docs-to-Delivery lineage, Delivery Pack behavior, browser exports, local storage, mock data, scoring, routing, or UI.

## RLS Readiness Summary

Future Studio/Docs RLS should support workspace-scoped generated artifact reads, cross-organization denial, cross-workspace denial, Buyer Viewer read-only behavior where approved, contributor/reviewer/approver gates, status-aware lifecycle controls, same-tenant source assessment enforcement, evidence and lineage boundaries, handoff boundaries, and later server-mediated export controls.

M5.2d does not implement or prove tenant isolation.

## AP Decisions Needed

- Decide whether Delivery/project ownership must be implemented before Studio/Docs ownership.
- Decide whether Studio artifacts are workspace-owned, project-owned, assessment-owned, or mixed.
- Approve whether `document_generations` should be promoted or replaced with a richer document repository model.
- Approve nullable-first ownership columns for future Studio/Docs migrations.
- Decide whether generated artifacts need separate metadata or version tables.
- Decide template scope and separation from provider prompt authority.
- Approve source Assess to Studio/Docs workspace mapping.
- Decide whether Docs-to-Delivery lineage becomes standalone authority or remains task metadata until Delivery ownership is complete.
- Confirm export records and storage object references remain deferred to the export-controls milestone.
- Approve future backfill, constraint validation, and RLS test timing.

## Risks Carried Forward

- Studio/Docs ownership depends on project/workspace authority that is not applied yet.
- Draft schema can be mistaken for applied migration authority.
- Browser/demo persistence remains useful for buyer-demo flow but is not hosted-pilot authority.
- Embedded generated artifact payloads are not durable evidence or lineage authority.
- RLS readiness columns do not enforce tenant isolation without policies and tests.
- Broad FKs and backfills can fail without approved workspace mapping.
- Export/storage controls and provider execution boundaries remain separate future work.
