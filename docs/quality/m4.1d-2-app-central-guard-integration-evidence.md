M4.1d-1 added the pure central View/RBAC guard helper but did not integrate it into `App.tsx`. App-level persisted state, direct handlers, project handoffs, and fallback effects could still move between views through local `setCurrentView` calls or module-only checks.

## App Integration Summary

`App.tsx` now imports `resolveViewAccess` and wraps it in local App-shell helpers:

- `resolveAppViewAccess(view, scope)`
- `applyGuardedView(view, requestedScope)`
- `setScopeIfChanged(scope)`

The integration passes current user, auth/org loading state, current organization, enabled modules, requested view, and requested scope into the central resolver before applying App-level view transitions.

Auth and organization loading are treated as wait states. The App shell does not redirect or overwrite persisted view/scope state while those inputs are still loading.

## AP Handoff Hardening Note

AP review identified that Assess-to-Studio source context could be cleared when the original handoff scope was not directly valid for Docs Forge, even if the central guard could safely land the user on Docs Forge through a compatible fallback scope.

The hardening keeps the fix local to `App.tsx`:

- Assess-to-Studio handoff now normalizes the requested Docs Forge scope before calling `applyGuardedView`.
- `MY_WORK`, `TEAM`, and `PROJECT` scopes are preserved as requested scopes.
- `ORGANIZATION` scope is normalized to `MY_WORK` for the Docs Forge transition.
- Valid source context is preserved when the normalized Docs Forge transition is allowed.
- Source context is still cleared when Docs Forge cannot be reached because Docs is disabled, the user lacks access, or the guard does not allow the normalized Docs Forge transition.

This does not change source context attached to generated artifacts after a valid Assess-to-Studio handoff.

## Fallback Behavior Summary

Denied App transitions now use `resolveViewAccess` fallback output:

- stale persisted view/scope falls back after auth/org loading resolves
- disabled modules fall back to the resolver-selected enabled module path
- invalid scope applies the fallback scope where the resolver supplies one
- missing permission applies the resolver-selected fallback view/scope
- Reports and Teams remain governed by existing deferred/decision-pending helper metadata

The existing organization-scope Admin workspace path is preserved. Workspace/Admin route semantics remain an AP decision area and were not newly implemented in M4.1d-2.
