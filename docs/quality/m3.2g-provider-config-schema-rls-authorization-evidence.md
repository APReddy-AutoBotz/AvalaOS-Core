## Authorization Summary

`docs/planning/m3.2g-provider-config-schema-rls-authorization-plan.md` records the final AP authorization blueprint for the first provider config schema/RLS migration PR.

Recommended first migration object list:

- `ai_provider_configs`
- `ai_provider_key_refs`
- `ai_workspace_provider_policies`
- `ai_provider_audit_events`

Deferred scope:

- prompt version table changes
- usage/cost expansion
- Admin UI
- runtime resolver
- provider-call integration
- external secret-manager lookup
- Supabase Function changes
- browser fallback removal
- Health work

Key AP amendments included:

- `secret_ref` means a service-only reference identifier, not raw secret material.
- `secret_ref` must never contain raw provider keys, encrypted provider keys, auth headers, browser-readable secret values, or secret-manager payloads.
- `ai_provider_configs` must not contain raw secrets, encrypted secrets, auth headers, browser-readable secret values, or secret-manager values.
- Browser/client code must never directly write sensitive provider/key/audit objects or resolve provider keys.
- Future Admin UI writes must be mediated by AP-approved server/control workflows, RLS, audit evidence, and secret-safety gates.
- `ai_workspace_provider_policies` keeps future-compatible naming while first blueprint uses `org_id` as the enforceable tenant boundary.
- Strict `workspace_id` FK remains deferred until AP approves workspace authority.
- `ai_provider_configs.key_ref_id` may be a nullable FK to `ai_provider_key_refs.id`.
- `ai_provider_key_refs` should not FK back to provider configs in the first migration blueprint.
