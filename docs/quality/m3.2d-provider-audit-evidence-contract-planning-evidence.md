- `docs/schema/ai_governance_jobs.sql` defines prior-context `ai_generation_jobs` and `ai_usage_events`.
- `ai_generation_jobs` can record job status, provider config reference, prompt template reference, prompt version, model, input/output refs, error message, and timestamps.

## Event Taxonomy Summary

The planning doc proposes these future event types only:

- `provider_resolution_started`
- `provider_resolution_allowed`
- `provider_resolution_blocked`
- `provider_resolution_failed`
- `provider_call_started`
- `provider_call_succeeded`
- `provider_call_failed_sanitized`
- `provider_policy_missing`
- `provider_disabled`
- `key_reference_missing`
- `key_reference_disabled`
- `key_resolution_failed`
- `prompt_version_missing`
- `usage_recorded`

For each event type, the plan records purpose, emission point, actor, tenant/workspace, safe provider/config/key-reference identifiers, prompt/template/version references, operation type, mode, policy result, failure class when applicable, safe UI/control-plane summary, and fields never logged.

## Common Event Envelope Summary

Candidate event/schema discussion is not approval to migrate, alter, depend on, or implement these structures in this milestone.

The proposed common event envelope includes:

- `event_id`
- `event_type`
- `correlation_id`
- `request_id`
- `actor_id`
- `tenant_id`
- `workspace_id`
- `mode`
- `operation_type`
- `provider_id`
- `provider_config_id`
- `key_reference_id` or AP-approved safe fingerprint only
- `prompt_template_id`
- `prompt_version_id`
- `policy_result`
- `failure_class`
- `sanitized_status`
- `usage_metadata`
- `created_at`
- `schema_version`

Raw secrets must never be included.

AP clarification recorded:

- Evidence events must prove that a gate was checked, but must not become a storage location for sensitive payloads.
- Evidence should record the result, reason class, safe identifiers, and timestamps, not the underlying secret, prompt body, response body, raw provider error, or hidden chain/payload.

## Success / Failure Event Summary

Success event planning covers:

- allowed provider resolution evidence
- provider-call success evidence
- usage metadata shell
- token/cost fields only when safe and non-misleading
- prompt/version binding
- policy proof
- human-readable control evidence

Failure/block event planning covers:

- fail-closed blocked events
- M3.2c sanitized failure classes
- retry posture
- safe UI message category
- AP-review evidence fields
- prohibited logs
- how blocked events prove governance without leaking secrets

## Fields Never Logged Summary

Future audit/evidence events must not log:

- raw provider secrets
- full prompt content unless separately AP-approved
- full model response unless separately AP-approved
- raw provider error body
- provider authorization headers
- browser raw keys
- `userApiKey`
- `localStorage` values
- secret-manager values
- unredacted stack traces
- regulated/compliance claims
- hidden chain/payload or reasoning content
- resolved key material
- service-role keys
- browser session tokens, cookies, or authorization headers
- raw source document bodies
- raw provider request bodies
- raw provider response bodies

## Schema Relationship Notes

This proposal does not approve schema/RLS changes or migrations.

Planning notes:

- `ai_usage_events` may be reusable for safe usage metadata but should not be the only audit record for resolver, provider config, key-reference, and policy gate evidence.
- A separate `ai_provider_audit_events` object may be needed for append-only provider resolution, policy, key-reference, prompt/version, and fail-closed evidence.
- `ai_generation_jobs` may later reference provider audit events by future AP-approved foreign key or correlation ID.
- Candidate foreign keys may involve provider config, key reference, prompt template/version, generation job, usage event, tenant/org, workspace, and actor records.
- Retention must preserve governance evidence without retaining sensitive payloads.
- RLS/read visibility must separate safe admin/control evidence from sensitive payloads.
- Service-role writer posture should prefer append-only server-side writes; browser/client direct writes should not be approved for provider audit events.
