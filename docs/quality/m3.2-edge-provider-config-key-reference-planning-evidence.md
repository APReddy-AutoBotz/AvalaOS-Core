## Current Architecture Inventory

- `services/aiMode.ts` defines exactly four runtime modes: `local-demo`, `internal-dev`, `pilot`, and `production`.
- `services/aiMode.ts` blocks invalid or missing mode configuration and blocks `pilot` / `production` when Edge AI is unavailable.
- `services/aiOrchestrator.ts` resolves mode policy before AI execution.
- `services/aiOrchestrator.ts` routes to `aiEdgeClient` when the policy boundary is Edge.
- `services/aiOrchestrator.ts` allows browser provider fallback only after the mode policy allows explicit `local-demo` / `internal-dev` fallback.
- `services/aiEdgeClient.ts` invokes Supabase Edge Functions and sends `providerType`, document/refinement inputs, and source metadata from the client to Edge.
- `services/geminiService.ts` reads `VITE_GROQ_API_KEY` and `VITE_GEMINI_API_KEY` only for the explicit demo/dev browser fallback path. It no longer accepts a user-provided raw browser key.
- `supabase/functions/_shared/ai.ts` currently supports `gemini` and `groq`.
- `supabase/functions/_shared/ai.ts` currently reads direct server environment keys with `Deno.env.get('GROQ_API_KEY')` and `Deno.env.get('GEMINI_API_KEY')`.
- `supabase/functions/_shared/ai.ts` currently uses direct provider calls for generation and refinement. Future M3.2 implementation should replace direct key lookup with server-side provider config and allowlisted key-reference resolution.
- `scripts/check-ai-boundary.mjs` scans repository files for AI/key boundary patterns, classifies allowed hits through `docs/quality/ai-boundary-static-scan-allowlist.json`, and fails closed on forbidden or stale runtime hits.
- `package.json` already includes `test:ai-boundary-static`; this milestone did not change package scripts or dependencies.

## Current Browser / Demo Fallback Risks

- Browser provider fallback still exists for explicit `local-demo` / `internal-dev` paths.
- `VITE_GROQ_API_KEY` and `VITE_GEMINI_API_KEY` remain classified by the scanner as demo/dev fallback references, not pilot/production permission.
- `localStorage` remains broad and must continue to be classified carefully so non-secret app state does not become a provider-key storage path.
- `StorageKeys.API_KEY` remains only as deprecated cleanup-only handling.
- Future implementation must not remove local-demo/internal-dev fallback unless separately approved.

## Current Server-Side Risks

- `supabase/functions/_shared/ai.ts` reads direct server env keys instead of resolving a tenant/workspace provider config and key reference.
- Edge generation/refinement accepts `providerType` from the client and checks only supported provider type before direct server key lookup.
- Missing direct server env keys currently fail with provider-specific missing key messages. Future behavior should classify missing key reference, missing provider config, disabled provider, unsupported provider, and policy denial as sanitized fail-closed errors.
- Existing job/usage evidence is not yet the full M3.2 provider config/key-reference evidence shape.

## Planned Architecture Summary

Future M3.2 implementation should define:

- provider registry with supported provider types, status, default model policy, and provider capability metadata
- provider key reference as non-secret reference-only data
- tenant/workspace provider policy enforced server-side before provider calls
- provider status states that fail closed when missing, disabled, pending, unsupported, or policy-disallowed
- fail-closed behavior before provider calls when auth, org/workspace membership, provider config, key reference, prompt/version reference, or evidence logging is unavailable
- audit/evidence event shape with provider config reference, provider type, model, prompt key/version, operation, usage metadata, status, output reference, and sanitized error class
- prompt/version reference tied to server-side prompt constants or later DB-backed prompt templates after approval
- sanitized errors that never expose raw provider secrets, resolved key values, auth headers, raw prompts, source bodies, or provider secret material
- human-readable control evidence that can later support read-only Admin Controls without raw-key display
