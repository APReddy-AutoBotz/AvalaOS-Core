## Root Cause

M4.1d found that AvalaOS Core has multiple ways to arrive at a view:

- persisted `currentView` / `currentScope`
- Sidebar filtering
- module enablement helpers
- direct `setCurrentView` handlers
- ModuleJourney
- App fallback effects

Those surfaces need a shared View/RBAC decision contract before App integration can safely centralize persisted and direct view transitions.

## Guard Helper Summary

`services/viewAccessGuard.ts` adds:

- `VIEW_ACCESS_METADATA`
- `resolveViewAccess`
- `hasViewPermission`
- typed guard reasons and severities
- typed metadata and result contracts

The helper is pure and deterministic. It does not mutate state, persist state, render UI, call providers, fetch remote data, or change scoring behavior.

## Guard Precedence Summary

Implemented precedence:

1. Auth loading: `auth_loading`, `wait`.
2. Unauthenticated: `unauthenticated`, `redirect`.
3. No organization: `no_organization`, setup fallback.
4. Empty enabled modules: `setup_required`, setup fallback.
5. Unknown/stale persisted view: `stale_persisted_view`.
6. Deferred/decision-pending view: `deferred_view` or `admin_decision_pending`.
7. Disabled module: `disabled_module`.
8. Invalid scope: `invalid_scope`.
9. Missing permission: `missing_permission`.
10. Allowed: `allowed`.

## Permission Semantics Summary

The helper preserves current Sidebar semantics:

- Admin role bypasses listed per-view permissions.
- Non-admin users need any one listed permission.
- Views with no required permissions still pass through auth, organization, module, scope, and deferred-view checks.

The helper does not introduce all-permission enforcement.
