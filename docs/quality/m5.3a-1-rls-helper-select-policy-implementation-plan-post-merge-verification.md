## SELECT Policy Plan Confirmation

The merged plan confirms explicit SELECT-only policy direction:

- `profiles`: own active, non-deleted profile only.
- `organization_members`: own active, non-deleted membership only, gated by active org helper.
- `workspace_memberships`: own active, non-deleted workspace membership only, gated by active workspace helper.
- `organizations`: readable only through active org membership.
- `workspaces`: readable only through active workspace membership.
- `roles`: org roles through active org membership; workspace roles through active workspace membership; system role visibility deferred to AP.

The plan excludes broad `FOR ALL`, INSERT, UPDATE, DELETE, hard DELETE, mutation, artifact, provider governance, and legacy draft table policies.

## SECURITY DEFINER Hardening Confirmation

The merged docs state that `SECURITY DEFINER` is provisional, not automatic. Future implementation must prove no recursive RLS failure, no broad role grants, no `search_path` hijack risk, no cross-tenant information leakage, no data-returning helper behavior, no disabled/deleted membership bypass, and no parent org/workspace disabled/deleted bypass.

## Recursion Risk Analysis Confirmation

The merged docs record these recursion risks:

- Helpers may query membership tables while those tables have RLS enabled.
- Membership policies may depend on helpers.
- Role policies may depend on membership visibility.
- Overbroad definer helpers can leak membership or role existence across tenants.

Recorded controls are minimal helper functions, no permission helpers in M5.3a-1, no role-helper dependency in membership visibility, no artifact table dependency, and required local validation before verified isolation can be claimed.
