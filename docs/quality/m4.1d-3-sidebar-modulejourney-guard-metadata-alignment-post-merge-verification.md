## Sidebar Before/After Summary

Before M4.1d-3:

- Sidebar nav items carried local module keys.
- Sidebar nav items carried local scope arrays.
- Sidebar nav items carried local permission arrays.
- Sidebar had local access logic for Admin bypass and permission matching.

After M4.1d-3:

- Sidebar keeps presentation metadata only: view, icon, label, and section grouping.
- Sidebar delegates each nav item access decision to `resolveViewAccess`.
- Disabled module, missing permission, deferred view, loading, setup, stale, and decision-pending results are hidden.
- Invalid-scope items stay visible but disabled with the existing available-scope tooltip pattern.
- The Avala Admin organization-scope button remains preserved as a deferred AP decision path.

## ModuleJourney Before/After Summary

Before M4.1d-3:

- ModuleJourney listed enabled modules directly.
- ModuleJourney marked active module through a separate view-to-module map.
- ModuleJourney navigated directly to `MODULE_HOME_VIEW` for each module.

After M4.1d-3:

- ModuleJourney keeps its existing lifecycle layout, labels, icons, and outcome text.
- ModuleJourney resolves each module home through `resolveViewAccess`.
- Allowed module homes remain clickable.
- Invalid-scope module homes remain visible but disabled with a title naming the valid scope.
- Missing-permission and other denied module homes are hidden.
- Active module calculation reads from `VIEW_ACCESS_METADATA`.
- ModuleJourney no longer navigates blindly to module home views.

## Central Guard Metadata Usage Summary

Sidebar and ModuleJourney now use:

- `resolveViewAccess`
- `VIEW_ACCESS_METADATA`
- current user, organization, module, and scope inputs
- the existing central guard permission semantics

This aligns navigation visibility and clickability with the same guard contract App uses for view/scope application.
