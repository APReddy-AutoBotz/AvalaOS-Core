## Secret Safety Confirmation

`providerSecretAdapter`:

- accepts only allowed resolver decisions
- blocks non-allowed decisions
- blocks `manual_placeholder`
- blocks `external_secret_reference`
- blocks unsafe environment reference shapes
- accepts only provider-specific server references using:
  - Groq references: `AVALA_PROVIDER_SECRET_GROQ_[A-Z0-9_]+`
  - Gemini references: `AVALA_PROVIDER_SECRET_GEMINI_[A-Z0-9_]+`
- rejects reserved platform, database, JWT, direct provider-key, and unrelated provider-key environment names before any environment read
- rejects provider-prefix mismatches before any environment read
- rechecks key reference org, provider, ID, status, resolver type, expiry, and deletion state
- treats key reference values as server environment variable names only
- returns raw key material only to the immediate server-side provider-call boundary

The test suite checks that safe failure responses and audit metadata do not include secret references, raw keys, auth headers, or provider payload fields.

## Resolver Exception Safety Confirmation

`providerResolverIntegration` now wraps the resolver call. If a resolver DB dependency throws before a decision exists, the Edge runtime returns a sanitized blocked response with a safe correlation ID and `provider_call_blocked`.

This fail-closed path:

- does not include raw Supabase/PostgREST errors
- does not include table names
- does not include stack traces
- does not include provider payloads
- does not include key references or key material
- does not call secret lookup
- does not call provider helpers
- does not create AI jobs
- can skip audit persistence because no safe resolver decision shell exists yet

## Audit Safety Confirmation

`providerResolverAuditDb`:

- inserts sanitized resolver decision events into `ai_provider_audit_events`
- reuses the M3.2l audit event shell as input
- revalidates metadata safety before insert
- maps allowed event shell status to the schema-compatible recorded status
- fails closed before secret lookup and provider handoff when required allowed-path audit persistence fails

Blocked decisions attempt sanitized audit persistence when safe context is available. If context is insufficient or persistence fails, the response still fails closed and does not proceed to secret lookup or provider handoff.
