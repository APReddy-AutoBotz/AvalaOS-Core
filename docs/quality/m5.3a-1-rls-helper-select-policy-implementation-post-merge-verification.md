## Helper Function Summary

Implemented helpers:

- `public.is_active_org_member(p_org_id uuid)`
- `public.is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`

Helper posture confirmed by static review:

- `LANGUAGE sql`
- `STABLE`, not `IMMUTABLE`
- `SECURITY DEFINER`
- Boolean-only output
- Minimum safe `SET search_path = pg_catalog`
- Schema-qualified table references
- Requires `auth.uid()`
- Requires active, non-deleted membership
- Requires active, non-deleted parent organization
- Requires active, non-deleted parent workspace for the workspace helper

## SECURITY DEFINER Decision

`SECURITY DEFINER` was used narrowly for the two boolean membership helpers to avoid recursive RLS bootstrap failures where membership, organization, workspace, and role SELECT policies depend on membership resolution while membership tables are themselves RLS-protected.

The helpers return no tenant metadata, role metadata, permission metadata, row payloads, counts, target values, or secret values.

## Helper Search Path And Grant Posture

Confirmed:

- Both helpers use `SET search_path = pg_catalog`.
- Helper table references remain schema-qualified.
- Helper function calls to `auth.uid()` remain schema-qualified.
- `EXECUTE` is revoked from `PUBLIC`.
- `EXECUTE` is revoked from `anon` when the role exists.
- `EXECUTE` is granted only to `authenticated` when the role exists.
- `SELECT` is granted to `authenticated` for the six scoped tables so RLS policies determine row access.
- No service-boundary role is targeted by M5.3a-1 policies.

## SELECT Policy Summary

Created six explicit `FOR SELECT TO authenticated` policies:

- `profiles`: own active, non-deleted profile only.
- `organization_members`: own active, non-deleted organization membership only, gated by active organization membership.
- `workspace_memberships`: own active, non-deleted workspace membership only, gated by active workspace membership.
- `organizations`: active organization members can read active, non-deleted organization metadata for their organization.
- `workspaces`: active workspace members can read active, non-deleted workspace metadata for their workspace.
- `roles`: active organization members can read active non-system organization-scoped roles, and active workspace members can read active non-system workspace-scoped roles.

Confirmed:

- No broad `FOR ALL` policies.
- No INSERT policies.
- No UPDATE policies.
- No DELETE policies.
- No mutation policies.
- No artifact, provider, legacy, storage/export, Delivery Pack, Monitor, lineage, or handoff policies.
- No permission helpers.
