## Target Architecture Summary

- Local, development, staging, and production environments must be separated with distinct ownership, values, storage, audit evidence, and promotion gates.
- Supabase Auth is the hosted pilot identity direction.
- Organizations are tenant authorities; workspace, user, role, and membership authority must be designed before schema and RLS implementation.
- Data access should be RLS-first, deny-by-default, and regression-tested by organization, workspace, role, artifact, and denial case.
- Evidence, lineage, handoff, generated artifact, delivery pack, export, and audit records should become durable Supabase-backed authorities.
- Export controls should use private storage, org/workspace-scoped paths, signed access, review metadata, retention policy, and audit events.
- Pilot and production provider execution should be server-side through Supabase Edge or an approved server runtime using key references, not browser provider secrets.

## Secret Hygiene Summary

- `.env*` files remain ignored and unstaged.
- M5.0 did not read, print, summarize, copy, or commit `.env` values or secrets.
- Browser-side provider secrets are prohibited for pilot and production behavior.
- Service-role credentials, database owner credentials, management tokens, storage-admin credentials, and provider secrets are server-only.
- No personal Supabase values, project URLs, project references, tokens, keys, organization identifiers, bucket names, or secrets were committed.
- Verification may confirm names, presence, absence, and class, but must not print secret values.
