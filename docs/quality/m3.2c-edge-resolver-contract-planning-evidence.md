## Current Edge Flow Inventory

- `supabase/functions/_shared/ai.ts` defines `getProviderKey(provider)`.
- `getProviderKey('groq')` reads `Deno.env.get('GROQ_API_KEY')`.
- `getProviderKey('gemini')` reads `Deno.env.get('GEMINI_API_KEY')`.
- Current supported Edge providers are `groq` and `gemini`.
- Current model env reads include `GROQ_MODEL` and `GEMINI_MODEL`.
- `generateDocumentWithProvider` and `refineSectionWithProvider` read server env keys before calling providers and fail when the selected server key is missing.
- `services/aiEdgeClient.ts` sends `providerType` and operation request bodies to Supabase Edge Functions.
- `services/aiOrchestrator.ts` and `services/aiMode.ts` enforce the M3.1a mode boundary: pilot/production require Edge AI, while `local-demo` / `internal-dev` retain approved browser fallback.

Current gaps recorded in the planning doc:

- no tenant/workspace provider policy
- no key-reference resolver
- no provider config lifecycle check
- no prompt/version binding contract
- no resolver audit/evidence event emission
- no provider operation policy

Existing direct server env-key reads remain unchanged in M3.2c.

## Proposed Resolver Contract Summary

The planning doc proposes `edgeProviderResolver` as a future conceptual module name only. It is not implemented in M3.2c.

Input summary:

- mode
- provider ID
- optional provider config ID
- operation type
- tenant/org ID
- workspace ID
- actor/user ID
- prompt/template/version reference
- requested model metadata
- correlation/evidence ID
- key-reference selector

Output summary:

- resolved provider metadata
- allowed operation/mode result
- prompt/version binding
- sanitized evidence context
- usage metadata shell
- server-side-only provider-call handoff

Failure output summary:

- sanitized failure class
- safe UI message category
- evidence fields
- retry posture
- `providerCallAllowed: false`
- no raw secret/provider payload

AP clarification recorded:

- The resolver must never return raw key material to general business logic.
- The future provider-call layer may receive server-side-only secret material only inside the Edge/server boundary.
- Raw key material must never be included in resolver evidence, errors, logs, return payloads, or UI-facing responses.

## Policy Order Summary

Future implementation must fail closed in this order:

1. Validate mode.
2. Authenticate actor.
3. Verify tenant/workspace membership.
4. Resolve provider policy.
5. Verify provider enabled/status.
6. Verify operation allowed.
7. Resolve key reference server-side.
8. Verify prompt/template/version availability.
9. Prepare sanitized audit/evidence context.
10. Only then allow provider call.
