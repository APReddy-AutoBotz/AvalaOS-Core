## Migration Summary

M5.2c adds the first Assess persistence and ownership slice needed for future workspace-aware RLS. The migration is additive and promotes only these Assess tables into the migration chain:

- `assess_processes`
- `assessments`
- `assessment_review_events`

The migration uses `CREATE EXTENSION IF NOT EXISTS pgcrypto`, `CREATE TABLE IF NOT EXISTS`, `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, and guarded `DO $$` blocks for constraints.

## Assess-First And Govern-Deferred Clarification

Although the milestone name includes Govern, this implementation is Assess-first.

- Assess base persistence and ownership inputs are implemented for future RLS readiness.
- Standalone Govern persistence is deferred.
- Standalone Decision Pack persistence is deferred.
- Govern and Decision Pack behavior remains runtime-derived through deterministic scores, Decision Pack rendering, exports, and UI panels until a later AP-approved persistence milestone.

## Current Inventory Summary

- `assess_processes`: previously docs/schema plus adapter only; draft posture had `org_id` and `owner_id`; lacked `workspace_id`, creator/updater, and archive/delete lifecycle.
- `assessments`: previously docs/schema plus adapter only; draft posture had `org_id` and `process_id`; lacked `workspace_id`, creator/updater, reviewer/approver fields, lifecycle, and audit correlation.
- `assessment_review_events`: previously docs/schema plus adapter only; draft posture had `org_id`, `assessment_id`, `process_id`, and `actor_id`; lacked `workspace_id` and correlation.
- Decision Pack and Govern are runtime-derived through scores, export services, and UI panels.
- Evidence refs and assumptions remain embedded JSON on `assessments`.

## Constraints And Indexes Summary

- Helper unique constraints on `(id, org_id)` are added for `assess_processes` and `assessments`.
- Broad required-column and status checks are added as `NOT VALID` where existing/manual rows may exist.
- Same-org composite FK direction is added with `NOT VALID` constraints:
  - `assessments(process_id, org_id)` to `assess_processes(id, org_id)`
  - `assessment_review_events(assessment_id, org_id)` to `assessments(id, org_id)`
  - `assessment_review_events(process_id, org_id)` to `assess_processes(id, org_id)`
  - `(workspace_id, org_id)` to `workspaces(id, org_id)` for nullable workspace ownership
- Actor columns reference `profiles(id)` using `NOT VALID` constraints.
- Lookup indexes cover org, workspace, status, deleted markers, process, assessment, actor, reviewer, approver, and correlation fields.

## FK And Backfill Strategy

- Ownership columns are nullable-first.
- `workspace_id` is nullable and not backfilled in M5.2c.
- `NOT VALID` FKs are used to avoid forcing validation before AP-approved mapping.
- Constraints are not validated in M5.2c.
- No `NOT NULL` ownership enforcement is added for existing/manual rows.
- No seed data is added.
- No backfill is performed.
- No committed hosted organization, workspace, project, bucket, or personal values are introduced.
- Canonical AP demo workspace mapping remains a later AP-approved task.
