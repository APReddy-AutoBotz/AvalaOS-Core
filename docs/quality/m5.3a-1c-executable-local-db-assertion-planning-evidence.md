## Recommended Execution Method

Use a JS runner plus SQL assertion files:

- JS runner proves local target before DB action.
- SQL assertion files hold executable scenario contracts.
- Target must be AP-approved local Supabase/Postgres only.
- Node `pg` may be used only after explicit local-target proof.
- `scripts/applySupabaseMigrations.mjs` is not the default assertion runner.
- `DATABASE_URL` is not silently accepted.

## Auth Simulation Recommendation

Preferred future method:

- Local SQL session claim simulation.
- Generated local UUIDs only.
- Anonymous scenario uses no subject claim.
- Authenticated scenarios use local authenticated role/session plus local subject claim.
- Disabled and non-member actors are modeled through local fixture state.
- Server-only fixture setup remains local-only and not browser-callable.

No real JWTs, token printing, service-role keys, row dumps, or claim-value output should be used.

## Fixture Setup Strategy

Future fixtures should include:

- Two local organizations.
- Two local workspaces per organization.
- Active member.
- Disabled organization member.
- Disabled workspace member.
- Non-member.
- Organization-scoped roles.
- Workspace-scoped roles.
- Same-organization and cross-organization rows.
- Same-workspace and cross-workspace rows.
- Deleted-state variants where useful.

Fixtures must use generated local data only and must not include personal IDs, hosted URLs, real project IDs, provider keys, service-role values, tokens, bucket names, `.env` values, or row payload output.
