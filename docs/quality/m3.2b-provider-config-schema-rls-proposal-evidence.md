## Proposed Schema / RLS Summary

The planning doc proposes future candidate objects only:

- `ai_provider_configs`
- `ai_provider_key_refs`
- `ai_workspace_provider_policies`
- `ai_provider_audit_events`
- `prompt_templates` / `prompt_versions`
- `ai_usage_events`

For each candidate object, the proposal records purpose, key fields, secret-handling rule, tenant/workspace relationship, expected writer, expected reader, RLS posture, audit/evidence relevance, and migration dependency.

RLS proposal summary:

- Browser/client code must never read raw provider secrets, resolved key values, secret manager payloads, service-role-only resolver fields, or raw key-reference internals.
- Authenticated users may read only safe tenant/workspace metadata when active membership allows it.
- Admin/control-plane writes should use approved server-side paths only.
- Edge/service-role access should resolve provider config, key references, policy, prompt/version, and evidence only inside the approved resolver workflow.
- Tenant/workspace ownership must be explicit; ambiguous ownership fails closed.
- Missing config, missing policy, denied policy, disabled status, invalid key reference, missing prompt/version, unavailable audit path, unclear RLS ownership, or scanner classification failure must fail closed.

## Secret / Key-Reference Summary

- Raw provider secrets never live in browser env, browser state, logs, evidence, product copy, ordinary application DB rows, or responses.
- Raw provider secrets must not be stored even encrypted in ordinary application DB rows unless AP separately approves a dedicated secret-storage model.
- Only server-side key references may be stored in ordinary application DB rows.
- Key references point to server-side secret material or secret manager identifiers.
- Missing, disabled, unauthorized, malformed, unsupported, expired, or invalid key references fail closed before provider calls.
