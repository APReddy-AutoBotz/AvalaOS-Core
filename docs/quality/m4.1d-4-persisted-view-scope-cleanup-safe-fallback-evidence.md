## Root Cause

`App.tsx` persisted `currentView` and `currentScope` through the generic persisted-state hook. The hook parses stored JSON and returns it as typed state, but restored browser-storage values can be stale or malformed at runtime.

Before M4.1d-4, the central guard could correct denied view/scope combinations after auth/org loading, but it did not first sanitize malformed persisted shapes. That left risk from unknown view strings, unknown scope types, malformed project/team scopes, and stale extra fields on simple scopes.

## Persisted State Risks Found

- Unknown or stale view values could be restored from persisted state.
- Unknown scope types could be restored and then reach App render paths.
- Project scope values could be missing `id` or `name`.
- Team scope values could be missing `id` or `name`.
- My Work and Organization scopes could carry stale extra fields.
- Valid-looking persisted view/scope pairs could still be denied by central guard after user, organization, module, permission, or scope changes.

## Sanitizer Summary

Added `services/viewStatePersistence.ts` with pure helpers:

- `isValidView`
- `normalizePersistedView`
- `isValidScope`
- `normalizePersistedScope`
- `areScopesEqual`
- `resolvePersistedViewScopeState`

Sanitizer behavior:

- Unknown view values normalize to `DASHBOARD`.
- Unknown scope types normalize to My Work scope.
- Malformed project/team scopes normalize to My Work scope.
- My Work and Organization scopes normalize to clean simple shapes.
- Valid project/team scopes retain `type`, `id`, and `name`.
- Input objects are not mutated.
- The final normalized pair goes through `resolveViewAccess` before it remains active.

## App Integration Summary

`App.tsx` now keeps raw persisted `VIEW` and `SCOPE` values at the storage boundary and derives sanitized runtime values for `currentView` and `currentScope`.

After auth/org loading completes, App calls `resolvePersistedViewScopeState` and writes back normalized or guard-fallback values only when needed.

The integration does not clear all browser storage and does not touch unrelated persisted keys.

## Fallback Behavior Summary

- Unknown/stale view: normalize to `DASHBOARD`, then resolve through central guard.
- Unknown or malformed scope: normalize to My Work scope, then resolve through central guard.
- Clean but guard-denied persisted pair: apply the resolver fallback view/scope.
- Loading auth/org state: wait and do not mutate from loading state.
- Organization-scope Admin Workspace: preserve existing App behavior pending AP decisions.
