## Design Confirmations

- Auth simulation recommendation: prefer local SQL session claim simulation with generated local UUIDs, no real JWTs, no token printing, no service-role keys, no browser-callable setup, and no row dumps.
- Fixture setup strategy: use two local orgs, two local workspaces per org, active and disabled members, non-member, scoped roles, same-org/cross-org rows, same-workspace/cross-workspace rows, and deleted-state variants. Use generated local data only.
- Local-target proof strategy: allow only localhost, loopback, Docker-local network, or AP-approved local token. Reject hosted URLs, remote hosts, silent DB URL acceptance, ignored secret-file fallback, and unknown args. Never print target values.
- Secret-safe output strategy: output only scenario counts, pass/fail/blocked totals, tenant-isolation yes/no, sanitized classifications, and target proof status without target value.
- SQL assertion file strategy: preserve the M5.3a-1b contract and add a separate future assertion SQL file in M5.3a-1d.
- Runner implementation strategy: update the existing harness script only in M5.3a-1d, keep it fail-closed until executable assertions exist, and return zero only when AP-approved local assertions actually run and pass.
- `supabase/config.toml` decision: optional and AP-gated; if approved later, it must be generic, portable, local-only, safe to commit, and contain no hosted or personal values.
- `applySupabaseMigrations` posture: inspect-only for this closure; not the default future assertion runner.
- Future implementation file strategy: update the harness script, add the assertion SQL file, add M5.3a-1d implementation evidence, and only add optional config/package changes with AP approval.
