# M5.3a-2w Controlled Post-Remediation Classifier Evidence Run Approval Gate

## Purpose

M5.3a-2w is a docs-only approval gate for a future controlled post-remediation classifier evidence run.

This milestone does not run the classifier, does not run startup, does not prove DB availability, schema availability, artifact SELECT isolation, tenant isolation, hosted readiness, or production readiness. It only defines what AP must approve before any future run of classifier classification mode after M5.3a-2v hardening.

## Closed Prerequisite

M5.3a-2v Local Startup Timeout With Oversized Output Remediation Implementation is closed and tagged.

- PR: `#123`
- Accepted/head commit: `6947a0313b2f048b96527d59a3413283483375e2`
- Merge commit: `a9499713082b4e71354e132bf2bdfbe7a3f5a3eb`
- Post-merge verification commit / tag target: `0f31ed7ac989005e2148628f46cb01bbe19ef7c3`
- Tag: `avalaos-core-m5.3a-2v-local-startup-timeout-with-oversized-output-remediation-implementation`

M5.3a-2v established:

- The classifier now has output-budget hardening.
- Timeout plus oversized output can be emitted only as the low-confidence redacted signal `local-startup-timeout-with-oversized-output`.
- The timeout/oversized classifier logic accepts either output-size bucket oversized or line-count bucket oversized while preserving all safety gates.
- Non-timeout over-budget output remains `unknown-local-startup-failure`.
- Unsafe output or cleanup failure fails closed to `classification-output-safety-failed`.
- No root-cause inference was added.

M5.3a-2w does not:

- Run classifier classification mode.
- Attempt local startup directly or indirectly.
- Run Supabase stack commands.
- Run Docker commands or Docker listings.
- Run DB commands or make DB connections.
- Run schema readiness commands.
- Run RLS harnesses.
- Run artifact assertion commands.
- Run migration or bootstrap commands.
- Run remote, hosted, or personal Supabase validation.
- Run provider calls.
- Prove local startup success.
- Prove local DB availability.
- Prove schema readiness.
- Verify artifact SELECT isolation.
- Newly verify tenant isolation.
- Claim hosted readiness.
- Claim production readiness.
- Infer root cause.

## Required AP Approvals Before Future Evidence Run

No future controlled evidence run may occur until AP explicitly approves all of the following for that future milestone:

- Exact milestone name for the run.
- Exact command label to be recorded in evidence.
- Whether the run may execute classifier classification mode.
- Whether the run may indirectly attempt startup through classifier behavior.
- Expected output categories.
- Maximum evidence content allowed.
- Cleanup expectations.
- Stop conditions.
- Final proof boundary language.

Approval must be specific to the future evidence run. This approval gate does not grant execution permission.

## Future Allowed Sanitized Evidence Categories

Future evidence may include only sanitized, redacted categories approved by AP:

- Command label only, not the literal command string.
- Run count.
- `startupAttemptStatus`.
- Classification.
- Confidence.
- Output safety result.
- Scratch cleanup result.
- Root cause inferred: `no`.
- Local DB availability outcome.
- Schema availability outcome.
- Artifact SELECT isolation outcome.
- Tenant isolation outcome.

- `lineCountBucket`
- `timeoutFlag`

- `noSecretsConfirmation`
- `noLocalPathConfirmation`

## Required Future Stop Conditions

A future controlled evidence run must stop and fail closed if:

- Classifier output safety fails.
- Scratch cleanup fails.
- Raw output would need to be inspected.
- Evidence would need local path or environment details.
- Evidence would expose target, hosted, DB, provider, token, path, payload, auth, claim, container, or image values.
- The result would imply root cause from redacted fields.
- The result would imply local DB availability.
- The result would imply schema readiness.
- The result would imply artifact SELECT isolation.
- The result would imply tenant isolation.
- The result would imply hosted readiness.
- The result would imply production readiness.

## Required Proof Boundary Language

Future evidence must preserve this boundary language unless AP explicitly approves stricter language:

- Local DB availability remains unresolved.
- Schema availability is not proven.
- Artifact SELECT isolation is not verified.
- Tenant isolation is not newly verified.
- Hosted readiness is not proven.
- Production readiness is not proven.
- Root cause is not inferred.
- The future controlled run, if later approved, can prove only the sanitized classifier result after M5.3a-2v hardening.
- The future controlled run cannot prove startup success, DB availability, schema readiness, artifact isolation, tenant isolation, hosted readiness, or production readiness.

## Future Evidence Run Shape

A future milestone may propose `M5.3a-2x Controlled Post-Remediation Classifier Evidence Run` only after AP approves this gate and explicitly authorizes the run scope.

The future run must:

- Preserve the existing classifier interface unless separately approved.
- Record a command label only, not a literal command string.
- Capture only approved sanitized evidence categories.
- Stop on any approved stop condition.
- Keep `local-startup-timeout-with-oversized-output` as a low-confidence redacted signal.
- Keep `unknown-local-startup-failure` as the safe fallback.
- Prevent low-confidence signal classifications from becoming root-cause claims.
- Record sanitized outcomes only.
