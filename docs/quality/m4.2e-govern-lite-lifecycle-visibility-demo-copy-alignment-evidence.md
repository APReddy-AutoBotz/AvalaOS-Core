## Current Lifecycle Gap

Before M4.2e, the visible top lifecycle and buyer-demo entry copy presented:

`Avala Assess -> Avala Studio -> Avala Delivery Lite -> Avala Monitor`

Govern Lite existed in Decision Pack, Process Detail, Assess-to-Studio source context, and Delivery Pack context, but it was not visible as its own lifecycle moment. That created a buyer-demo risk: governance could look absent even though the control context existed.

## Implemented Lifecycle And Copy Changes

M4.2e updates the visible buyer-demo lifecycle to:

`Avala Assess -> Avala Govern Lite -> Avala Studio -> Avala Delivery Lite -> Avala Monitor`

Implementation details:

- Added `components/shared/moduleJourneyModel.ts` as a pure lifecycle copy/model helper.
- `ModuleJourney` now uses the helper and inserts Avala Govern Lite between Assess and Studio when both Assess and Studio are visible in the guarded journey.
- Govern Lite is rendered as an informational lifecycle step, not a new route target.
- Login buyer-demo value pillars now include Avala Govern Lite between Avala Assess and Avala Studio.
- Login demo copy now references the role-aware flow across Assess, Govern Lite, Studio, Delivery Lite, and Monitor.
- Process Detail Govern Lite card copy now describes Govern Lite as the control-card step between Assess and Studio.
- Process Detail handoff copy now references reviewed Decision Pack and Govern Lite context for Studio artifacts.
- The M4.2d runbook now expects Govern Lite in the lifecycle strip and no longer describes it as missing from the visible lifecycle.

## Govern Lite Integration Classification

Govern Lite was added as a visual-only lifecycle step backed by existing Assess/Decision Pack context.

It is not metadata-backed as a product module. It is not a new route. It is not a Sidebar entry. It does not call `onNavigate`. It points the buyer narrative to the existing Decision Pack and Govern Lite card context in Process Detail.
