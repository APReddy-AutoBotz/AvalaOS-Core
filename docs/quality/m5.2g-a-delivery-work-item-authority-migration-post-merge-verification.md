## Migration Summary

M5.2g-a creates or promotes only `delivery_work_items` as a narrow AvalaOS Delivery work-item authority table.

The table is scoped to organization, nullable-first workspace, and project authority. It is intended to support future workspace-aware RLS and Delivery ownership tests, not to replace external project management systems or implement runtime work execution.

## Table Created Or Promoted

- `delivery_work_items`

No epics, sprints, legacy tasks, task comments, task activity, timesheets, Delivery Pack, Monitor, export, lineage, handoff, Studio/Docs, provider, seed, or backfill tables were created or promoted.

## Ownership And Compatibility Fields

The migration includes ownership and compatibility inputs for:

- tenant/workspace/project scope: `org_id`, `workspace_id`, `project_id`
- compatibility bridge: `app_id`
- source linkage: `document_generation_id`, `source_process_id`, `source_assessment_id`
- actors: `created_by`, `updated_by`, `owner_id`, `assigned_to`, `reporter_id`
- work-item content: `title`, `description`
- lifecycle/classification: `status`, `priority`, `type`, `archived_at`, `deleted_at`
- compatibility payloads: `source_lineage`, `metadata`
- audit linkage: `audit_correlation_id`
- timestamps: `created_at`, `updated_at`

The migration does not force runtime adapters to use these columns yet.

## app_id Hardening Confirmation

`app_id` is nullable and remains a compatibility bridge only.

Confirmed posture:

- `app_id` is not tenancy authority.
- `app_id` is not globally unique.
- `app_id` is not a Supabase project identifier.
- `app_id` is not a replacement for `id`.
- The partial uniqueness guard is scoped to `(org_id, project_id, app_id)` for active/non-deleted rows.

## metadata And source_lineage Hardening

`metadata` and `source_lineage` are compatibility payloads only.

They are not durable evidence, approval, export/storage, review, cross-tenant reference, or standalone lineage authority. Migration comments prohibit secrets, provider keys, service-role values, raw private tokens, personal hosted-project values, and unsupported compliance claims in these payloads.

Extra runtime fields such as dependencies, subtasks, user stories, ordering, and extra assignees remain non-authority compatibility payloads unless AP approves normalized persistence later.
