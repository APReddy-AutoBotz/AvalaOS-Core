## Guard Helper Summary

M4.1d-1 added a pure central View/RBAC helper in `services/viewAccessGuard.ts`.

The helper includes:

- `VIEW_ACCESS_METADATA`
- `resolveViewAccess`
- `hasViewPermission`
- typed access reasons
- typed guard severities
- explicit fallback view and optional fallback scope output

The helper is pure and deterministic. It does not mutate app state, persist app state, render UI, call providers, fetch remote data, or change scoring behavior.

## Guard Precedence Summary

The accepted helper implements the M4.1d guard precedence:

1. Auth loading returns `auth_loading` with `wait`; it does not redirect.
2. Missing user after auth loading returns `unauthenticated`.
3. Missing organization returns `no_organization`.
4. Empty enabled-module configuration returns `setup_required`.
5. Unknown or stale persisted view returns `stale_persisted_view`.
6. Deferred or decision-pending view returns `deferred_view` or `admin_decision_pending`.
7. Disabled module returns `disabled_module`.
8. Invalid scope returns `invalid_scope`.
9. Missing permission returns `missing_permission`.
10. Passing checks returns `allowed`.

## Permission Semantics Summary

M4.1d-1 preserves the first implementation permission semantics from Sidebar:

- Admin role bypasses listed per-view permissions.
- Non-admin users are allowed when they have any one listed permission.
- Views with no required permissions still pass through auth, organization, module, scope, and deferred-view checks.

The helper does not introduce all-permission enforcement.
