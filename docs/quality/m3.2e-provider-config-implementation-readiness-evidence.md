## Consolidated Architecture Summary

M3.2e reconciles M3.2 through M3.2d into a single planning/readiness gate.

Consolidated direction:

- provider registry remains server-side
- provider config records contain safe metadata only
- key references point to server-side secret material or secret manager identifiers
- workspace/tenant provider policy gates provider use by mode and operation
- Edge resolver contract checks mode, auth, membership, policy, provider status, operation, key reference, prompt/version, and evidence before provider calls
- audit/evidence event contract records safe proof of gate checks
- prompt/version binding must exist before future provider calls
- usage metadata must come from server-side provider responses
- M3.2c sanitized failure classes are the future baseline
- M3.1c static scan must classify any new implementation paths exactly

## Recommended First Implementation PR

Recommended option now: no implementation yet; more AP decision work is needed.

After AP resolves the readiness gaps, the safest first real implementation PR should be a schema/RLS migration PR with no runtime provider calls.

Rationale:

- A schema/RLS-only PR is smaller and more reversible than runtime provider changes.
- It can be tested without enabling new AI behavior.
- It keeps `supabase/functions/_shared/ai.ts` unchanged until a later resolver PR.
- It can establish reference-only fields and RLS posture before key-reference resolution exists.
- It gives future scanner evidence a stable backend contract before runtime files change.

## Secret-Safety Gate

The readiness gate records these rules:

- raw provider secrets must never be stored in ordinary application DB rows, even encrypted, unless AP separately approves a dedicated secret-storage model
- raw provider secrets must never be returned to browser/client
- raw provider secrets must never be logged
- raw provider secrets must never appear in evidence
- key references must point to server-side secret material or secret manager identifiers
- provider-call layer may receive secret material only inside Edge/server boundary
- no secret material may cross into general business logic

## Evidence-Safety Gate

The readiness gate records these rules:

- evidence proves gate checks
- evidence does not store sensitive payloads
- evidence records result, reason class, safe identifiers, and timestamps
- evidence must not store raw secret, prompt body, response body, raw provider error, hidden chain/payload, `localStorage` values, `userApiKey`, or unredacted stack traces
