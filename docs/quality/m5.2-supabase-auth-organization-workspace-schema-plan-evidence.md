## Current Auth, Org, And Workspace Inventory

- Supabase Auth exists through the browser Supabase client and `authAdapter` when public config is present.
- Local buyer-demo fallback stores the current user in browser storage and uses demo login profiles.
- Supabase user mapping currently overlays demo persona data by email and does not load canonical DB-backed profile, membership, role, and permission state.
- Organization loading uses `organizations`, `organization_members`, and `roles` when Supabase is configured.
- Demo mode exposes one canonical organization, `Avala Demo Enterprise`, with members derived from demo users.
- Workspace is not a first-class database authority. Current `ScopeType` values are navigation context, and `View.WORKSPACE` is a Docs/Admin surface.
- Current schema contracts are mostly `org_id` scoped. Provider governance has future workspace-named surfaces, but canonical workspace authority remains deferred.
- Browser storage remains demo-only persistence for user, scope, view, organization, audit-adjacent logs, handoff ledger, Assess, Docs, Delivery, and transitional settings.

## Proposed Entity Model Summary

- Supabase `auth.users` is the hosted pilot identity authority.
- `profiles` or `user_profiles` stores app user metadata only.
- `organizations` remains the tenant root.
- `organization_members` or `organization_memberships` becomes active org membership authority.
- `workspaces` becomes the organization child and primary hosted-pilot work boundary.
- `workspace_memberships` becomes workspace access authority.
- Roles require explicit AP decision on global, org-scoped, workspace-scoped, or mixed scope.
- Permissions need AP decision on JSONB role storage versus normalized `permissions` / `role_permissions`.
- Future module records should carry `org_id` and, for most work artifacts, `workspace_id`.

## Proposed Role And Workspace Model Summary

- Display persona, Supabase identity, app profile, organization membership, workspace membership, role assignment, and permission set are separate concepts.
- Platform Admin maps to org admin/security and user-management authority.
- AP Process Owner maps to workflow owner/reviewer authority for Assess and Govern workflows.
- Process Analyst maps to workspace contributor authority.
- Delivery Lead maps to delivery workspace lead authority.
- Control Reviewer maps to control and evidence reviewer authority.
- Automation Contributor maps to implementation contributor authority without provider-secret or platform security authority.
- Buyer Viewer maps to read-only viewer authority.
- A user may belong to multiple organizations if AP approves the switching behavior.
- An organization may have multiple workspaces.
- Workspace should become the primary data boundary for Assess, Govern, Studio, Delivery, and Monitor hosted-pilot records.
