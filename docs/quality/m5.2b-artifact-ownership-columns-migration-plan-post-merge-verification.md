## Artifact Ownership Plan Summary

M5.2b correctly keeps the implementation as planning first because the ownership blast radius spans multiple domains and several artifact tables are still draft or derived. The recommended ownership vocabulary is:

- Tenant ownership: `org_id`, nullable-first `workspace_id`.
- Change actors: `created_by`, `updated_by`.
- Business ownership: `owner_id`.
- Governance actors: `reviewer_id`, `approver_id`.
- Cross-source traceability where needed: `source_org_id`, `source_workspace_id`.
- Lifecycle: `status`, `deleted_at`, `archived_at`.
- Audit linkage: `correlation_id` or `audit_correlation_id`.

The plan does not force every column onto every table. It recommends domain-specific ownership columns and the first implementation slice in Assess/Govern.

## FK And Backfill Summary

- Add nullable ownership columns first.
- Prefer `NOT VALID` foreign keys for broad artifact tables where existing data or draft authority may not yet be backfilled.
- Validate constraints only after AP-approved mapping and data review.
- Add `NOT NULL` only after backfill is verified.
- Avoid destructive updates, seed data, personal hosted IDs, committed environment-specific IDs, and inferred workspace mappings.
- Avoid foreign keys on draft/demo-only tables until those tables become migration-chain authority.

## RLS Readiness Summary

The M5.2b records prepare M5.3a by defining how artifact ownership columns will support:

- Workspace-scoped reads by active workspace members.
- Cross-org denial through `org_id`.
- Cross-workspace denial through `workspace_id`.
- Buyer Viewer read-only policies.
- Reviewer and approver gates using explicit actor and role/permission inputs.
- Export, evidence, lineage, and handoff tenant boundaries.
- Provider metadata boundaries without exposing secret values.
- Audit append-only behavior with actor and correlation context.

M5.2b does not implement RLS, does not prove tenant isolation, and does not claim hosted pilot completion.
