## Root Cause

M4.1d-2 guarded App-shell view/scope transitions, but Sidebar and ModuleJourney still had independent navigation rules:

- Sidebar duplicated module, scope, and permission metadata in local navigation arrays.
- Sidebar evaluated local permission access instead of the central guard result.
- ModuleJourney displayed enabled modules and navigated directly to module home views without checking current scope or permissions.
- App transitions were centrally guarded, but navigation display and clickability could still drift from the same contract.

## Sidebar Before/After Summary

Before:

- Sidebar nav items carried local module keys.
- Sidebar nav items carried local scope arrays.
- Sidebar nav items carried local permission arrays.
- Sidebar had local access logic for Admin bypass and permission matching.

After:

- Sidebar keeps presentation metadata only: view, icon, label, and section grouping.
- Sidebar calls `resolveViewAccess` for each nav item using the current user, loading state, organization, enabled modules, target view, and current scope.
- Disabled module, missing permission, deferred view, loading, setup, stale, and decision-pending results are hidden.
- Invalid-scope items remain visible but disabled with the existing available-scope tooltip pattern.
- Admin bypass OR any one listed permission remains enforced by the central guard helper.
- The Avala Admin organization-scope button is preserved as a deferred AP decision path.

## ModuleJourney Before/After Summary

Before:

- ModuleJourney listed enabled modules directly.
- ModuleJourney marked active module through a separate view-to-module map.
- ModuleJourney navigated directly to `MODULE_HOME_VIEW` for each module.

After:

- ModuleJourney still uses enabled modules for the operating lifecycle layout.
- ModuleJourney resolves each module home through `resolveViewAccess`.
- Allowed module homes remain clickable.
- Invalid-scope module homes remain visible but disabled with a title that names the valid scope.
- Missing-permission and other denied module homes are hidden.
- Active module calculation reads from `VIEW_ACCESS_METADATA`.
- ModuleJourney no longer navigates blindly to module home views.

## Central Guard Alignment Summary

Sidebar and ModuleJourney now use:

- `resolveViewAccess`
- `VIEW_ACCESS_METADATA`
- the existing central guard permission semantics
- current user/org/module/scope inputs

This keeps navigation decisions aligned with the same guard contract App uses for view/scope application.
