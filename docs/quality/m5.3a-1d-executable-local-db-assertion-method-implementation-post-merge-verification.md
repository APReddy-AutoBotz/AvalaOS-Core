## Executable Assertion Method Summary

The accepted implementation updates the existing M5.3a-1 local harness so it can load an executable assertion SQL file only after explicit local-target proof. The harness remains fail-closed by default and reports sanitized counts-only output.

The implementation preserves the M5.3a-1b scenario contract and adds `docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql` as the executable assertion file. The current baseline still has no RLS helpers or SELECT policies, so the harness verifies plumbing and safe failure behavior only.

## Local-Target Proof Behavior

Confirmed behavior:

- Requires explicit local-target proof before DB action.
- Rejects hosted URLs, remote hosts, silent environment DB targets, ignored secret-file fallback, and unknown target args.
- Never prints target values.
- Never prints DB URLs, `.env` values, service-role values, provider keys, JWT/private tokens, hosted URLs, bucket names, personal IDs, claim values, auth headers, or row payloads.
- Outputs only scenario counts, pass/fail/blocked totals, tenant isolation verified yes/no, sanitized classifications, and local target proof status without target value.
- Returns non-zero when target proof is missing, target is rejected, local DB connection is unavailable, assertions cannot run, assertions fail, cleanup fails, or output sanitization would fail.
- Returns zero only when AP-approved executable local DB assertions run and pass.

## Auth Simulation Behavior

The assertion file includes local session-claim simulation checks using generated local UUID values only. It checks anonymous-claim clearing and local subject-claim setup without real JWTs, token printing, service keys, browser-callable setup, or row payload output.

Policy-level role switching and final tenant access assertions remain blocked until AP-approved helpers and SELECT policies exist.
