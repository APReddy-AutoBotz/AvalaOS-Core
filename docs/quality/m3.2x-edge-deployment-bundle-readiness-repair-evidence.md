## Root Cause Summary

M3.2w deploy attempts reached remote Supabase bundling and failed with a sanitized module-resolution diagnostic for `_shared/providerResolver` from `_shared/providerResolverIntegration.ts`.

M3.2x import-graph inspection found extensionless sibling imports in deployed shared resolver modules. The specific chain included:

- `providerResolverIntegration.ts` importing `./providerResolver`.
- `providerResolverIntegration.ts` importing other resolver shared modules without `.ts`.
- `providerResolver.ts`, resolver DB/audit modules, and the secret adapter importing sibling shared modules without `.ts`.

This is compatible with the local Node-oriented TypeScript test emit, but it is not compatible with Deno/Supabase Edge remote bundling, which requires explicit local file extensions.

## Minimal Fix Summary

The runtime repair adds explicit `.ts` extensions to local imports in the deployed shared resolver chain only.

The test-only repair adds `--rewriteRelativeImportExtensions` to the existing resolver-related TypeScript emit commands so local CommonJS test output rewrites `.ts` imports to emitted `.js` files.

No resolver decision formulas, provider policies, secret lookup rules, audit sanitization rules, provider calls, Edge entrypoint request/response behavior, schema, UI, Health, scoring, browser fallback, dependencies, usage behavior, or cost behavior were changed.
