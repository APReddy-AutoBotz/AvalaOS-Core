## Current-State Findings

- `App.tsx` persists `currentScope` and `currentView`.
- `App.tsx` applies current user defaults for scope/view after user changes.
- `isViewEnabled` and `isModuleEnabled` enforce module enablement only.
- `Sidebar.tsx` separately applies module, permission, Admin role, and scope checks.
- Sidebar permissions currently mean Admin role bypass OR any one listed permission.
- `App.tsx` contains multiple direct `setCurrentView` flows outside Sidebar checks.
- `ModuleJourney` navigates by enabled module home only.
- `REPORTS` and `TEAMS` have local fallback behavior.
- `WORKSPACE` is overloaded across Docs artifact workspace and organization/Admin-adjacent navigation.
- Organization setup prevents or normalizes empty module selections, but this is not a central guard.

## Route/RBAC Risks Found

| Risk | Evidence | Impact |
| --- | --- | --- |
| Sidebar visibility is not central authorization. | Sidebar filters nav items, but App handlers can set views directly. | A hidden nav item is not enough to prove access control. |
| Module enablement is not permission enforcement. | `isViewEnabled` checks only module mapping. | A user can be sent to an enabled module without validating view permission. |
| Persisted state can be stale. | `StorageKeys.SCOPE` and `StorageKeys.VIEW` restore prior app state. | User/org/module changes can revive outdated view/scope combinations. |
| Fallback logic is local and incomplete. | App redirects Reports, Teams, and disabled modules through separate conditions. | Future behavior can diverge by entry point. |
| Workspace/Admin semantics are overloaded. | Organization scope renders setup/Admin, while `WORKSPACE` is also Docs artifact workspace. | Product and route semantics need AP decision before enforcement. |

## Non-Sidebar Entry Point Risks

| Entry point | Risk |
| --- | --- |
| `handlePrimaryAction` | Chooses Docs Forge, Process Catalog, Boards, or Dashboard from enabled modules without central permission/scope resolution. |
| Project selector / Docs Forge entry | Sets project scope and opens Docs Forge outside Sidebar filtering. |
| Assess-to-Studio handoff | Opens Docs Forge from Assess handoff outside Sidebar filtering. |
| `ModuleJourney` | Navigates to module home views from module enablement only. |
| Persisted `currentView` / `currentScope` restore | Can restore stale state before central validation exists. |
| Direct `setCurrentView` handlers | Many App flows bypass shared route/view metadata. |
| Fallback effects in `App.tsx` | Redirect selected views through local exceptions instead of a single guard reason. |

## Proposed Guard Summary

Future implementation should add a pure `resolveViewAccess` helper that receives user, auth loading state, organization, enabled modules, target view, and scope. It should return allowed status, fallback view/scope, reason, module, required permissions, required scope, and guard severity.

Guard precedence must separate auth loading from unauthenticated:

1. Auth loading waits and does not redirect.
2. Unauthenticated denies to auth flow.
3. No organization/onboarding required.
4. Deferred or decision views.
5. Disabled module.
6. Invalid scope.
7. Missing permission.
8. Allowed.

The first implementation default must preserve current Sidebar permission semantics: Admin role bypass OR any one listed permission grants access.
