## Schema/RLS Migration Summary

The migration adds provider governance schema/RLS for:

1. `ai_provider_key_refs`
2. `ai_provider_configs`
3. `ai_workspace_provider_policies`
4. `ai_provider_audit_events`

The migration uses `org_id` as the enforceable tenant boundary. `ai_workspace_provider_policies.workspace_id` remains nullable without a strict FK because no AP-approved workspace authority exists yet. `ai_provider_key_refs` has no FK back to provider configs. `ai_provider_configs.key_ref_id` is nullable and references `ai_provider_key_refs(id)`.

## Secret-Safety Confirmation

- `secret_ref` is documented as a service-only reference identifier.
- `secret_ref` must not contain raw provider keys, encrypted provider keys, auth headers, browser-readable secret values, or secret-manager payload values.
- `ai_provider_configs` is documented as prohibited from storing raw secrets, encrypted secrets, auth headers, browser-readable secret values, or secret-manager payload values.
- Browser/client code does not resolve provider keys in M3.2h.
- Browser/client code does not directly write sensitive provider/key/audit objects in M3.2h.

## RLS Confirmation

- `ai_provider_key_refs`: RLS enabled; no member read policy; no broad authenticated write policy.
- `ai_provider_configs`: RLS enabled; active org members can SELECT non-deleted metadata; no browser/client write policy.
- `ai_workspace_provider_policies`: RLS enabled; active org members can SELECT non-deleted metadata; no browser/client write policy.
- `ai_provider_audit_events`: RLS enabled; member read deferred; no browser/client write policy.
