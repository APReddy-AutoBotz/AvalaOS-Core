## Migration Summary

M5.2f adds the first Delivery/project authority schema slice by creating or promoting only the `projects` table in the applied migration chain.

The migration is additive. It does not create work-item, Studio/Docs, Monitor, Delivery Pack, export, lineage, handoff, epics, sprints, task, comment, activity, or timesheet authority.

The migration uses `CREATE EXTENSION IF NOT EXISTS pgcrypto`, `CREATE TABLE IF NOT EXISTS`, `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, guarded constraint blocks, and staged `NOT VALID` constraints where existing/manual non-production rows may exist.

## Table Created Or Promoted

- `projects`

`projects` represents an AvalaOS governed Delivery project under organization and workspace ownership. Avala Portfolio remains a view over `projects` for now. The table is not a generic project-management replacement, not live runtime execution authority, and not bot execution authority.

## App ID Semantics

`app_id` is nullable and exists only as a UI/demo bridge for current project identifiers. It is not tenancy authority, not a Supabase project identifier, and not globally unique.

The migration creates a partial unique index on `(org_id, app_id)` only where `app_id IS NOT NULL AND deleted_at IS NULL`.

## Constraints And Indexes Summary

The migration adds a composite helper key on `(id, org_id)` for future same-organization references.

Required semantics are staged through `NOT VALID` checks for `org_id`, non-empty `name`, and `status`.

Lifecycle and status checks were added because inspected runtime and draft values align with the safe sets:

- `lifecycle_stage`: `Planning`, `Analysis & Design`, `Development`, `Testing`, `Deployment`, `Maintenance`
- `health_status`: `On Track`, `At Risk`, `Off Track`
- `status`: `active`, `paused`, `archived`, `completed`, `cancelled`

Indexes were added for organization/status/deleted lookup, workspace/status/deleted lookup, owner lookup, delivery lead lookup, source process lookup, source assessment lookup, and audit correlation lookup.

## FK And Backfill Strategy

The migration adds staged `NOT VALID` foreign keys:

- `org_id` to `organizations(id)` with default delete behavior for tenant artifact history
- `(workspace_id, org_id)` to `workspaces(id, org_id)`
- `owner_id`, `delivery_lead_id`, `created_by`, and `updated_by` to `profiles(id)` with `ON DELETE SET NULL`
- `(source_process_id, org_id)` to `assess_processes(id, org_id)`
- `(source_assessment_id, org_id)` to `assessments(id, org_id)`

Tenant artifact ownership and source references intentionally avoid broad hard-delete cascade. Organization, workspace, source process, and source assessment references use staged default delete behavior so history cannot be removed through a parent hard delete without a later explicit data-retention decision. Actor references use `ON DELETE SET NULL`.

No constraints are validated in M5.2f. No backfill is performed. No seed data is added. No committed hosted IDs, personal IDs, project URLs, bucket names, or environment-specific values are introduced.

`workspace_id`, actor fields, and source references remain nullable until AP approves canonical workspace/project mapping and backfill.

## RLS Enabled With No Policies

RLS is enabled on `projects`.

No RLS policies are created in M5.2f. This is fail-closed readiness only. Tenant isolation is not implemented or proven in this milestone.

Configured Supabase Delivery/Docs runtime paths that query `projects` may fail closed until M5.3a policies and separately approved runtime adapter work exist.

## Studio/Docs Dependency Unlock

Project authority gives later Studio/Docs ownership work a stable applied migration-chain target for `document_generations.project_id` planning. It does not create `document_generations`, Studio/Docs ownership columns, or document persistence authority.
