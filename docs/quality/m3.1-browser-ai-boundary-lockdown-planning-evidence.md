## Risks Found

- `services/aiOrchestrator.ts` still falls back to browser provider execution when Edge AI is not enabled.
- `services/geminiService.ts` can create browser provider classes and reads Vite-exposed provider key variables.
- `services/geminiProvider.ts` sends browser requests directly to Gemini with the provider key in the request URL.
- `services/groqProvider.ts` sends browser requests directly to Groq with the provider key in the authorization header.
- `App.tsx` keeps `userApiKey` state and passes it through multiple UI surfaces even though the legacy localStorage key is cleared on mount.
- `services/storage.ts` still defines `StorageKeys.API_KEY`.
- `setup-api-key.js` remains an API-key/localStorage setup helper and must be removed, archived, or clearly retired in future M3.1 implementation.
- `components/docs/DocsForgeView.tsx` reads uploaded files in the browser and passes file text into the orchestrator.
- `components/shared/LandingPage.tsx` accepts PDF/DOCX uploads while current browser extraction is not production-ready for binary formats.
- `components/shared/LandingPage.tsx` exposes provider selection in the current Avala AI Controls area; future pilot/production UI must avoid browser raw-key entry.
- `AiInsightsWidget` and `SprintPlanningView` receive AI provider/key props, although their current AI behavior is disabled in pilot-safe client paths.
- No explicit `local-demo | internal-dev | pilot | production` client mode contract exists yet.

## AP Amendments Applied

- Client-side mode gating is documented as only a product/UI/orchestrator guard.
- Server-side Edge enforcement remains mandatory for pilot and production.
- Allowed modes are exactly `local-demo`, `internal-dev`, `pilot`, and `production`.
- Local development may default to `local-demo` only when Vite/dev context is explicit.
- Deployed/non-dev builds must fail closed or require explicit mode.
- Invalid mode must fail closed.
- Pilot and production must use Edge AI only.
- This planning record does not change `package.json`.
- Future M3.1 implementation may request a script-only `test:ai-boundary` package change.
- No dependency or package-lock changes are planned.
- Future scan evidence must list every remaining AI-boundary scan hit with file path, line or pattern, allowed reason, and whether it is allowed only in `local-demo`, `internal-dev`, test, or control docs.
- PDF/DOCX/browser extraction mismatch is inventoried in M3.1 and remediated in M3.4 Source Safety unless AP separately approves earlier work.
- Future M3.1 implementation must remove, archive, or clearly retire `setup-api-key.js`.
- No active localStorage API-key setup helper should remain for pilot/production.
