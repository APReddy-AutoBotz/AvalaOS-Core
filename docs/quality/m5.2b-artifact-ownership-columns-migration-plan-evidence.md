## Artifact Ownership Model Summary

The M5.2b plan defines a standard ownership vocabulary:

- `org_id`
- `workspace_id`
- `created_by`
- `updated_by`
- `owner_id`
- `reviewer_id`
- `approver_id`
- `source_org_id`
- `source_workspace_id`
- `status`
- `deleted_at`
- `archived_at`
- `correlation_id` or `audit_correlation_id`

The plan does not force every column onto every table. It recommends per-domain ownership, with Assess/Govern as the first implementation slice and evidence, lineage, audit, export, and provider workspace-scope decisions deferred until their persistence authority is approved.

## FK And Backfill Strategy Summary

- Add nullable ownership columns first.
- Prefer `NOT VALID` FKs for broad artifact tables.
- Validate constraints after AP-approved data mapping.
- Add `NOT NULL` only after backfill is verified.
- Avoid FKs for draft-only or not-yet-canonical tables.
- Avoid seed data, broad updates, personal identifiers, hosted project identifiers, and destructive backfills.

## RLS Readiness Summary

Ownership columns will support future M5.3a policy implementation by enabling:

- Workspace-scoped reads by active workspace members.
- Cross-org denial through `org_id`.
- Cross-workspace denial through `workspace_id`.
- Buyer Viewer read-only policies.
- Reviewer and approver role gates.
- Export, evidence, lineage, handoff, and audit tenant boundaries.
- Provider metadata boundaries without secret exposure.
- Audit append behavior with actor and correlation context.

M5.2b does not implement RLS and does not prove tenant isolation.

## Risks Carried Forward

- Broad schema blast radius if every artifact domain is migrated in one PR.
- FK/backfill failure if workspace mapping is inferred instead of AP-approved.
- Buyer-demo compatibility risk if runtime depends on new ownership columns too early.
- Ambiguity between draft SQL contracts and the real migration chain.
- Existing provider-governance records are mostly org-scoped.
- False confidence before RLS policies and isolation tests exist.
- Migration ordering and hosted migration-history risk.
- Personal value leakage risk if validation uses local ignored files or remote projects.
