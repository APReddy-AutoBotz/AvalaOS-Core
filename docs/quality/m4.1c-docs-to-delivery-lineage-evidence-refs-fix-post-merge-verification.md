## Root Cause

M4.0a critical finding `M4A-C03` found that generated work items imported from Avala Studio into Avala Delivery Lite did not retain source lineage or evidence reference ids. M4.1b gave generated artifacts an Assess source context, but the Delivery import path still did not copy that source context onto imported tasks. The handoff ledger also recorded generated work item titles as `evidenceRefs`, which were not actual evidence reference ids.

## Docs-to-Delivery Before And After

Before M4.1c:

- Generated work item import created Delivery epics and tasks without task-level lineage.
- Delivery tasks did not carry document generation, assessment, process, or handoff ledger refs.
- Handoff ledger `evidenceRefs` contained generated work item titles rather than upstream evidence ref ids.
- Delivery Pack logic could read `task.sourceLineage`, but the import path did not populate or persist it.

After M4.1c:

- Generated Story/Task imports receive compact `TaskSourceLineageMetadata`.
- Existing task metadata persistence round-trips `task.sourceLineage`.
- Handoff ledger entries record real upstream evidence ref ids when available.
- Generated artifacts from valid Assess handoff retain their own `sourceContext`.
- Docs-only imports remain partial or missing lineage and do not receive fake Assess-backed evidence.
- Generated Epic rows remain grouping records because current epic persistence has no metadata field.

## Lineage Model Summary

M4.1c added a compact, non-authoritative task lineage model. It points to source records; it does not rescore, re-approve, or certify work items.

Lineage field groups:

- Source identity: `sourceModule`, `upstreamSourceModule`, `sourceType`, `sourceLabel`, `sourceStatus`.
- Generation identity: `documentGenerationId`, `sourceGenerationId`, `sourceArtifactKey`, `sourceArtifactTitle`, `sourceWorkItemTitle`.
- Assess source identity: `processId`, `assessmentId`, `sourceContextLabel`, `sourceDecisionPackRef`, `sourceHandoffPackRef`.
- Handoff identity: `handoffLedgerEntryIds`.
- Evidence identity: `evidenceRefs`, `assumptionRefs`.
- Gap visibility: `lineageCompleteness`, `lineageNotes`, `createdAt`.

Completeness behavior:

- `complete`: document generation id, Assess process/assessment ids, and evidence ref ids exist.
- `partial`: a document generation id or source context exists, but one or more source/evidence components are incomplete.
- `missing`: neither source context nor document generation id exists.

## Evidence Refs Summary

- Assess-backed generated artifacts carry upstream evidence ids from `GeneratedArtifacts.sourceContext.evidenceRefs`.
- Imported Story/Task records receive those evidence ids through `task.sourceLineage.evidenceRefs`.
- The docs-to-delivery handoff ledger records the same upstream evidence ref ids rather than generated work item titles.
- Assumption ids are retained separately through `task.sourceLineage.assumptionRefs`.
- Evidence refs are compact source pointers, not proof that export policy has passed.

## Docs-Only Partial Lineage Hardening

Docs-only generated work items remain importable, but they are explicitly not treated as Assess-backed evidence:

- If a document generation id exists but no Assess `sourceContext` exists, lineage remains `partial`.
- Docs-only partial lineage includes the explicit note: `Docs-only generation is not Assess-backed evidence.`
- This avoids making docs-only generated work look equivalent to an Assess-backed handoff.
