## Planning Summary

M5.2g records that applied migration-chain authority currently includes tenant/workspace authority, Assess authority, `projects`, and `document_generations`.

Delivery work-item authority remains unimplemented. The applied migration chain does not include `delivery_work_items`, `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, `timesheet_entries`, Delivery Pack records, Monitor signal records, lineage authority, handoff authority, or export records.

Draft schema and runtime adapters still reference Agile-like tables and `app_id` mapping. Mock/demo data includes delivery tasks, assignees, reporters, dependencies, user stories, source lineage, Delivery Pack refs, handoff refs, and evidence refs.

## App ID Hardening Confirmation

If AP approves `app_id`, it is planned only as a nullable compatibility bridge.

`app_id` is not tenancy authority.

`app_id` is not globally unique.

`app_id` is not a Supabase identifier.

`app_id` is not a replacement for `id`.

`app_id` is not sufficient for authorization.

Proposed uniqueness should be scoped to active, non-deleted rows by at least `(org_id, project_id, app_id)`.

If AP rejects `app_id`, future runtime adapter work must map through UUID authority.

## Metadata And Source Lineage Hardening Confirmation

`metadata` and `source_lineage` are compatibility payloads only.

They are not durable evidence authority.

They are not durable approval authority.

They are not durable lineage authority.

They are not export or storage authority.

They are not review authority.

They are not cross-tenant reference authority.

They must not store secrets, provider keys, service-role values, raw private tokens, personal hosted values, or unsupported certification or compliance claims.

Extra assignees, dependencies, subtasks, user stories, and order rank remain non-authority UI/runtime compatibility until AP approves normalized persistence.

## FK And Delete Behavior Proposal Confirmation

Future tenant/source/project/document FKs should use default/no-cascade delete behavior:

- `org_id -> organizations(id)`
- `(workspace_id, org_id) -> workspaces(id, org_id)`
- `(project_id, org_id) -> projects(id, org_id)`
- `(document_generation_id, org_id) -> document_generations(id, org_id)`
- `(source_process_id, org_id) -> assess_processes(id, org_id)`
- `(source_assessment_id, org_id) -> assessments(id, org_id)`

Actor/profile FKs should use `ON DELETE SET NULL` where appropriate:

- `created_by`
- `updated_by`
- `owner_id`
- `assigned_to`
- `reporter_id`

Future M5.2g-a should use staged `NOT VALID` FKs, avoid constraint validation, avoid seed/backfill, and avoid destructive cleanup unless AP separately approves it.
