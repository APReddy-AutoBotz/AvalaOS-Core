## Migration Summary

M5.2g-a adds one additive migration that creates or promotes only `delivery_work_items` as the first AvalaOS Delivery work-item authority table.

The migration uses `CREATE EXTENSION IF NOT EXISTS pgcrypto`, `CREATE TABLE IF NOT EXISTS`, compatibility `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, guarded `DO $$` constraint blocks, `CREATE INDEX IF NOT EXISTS`, and `ALTER TABLE delivery_work_items ENABLE ROW LEVEL SECURITY`.

No runtime code depends on this table in M5.2g-a.

## Table Created Or Promoted

- `delivery_work_items`

No `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, `timesheet_entries`, Delivery Pack, Monitor, lineage, handoff, export, storage, Document Vault, provider, or policy tables are created or promoted.

## Option B Confirmation

M5.2g-a implements the AP-accepted Option B direction from M5.2g: one narrow AvalaOS-specific `delivery_work_items` authority table first.

The migration does not promote the draft Agile-like table family. AvalaOS Delivery remains a governed handoff and work-item surface, not a broad project-management replacement.

## App ID Hardening

`app_id` is included as a nullable compatibility bridge only.

`app_id` is not tenancy authority, not globally unique, not a Supabase identifier, not a replacement for `id`, and not sufficient for authorization.

The migration adds active/non-deleted scoped uniqueness on `(org_id, project_id, app_id)` only when `app_id IS NOT NULL AND deleted_at IS NULL`.

If AP later rejects the compatibility bridge, runtime adapter work must map through UUID authority instead.

## Metadata And Source Lineage Hardening

`metadata` and `source_lineage` are compatibility payloads only.

They are not durable evidence authority, durable approval authority, durable lineage authority, export or storage authority, review authority, handoff authority, or cross-tenant reference authority.

They must not store secrets, provider keys, service-role values, raw private tokens, personal hosted values, or unsupported certification claims.

Extra assignees, dependencies, subtasks, user stories, order rank, and UI/runtime values remain non-authority compatibility data until AP approves normalized persistence.

## FK And Delete Behavior Summary

Tenant, source, project, and document FKs use staged `NOT VALID` constraints with default/no-cascade delete behavior:

- `org_id -> organizations(id)`
- `(workspace_id, org_id) -> workspaces(id, org_id)`
- `(project_id, org_id) -> projects(id, org_id)`
- `(document_generation_id, org_id) -> document_generations(id, org_id)`
- `(source_process_id, org_id) -> assess_processes(id, org_id)`
- `(source_assessment_id, org_id) -> assessments(id, org_id)`

Actor/profile FKs use `ON DELETE SET NULL`:

- `created_by -> profiles(id)`
- `updated_by -> profiles(id)`
- `owner_id -> profiles(id)`
- `assigned_to -> profiles(id)`
- `reporter_id -> profiles(id)`

The migration does not use `ON DELETE CASCADE`.
