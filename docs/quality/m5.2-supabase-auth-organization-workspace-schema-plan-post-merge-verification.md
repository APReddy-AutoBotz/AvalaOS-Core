## Current Repo Inventory Summary

- Supabase Auth exists through the browser Supabase client and the auth adapter when public config is present.
- The current runtime maps Supabase users into app users and can overlay demo persona data by email.
- Runtime profile, membership, role, and permission loading is not yet canonical DB authority.
- Organizations are the current tenant root, with existing adapter support for `organizations`, `organization_members`, and `roles` when Supabase is configured.
- Workspace is not yet a first-class hosted-pilot authority; current workspace-like surfaces are navigation, scope, provider placeholder, or module concepts.
- Most existing schema contracts are org-scoped. Future work needs explicit workspace authority and ownership mapping.
- Browser/demo storage remains demo-only persistence for current user, scope, view, organization, audit-adjacent logs, handoff ledger, Assess, Docs, Delivery, and transitional settings.
- Evidence, lineage, handoff, export, audit, and provider governance records still need server-authoritative persistence, tenant ownership, and RLS planning before hosted pilot use.

## Enterprise Gaps Summary

- Canonical DB-backed app profile, organization membership, workspace membership, role assignment, and permission authority remain pending.
- Workspace authority needs AP approval before schema and RLS planning can safely proceed.
- Current display personas and demo permissions are not hosted-pilot authorization.
- Existing org-scoped schema posture is insufficient for workspace isolation without further table ownership mapping.
- Existing draft RLS work needs a table-by-table design and negative test matrix aligned to active membership and role authority.
- Export, evidence, lineage, handoff, audit, provider policy, and storage records need org/workspace ownership direction before implementation.

## Target Architecture Summary

- `auth.users` is the hosted-pilot identity authority.
- `profiles` or `user_profiles` holds app metadata only and does not own role authority by itself.
- `organizations` is the tenant root.
- `organization_members` or `organization_memberships` is the active org membership authority.
- `workspaces` is the organization child and proposed primary hosted-pilot work boundary.
- `workspace_memberships` is the workspace access authority.
- Roles require AP decision on global, org-scoped, workspace-scoped, or mixed scope.
- Permissions require AP decision on JSONB role storage versus normalized `permissions` and `role_permissions`.
- Future records should carry `org_id`, and most work artifacts should also carry `workspace_id`.
- Display persona, Supabase identity, app profile, org membership, workspace membership, role assignment, and permission set are separate concepts.

## RLS Readiness Summary

M5.3 should remain a design and test-plan milestone unless AP separately approves implementation. The M5.2 records hand off:

- Table-by-table RLS matrix requirements.
- Active org membership rule.
- Active workspace membership rule.
- Role and permission authority rule.
- Cross-org and cross-workspace denial tests.
- Suspended or removed membership denial tests.
- Server-only service-role exception boundaries.
- Browser prohibition for service-role credentials, database URLs, management tokens, provider secrets, and personal hosted values.

## AP Decisions Needed

- Final organization and workspace hierarchy.
- Whether workspace is mandatory for all hosted-pilot records.
- `organization_members` versus `organization_memberships` naming.
- Profile table naming and shape.
- Role model: global, org-scoped, workspace-scoped, or mixed.
- Whether permissions remain JSONB on roles or normalize into `permissions` and `role_permissions`.
- Workspace switching behavior for users with multiple organizations or workspaces.
- Whether UI scope needs a future workspace scope.
- Demo tenant and workspace seed policy.
- Invitation and SSO timing.
- Whether existing guard permissions need canonical DB mapping before RLS.
- Whether a later M5.2a implements migrations before M5.3.

## Risks Carried Forward

- Workspace authority may require broad FK, backfill, adapter, and RLS planning.
- Supabase user mapping currently overlays demo personas instead of loading DB-backed profile, membership, role, and permission state.
- Multi-organization and multi-workspace switching can create UX and authorization ambiguity.
- Draft RLS policy work may need redesign for workspace isolation.
- Current `OrgRole` and demo permission arrays are too coarse for hosted-pilot authorization.
- Provider governance has future workspace-named surfaces, but canonical workspace authority is not yet approved.
