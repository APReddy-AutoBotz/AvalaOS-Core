## Original Audit Failure Summary

The post-merge M4.1d-1 verification on `main` failed only at:

```text
npm audit --audit-level=moderate
```

Original audit JSON reported 3 high-severity advisories:

| Package | Direct | Vulnerable range | Chain / effect | npm fixAvailable |
| --- | --- | --- | --- | --- |
| `@vitejs/plugin-react` | Yes | `4.0.0-beta.0 - 5.1.4` | via `vite` | `true` |
| `vite` | Yes | `4.2.0-beta.0 - 8.0.3` | via `esbuild` | `vite@8.0.16`, semver major |
| `esbuild` | No | `0.17.0 - 0.28.0` | transitive through `vite` | `vite@8.0.16`, semver major |

The advisory was `GHSA-gv7w-rqvm-qjhr`, titled `esbuild: Missing binary integrity verification in Deno module enables remote code execution via NPM_CONFIG_REGISTRY`.

## Remediation Sequence

First, plain `npm audit fix` was run without `--force`. It reduced the audit from 3 high advisories to 2 by updating only lockfile resolutions inside existing ranges:

- `@vitejs/plugin-react`: `5.1.2` -> `5.2.0`
- `vite`: `6.4.2` -> `6.4.3`

The remaining audit JSON still required `vite@8.0.16` as a semver-major fix. AP then approved controlled semver-major dev-tooling remediation while continuing to prohibit `npm audit fix --force`.

The final controlled update was:

```text
npm install --save-dev vite@8.0.16 --save-exact
```

This changed the explicit Vite dev dependency only. `@vitejs/plugin-react` was not manually upgraded because the existing `5.2.0` resolved version already declares Vite 8 peer compatibility.
