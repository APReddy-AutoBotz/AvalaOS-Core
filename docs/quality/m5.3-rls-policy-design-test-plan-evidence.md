## RLS Design Principles Summary

- Deny by default.
- Use `auth.uid()` as identity anchor.
- Require active org membership.
- Require active workspace membership for workspace-scoped records after workspace schema exists.
- Require explicit role and permission checks.
- Deny anonymous tenant-data access.
- Deny cross-org and cross-workspace access unless AP approves a specific exception.
- Keep service-role exceptions server-only and audited.
- Keep provider secrets out of browser-visible records.

## Helper Strategy Summary

- Prefer small SQL helper functions after schema approval.
- Plan future `is_active_org_member(org_id)`.
- Plan future `is_active_workspace_member(workspace_id)`.
- Plan future role and permission helpers for org/workspace scope.
- Use direct policy joins only for simple reads.
- Use security definer helpers only when needed to avoid recursive RLS or support audited server workflows.
- Defer materialized permission views unless performance tests require them.
- Carry forward recursive RLS risk.

## Fixture And Test Strategy Summary

- Future fixture model uses two orgs, two workspaces per org, active member, disabled member, non-member, Platform Admin, Process Owner, Control Reviewer, Delivery Lead, Automation Contributor, and Buyer Viewer.
- Record classes include Assess, Govern, Studio, Delivery, evidence, lineage, export, audit, and provider governance.
- Future tests should include SQL-level policy tests and JS integration tests.
- Anon, authenticated, admin-like, and service-role clients must remain separated.
- Test output should record pass/fail status and counts only.
- CI timing remains an AP decision.

## Risks Carried Forward

- Existing draft policies are mostly org-scoped and may create false confidence if treated as workspace-ready.
- Workspace authority requires broad FK, backfill, and planning across Assess, Govern, Studio, Delivery, Monitor, evidence, lineage, handoff, audit, export, and provider records.
- Recursive RLS or complex joins can make policies fragile.
- Role and permission strings in demo/runtime guards may not map cleanly to DB permission authority.
- Service-role exceptions can become overbroad without server-only execution and audit context.
- RLS tests can leak sensitive values if scripts are not pass/fail and counts-only.
