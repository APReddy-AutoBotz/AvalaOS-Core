## Helper Function Plan

Future helpers are limited to `public.is_active_org_member(p_org_id uuid)` and `public.is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`. The refresh requires SQL helpers to be `LANGUAGE sql`, `STABLE`, boolean-only, hardened by `search_path`, schema-qualified where possible, and granted only to `authenticated` after revoking execute from `PUBLIC` and `anon`. `SECURITY DEFINER` remains AP-approved only if needed to avoid recursive RLS failure.

## SELECT Policy Plan

The refresh keeps future policy SQL SELECT-only and deny-by-default. It excludes broad `FOR ALL`, INSERT, UPDATE, DELETE, mutation, artifact, provider, and legacy policies.

Planned SELECT directions:

- `profiles`: own active, non-deleted profile only.
- `organization_members`: own active, non-deleted membership only, with active non-deleted parent organization.
- `workspace_memberships`: own active, non-deleted workspace membership only, with active non-deleted parent organization and workspace.
- `organizations`: active organization membership only.
- `workspaces`: active workspace membership only.
- `roles`: organization roles through active org membership; workspace roles through active workspace membership; system role visibility AP-deferred.

## Recursion And Security Analysis

The refresh records recursion risk where helpers query membership tables while membership tables are RLS-protected, membership policies depend on helpers, and role policies depend on membership visibility. It recommends a minimal helper pattern, direct own-profile reads, AP-approved definer helpers only if needed, no permission helpers, no artifact dependencies, hardened grants, hardened `search_path`, and executable local assertions before any local tenant-isolation claim.
