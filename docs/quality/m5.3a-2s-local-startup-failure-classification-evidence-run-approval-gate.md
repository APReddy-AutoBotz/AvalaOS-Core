# M5.3a-2s Local Startup Failure Classification Evidence Run Approval Gate

## Purpose

M5.3a-2s exists only to define what AP must approve before any future local startup failure classification evidence run can occur.

This milestone is an approval gate. It is not the evidence run, does not authorize execution by itself, and does not change classifier behavior, scripts, package scripts, SQL, migrations, policies, assertions, harnesses, runtime, UI, CI, provider, deployment, export, scoring, demo, Reports, Teams, Admin, or Health files.

## Non-Goals

M5.3a-2s does not:

- Run the classifier.
- Attempt local startup.
- Run Supabase stack commands.
- Run Docker commands.
- Run DB commands or make DB connections.
- Run schema bootstrap or readiness commands.
- Run RLS harnesses.
- Run artifact assertions.
- Run migration or bootstrap commands.
- Run remote, hosted, or personal validation.
- Run provider calls.
- Prove local startup success.
- Prove local DB availability.
- Prove schema availability.
- Verify artifact SELECT isolation.
- Newly verify tenant isolation.
- Claim hosted readiness.
- Claim production readiness.
- Identify root cause.

## Required AP Approvals Before Any Future Run

No future evidence run may occur until AP explicitly approves all of the following:

- Classifier rerun approval.
- Local startup attempt approval.
- Exact command label approval.
- Stop condition approval.
- Capturable evidence field approval.
- Continued prohibition on raw output, local values, target values, and secret-like values in evidence.
- Continued prohibition on remote, personal, or hosted validation unless a separate milestone explicitly approves it.

Approval must be specific to the future evidence run. This approval gate does not grant execution permission.

## Future Allowed Evidence Categories

Future evidence may capture only redacted, signal-level categories approved by AP. Allowed categories are limited to:

- Command label.
- Command class.
- Exit result class.
- Timeout flag.
- Redacted line-count bucket.
- Redacted duration bucket.
- Safety gate result.
- Scratch cleanup result.
- No-secrets confirmation.
- No-local-path confirmation.
- No-raw-log confirmation.
- Fallback classification result.
- Low-confidence classification result.

Future evidence must not include actual command strings, local paths, raw output, target values, local environment values, hosted identifiers, container IDs, image IDs, local config values, row payloads, auth headers, claim values, or machine-specific values.

## Stop Conditions

A future evidence run must stop immediately if any of these conditions occur:

- Any secret-like value could be exposed.
- Raw logs would need to be copied.
- Local paths would need to be written into evidence.
- Hosted, personal, or remote validation becomes necessary.
- DB, schema, or RLS execution would be required without separate approval.
- Classifier output attempts to infer root cause from low-confidence signals.
- Evidence would require actual command strings, target values, local environment values, raw output, or machine-specific values.
- Scratch cleanup fails.
- Output safety checks fail.
- Redacted category generation cannot complete without copying raw or local values.

## Proof Boundary

Even after this approval gate, nothing is newly proven until a separately approved evidence run is executed and reviewed.

This milestone proves only that the approval requirements, evidence categories, stop conditions, and proof boundaries are documented. It does not prove startup success, local DB availability, schema availability, artifact SELECT isolation, tenant isolation, hosted readiness, production readiness, or root cause.

## Future Evidence Run Shape

A future milestone may propose a controlled evidence run only after AP approves this gate and explicitly authorizes the run scope.

The future run must:

- Preserve the existing classifier interface unless separately approved.
- Use only approved command labels.
- Capture only approved redacted evidence categories.
- Stop on any approved stop condition.
- Keep fallback behavior safe.
- Keep low-confidence signal classifications from becoming root-cause claims.
- Record sanitized outcomes only.
