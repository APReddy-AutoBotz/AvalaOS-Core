## Current Schema / Context Inventory

- `ai_provider_configs` with org scope, provider type, display name, `key_reference`, default model, model policy, status, creator, timestamps, index, RLS, and member-read policy.

Candidate schema discussion is not approval to migrate, alter, depend on, or implement these structures in this milestone.

## Current Edge Env-Key Usage Inventory

`supabase/functions/_shared/ai.ts` currently:

- supports `gemini` and `groq` through `ProviderType`
- resolves provider support through `assertSupportedProvider`
- reads direct server environment keys with `Deno.env.get('GROQ_API_KEY')` and `Deno.env.get('GEMINI_API_KEY')`
- reads model names with `Deno.env.get('GROQ_MODEL')` and `Deno.env.get('GEMINI_MODEL')`
- calls Groq and Gemini directly for generation/refinement
- returns usage metadata from provider responses
- throws missing key errors when direct server env keys are unavailable

M3.2a did not change this behavior. Future implementation should replace direct provider-key lookup with a server-side provider config/key-reference resolver after AP approval.

## Current Client / Mode Boundary Inventory

- `services/aiMode.ts` defines `local-demo`, `internal-dev`, `pilot`, and `production`.
- `services/aiMode.ts` blocks `pilot` and `production` when Edge AI is unavailable.
- `services/aiOrchestrator.ts` routes Edge-allowed calls through `aiEdgeClient`.
- `services/aiOrchestrator.ts` allows browser fallback only for explicit `local-demo` / `internal-dev` policy.
- `services/aiEdgeClient.ts` sends `providerType` and request metadata to Supabase Edge Functions. It does not send raw provider keys.

## Proposed Contract Summary

The planning doc proposes future server-side concepts for:

- provider registry
- provider config record
- provider key reference
- workspace/tenant provider policy
- provider enabled/disabled status
- allowed modes
- default provider behavior
- fail-closed response model
- sanitized error classes
- audit/evidence event fields
- prompt/template version reference
- usage metadata fields

Key proposal points:

- Pilot/production must resolve provider config and key references server-side only.
- Raw provider secrets must never be stored in browser env, browser state, logs, database rows, evidence rows, product copy, or generated artifacts.
- Default provider selection must fail closed when missing, ambiguous, disabled, unauthorized, or policy-disallowed.
- Provider calls must not happen until auth, tenant/workspace membership, provider policy, provider status, key-reference resolution, prompt/version reference, and audit/evidence readiness pass.

## Proposed Schema / RLS Summary

Candidate future structures discussed:

- `ai_provider_configs`
- `ai_provider_key_refs`
- `ai_workspace_provider_policies`
- `ai_provider_audit_events`
- `prompt_templates` or prompt versions
- `ai_usage_events`

RLS/access proposal:

- Tenant/workspace members may read safe provider config metadata only when active membership allows it.
- Admin/control roles may write provider config metadata only through approved server-side paths.
- Browser/client code can never read raw secrets, resolved key values, secret manager payloads, or service-role-only resolver fields.
- Edge/service-role access may resolve key references only after user auth, membership, provider policy, provider status, allowed mode, and operation checks pass.
- Missing policy, unclear ownership, disabled status, missing key reference, invalid key reference, missing prompt/version, or missing evidence fields must fail closed.

## Edge Resolver Contract Summary

Future Edge resolver inputs should be safe request metadata only:

- authenticated user context
- tenant/workspace identifier
- operation type
- requested provider type or provider config identifier when allowed
- deployment mode/context resolved server-side
- source/input references, not raw secrets

Future Edge resolver checks should run before provider calls:

- authenticate user
- resolve active tenant/workspace membership
- resolve mode/deployment context
- resolve provider config by tenant/workspace, operation, mode, type, status, and policy
- resolve provider key reference server-side
- bind prompt/template key/version
- prepare required audit/evidence
- call provider only after all checks pass
- record usage/completion/failure evidence with sanitized fields

Fields never logged:

- raw provider secrets
- resolved secret values
- authorization headers
- service-role keys
- browser/client secrets
- secret manager payloads
- raw prompts or source document bodies
- provider payloads containing secret material
