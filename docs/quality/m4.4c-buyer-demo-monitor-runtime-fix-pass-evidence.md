The page rendered visibly, but the buyer-facing close was not clean enough for release-candidate closure.

## Root Cause

`BurndownChartWidget` assumed an active sprint always had positive finite story-point estimates. Sarah's buyer `My Work` context has an active sprint available but no active-sprint task estimates in the filtered task list.

That made `totalStoryPoints` equal `0`, then `maxY` equal `0`, and the chart's Y-coordinate calculation divided `0 / 0`. The result propagated into SVG text, line, path, and circle coordinates as `NaN`.

The same chart math also needed a defensive guard for one-day sprint windows because `maxX` would be `0`.

## Fix Implemented

- Extracted burndown chart calculation into `buildBurndownChartModel`.
- Added finite-number guards for dimensions, sprint duration, total story points, chart points, ticks, and SVG paths.
- Returned `null` when the burndown model cannot produce finite SVG coordinates.
- Rendered a clear empty state for active sprints without finite story-point estimates: `No sprint estimates`.
- Added a focused regression test covering:
  - Sarah-like active sprint with no task estimates
  - zero story-point tasks
  - one-day sprint duration
  - valid sprint data producing finite coordinates and paths
- Added `test:monitor-burndown-svg` and wired it into `npm run test`.

No product wording from M4.4a was changed.
