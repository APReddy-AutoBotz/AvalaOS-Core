## Current Delivery Schema Inventory Summary

The applied migration chain currently has tenant authority, Assess ownership, and provider governance only. No Delivery/project authority tables exist in `supabase/migrations`.

Draft schema defines `projects`, `epics`, `sprints`, `tasks`, `task_comments`, `task_activity_events`, and `timesheet_entries` with `org_id` and no `workspace_id`.

Runtime Delivery uses `deliveryAdapter` to target draft Delivery tables when Supabase is configured, using `org_id` and stable `app_id` mapping. Task assignees, reporter, dependencies, ordering, user stories, comments, activity, and source lineage are partially in runtime metadata or draft child-table sync.

Delivery Pack is a derived TypeScript read model, not a database table. Monitor project/task signals are derived from project/task/demo state and widgets, not live telemetry tables.

## Docs-Only Decision Summary

M5.2e remains docs-only because Delivery schema is draft-only and project authority must be implemented before work-item authority, Studio/Docs ownership, Delivery Pack persistence, Monitor persistence, and workspace-aware RLS can safely depend on Delivery records.

No migration is created in M5.2e.

## Project Authority Recommendation

M5.2f should make `projects` the first Delivery authority table. Project authority should remain narrow: a governed delivery project under organization/workspace ownership, not a generic project-management expansion.

Portfolio should remain a view over Delivery projects for now, not a separate persisted authority.

Epics, sprints, tasks, comments, activity, timesheets, Delivery Pack records, Monitor signals, handoff/lineage authority, and export ownership should stay deferred until project authority exists.

## Proposed Ownership Model Summary

Future project authority should plan `org_id`, `workspace_id`, `app_id`, `name`, `description`, `owner_id`, `delivery_lead_id`, `created_by`, `updated_by`, `source_process_id`, `source_assessment_id`, `lifecycle_stage`, `health_status`, `status`, `archived_at`, `deleted_at`, and `audit_correlation_id`.

Future work-item authority should later add only needed project/workspace ownership, assignee/reporter actors, source lineage refs, lifecycle/status, and correlation fields.

## FK And Backfill Strategy

Future implementation should use nullable ownership columns first, `NOT VALID` FKs where safe, references to `organizations(id)`, composite `workspaces(id, org_id)`, `profiles(id)` for actors, and M5.2c Assess tables for source refs where same-organization constraints can be enforced.

Do not hard-FK to `document_generations` until Studio/Docs is applied migration authority.

Do not add `NOT NULL` ownership requirements, seed data, broad updates, personal IDs, hosted IDs, or destructive backfill before AP-approved mapping.

## Migration Safety Strategy

M5.2e creates no migration and changes no schema.

Future implementation should be additive only, avoid drops/renames/destructive updates, avoid RLS policies until the RLS implementation milestone, avoid runtime dependency until adapters are approved, avoid remote Supabase validation, and avoid `scripts/applySupabaseMigrations.mjs` unless AP approves a local-only path.

## Buyer-Demo Compatibility Strategy

Existing mock/local Delivery flow remains intact. Runtime adapters do not depend on future columns. AP delivery work items remain local/demo-backed until future persistence adapters are approved.

M5.2e changes no scoring, Decision Pack rendering, Govern copy, Studio handoff, delivery lineage, document export, Delivery Pack, Monitor signals, burndown widget, demo data, routing, or UI.

## RLS Readiness Mapping

Delivery ownership will support future workspace-scoped project and work-item reads, cross-organization denial, cross-workspace denial, Buyer Viewer read-only access where approved, Delivery Lead update gates, contributor own-task update gates, reviewer/approver gates, source-lineage and evidence boundary checks, Docs-to-Delivery handoff boundaries, Delivery Pack and Monitor boundaries after persistence models exist, and export boundaries under export controls.

M5.2e does not implement RLS policies or prove tenant isolation.
