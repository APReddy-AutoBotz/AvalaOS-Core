## Current-State Route/RBAC Diagnosis

M4.1d confirmed that AvalaOS Core currently uses `View` state and `Scope` as the primary app-shell navigation model rather than URL route authorization. The current route/RBAC risk is therefore a central View/RBAC guard gap:

- `currentView` and `currentScope` are persisted and can restore stale states.
- Module checks through `isViewEnabled` and `isModuleEnabled` validate module enablement only.
- Sidebar filtering separately checks module, scope, permissions, and Admin role.
- App render and transition paths can set views without going through Sidebar filtering.
- Deferred or overloaded areas are handled through local exceptions rather than one shared decision model.

## Proposed Guard Model Summary

M4.1d designed a future pure resolver concept:

```ts
resolveViewAccess({
  user,
  authLoading,
  organization,
  enabledModules,
  view,
  scope,
})
```

The planned return shape includes:

- `allowed`
- `fallbackView`
- optional `fallbackScope`
- `reason`
- `module`
- `requiredPermissions`
- `requiredScope`
- `guardSeverity`

Guard precedence separates auth loading from unauthenticated access:

1. Auth loading waits and does not redirect.
2. Unauthenticated access denies to the auth flow.
3. No organization or setup-required state routes to onboarding/setup.
4. Deferred or decision views follow AP-approved behavior.
5. Disabled module falls back to a safe enabled module home.
6. Invalid scope falls back to a valid scope/view pair.
7. Missing permission falls back or hides according to guard severity.
8. Allowed state renders the requested view.

The design preserves current Sidebar permission semantics for the first implementation: Admin role bypass OR any one listed permission grants access.

The matrix also records Sidebar-derived view scopes and permissions where metadata exists. Views without Sidebar entries need future shared metadata before central enforcement. `REPORTS`, `TEAMS`, `WORKSPACE`, Admin, and AI Controls remain deferred/AP decision areas.

## Non-Sidebar Entry Point Risk Summary

M4.1d explicitly documented non-Sidebar entry points that need future guard integration:

- `handlePrimaryAction`
- project selector / Docs Forge entry
- Assess-to-Studio handoff
- `ModuleJourney`
- persisted `currentView` / `currentScope` restore
- direct `setCurrentView` handlers
- fallback effects in `App.tsx`

These entry points are not bugs by themselves, but they prove that Sidebar visibility is not a sufficient authorization model.

## Implementation Sequence

M4.1d defines the future implementation sequence:

1. `M4.1d-1`: central View/RBAC guard helper and focused tests.
2. `M4.1d-2`: `App.tsx` integration for persisted and direct view state.
3. `M4.1d-3`: Sidebar and ModuleJourney alignment to shared metadata.
4. `M4.1d-4`: persisted view/scope cleanup and safe fallback.
5. `M4.1d-5`: Reports, Teams, Workspace/Admin, and AI Controls deferred-route handling after AP decisions.
