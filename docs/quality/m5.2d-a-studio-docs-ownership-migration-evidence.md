## Migration Summary

M5.2d-a creates or promotes only `document_generations` as the first applied Studio/Docs authority table.

The migration is additive and uses `CREATE EXTENSION IF NOT EXISTS pgcrypto`, `CREATE TABLE IF NOT EXISTS`, compatibility `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`, guarded constraint blocks, `CREATE INDEX IF NOT EXISTS`, staged `NOT VALID` checks and foreign keys, legacy draft policy/FK hardening, and RLS enablement without policies.

The current adapter contract is preserved:

- `id`
- `org_id`
- `project_id`
- `template_id`
- `generated_at`
- `artifacts`
- `created_at`

## Table Created Or Promoted

- `document_generations`

## Current Adapter Compatibility

`services/adapters/docsAdapter.ts` reads and writes `document_generations` through `org_id`, `project_id`, `template_id`, `generated_at`, and `artifacts`.

The migration preserves those fields and does not add `app_id`, `project_app_id`, or document app identifiers. The adapter continues to resolve the project UUID through `projects.app_id`, then save `project_id` on the document generation row. New ownership/readiness columns remain nullable-first or defaulted, so no runtime adapter change is required in this milestone.

Configured Supabase Docs paths may still fail closed because RLS is enabled with no policies. That is an expected readiness posture until AP approves RLS policies and tests.

## Status Model Decision

The status model is intentionally narrow:

- `generated`
- `draft`
- `archived`
- `failed`

The migration does not add `approved`, `rejected`, or `ready_for_review` statuses because durable review-event authority is deferred. Current approval details remain embedded in the generated `artifacts` payload and are not promoted to standalone review authority in this slice.

Required semantics are staged through `NOT VALID` checks for:

- `org_id IS NOT NULL`
- `project_id IS NOT NULL`
- non-empty `template_id`
- `status IS NOT NULL`
- status value membership in the approved set

No broad `ALTER COLUMN SET NOT NULL` is used for ownership fields.

## FK And Delete Behavior Summary

Tenant, project, and source references avoid broad hard-delete cascade:

- `org_id` references `organizations(id)` with default delete behavior.
- `(workspace_id, org_id)` references `workspaces(id, org_id)` with default delete behavior.
- `(project_id, org_id)` references `projects(id, org_id)` with default delete behavior.
- `(source_process_id, org_id)` references `assess_processes(id, org_id)` with default delete behavior.
- `(source_assessment_id, org_id)` references `assessments(id, org_id)` with default delete behavior.

Actor references may be nulled:

- `created_by` references `profiles(id)` with `ON DELETE SET NULL`.
- `updated_by` references `profiles(id)` with `ON DELETE SET NULL`.

All foreign keys are staged as `NOT VALID`. No constraints are validated in this milestone.

## Legacy Draft Schema Hardening

Older draft/manual non-production `document_generations` tables may have broad org-scoped Docs policies or legacy single-column/cascade FKs. M5.2d-a now hardens those cases without broadening scope.

Known legacy Docs policies removed if present:

- `Users can see org docs`
- `Users can modify org docs`

Known legacy FK behavior hardened:

- Drops legacy single-column `document_generations_project_id_fkey` if present so M5.2d-a can rely on same-organization `(project_id, org_id) -> projects(id, org_id)` authority.
- Drops existing M5.2d-a named tenant/source/project FKs only when the existing FK has cascade delete behavior, allowing the final FK creation block to recreate them with default/non-cascade behavior.
- Applies this cascade hardening only to `document_generations_org_id_fkey`, `document_generations_workspace_org_fkey`, `document_generations_project_org_fkey`, `document_generations_source_process_org_fkey`, and `document_generations_source_assessment_org_fkey`.

This hardening does not drop data, does not drop columns, does not backfill, does not validate constraints, and does not implement RLS policies. It preserves fail-closed readiness by removing only known legacy draft policies and only the FK shapes needed to preserve non-cascade, same-org authority.
