## App Integration Summary

M4.1d-2 integrated the M4.1d-1 central View/RBAC guard helper into the App shell. `App.tsx` now resolves App-level view and scope transitions through `resolveViewAccess` before applying the requested view, fallback view, or fallback scope.

Auth and organization loading remain wait states. The App shell does not overwrite persisted view or scope while the guard inputs are still loading.

Existing organization-scope Admin Workspace behavior was preserved because Workspace/Admin route semantics remain an AP decision area.

## Fallback Behavior Summary

Denied App transitions now use the central guard fallback output:

- auth or organization loading waits without redirecting
- unauthenticated or missing organization state is denied to the existing auth/onboarding flow
- disabled modules fall back to an accessible enabled module path where available
- invalid scopes apply the resolver fallback scope when supplied
- missing permissions apply the resolver-selected fallback view and scope
- Reports and Teams remain deferred or decision-pending according to existing guard metadata

## Assess-To-Studio Handoff Hardening Summary

The accepted AP hardening normalizes Docs Forge target scope before applying the guarded Assess-to-Studio transition:

- `MY_WORK`, `TEAM`, and `PROJECT` scopes are preserved for Docs Forge.
- `ORGANIZATION` scope is normalized to `MY_WORK` for the Docs Forge transition.
- Valid Assess source context is preserved when Docs Forge can be reached through the normalized scope.
- Source context is cleared when Docs Forge cannot be reached.
- Generated artifacts from a valid handoff retain their own source context for later lineage work.
