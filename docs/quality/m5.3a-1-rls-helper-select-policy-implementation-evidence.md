## Helper Functions

Implemented helpers:

- `public.is_active_org_member(p_org_id uuid)`
- `public.is_active_workspace_member(p_workspace_id uuid, p_org_id uuid)`

Helper posture:

- `LANGUAGE sql`
- `STABLE`, not `IMMUTABLE`
- Boolean-only return value
- `SECURITY DEFINER`
- Hardened `search_path = pg_catalog, public, auth`
- Schema-qualified table references
- Requires `auth.uid()`
- Requires active, non-deleted membership
- Requires active, non-deleted parent organization
- Requires active, non-deleted parent workspace for workspace helper

`SECURITY DEFINER` was used intentionally to avoid recursive RLS bootstrap failures where membership, organization, workspace, and role SELECT policies depend on membership resolution while the membership tables themselves are RLS protected. The helpers return no tenant metadata, role metadata, permission metadata, row payloads, counts, target values, or secret values.

## Grants

- `EXECUTE` revoked from `PUBLIC` for both helpers.
- `EXECUTE` revoked from `anon` when the role exists.
- `EXECUTE` granted only to `authenticated` when the role exists.
- `SELECT` granted to `authenticated` for the six scoped tables so RLS policies, not raw table grants, determine row access.
- No service-boundary role is targeted by any M5.3a-1 policy.

## SELECT Policy Summary

Created six explicit `FOR SELECT TO authenticated` policies:

- `profiles`: users can read only their own active, non-deleted profile.
- `organization_members`: users can read only their own active, non-deleted organization membership, gated by active organization membership.
- `workspace_memberships`: users can read only their own active, non-deleted workspace membership, gated by active workspace membership.
- `organizations`: active organization members can read active, non-deleted organization metadata for their organization.
- `workspaces`: active workspace members can read active, non-deleted workspace metadata for their workspace.
- `roles`: active organization members can read active organization-scoped roles, and active workspace members can read active workspace-scoped roles. System role visibility remains AP-deferred.

No broad `FOR ALL`, INSERT, UPDATE, DELETE, hard-delete, soft-delete, review, approval, export, delivery, artifact, provider, or legacy draft-table policies were created.

## Assertion SQL Update

`docs/schema/m5.3a_1_rls_membership_isolation_assertions.sql` was updated from the prior blocked skeleton into executable local assertion SQL for the six scoped tables.

The assertion file:

- Declares the same 16 scoped scenarios.
- Creates generated local UUID fixtures.
- Inserts fixture rows into local `auth.users` and the six scoped authority tables inside a single transaction.
- Simulates local anonymous and authenticated subject claims with local session settings.
- Switches to the local `authenticated` role for policy checks.
- Returns counts only: declared, executed, pass, fail, blocked, tenant isolation verified, harness status, and sanitized classifications.
- Rolls back all fixture rows.

Although the plan preferred temp-table-only fixtures, final policy assertions must query the actual RLS-protected tables to prove PostgreSQL RLS behavior. The implemented assertion method therefore uses transaction-scoped real-table fixture rows plus `ROLLBACK`; it creates no durable seed data, demo data, runtime data, backfill, or constraint validation.
