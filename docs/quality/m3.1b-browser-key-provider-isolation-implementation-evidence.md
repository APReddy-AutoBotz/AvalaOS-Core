## Scope Summary

M3.1b reduced and isolated browser raw-key/provider prop paths after the M3.1a mode boundary.

Implemented:

- Deleted `setup-api-key.js`.
- Removed `vite.config.ts` browser-facing provider-secret injection for `process.env.API_KEY` and `process.env.GEMINI_API_KEY`.
- Removed `userApiKey` state and prop flow from active Docs Forge generation, Workspace refinement, Project/Sprint, dashboard, and AI Insights surfaces.
- Removed `userApiKey` from `aiOrchestrator.generateArtifacts` and `aiOrchestrator.refineSection` call signatures.
- Removed `userApiKey` and `allowUserKey` from `getAiProviderApiKey` and `getAiProvider`.
- Kept browser Gemini/Groq provider classes in place only behind the existing M3.1a `local-demo` / `internal-dev` fallback policy.
- Kept `StorageKeys.API_KEY` as deprecated cleanup-only because `App.tsx` still removes the old stored key.

Not implemented:

- No Edge provider config/key-reference resolver.
- No static scan automation.
- No provider class removal.
- No provider file movement or rename.
- No package, package-lock, dependency, schema/RLS, Supabase Function, Health, scoring, Admin UI, runtime-agent, MCP/A2A, external execution, or compliance changes.
